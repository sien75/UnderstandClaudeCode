Supporting Services: Turning Configuration and Extensions into Controllable, Communicable, Observable Agent Capabilities

This set of crates does not directly determine how the agent reasons, but instead connects local configuration, plugin marketplace, skills, Hooks, MCP, network access, cloud tasks, and telemetry into the agent runtime. For users, the trigger points are usually enabling a plugin, adding a marketplace, starting a task, allowing network access, or configuring an observability exporter; the system then handles discovery, validation, materialization, registration, execution, and reporting back.

Input
Layered configuration, plugin or skill declarations, Hook events, MCP calls, model API requests, network policy, and cloud task commands
Output
Registered agent capabilities, policy-constrained external communication, resumable cloud task results, and log, metric, and analytics events

The plugin data flow starts from merging configuration layers with remote installation status. The loader reads each plugin's manifest and combines skills, MCP servers, App connectors, and Hooks into a unified result; the manager serializes refresh, caching, and installation, materializing the source first during installation, then atomically writing to the plugin repository and user configuration. Product restrictions are enforced during the "admission" phase; the runtime only consumes what has already been admitted.

The extension API consolidates different capabilities into a typed contributor registry: configuration, context, tools, MCP, approval, thread, and turn lifecycle each have independent slots. Hooks are another explicit lifecycle boundary; they serialize events, pass them to discovered commands for execution according to matching rules, and allow pre-hooks to reject tools or modify input. Permission requests can thus be decided both by built-in policy and reviewed by external Hooks.

The communication side is divided into three layers: codex-api encodes internal requests into HTTP protocols such as Responses, WebSocket, search, and image and parses streams; rmcp-client handles stdio, in-process, or streamable HTTP MCP sessions and OAuth; network-proxy uniformly applies domain, Unix socket, MITM, and dynamic approval policies before actually going outbound. Network rejections are recorded by the observer rather than silently failing.

The remaining support chains each bear clear boundaries: config merges system, user, project, cloud requirements, and runtime overrides, preserving source and diagnostics; core-skills discovers, filters, renders, and injects skills by mention; connectors merge App metadata from directory and cache; cloud-tasks provides cloud task creation, environment detection, polling, and diff display; external-agent module migrates other agents' MCP, Hooks, commands, and sessions; analytics coalesces raw facts into product events; otel installs log, trace, and metric exporters, flushing on shutdown. The small crates under utils provide path, PTY, streaming text parsing, images, templates, caching, sleep suppression, and other stateless foundational capabilities.

  • Configuration is the common root of permissions and lifecycle: plugin admission, skill disabling, Hook trust, and network constraints all derive from traceable configuration layers.
  • Remote plugins, MCP OAuth, model API, cloud tasks, and OTLP exporters are all external communication boundaries; failures are converted into domain errors or diagnostic information.
  • Plugin caching and remote refresh use serialized state to avoid concurrent refresh overwrites; telemetry and analytics events use queuing and deduplication limits for capacity control.
  • Tests, generated schema/protobuf, and fixed templates are not expanded as runtime implementations; small utility crates have been mapped by public API and ownership.
