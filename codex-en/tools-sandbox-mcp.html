Tool Execution, Approval, Sandbox, and MCP

When the model requests running a local command, modifying a file, calling an MCP or extension tool, the system converts response items into unified tool calls, dispatches them through the registry and lifecycle hooks, then selects approval and sandbox according to permission policy; the results are finally collated into model-consumable output. This module only drills down into the tool domain and does not duplicate the CLI main chain.

Input
Function/custom tool calls produced by the model, user permission policy, working directory and environment
Output
Controlled execution of command output, file changes, MCP/extension results, or explicit rejection and sandbox errors
graph TD
  response-item["Model returns tool call items"] --> decode-call{"Identify function, custom, or search call"}
  decode-call -->|not a client tool| ignore-call["Do not execute locally"]
  decode-call -->|executable| registry-dispatch["Look up executor by namespace and name"]
  registry-dispatch --> exposure-check{"Is the tool exposed and do parameter types match?"}
  exposure-check -->|No| unsupported["Return unsupported tool error"]
  exposure-check -->|Yes| pre-hook["Run tool start and pre-hooks"]
  pre-hook --> invoke-handler["Enter Shell, Patch, MCP, Code Mode, or Extension handler"]
  invoke-handler --> post-hook["Record result and run post-hooks"]
  post-hook --> model-output["Transform and truncate for model output"]

The registry simultaneously controls tool visibility to the model, whether parallel execution is allowed, and cancellation semantics. Executors contributed by extensions are added to the routing table on each turn construction, so built-in and extension tools share the same lifecycle boundary.

graph TD
  local-request["Local command or patch forms a run request"] --> approval-rule{"Calculate approval requirements"}
  approval-rule -->|forbidden| reject-now["Reject immediately"]
  approval-rule -->|needs approval| approval-cache{"Check cache or request user/Guardian"}
  approval-cache -->|rejected| reject-review["Return rejection reason"]
  approval-cache -->|approved| choose-sandbox["Select initial sandbox based on policy and permissions"]
  approval-rule -->|skip| choose-sandbox
  choose-sandbox --> network-gate["Establish managed network approval and cancellation signal"]
  network-gate --> transform-command["Convert command to platform sandbox request"]
  transform-command --> execute-process["Start process and capture output"]
  execute-process --> attempt-ok{"Execution successful?"}
  attempt-ok -->|success| finish-network["Finalize network approval and return result"]
  attempt-ok -->|sandbox refused and can escalate| retry-check{"Does policy allow retry without sandbox?"}
  retry-check -->|No| sandbox-error["Return sandbox error"]
  retry-check -->|Yes| escalated-run["Reuse approval, escalate privileges and retry"]
  escalated-run --> finish-network
  execute-process --> linux-path["Linux: bubblewrap/Landlock/seccomp"]
  execute-process --> windows-path["Windows: restricted token, ACL, job object"]
  execute-process --> unix-escalation["Unix: can request execve escalation via socket"]

Shell first parses command safety and approval keys, and retains rejected read rules; Patch first parses, validates target paths and write permissions, then applies file-by-file changes, returning already committed deltas even on failure, avoiding reporting a partial success as completely unmodified.

  extension-lifecycle --> extension-result["Convert extension output back to core turn item"]
  • Permission boundary: approval requirements, additional file permissions, network permission, and platform sandbox are layered; privilege escalation removes the Codex-managed proxy environment to avoid leaking controlled network configuration into unsandboxed processes.
  • Communication boundary: the MCP connection manager handles server startup, tool discovery, invocation, and resource reading; mcp-server conversely wraps the Codex session as MCP tools and forwards approval through the message handler.
  • Lifecycle: Code Mode cells can execute, wait, and terminate; extensions can contribute tools and observe start/completion events; the unified exec session supports continuous processes and subsequent stdin.
