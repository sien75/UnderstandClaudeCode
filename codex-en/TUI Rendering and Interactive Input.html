TUI Rendering and Interactive Input

This module covers the entire interaction chain in Codex's terminal interface (TUI): how the user types, submits, answers agent questions, and resumes old sessions, plus how the interface is drawn line by line after each action. The core container is BottomPane (the bottom panel): it holds the permanent ChatComposer (chat input box) and a temporary view stack (popups/modals). All keyboard events arrive here first, and it decides "who gets this key." At render time it assembles itself into a flex layout: status line + pending-input preview + input box (or the topmost popup). Session recovery follows a separate alt-screen flow (resume_picker).

The main perspective is data flow: from key press in, to pixels on screen out. Auxiliary perspectives are state machines (view-stack push/pop, RequestUserInputOverlay multi-question state machine) and communication boundaries (AppEvent to the app layer, TuiEvent/background events back in).

Key concepts at a glance

BottomPane (bottom panel)
The manager of the bottom interaction area. Holds the permanent ChatComposer plus a view stack. Keyboard events arrive here first; it decides which surface should receive them. It does not handle process-level decisions like quit/interrupt; those belong to the upper-level ChatWidget.
ChatComposer (chat composer)
The real multiline input box, containing a TextArea (plain text editor) + popups (@ mentions, / slash commands, skills) + history navigation + a bottom footer hint. After submission it produces InputResult::Submitted.
BottomPaneView trait (view stack interface)
Every popup or modal that can be layered over the composer implements this trait: approval overlay, user-input-request overlay, MCP elicitation, list selector, etc. The trait defines keyboard handling, completion semantics, Ctrl+C routing, paste-burst coordination, timed ticks, and a hook for whether a new request can be consumed by an already-visible view.
RequestUserInputOverlay (user-input-request modal)
Pops up when the agent proactively asks the user multiple-choice or free-text questions. Reuses ChatComposer as a notes input box. Internally it is a multi-question state machine: each question is optional and can have a note; after submitting the current question it jumps to the next; on the last question it submits all answers back to the app.
TranscriptState (conversation stream state)
The state object inside ChatWidget that manages the currently active cell (the streaming segment), historical markdown copy buffer, and plan-increment buffer. At render time the active cell is wrapped as TranscriptAreaRenderable and drawn.
SessionPicker (session recovery selector)
A standalone alt-screen full-screen interface that lists historical session threads from the app-server. Cursor-paginated fetching, local search filtering, Ctrl+T to view transcript preview, Enter to resume the selected session.

Flow 1: BottomPane keyboard routing and rendering

Every key press enters BottomPane::handle_key_event. The core decision is: is there an active view (popup) right now? If yes, hand the key to the topmost view; otherwise hand it to the composer. Special keys (interrupt_turn, Ctrl+C, Esc) have dedicated priority handling. Rendering goes through BottomPane::as_renderable, which vertically flex-stacks "status line / unified footer / pending preview / composer," or directly draws the view when one is present.

Flow 2: BottomPane rendering data flow (no popup)

ChatWidget's as_renderable (rendering.rs) splits the chat area into two large blocks: above is the transcript active cell (streaming/history), below is BottomPane. BottomPane itself vertically flex-assembles: status line → unified exec footer → pending thread approvals → pending input preview → composer. Each layer is a Renderable and is finally drawn into the Buffer by ratatui.

Flow 3: RequestUserInputOverlay multi-question state machine

When a tool call requires user input (multiple choice + free text), the app layer sends ToolRequestUserInputParams. BottomPane calls push_user_input_request to push the overlay onto the view stack and disable the composer. Inside, the overlay is a multi-question state machine: focus switches between Options (option list) and Notes (notes composer); after submitting the current question it jumps to the next; on the last question or when unanswered questions are detected it enters a confirmation sub-flow, and finally sends the answers HashMap back to the app.

Flow 4: Session recovery selector (resume_picker)

When the user picks a historical session to resume, a separate alt-screen flow runs. The entry run_resume_picker_with_app_server enters the alternate screen, spawns a background task to asynchronously paginate the thread list, and the main loop tokio::select waits for both TUI events and background events. Keys fall into three categories: navigation (up/down/page/home/end), search (ordinary characters enter the query), and actions (Ctrl+T view transcript, Ctrl+E expand, Ctrl+O toggle density, Enter resume, Esc/Ctrl+C exit).

Auxiliary: BottomPaneView trait (unified interface for the view stack)

Every popup that can be layered over the composer implements BottomPaneView. This trait is what lets the view stack be stacked, consumed by new requests, and timed. It unifies cross-popup behavior such as completion semantics (Accepted/Cancelled), Ctrl+C priority consumption, paste-burst coordination, and app-server request retraction. BottomPane only calls the corresponding trait methods in handle_key_event / push_* / dismiss_app_server_request without caring about the concrete view type.

  • handle_key_event / on_ctrl_c / handle_paste: input routing
  • is_complete + completion: completion semantics, deciding whether to cascade-dismiss child views on pop
  • try_consume_approval_request / try_consume_user_input_request / try_consume_mcp_server_elicitation_request: let an already-visible view of the same kind swallow a new request instead of stacking another layer
  • dismiss_app_server_request: another client has answered the same request, so the local view retracts synchronously
  • pre_draw_tick + next_frame_delay: ticks for time-driven UI (countdowns, animations)
  • terminal_title_requires_action: tells the terminal title bar to show "action required" instead of an ordinary spinner

Auxiliary: Approval delay (approval_prompt_typing_idle_delay)

When the agent requests approval (execute command / permission), if the user is typing, BottomPane does not immediately pop an approval modal and interrupt input. Instead it queues the request in delayed_approval_requests and promotes it to a modal only after the user has stopped typing for one second. Every visible-character key refreshes last_composer_activity_at, resetting the delay. pre_draw_tick calls maybe_show_delayed_approval_requests_at to check whether the delay has elapsed.

Auxiliary: Streaming and transcript connection

The "upper block" of the rendering layer (transcript) is fed by ChatWidget's streaming module. Agent message deltas enter handle_streaming_delta and accumulate in stream_controller; when complete, flush_answer_stream_with_separator merges the streaming cell into a single AgentMarkdownCell and emits AppEvent::ConsolidateAgentMessage so history can be re-rendered. TranscriptState maintains active_cell_revision (cache invalidation counter), agent_turn_markdowns (copyable per-turn markdown), plan-increment buffer, etc.

Auxiliary: TextArea editing capabilities

textarea.rs is a multiline editor with built-in Vim mode. Core fields: raw text + cursor_pos + wrap_cache (soft-line cache) + elements (user text elements such as mention byte ranges) + kill_buffer (clipboard, character-level vs line-level). Input splits into two paths: Emacs style (input_with_keymap, bound to EditorKeymap) and Vim style (handle_vim_input → normal/insert/operator/text-object). set_text_inner is the entry for full text replacement: it rebuilds elements, clamps the cursor, clears wrap_cache, but preserves kill_buffer (so users can still Ctrl+Y to recover after submit/clear).

Notes (supplementary)

  • Layered routing: BottomPane only decides "who gets the key"; process-level decisions (final semantics of quit/interrupt) are left to ChatWidget. Ctrl+C is consumed by the view first (close popup), then by composer's history-search cancel, then by clearing the draft, and only then does ChatWidget treat it as quit/interrupt. This priority is critical in on_ctrl_c and handle_key_event.
  • Composer reuse: RequestUserInputOverlay does not create its own input box; it creates a ChatComposer (plain_text config, popups/slash/images disabled), so notes input style, paste behavior, and keymap match the main input box.
  • Paste-burst: Large pastes are sliced into burst frames and flushed progressively; during this time BottomPane schedules short delayed redraws to leave a window for flushing. The view trait also has is_in_paste_burst / flush_paste_burst_if_due so modal composers participate.
  • Auto-resolution: RequestUserInputOverlay supports the agent-set autoResolutionMs—60 s hidden grace + 60 s visible countdown, after which it auto-submits empty answers. Any key press or paste snoozes it.
  • Render caching: TranscriptAreaRenderable uses active_cell_revision for cache invalidation; BottomPane's set_* methods all follow a "compare first, request_redraw only if changed" pattern to avoid unnecessary redraws on thread switches.
  • Alt-screen isolation: resume_picker uses AltScreenGuard to enter the alternate screen; Drop switches back, ensuring the main chat interface is not polluted by the picker's full-screen list.
