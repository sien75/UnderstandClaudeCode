App Server and CLI Management Surface

This module describes how desktops, IDEs, scripts, or administrators control sessions, turns, accounts, plugins, MCP, remote control, and isolated execution through Codex's public JSON-RPC API and command line. External input is first restored into typed requests by the protocol layer, then passes through connection initialization, experimental capabilities, and permission boundaries, and is finally serialized by resource scope and delivered to the corresponding processor; results and ongoing events are returned along the original connection.

Input
CLI subcommands, or JSON-RPC requests and notifications from stdio, WebSocket, Unix socket, or remote control channels
Output
Typed responses, errors, and ongoing notifications; or a remote execution environment constrained by session and filesystem boundaries

The backbone of the public API is not a set of independent handlers. Protocol macros first bind parameters, responses, experimental flags, and serialization scope to each method; the message processor then performs connection-level checks and dispatches calls to thread, turn, account, plugin, MCP, filesystem, or process processors. This ensures that write operations on the same thread or process do not cross each other, while unrelated resources can still be processed concurrently.

The communication layer places the same message semantics on different carriers: local CLI commonly uses stdio, desktop applications and daemons can use Unix socket or WebSocket, and remote control forwards connections after authentication, pairing, and desired state synchronization. The transport layer is only responsible for connection, framing, and outbound backpressure; request semantics remain under the unified message processor.

The CLI management surface can both directly start an interactive Codex session and manage a long-running App Server. The `app-server` subcommand handles the service lifecycle, remote control toggle, proxy, and protocol artifacts; the independent `exec-server` is designed for executors, wrapping processes, filesystems, and HTTP requests into JSON-RPC, and constraining resources with initialization sessions, process registration tables, and sandbox context.

  • Type boundary: v2 request fields use camelCase on the wire, with the exception of configure RPC which keeps snake_case consistent with config.toml; method parameters, responses, and notifications can jointly generate TypeScript and JSON Schema.
  • Permission boundary: Initialization capabilities determine whether experimental APIs are visible; the connection permission gate restricts high-risk methods; exec-server additionally checks initialization order, session ownership, and filesystem sandbox context.
  • Compatibility boundary: The protocol still retains v1 mapping, but new APIs are concentrated in v2; clients can either embed the in-process service or connect to a remote one via Unix socket or WebSocket.
  • Execution results may not be one-shot responses: long command output, turn items, approval requests, account status, and remote control status can all be pushed via server notifications.
