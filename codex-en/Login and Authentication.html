Login and Authentication

This group of crates solves the core problem: how users tell Codex "who I am and what I can do", and how Codex securely stores that credential, refreshes it on demand, automatically recovers from expiration or 401 errors, and cleanly revokes it on logout. The entry point is codex-login, which exposes AuthManager as the single source of truth for credentials across the program; surrounding crates each handle one part: agent-identity manages machine-agent identity JWTs, keyring-store/secrets handles OS keyring and encrypted file storage, aws-auth handles Bedrock SigV4 signing, the chatgpt crate is the consumer—it takes the token from AuthManager and calls the ChatGPT backend APIs—and install-context determines how Codex itself was installed (affecting originator headers and bundled resource paths).

From the user's perspective, this scenario supports: ① after installing Codex, first-time codex login via browser OAuth or device code login; ② directly running codex login --api-key / pasting in a PAT starting with at- / pasting in an AgentIdentity JWT; ③ setting the CODEX_API_KEY or CODEX_ACCESS_TOKEN environment variables to run without logging in; ④ an external host (IDE/plugin) temporarily injecting a ChatGPT token or custom bearer command through the ExternalAuth trait; ⑤ automatic refresh or 401 recovery by AuthManager when a token expires or is rejected by the server; ⑥ codex logout to revoke and clear local credentials.

Module Inventory

codex-login (auth/manager.rs)
Authentication central hub. Merges three sources—persistent auth.json, environment variables, and optional external provider—according to a fixed priority into a CodexAuth snapshot cache, and provides token retrieval, refresh, logout, and 401 recovery. Input: codex_home, storage mode, environment variables, optional external provider. Output: currently valid CodexAuth (or None = not logged in).
codex-login (auth/storage.rs)
How credentials are persisted. Selects one backend according to storage mode: file / direct OS keyring / encrypted age file / auto fallback / in-process ephemeral, unified behind load/save/delete interfaces. Input: serialized AuthDotJson bytes. Output: the same bytes read back. Note: keyring account name is a truncated sha256 of the codex_home path, preventing collisions between instances.
codex-login (server.rs + device_code_auth.rs + pkce.rs)
Two interactive login paths. The browser path starts a local server on port 1455 to receive the OAuth callback; the device code path is used when no browser is available, prompting the user to open a link on another device and enter a code. Both paths end by calling exchange_code_for_tokens to obtain id/access/refresh tokens and persist them. Input: issuer, client_id, PKCE. Output: written auth.json.
codex-login (auth/personal_access_token.rs / agent_identity.rs / bedrock_api_key.rs / external_bearer.rs / access_token.rs)
Non-OAuth login branches. PAT calls /whoami to fill in account metadata; AgentIdentity JWT is verified against JWKS and then registered for a task; Bedrock API key is persisted directly; external bearer runs a user-configured command and uses stdout as the token. Input: respective raw credential string. Output: a structure that fits into CodexAuth.
codex-login (auth/revoke.rs)
Revokes OAuth tokens on logout. Prefers refresh_token, falls back to access_token; revocation failure does not block local cleanup. Input: token from auth.json. Output: a POST to auth.openai.com/oauth/revoke.
codex-agent-identity
Cryptographic layer for machine-agent identity. Generates ed25519 key pairs, decodes JWTs, fetches JWKS for verification, signs tasks into AgentAssertion headers, registers tasks to obtain task_id (with curve25519 decryption when needed). Input: JWT or PKCS8 private key. Output: verified claims or signed header. Note: this is for automated agents, not human login.
codex-keyring-store
Thin wrapper around the OS keyring. Wraps the keyring crate's load/save/delete into a KeyringStore trait with a Mock implementation for tests. Input: service+account. Output: plaintext string. Note: the secrets crate and auth storage both rely on it as the underlying layer.
codex-secrets
Local encrypted secrets store. Uses age (scrypt) to encrypt a JSON map into files such as codex_auth.age, with the encryption key in the OS keyring; auth's SecretsKeyringAuthStorage uses it as "double-insurance" encrypted storage. Input: scope+name+value. Output: decrypted string. Note: the CodexAuth namespace is kept separate for CLI authentication credentials.
codex-aws-auth
AWS Bedrock SigV4 signing. Resolves credentials via the AWS SDK default chain (profile/region/env/IMDS), then signs outbound requests with SigV4. Input: AwsAuthConfig + request to be signed. Output: request headers with Authorization/x-amz-date. Note: this is entirely parallel to the OAuth flow and does not go through AuthManager.
codex-chatgpt
One of the authentication consumers. Uses AuthManager to get a ChatGPT token and then calls backend APIs: list connectors, pull task diff, check workspace plugins. Input: Config + CodexAuth. Output: deserialized business objects. Note: it only reads authentication state, never writes.
codex-install-context
Detects how Codex itself was installed (standalone/npm/bun/brew/other) and the bundled resource paths. Input: current_exe, environment flags. Output: InstallMethod + resource directory. Note: not directly related to authentication, but originator headers and CA resource paths affect how login requests are sent.

Data Flow 1: Credential Loading at Startup (Priority Chain)

This chain answers "when the program starts, which credential does it actually use?" Priority is fixed: CODEX_API_KEY env var > in-process ephemeral storage (externally injected ChatGPT token) > CODEX_ACCESS_TOKEN env var (login-less PAT/AgentIdentity) > persistent storage (file/keyring/secrets). This ordering ensures: credentials temporarily injected by an external host override local login, and local login overrides environment variables. Ephemeral ranks above CODEX_ACCESS_TOKEN because external host injection uses ephemeral storage.

    A2 -- yes --> A3[Use directly as API Key]
    A2 -- no --> A4{Ephemeral in-memory storage has credentials?}
    A4 -- yes --> A5[Read auth.json from ephemeral]
    A4 -- no --> A6{CODEX_ACCESS_TOKEN env var set?}
    A6 -- yes --> A7[Classify access token by prefix]
    A6 -- no --> A8{Requesting ephemeral mode?}
    A8 -- yes --> A9[Return not logged in None]
    A8 -- no --> A10[Read auth.json from persistent storage]
    A5 --> A11[Parse into CodexAuth by auth_mode]
    A10 --> A11
    A7 --> A12{PAT or AgentIdentity JWT?}
    A12 -- prefix at- is PAT --> A13[Call /whoami to fill account metadata]
    A12 -- no --> A14[Treat as AgentIdentity JWT and verify]
    A11 --> A15[Validate workspace restrictions]
    A13 --> A15
    A14 --> A15
    A3 --> A16[Cache in memory as CachedAuth for global retrieval]
    A15 --> A16

Data Flow 2: Browser OAuth Login

This path is taken when the user runs codex login (without extra arguments). Codex starts a tiny_http server on local port 1455 (if occupied it sends a cancellation request and retries; if that also fails it falls back to 1457), generates PKCE and state, builds the authorization URL, and opens it with webbrowser. After the user completes ChatGPT login in the browser, the OpenAI authorization server redirects the code to /auth/callback. The local server exchanges the code at /oauth/token for id/access/refresh tokens, then uses id_token in a token exchange to obtain an API key, finally stores the whole set as auth.json and redirects the browser to a /success page to finish. Any step failure renders an error page and exits with an error; user cancellation goes through /cancel.

Data Flow 3: Device Code Login

This path is used in browserless environments (SSH/containers). Codex first requests a user code and polling interval from /deviceauth/usercode, prints the link and code for the user to complete login on another device with a browser; at the same time it begins polling /deviceauth/token. The server returns 403/404 while the user has not completed login, and Codex retries at the interval for up to 15 minutes. Once the user confirms, the server returns authorization_code and code_verifier; Codex then uses the standard exchange_code_for_tokens to exchange tokens, validates workspace, and persists. Compared to the browser chain, it lacks the local callback server and the API key exchange step.

    C6 --> C7[Poll /deviceauth/token]
    C7 --> C8{Polling result?}
    C8 -- 403/404 not completed --> C9{Exceeded 15 minutes?}
    C9 -- no --> C7
    C9 -- yes --> C10[Report timeout]
    C8 -- success --> C11[Get authorization_code and code_verifier]

Data Flow 4: Token Refresh and 401 Recovery State Machine

This is the most critical runtime piece of authentication. UnauthorizedRecovery is an explicit state machine driven only when a request receives a 401 (one next() call per retry). It has two major modes: Managed (locally managed ChatGPT login) goes through Reload → RefreshToken: first reload from disk (in case another Codex process already refreshed the token), and only if credentials haven't changed proceed to auth.openai.com refresh; on success persist the new token and reload cache. External (externally injected ChatGPT token or custom bearer command) takes one step: call the external provider's refresh() to get a new token, store it in ephemeral, and reload. API key / PAT / Bedrock credentials that cannot be refreshed bypass recovery entirely, and 401s propagate as-is.

Additionally, every auth() call has a proactive refresh branch: if the access_token expires within 5 minutes, or last_refresh is older than 8 days, it triggers a refresh first to avoid waiting for a 401. Permanent refresh failures are cached, and the same credential will not retry network calls, instead failing fast.

    D23 -- yes --> D11
    D23 -- no --> D24[Return cached credentials]
    D1[API receives 401 triggers recovery] --> D2{Credential type?}
    D2 -- managed ChatGPT login --> D3[Mode=Managed start=Reload]
    D2 -- external ChatGPT token / external bearer --> D4[Mode=External start=ExternalRefresh]

Data Flow 5: Logout and Revocation

Logout has two paths: normal logout only deletes local storage; logout_with_revoke first attempts to revoke OAuth tokens with OpenAI (preferring refresh_token, otherwise access_token) and then deletes local storage. Revocation failure only warns and does not block—local credentials are still deleted, because revocation is best-effort and network issues should not prevent the user from logging out. Deletion covers both ephemeral and persistent storage, ensuring externally injected temporary credentials are also cleared. Finally, a reload is always performed to clear the in-memory cache.

graph TD
    E1[User logs out] --> E2{Revoke token?}
    E2 -- normal logout --> E3[Delete ephemeral and persistent storage]
    E2 -- logout with revoke --> E4[Read current auth.json]
    E4 --> E5{Has revocable token?}
    E5 -- refresh_token non-empty --> E6[POST /oauth/revoke to revoke refresh_token]
    E5 -- no access_token non-empty --> E7[POST /oauth/revoke to revoke access_token]
    E5 -- neither --> E8[Skip revocation]
    E6 --> E9{Revocation succeeded?}
    E7 --> E9
    E9 -- no --> E10[Warn but continue]
    E9 -- yes --> E11[Delete ephemeral and persistent storage]
    E10 --> E11
    E3 --> E12[Reload cache clear login state]
    E11 --> E12

Data Flow 6: Credential Storage Backend Selection

Where auth.json is actually stored is determined by the AuthCredentialsStoreMode + AuthKeyringBackendKind configurations. File mode is codex_home/auth.json (0600 permissions). Keyring mode splits into Direct (straight into OS keyring) and Secrets (encrypted age file with key in keyring). Auto mode prefers keyring and falls back to file on failure. Ephemeral mode uses an in-process HashMap that is lost when the process exits (external host temporary credentials use this). The keyring account name is a truncated sha256 of the codex_home path, ensuring different codex_home directories on the same machine do not interfere.

graph TD
    F1[Create credential storage backend] --> F2{Storage mode?}
    F2 -- File --> F3[auth.json file permissions 0600]
    F2 -- Keyring --> F4{Keyring backend?}
    F4 -- Direct --> F5[Store directly in OS keyring service=Codex Auth]
    F4 -- Secrets --> F6[Encrypted age file codex_auth.age key in keyring]
    F2 -- Auto --> F7[Try keyring first, fall back to file on failure]
    F2 -- Ephemeral --> F8[In-process HashMap lost on exit]

Edge: External Bearer Command Refresh

A custom model provider can configure a shell command as an authentication source (BearerTokenRefresher). It implements the ExternalAuth trait; resolve() caches stdout as a bearer token according to the configured refresh_interval; refresh() ignores the cache and re-runs the command. This chain is entirely independent of ChatGPT OAuth refresh, taking the same External path in 401 recovery but without touching disk or storing ephemeral.

Edge: AgentIdentity Task Signing

AgentIdentity is not a human login; it is for automated agents. The JWT carries agent_runtime_id and a PKCS8 private key; during loading, in addition to verification, it signs agent_runtime_id:timestamp and POSTs to /v1/agent/{id}/task/register to register a task. The server may return a plaintext task_id or a curve25519-encrypted one (decrypted using a curve25519 private key derived from the ed25519 private key). Afterward, every request carries an AgentAssertion header containing a signature over agent_runtime_id:task_id:timestamp. This system is entirely parallel to OAuth tokens and does not participate in refresh/revoke.

Edge: AWS Bedrock SigV4 Signing

Bedrock uses standard AWS SigV4 and has nothing to do with OpenAI's OAuth system. AwsAuthContext::load resolves the credentials provider and region using the AWS SDK default chain (profile/region/env/IMDS), and each outbound request is signed with sign_request to add Authorization, x-amz-date, and x-amz-security-token headers. Credentials provider timeout/transient errors are retryable; configuration errors are not.
