SDK Public API and Local Runtime Distribution

SDK users enter Codex through stable TypeScript exports or Python top-level objects. Both implementations ultimately need to find a Codex executable matching the current platform, but TypeScript resolves the binary via npm optional dependencies and starts a process per turn, while Python prefers using a standalone wheel runtime package and lets an App Server connection carry account, thread, turn, and goal operations.

Input
SDK configuration, account and thread operations, user input, goal budget, and the current operating system and CPU architecture.
Output
Typed synchronous/asynchronous SDK results; or an installable wheel containing the correct platform's Codex CLI for the SDK to launch at runtime.

The TypeScript entry point only exports selected public types and classes. The client resolves npm platform packages at construction time; at runtime it converts nested configuration into TOML overrides understandable by the CLI, and prepends bundled tool directories to PATH.

Python's public layer provides both synchronous and asynchronous facades; the async version offloads blocking client work to a thread. In addition to regular turns, goal operations collapse multiple physical turns into one logical flow: it truly ends only when the goal pauses, blocks, reaches its limit, or completes, and maintains consistent logical turn numbering across interruptions and turn-rolling races.

Python's CLI binary is independently published as a platform wheel. The build hook prohibits source distributions and pins non-pure-Python platform tags; the development/release helper pipeline derives the version from the SDK's exact dependency pin, downloads the corresponding GitHub Release assets, stages the runtime package and force-installs it, then finally verifies version and binary availability again.

  • TypeScript's type schema defines a discriminated union of events and thread items; Python's protocol model is primarily provided by generated code, and this report marks generated files as ignored, only delving into handwritten adaptation, validation, and lifecycle logic.
  • Python's approval mode configures both sandbox and approval policy simultaneously, avoiding dangerous combinations being scattered to the caller; initialization metadata and version resolution ensure the server can identify the SDK source.
  • Runtime download includes multiple levels of fallback: anonymous direct link, token-authenticated GitHub API, and finally authenticated gh. Any step failure retains diagnostic context rather than silently selecting the wrong platform.
