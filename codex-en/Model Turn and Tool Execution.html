Model Turns and Tool Execution

When a user initiates a request from the TUI, exec, app-server, or SDK, it ultimately enters the same core pipeline: the request is first turned into a submission with an ID, the session loop receives it in order, creates the TurnContext for this turn, writes the user input, attached context, skill and plugin prompts into history, and then initiates streaming sampling to the model. If the model only replies with text, the turn ends; if the model requests to call a tool, the system first executes the tool according to current permissions and the tool registry, appends the tool output back to the model input, and continues to the next round of sampling until the model explicitly ends or there is no pending input.

Input
User messages, attached context, thread settings, permission configuration, current workspace, and available tools
Output
A client-facing event stream: reasoning/text deltas, tool start and completion, token usage, diff, final assistant message or error

  enqueue-submission --> session-loop["Session loop dispatches Op in order"]
  session-loop --> create-turn["Creates TurnContext for this turn"]
  create-turn --> steer-or-start["Decides whether to append to active turn or start new task"]
  steer-or-start --> record-input["Records user input and attached context"]
  record-input --> inject-skills["Injects skills, plugins, and extended context"]
  inject-skills --> build-history["Builds model input from history"]
  build-history --> build-tools["Exposes tools according to this turn's config"]
  build-tools --> stream-model["Initiates streaming sampling to model"]
  stream-model --> handle-text["Text/reasoning deltas converted to client events"]
  stream-model --> handle-tool["Model tool call enters ToolRouter"]
  handle-tool --> dispatch-tool["Executes tool according to registry and permissions"]
  dispatch-tool --> tool-output["Tool output written back to model input"]
  tool-output --> build-history
  stream-model --> complete-response["response.completed summarizes usage and whether to continue"]
  complete-response --> need-follow-up["Checks for remaining tools, pending input, or compression continuation"]
  need-follow-up --> build-history
  need-follow-up --> finish-turn["Runs stop hook and ends turn when no follow-up work"]
  finish-turn --> emit-final["Emits final events, tokens, and diff"]
  • The Codex facade only exposes queue interfaces for submitting and receiving events; concrete behavior is dispatched by the background session loop after consuming an Op.
  • TurnContext is the source of truth for a turn, containing the model, permissions, sandbox, environment, skills, dynamic tools, output schema, and telemetry.
  • Tools are not executed directly by the model; model output is first parsed into ToolCall, then ToolRouter finds the corresponding executor from the registry.
  • A turn may loop multiple times: tool output, automatic compression, stop hook continuation, or mid-turn user input all cause the system to rebuild model input and continue sampling.
