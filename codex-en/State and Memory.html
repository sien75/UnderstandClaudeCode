State and Memory

This group of crates handles all read/write access to Codex's "persistent state" in the local process: session thread metadata, generated memories, global message history, and the parent-child derivation topology graph between agents. You can think of it as everything the Codex CLI leaves behind after a conversation and can reuse next time. Technically it is 4 SQLite databases + a set of JSONL files + a git-baseline workspace, implemented with Rust async + sqlx.

Five crates are involved:

  • codex-rs/state — The core state runtime, StateRuntime, holding 4 SQLite connection pools (state/logs/goals/memories), providing read/write APIs for thread metadata, goals, memory tasks, backfill, and log persistence.
  • codex-rs/memories — Memory read path (read sub-crate: injection, reference resolution, usage statistics) and write path (write sub-crate: two-phase memory extraction pipeline phase1/phase2 at startup).
  • codex-rs/message-history — Append and trim the global message history file history.jsonl, safe for cross-process concurrency.
  • codex-rs/thread-store — Storage-neutral thread persistence abstraction, ThreadStore trait + local file/SQLite implementation LocalThreadStore, managing rollout JSONL writes and thread metadata sync.
  • codex-rs/agent-graph-store — Storage-neutral trait for thread derivation parent-child topology + local implementation, directly reusing StateRuntime's spawn edge table underneath.

What Users / the System Can Do

  • Start a new session → state writes the threads table + thread-store creates rollout JSONL + message-history appends user input
  • Continuously append output during a session → thread-store append_items writes rollout, state.touch_thread_updated_at updates timestamp
  • End / resume a session → thread-store load_history reads rollout, state reads thread metadata
  • Run the memory pipeline automatically at startup → memories/write phase1 extracts raw memory from old rollouts, phase2 consolidates into MEMORY.md
  • List / search historical threads → state.list_threads / thread-store.search_threads
  • Archive / delete / unarchive threads → corresponding thread-store operations + state sync
  • Agent spawns a child agent → agent-graph-store records parent-child edge, state.upsert_thread_spawn_edge
  • Record usage and prune memories → state.memories.record_stage1_output_usage / prune_stage1_outputs_for_retention
  • Persist tracing logs → state.log_db acts as a tracing Layer that writes logs DB in the background in batches

Key Flow 1: StateRuntime Initialization (Opening 4 SQLite Databases)

StateRuntime::init is the entry point for all upper-layer state access. It opens the state, logs, goals, and memories SQLite databases in a fixed order; any failure rolls back already-opened connection pools. Each database follows the same three steps: create_if_missing + WAL mode → connection pool (max 5 connections) → run migrations. After opening, it also: ensures backfill_state has an initial row, reads the maximum updated_at/recency_at timestamps from the threads table and caches them in AtomicI64 (subsequent hot paths read directly from memory), and runs a logs DB startup maintenance (pruning excess logs).

Communication boundary: the four databases each have independent connection pools; logs is split into its own database to reduce write-lock contention with the main state database (WAL + busy_timeout 5s). The recovery path (recovery.rs) recognizes SQLite corruption errors and supports rebuilding after backup.

Key Flow 2: Memory Two-Stage Pipeline (phase1 + phase2)

This is the most complex data flow in the whole scenario. start_memories_startup_task is called when the root session starts, skipping temporary sessions, sessions without the MemoryTool feature enabled, and child-agent sessions. After passing the gate, it spawns a tokio task that runs phase1 and phase2 sequentially.

Phase 1 (Extraction): Claim expired threads' stage-1 tasks from state.memories (based on updated_at time window + memory_mode=enabled + excluding the current thread). After claiming tasks, it concurrently (up to 8) calls the model to compress rollout history into a raw_memory + rollout_summary, and writes the results back to the stage1_outputs table on success. Task scheduling uses SQLite INSERT...ON CONFLICT as an optimistic lock claim, with lease/retry/backoff to avoid duplicate work across processes.

Phase 2 (Consolidation): First claim a globally unique lock (memory_consolidate_global job), then prepare_memory_workspace (create a git baseline for the memories directory), fetch the phase2 input selection from the DB, rebuild raw_memories.md and rollout_summaries/, generate a workspace diff file, then spawn an independent consolidation agent (via ThreadManager starting a new thread) for the model to read the diff + raw memories and edit MEMORY.md. After the agent completes, reset the baseline and mark the job as succeeded.

State machine: stage1 job has running/done/failed states, with lease_until/retry_at/retry_remaining fields controlling retries. phase2 is a global singleton job (memory_consolidate_global, job_key='global'); after success there is a 6-hour cooldown (PHASE2_SUCCESS_COOLDOWN_SECONDS). memory_mode has enabled/polluted states; polluted means the thread is marked for re-extraction.

Key Flow 3: Thread Persistence (thread-store + state Collaboration)

The ThreadStore trait is the storage-neutral thread persistence boundary, defining operations such as create/resume/append/persist/flush/shutdown/discard/load_history/read/list/search/update_metadata/archive/unarchive/delete. LocalThreadStore is the local implementation: rollout JSONL files serve as the durable replay format (readable without SQLite), while the SQLite state DB provides a queryable metadata index.

LiveThread is the handle for an active thread, holding Arc + ThreadMetadataSync. ThreadMetadataSync derives metadata updates from canonical rollout items (cwd, preview, first_user_message, title, git info) and writes them back to state in batches according to a throttling interval (5s). LiveThreadInitGuard is used for rollback during session initialization: when dropped, it discards the live writer without forcing a flush.

Communication boundary: append_items only writes the rollout and does not infer metadata; metadata patches are explicitly issued by the upper layer via update_thread_metadata after observation. This keeps the storage implementation simple and leaves policy in the live layer.

Key Flow 4: Message History File (history.jsonl)

message-history is an independent, global, append-only JSONL file (~/.codex/history.jsonl), one line per {session_id, ts, text}. append_entry uses O_APPEND + a single write syscall for atomicity, plus an advisory file lock + retries (10 times, 100ms interval) to avoid cross-process interleaving from multiple TUI processes. After writing, it enforces_history_limit inside the lock: if the file exceeds max_bytes, old lines are dropped from the beginning and trimmed to the soft cap (80%) to avoid immediate re-trimming next time.

graph TD
    D1[append_entry] --> D2{persistence=None?}
    D2 -->|Yes| D3[Return immediately]
    D2 -->|No| D4[Build JSON line + \n]
    D4 --> D5[Open file O_APPEND 0o600]
    D5 --> D6[spawn_blocking to acquire lock]

    D8 -->|No| D10[Error WouldBlock]
    D7 -->|Success| D11[seek End + write_all]
    D11 --> D12[enforce_history_limit trim]
    D12 --> D13[Return Ok]

Key Flow 5: Agent Spawn Topology (agent-graph-store)

The AgentGraphStore trait defines operations for thread derivation parent-child edges: upsert edge, update status, list direct children, BFS list all descendants. LocalAgentGraphStore directly delegates to StateRuntime's methods of the same name; underneath it is the state DB's threads_spawn_edges table. Each child thread has at most one incoming edge; list_descendants is sorted by depth-first then thread id for stable output, making it easy for the upper layer to merge live in-memory state with persistent state.

Key Flow 6: Tracing Log Persistence (log_db Layer)

The log_db module is a tracing_subscriber::Layer that captures tracing events, formats them into LogEntry, sends them through a bounded mpsc queue (capacity 512) to a background task, and INSERTs them into the logs DB in batches (128 entries/batch or 2s interval). This keeps the logging path off the business hot path. state also prunes logs per thread_id bucket (max 10MiB / 1000 lines per bucket) and runs maintenance once at startup.

Module Inventory

state (StateRuntime)
Unifies 4 SQLite databases + backfill + thread metadata + goals + memory task scheduling into a single runtime entry point. Input: codex_home path + provider; Output: Arc<StateRuntime>, readable/writable for all persistent state.
state.log_db
Turns the tracing event stream into batched rows in the logs DB. Input: tracing Event; Output: logs table INSERT.
state.runtime.memories (MemoryStore)
Turns memory extraction/consolidation task scheduling into state-machine mutations on the jobs and stage1_outputs tables. Input: thread updated_at timestamps; Output: task claim/success/failure state machine.
memories/write
Turns old rollout history into the MEMORY.md file. Input: rollout JSONL + model inference; Output: raw_memories.md, rollout_summaries/, and MEMORY.md under the memories/ directory.
memories/read
Turns agent reads of memory files into usage statistics + reference resolution. Input: bash command string / model output citation; Output: MemoriesUsageKind tag + MemoryCitation struct.
thread-store
Turns the session lifecycle (create/append/flush/shutdown/resume) into rollout JSONL files + threads table metadata. Input: CreateThreadParams/RolloutItem; Output: persisted rollout + StoredThread metadata.
message-history
Turns a user input text into one line of history.jsonl. Input: text + conversation_id; Output: appended to the global JSONL file.
agent-graph-store
Turns agent derivation relationships into parent-child edges in the threads_spawn_edges table. Input: parent/child ThreadId + status; Output: persisted topology edge.

Notes

  • The four SQLite database filenames are fixed: state_5.sqlite / logs_2.sqlite / goals_1.sqlite / memories_1.sqlite, located under codex_home. The CODEX_SQLITE_HOME environment variable can override the home directory.
  • All databases use WAL + synchronous=Normal + busy_timeout=5s + incremental auto-vacuum, balancing performance and crash safety.
  • stage1 task claim uses a single INSERT...ON CONFLICT...WHERE SQL statement as an optimistic lock, avoiding application-level locks and remaining cross-process safe.
  • The memory workspace uses a git baseline repository for diffs, so the phase2 agent can see signals of "which memories were deleted".
  • ThreadMetadataSync throttling (5s) avoids hitting SQLite on every append, reducing write amplification.
  • history.jsonl uses O_APPEND + single write + advisory lock; POSIX guarantees atomic writes up to PIPE_BUF bytes, making it cross-process safe.
  • LocalThreadStore's rollout JSONL is a compatibility layer readable without SQLite; SQLite is only an acceleration index for queries, preserving SQLite-less repair capability by design.
