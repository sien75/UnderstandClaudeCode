Configuration, Authentication, and Model Connection

After a user starts Codex, layered configurations are merged into a runtime configuration constrained by managed policies; the system then selects a model service, reads or refreshes login credentials, discovers available models, and finally passes authenticated requests to the HTTP/SSE transport layer. The construction of model turn content belongs to the upper boundary; this module stops at "the connection is ready to send inference requests."

Input
System, user, project, session and command-line configuration, credentials from CODEX_HOME or system keychain, and optional external authentication commands
Output
A determined model provider, a currently valid identity, available model metadata, and an authenticated HTTP client capable of accessing the model or ChatGPT backend

The first data flow decides "where to request." Higher-priority configuration overrides lower-priority configuration, but requirements and permission constraints can still reject disallowed final values.

The authentication data flow deliberately maintains a single snapshot: running components do not each read auth.json independently. Files, system keychains, and temporary external credentials share a storage interface; auto mode prefers the keychain and falls back to the file on failure.

Requests encountering a 401 do not retry indefinitely. Managed ChatGPT identities reload from storage once, then go through OAuth refresh; external identities only notify the host or re-run the authentication command once; API Keys and non-refreshable personal tokens bubble the error directly to the caller.

The provider combines configuration and identity into runtime capabilities. Bedrock uses a dedicated implementation; general compatible services decide whether to use OpenAI identity, static token, environment variable, or external command based on configuration. The model catalog can be fully fixed by configuration, or coordinated between bundled, disk cache, and remote /models.

The final communication boundary is divided into two types: the ChatGPT backend client handles path style, account/FedRAMP and other business headers; the generic Codex client handles proxy, custom CA, compression, trace headers, normal responses, and SSE byte streams. Both inject Authorization from a unified authentication provider, avoiding token logic duplication at business call sites.

  • Permission boundary: managed requirements constrain user choices before Config is formed; login mode and ChatGPT workspace can also be enforced by organization configuration.
  • Secret boundary: authentication debug output does not expose tokens; keychain mode entrusts long-term credentials to the operating system, external bearer can exist only within the process.
  • Cache boundary: model cache determines freshness by client version and TTL, and preserves ETag; current implementation notes that provider identity is not yet included in cache eligibility determination.
  • Failure strategy: model catalog network failure retains the existing catalog; HTTP non-success status preserves sufficient context for the upper layer to classify, and the 401 recovery state machine is strictly bounded.
