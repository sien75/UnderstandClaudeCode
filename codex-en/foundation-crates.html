Foundation Crates: Turning Configuration and Requests into Executable, Constrained Codex Behavior

These independent crates do not directly own a conversation, yet together they support every conversation: they first fold defaults, user configuration, and managed configuration into a stable feature set, then select tools and execution backends accordingly; commands are wrapped into platform sandbox only after policy matching and permission convergence. Bypass capabilities handle cloud policy caching, Git workspace recognition, skill installation, secret protection, feedback collection, prompt assembly, and terminal identification.

Input
Local and cloud configuration, model tool calls, commands and working directory, additional permissions granted by the caller
Output
A set of determined feature toggles, model-visible tools, policy decisions, and sandbox requests safe for operating system execution

  config-in["Read defaults, base configuration, config profiles, and temporary overrides"] --> resolve-features["Merge feature toggles by priority"]
  resolve-features --> normalize-features{"Are there dependent features that need to be completed?"}
  normalize-features -->|Yes| enable-dependency["Complete collaboration or code mode dependencies"]
  normalize-features -->|No| choose-tools["Directly select model tools"]
  enable-dependency --> choose-tools
  choose-tools --> shell-enabled{"Is Shell and unified execution enabled?"}
  shell-enabled -->|No| classic-shell["Use classic backend or disable Shell"]
  shell-enabled -->|Yes| composition-enabled{"Is unified execution combined with zsh allowed?"}
  composition-enabled -->|Yes| zsh-unified["Select unified PTY + zsh bridge"]
  composition-enabled -->|No| direct-unified["Select direct unified PTY, or safe fallback"]

The feature registry also records lifecycle: stable capabilities can be enabled by default, experimental capabilities enter the menu, in-development capabilities remain off, and removed keys only exist for backward compatibility with old configuration. This prevents the upper layer from repeatedly explaining old keys everywhere, and avoids a combined toggle accidentally enabling its dependent capabilities.

graph TD
  command-in["Receive command and optional policy file"] --> parse-policy["Parse Starlark policy and validate examples"]
  parse-policy --> policy-valid{"Is the policy valid?"}
  policy-valid -->|No| reject-policy["Return error with file and line number"]
  policy-valid -->|Yes| exact-match["Match prefix rules by first program name"]
  exact-match --> found-rule{"Found a definite rule?"}
  found-rule -->|Yes| rule-decision["Summarize allow, prompt, or deny"]
  found-rule -->|No| host-match["Try to resolve host executable alias"]
  host-match --> host-found{"Host rule matched?"}
  host-found -->|Yes| rule-decision
  host-found -->|No| heuristic["Fallback using caller heuristic"]
  rule-decision --> normalize-permissions["Normalize temporary network and file permissions"]
  heuristic --> normalize-permissions
  normalize-permissions --> permission-valid{"Are permission paths and globs valid?"}
  permission-valid -->|No| reject-permission["Reject unsafe permission expansion"]
  permission-valid -->|Yes| intersect-permissions["Keep only the intersection of requested and granted permissions"]
  intersect-permissions --> choose-sandbox{"Is platform sandbox mandatory?"}
  choose-sandbox -->|No| unsandboxed["Retain original command execution form"]
  choose-sandbox -->|Yes| platform-sandbox["Select Seatbelt, bubblewrap/Landlock, or Windows sandbox"]
  platform-sandbox --> transform-request["Generate execution request after injecting policy, working directory, proxy, and certificates"]

The permission chain follows the principle of "cannot grant more than authorized": additional permissions are first deduplicated and paths normalized, read permission globs only allow expression of denials; the authorization result is then intersected with the original request, retaining still-constraining deny items. The sandbox manager also decides whether native sandbox must be enabled based on platform, network requirements, and explicit preference.

  • The startup snapshot of cloud configuration is not replaced by background refresh midway, avoiding policy semantic drift within the same run.
  • Skill installation uses embedded content fingerprints and markers; no repeated writes are needed when content has not changed.
  • Git, file search, feedback, agent, and installation context crates all maintain narrow APIs, allowing core/app-server to orchestrate without duplicating platform details.
  • Execpolicy-legacy only maintains compatibility with old policy format; new command policies are handled by execpolicy's parsing, rule, and decision model.
