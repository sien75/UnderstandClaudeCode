Code Mode (code-mode)

Code mode gives the model a "code-writing sub-brain." In normal chat the model can only call one tool at a time (shell, file read, MCP…), wait for the result, and then decide the next step. In code mode the model instead calls a tool named exec and drops a piece of JavaScript source into it. The JS runs in an isolated V8 sandbox where a global tools object is mounted—the same tools the model already has access to. So the model can loop, branch, assemble, call multiple tools in parallel, store intermediate variables, and finish the job like a script, then feed the accumulated text/image output back to the model.

From the user's perspective, code mode requires no extra button presses in the IDE/editor. It is transparent to the user—an "enhanced tool-calling" capability for the model. When the model decides "this needs several tools combined with logical branches," it calls exec to write a JS snippet. The script may finish quickly (returning the result in one go) or run for a while—in which case exec first "yields" some intermediate output, and the model then calls the wait tool to keep pulling subsequent output until the script ends. What the user sees is the model working and reporting progress; under the hood it is JS orchestrating everything.

This capability is implemented by three collaborating crates (this module covers only the data flow inside these three crates):

  • code-mode-protocol (protocol layer): defines cross-boundary data types and traits—ExecuteRequest/WaitRequest, RuntimeResponse, CodeModeSession/CodeModeSessionProvider/CodeModeSessionDelegate traits, plus tool-description generation (build_exec_tool_description renders enabled_tools into JS declarations for the model). Pure data/contracts, no runtime logic.
  • code-mode (core implementation): CodeModeService implements CodeModeSession and manages the cell lifecycle (one exec call = one cell). Internally CellActor is a tokio async actor that orchestrates between the "V8 thread" and "host callbacks"; the runtime submodule runs a V8 isolate on a dedicated OS thread, installs global functions, executes JS modules, and passes tool calls made from JS back to the actor over a channel.
  • code-mode-host (host binary): currently just an empty main.rs (fn main(){}). It is a placeholder for a future "remote host process"—the protocol comments on CodeModeSessionProvider say implementations can share a remote host across processes. The provider actually used today is InProcessCodeModeSessionProvider (in the code-mode crate), meaning cells run inside the Codex process itself.

Boundary note (these files are outside this module's scope, but data flows through them): the Codex main process (core crate) tools/code_mode/execute_handler.rs is the entry point for the exec tool. It calls parse_exec_source to parse the source, constructs an ExecuteRequest, and calls CodeModeSession::execute; tools/code_mode/wait_handler.rs handles the wait tool; tools/code_mode/delegate.rs's CodeModeDispatchBroker implements CodeModeSessionDelegate and dispatches nested tool calls (invoke_tool) from the code-mode runtime to real Codex tools (shell, MCP, file reads, etc.). In other words: tools.foo() inside the JS sandbox → runtime RuntimeEvent::ToolCall → cell actor → ServiceCellHost::invoke_tool → delegate.invoke_tool → real core tool → result returns the same way → resolve_tool_response stuffs the result back into the JS Promise. This is the core communication boundary of code mode.

Module breakdown (one sentence each)

code-mode-protocol / description.rs
Turns the "tool table the model can use" into the "exec tool description + TS declarations shown to the model." Input: enabled_defs of tools; Output: an assembled tool-description string (containing declare const tools: {...}). Also responsible for parse_exec_source parsing the first-line // @exec: {...} pragma, and for rendering JSON Schema into TypeScript types so the model understands tool input/output shapes.
code-mode-protocol / session.rs + runtime.rs + response.rs
Defines cross-boundary contract types. Inputs/outputs are business semantics: ExecuteRequest = a request to "run this JS"; RuntimeResponse::Yielded/Terminated/Result = runtime yielded/terminated/final result; CodeModeSession trait = a session that can execute/wait/terminate/shutdown; CodeModeSessionDelegate trait = the host must provide invoke_tool/notify/cell_closed callbacks.
code-mode / service.rs (CodeModeService)
Turns external requests into cell lifecycle management. Input: ExecuteRequest/WaitRequest; Output: StartedCell/WaitOutcome. Maintains stored_values (cross-cell shared key-value store, used by JS store/load), the cells table (cell_id → CellHandle), and the shutting_down flag. ServiceCellHost is the CellHost trait implementation that bridges cell-actor callbacks to CodeModeSessionDelegate.
code-mode / cell_actor (CellActor + run_cell)
The "director" of one exec call. Input: CellRequest + host; Output: CellHandle + CellEvent stream. It is a tokio select loop watching four sources: the command channel (observe/terminate), yield timeouts, runtime events (Started/Pending/ContentItem/YieldRequested/ToolCall/Notify/Result), and background tasks (notification/tool JoinSet). It decides when to flush accumulated output to the observer based on ObserveMode: YieldAfter = flush when timed out, PendingFrontier = flush when the runtime pauses at the pending frontier. This is the state-machine core of the whole chain.
code-mode / runtime (mod.rs + globals + callbacks + module_loader + timers + value)
Runs a V8 isolate on a dedicated OS thread to execute JS. Input: JS source + tool table + stored_values; Output: RuntimeEvent stream. globals installs global functions (removes dangerous globals like console/Atomics/WebAssembly); callbacks are the V8 callbacks for each global (tool/text/image/store/load/notify/setTimeout/yield_control/exit); module_loader compiles ES modules, checks Promise completion, and resolves tool Promises; timers implements setTimeout on a separate thread; value handles V8↔JSON serialization and image-output normalization (rejects remote http(s) images, only accepts data: URIs).
code-mode-host / main.rs
Placeholder empty main, reserved for a future remote host process.

Key state machines and boundary annotations

  • Cell lifecycle state machine (cell_actor/mod.rs run_cell): a cell progresses through Running (an observer is waiting) → either Yielded (waiting for the next wait after yielding) → ... → Completed/Terminated. Only one observer may exist at a time (a second observe returns Busy); terminate has the highest priority and cancels the token + terminate_execution to forcibly tear down V8. completed_event is buffered; if no observer is present it is stored and delivered directly to the next observer.
  • Runtime pause mode (runtime/mod.rs next_runtime_command): in PauseUntilResumed mode, when JS reaches pending (no result and no tool call outstanding) it emits RuntimeEvent::Pending and blocks waiting for a control command—Continue = proceed to the next RuntimeCommand, Resume = keep spinning for new events, Terminate = force exit. This is used for the "model wants to see where JS is" PendingFrontier observation mode.
  • Communication boundary 1: V8 thread ↔ cell actor (runtime/mod.rs): uses std::mpsc (RuntimeCommand/RuntimeControlCommand in, V8 thread receives) and tokio mpsc (RuntimeEvent out, actor receives). The V8 thread is a synchronous blocking thread; the actor is a tokio task; the two are decoupled by channels.
  • Communication boundary 2: cell actor ↔ CodeModeService (cell_actor/types.rs CellHandle): CellHandle holds an mpsc::UnboundedSender<CellCommand>; observe/terminate/shutdown all send commands to the actor, and the actor returns CellEvents via oneshot.
  • Communication boundary 3: code-mode ↔ core host (service.rs ServiceCellHost + protocol CodeModeSessionDelegate): every tools.foo() call inside JS eventually reaches delegate.invoke_tool; the delegate is implemented in core (CodeModeDispatchBroker) and dispatches the call to the real tool executor, with results returning the same way. This is the only bridge between "JS inside the sandbox" and "real system capabilities."
  • Security boundary (runtime/globals.rs + value.rs): removes console/Atomics/SharedArrayBuffer/WebAssembly globals; rejects remote image URLs (only data: URIs accepted); dynamic import throws "Unsupported import"; store only accepts JSON-serializable values. The sandbox has no Node, no fs, no network.
