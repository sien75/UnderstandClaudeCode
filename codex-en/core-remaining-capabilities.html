Codex Core Remaining Agent Capabilities Overview

This module fills in the "agent infrastructure" of Codex Core that does not belong to single tool calls: after an external caller starts or continues a thread, the system organizes the environment, permissions, skills, and plugins into model context; compresses history when the conversation grows too long; spawns and resumes sub-agents when parallel work is needed; submits high-risk actions to the guardian for review; and retries remote response failures while maintaining session continuity. The main thread is how data flows from thread configuration into the model, and then into events, approvals, or new thread states.

Input
Thread configuration, user messages, current environment and permissions, historical response items, model stream events, and pending high-risk actions
Output
Bounded and consistent model context, resumable sub-agent threads, actions reviewed by policy and guardian, and events continuously streamed back to the client

Context main chain: Core's public entry organizes thread services; at the start of each turn, changes to environment and permissions are encoded as context items, then call-output pairs are normalized. When approaching the window limit, local or remote compression preserves key user intent, recent tool results, and initial constraints, replacing old history with a summary.

  compact-choice -->|Yes| remote-compact["Call remote compression endpoint"]
  remote-compact --> remote-v2{"Enable new retention strategy?"}
  remote-v2 -->|No| install-summary["Filter response items and install replacement history"]
  remote-v2 -->|Yes| truncate-retained["Limit retained messages and image budget"]
  truncate-retained --> install-summary
  local-compact --> install-summary
  install-summary --> next-turn["Re-inject necessary initial context for next turn"]

Agent lifecycle main chain: after the parent thread submits an independent task, the controller first checks depth, concurrency, and persistence capacity, then chooses to create a new thread or fork from parent history. Child threads inherit the environment and command policies that are allowed to propagate; completion, pause, resume, and interruption all return to the parent thread through unified registration status, and old rollouts can reconstruct the entire tree of unclosed agent relationships.

Permission and reliability main chain: commands are first evaluated against policy files and managed permission configuration; risk actions requiring human confirmation can be routed to guardian. Guardian uses an isolated review session, can reuse the same risk context, and sets a deadline; rejection is recorded in the thread state, only temporary failures trigger retries. Common Responses stream errors also distinguish between retryable, context-limit-exceeded, and terminal errors, avoiding blind replay.

  • Skills and plugins do not directly concatenate arbitrary text: the system first discovers, deduplicates, and budgets metadata, then generates bounded injection based on explicit mention or implicit matching, avoiding unbounded context growth.
  • Environment snapshots distinguish between thread-level selection and per-turn snapshots; sub-agents only inherit the portion that policy allows to propagate; permission changes form new context items rather than rewriting old history.
  • Compression preserves real user messages, filters old summaries and call items unsuitable for continued delivery; the new remote compression also imposes separate limits on text tokens and image counts.
  • Guardian review sessions have reuse keys, cancellation, and deadlines, preventing approval from blocking the main thread; rejection status affects subsequent same-turn actions.
  • Other production implementations (MCP resources, hooks, network proxy, live sessions, rollout, shell and unified exec, etc.) have been confirmed file-by-file with their responsibilities covered; tests, snapshots, static prompt templates, and artifacts are not expanded as business implementations.
