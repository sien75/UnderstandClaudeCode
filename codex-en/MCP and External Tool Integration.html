MCP and External Tool Integration

This section covers how Codex talks to "the outside world." Model Context Protocol (MCP) is a JSON-RPC protocol, and Codex plays two roles in it. On one hand it is an MCP client: it connects to the various external MCP servers declared in config.toml (such as GitHub, filesystem, or custom tools), pulls back the tools and resources those servers provide, and feeds them to the model. On the other hand it is an MCP server: it packages its own "coding agent" capabilities as the codex and codex-reply tools and exposes them to external MCP clients such as Claude or an IDE. In addition, there is a plugin/extension mechanism that lets third-party packages declare their own MCP servers, contribute tools, and hook into the conversation lifecycle.

What users and external systems can do

  • Declare MCP servers in config.toml (stdio child process or streamable HTTP); Codex auto-starts, handshakes, and lists tools on launch → MCP client integration
  • Perform OAuth login against an external MCP server (codex mcp login <server>) and store tokens in keyring or a file → MCP client integration
  • The model calls mcp__<server>__<tool> tools in conversation; Codex routes them to the corresponding server → MCP client integration
  • External MCP clients (Claude/IDE) connect to the codex-mcp process over stdio, call the codex tool to start a coding task, and receive codex/event event streams and the final result → MCP server exposure
  • External clients use the codex-reply tool with a threadId to continue an existing session → MCP server exposure
  • External clients receive elicitation/create requests when execution/patching needs approval, and reply agree/reject → MCP server exposure
  • Plugin packages declare MCP servers in their own mcp_servers files; Codex loads and merges them into the runtime catalog → Plugin/extension integration
  • Extensions contribute tools and mount onto the conversation/tool lifecycle via the Extension API → Plugin/extension integration

Overall architecture: three roles

The whole scope spans six crates with clear layered responsibilities: codex-mcp is the "brain" that turns configuration into a runtime server catalog, aggregates concurrent client connections, and uniformly exposes tool/resource APIs; rmcp-client is the "transport layer" that wraps the connection lifecycle of a single MCP server (stdio process, HTTP, OAuth); mcp-server is Codex's implementation when it acts as a server, handling external JSON-RPC and driving codex-core to run a task; connectors merges tools from multiple sources, filters them, and performs policy adjudication; plugin defines plugin manifests and loading; ext contains built-in extensions (skills, memories, web-search, goal, image-generation, mcp executor plugin, guardian) and the Extension API itself.

Flow 1: Codex as MCP client — from configuration to callable tools

This main line answers: "I wrote an MCP server in config.toml, and now the model can call its tools—what happened in between?" The entry point is codex-core session initialization (outside this scope), which calls codex-mcp's exposed effective_mcp_servers to turn the server catalog in config into a "pending startup" runtime view, then uses McpConnectionManager::new to spin up all clients concurrently.

The key "state machine" lives inside RmcpClient: ClientState has three states—Connecting (handshaking, holding pending transport) → Ready (holding RunningService and optional OAuthPersistor) → Closed. Every operation goes through the run_service_operation pipeline: refresh OAuth token first, with transient retry (250 ms / 1000 ms backoff), and on HTTP 404 session expiry rebuild the transport under the session_recovery_lock semaphore using the saved TransportRecipe, then replay the operation once. The Elicitation pause state (ElicitationPauseState) pauses the operation timeout clock while the server is eliciting input from the user, so the user isn't interrupted by a timeout while thinking.

OAuth login subflow: HTTP servers first determine authentication status before the handshake (determine_streamable_http_auth_status): bearer token env var present → BearerToken; usable stored OAuth token → OAuth; stored token needs authorization → NotLoggedIn; otherwise perform RFC 8414 /.well-known/oauth-authorization-server discovery; if discovery succeeds → NotLoggedIn, otherwise → Unsupported. When the user runs codex mcp login it goes through perform_oauth_login: start a local callback HTTP service, build the authorization URL, open the browser/print the link, wait for the callback code, exchange it for tokens, and store them in the keyring (with a file fallback ~/.codex/.credentials.json).

Flow 2: Codex as MCP server — external clients drive coding tasks

This main line answers: "Claude or an IDE calls codex tools through MCP to write code—what happens?" The entry point is mcp-server's run_main (main.rs is just an arg0 dispatch shell). It spawns three tokio tasks: stdin reader, message processor, and stdout writer, chained by a bounded mpsc channel. MessageProcessor is the core dispatcher; it holds codex-core's ThreadManager and translates external JSON-RPC requests into operations on the codex-core session.

This is a typical session-lifecycle state machine: uninitialized → receives initialize → initialized (only allowed once, second returns invalid_request). Each codex tool call creates a new Codex thread and enters a "running event loop"; TurnComplete/Error/Err are terminal states, but the thread itself is not destroyed—it remains for codex-reply to continue. Multiple threads can run concurrently, so every codex/event carries a threadId for multiplexing. running_requests_id_to_codex_uuid is the single source of truth for cancellation and terminal cleanup.

Bidirectional approval communication is the highlight of this flow: when codex-core wants to execute a command or apply a patch, it does not ask the user itself. Instead it sends an elicitation/create JSON-RPC request to the external client via OutgoingMessageSender::send_request, registers a oneshot callback, and the event loop continue keeps pulling events (non-blocking). After the external client replies, process_response wakes the callback, and the callback task submits Op::ExecApproval/Op::PatchApproval back to codex-core. Patch approval proactively Denies when the channel is disconnected, making it more conservative than command approval.

Flow 3: Plugin and extension integration — how third-party capabilities plug into Codex

This main line answers: "A plugin package declares its own MCP servers, skills, app connectors, and hooks—how do these flow into Codex's runtime?" It splits into two parallel paths: the plugin manifest path (plugin crate, responsible for statically loading MCP server/skill/app declarations) and the Extension API path (ext/extension-api, responsible for runtime hooking of extensions into the conversation/tool lifecycle).

There are three boundaries worth marking here: Communication boundary at ExtensionRegistry—host and extensions communicate only through trait objects (Arc<dyn ...Contributor>), extension private state is isolated by ExtensionData (a TypeId-keyed Any map) across session/thread/turn scopes, and extensions cannot access host internal structures. State-machine boundary at lifecycle contributors—threads have start/resume/idle/stop, turns have start/stop/abort/error, and tool calls have start/finish (Completed/Blocked/Failed/Aborted), forming nested state layers. Adjudication table at AppToolPolicyEvaluator—the "enable + approval mode" for each app tool is adjudicated by a fixed priority chain: admin requirement (immutable) > tool-level config > app default > destructive/open-world prompt, producing one of four levels: Auto/OnFailure/Always/OnRequest.

One sentence per module

codex-mcp (brain)
Turns the server catalog in McpConfig into a set of concurrent AsyncManagedClients, aggregates tools/resources, and uniformly exposes them to codex-core. Input: configuration + auth; Output: callable tool list + resource reads + tool-call routing.
codex-mcp/rmcp_client (single-client lifecycle)
Turns one EffectiveMcpServer into a handshake-complete, tool-listed, routable ManagedClient. Input: server config + runtime context; Output: ManagedClient or StartupOutcomeError.
codex-mcp/connection_manager (aggregator)
Aggregates multiple ManagedClients into one external interface, routes (server,tool) calls, and manages startup-state events. Input: server map; Output: list_all_tools/call_tool/list_resources/read_resource.
codex-mcp/mcp (config layer)
Turns ResolvedMcpCatalog and auth policy into the EffectiveMcpServer runtime view, handling OAuth state computation and tool-name prefixing. Input: McpConfig; Output: effective_mcp_servers + status snapshot.
codex-mcp/elicitation + auth_elicitation
Turns server-originated CreateElicitation requests into Codex protocol events, auto-accepting/rejecting or handing to the user per approval policy. Input: server elicitation; Output: ElicitationResponse or Event.
codex-mcp/plugin_config
Parses plugin mcp_servers files (two formats) into a standardized McpServerConfig and binds env-var sources. Input: plugin root + file content; Output: PluginMcpConfigParseOutcome.
rmcp-client (transport layer)
Turns a TransportRecipe into a reconnectable RmcpClient, wrapping stdio/HTTP/OAuth transports. Input: transport recipe; Output: initialize/list_tools/call_tool/shutdown.
rmcp-client/stdio_server_launcher
Turns a StdioServerCommand into an rmcp Transport, using tokio child process locally and ExecBackend remotely. Input: command line; Output: Transport byte stream + process handle.
rmcp-client/http_client_adapter
Turns rmcp JSON-RPC operations into HTTP requests (SSE/single-message), handling 401/403/404. Input: JSON-RPC message; Output: HTTP request + response stream.
rmcp-client/oauth + perform_oauth_login
Turns a server URL into a browser OAuth authorization-code flow and stores tokens in keyring (with file fallback). Input: URL + scope; Output: StoredOAuthTokens.
mcp-server (Codex as server)
Turns external JSON-RPC requests into operations on codex-core threads and turns Event streams into codex/event notifications. Input: stdio JSON-RPC; Output: tool results + event stream + elicitations.
connectors (app connector discovery)
Merges ChatGPT app catalog + reachable tools + plugin declarations by ID and adjudicates each tool's enablement and approval. Input: directory API + tool metadata; Output: merged AppInfo + AppToolPolicy.
plugin (plugin model)
Turns plugin manifests into LoadedPlugins and aggregates effective resource unions. Input: manifest + disk root; Output: MCP servers/skills/apps/hooks unions.
ext/extension-api (extension contract)
Defines eleven Contributor trait types and three-layer ExtensionData state; the host queries through ExtensionRegistry. Input: extension install registration; Output: tools/prompts/MCP overlays/lifecycle callbacks.
ext/mcp (MCP executor plugin)
When a thread selects an executor plugin, parses its stdio MCP server and contributes it as a SelectedPlugin overlay. Input: selected plugin root; Output: McpServerContribution::SelectedPlugin.
ext/skills
Reads skill main prompts mentioned by the user and injects them into turn input, contributing skill list/read tools. Input: user input + skill roots; Output: prompt fragments + skill tools.
ext/memories
Contributes memory developer instructions and list/read/search/add_note tools (stored locally in codex_home). Input: config; Output: prompt fragments + memory tools.
ext/web-search
Contributes the web.run tool (OpenAI model-provider backend). Input: config + auth; Output: WebSearchTool.
ext/goal
Manages persistent goal state across threads/turns, performs budget accounting and progress counting, and contributes goal tools. Input: state DB; Output: goal tools + steering + accounting.
ext/image-generation
Contributes the image_gen.imagegen tool (Codex images backend). Input: config + auth; Output: ImageGenerationTool.
ext/guardian
Captures the thread id and host-injected AgentSpawner for forking review subagents. Input: thread id; Output: thread-local forked-from context.

Key findings

  • Dual identity: Codex is both client (codex-mcp + rmcp-client) and server (mcp-server) in the MCP protocol. The two sides share rmcp's JSON-RPC model but have completely different responsibilities—the client side aggregates external tools for the model, while the server side exposes Codex coding capabilities to external clients.
  • Three-level cache: codex_apps tools have a startup snapshot cache (immediately available at startup) + runtime cache (listed_tools hit) + hard refresh (hard_refresh_codex_apps_tools_cache bypasses cache), solving the slow-remote-app-tool-list problem.
  • Untrusted metadata stripping: connector_* metadata from non-codex_apps servers is stripped by sanitize_tool_connector_metadata, preventing external servers from impersonating connector identities; only trusted codex_apps servers retain it.
  • Session recovery: RmcpClient rebuilds the transport under session_recovery_lock using the saved TransportRecipe when it sees HTTP 404, then replays the operation transparently to callers.
  • Elicitation pause clock: ElicitationPauseState pauses the operation timeout clock while the server is eliciting user input, so the user isn't interrupted by a timeout while thinking.
  • Bidirectional approval channel: mcp-server does not ask the user directly; it sends elicitation/create to the external client, the event loop remains non-blocking, and the client reply wakes the oneshot callback via process_response; patch approval proactively Denies when the channel is disconnected.
  • Plugin MCP name collisions: effective_mcp_servers uses first-definition-wins with RegistrationPrecedence priority layers (Plugin > SelectedPlugin > Config > Compatibility > Extension).
  • Three-layer extension state isolation: ExtensionData isolates extension private state across session/thread/turn scopes using a TypeId-keyed Any map; extensions cannot access host internal structures.
  • mcp-server is an MVP: resources/prompts/logging/completion are stubs that only log; only tools-related methods and elicitation are implemented; tasks/* directly returns METHOD_NOT_FOUND.
