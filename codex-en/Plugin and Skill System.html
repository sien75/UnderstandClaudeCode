Plugins and Skills System

This is a data-flow analysis of Codex's four extension mechanisms—plugins, skills, tools, and hooks. These four systems transform Codex from a static CLI into a runtime that can be continuously injected with new capabilities by users, marketplaces, and remote backends. The entry point is codex-rs/core-plugins/src/lib.rs, which exposes the plugin manager (PluginsManager) to upper layers. The plugin manager is the hub connecting skills, tools, and hooks: once a plugin is loaded, it simultaneously contributes skill roots, MCP server configurations, App connector declarations, and hook event sources, which flow to the skills, tools/hooks subsystems respectively.

One-sentence summary
Take three inputs—"user configuration + marketplace catalog + remote directory"—and produce four outputs: "loaded skills / MCP / App / hooks", consumed by the model and executor.
Input (business meaning)
The [plugins] section in config.toml, cached plugin directories under ~/.codex/plugins/cache, the list of installed plugins returned by the remote backend, and plugin sources registered in the marketplace marketplace.json.
Output (business meaning)
PluginLoadOutcome—a set of LoadedPlugins, each carrying skill roots, an MCP server table, App declarations, and hook sources; plus a PluginCapabilitySummary for the model.
Notes
The plugin ID format is always <plugin_name>@<marketplace_name>, and the cache directory is laid out as plugins/cache/<marketplace>/<plugin>/<version>. OpenAI official curated marketplaces (openai-curated / openai-api-curated) use a special git sync + truncated sha versioning path; non-curated marketplaces use semver comparison.

What the Four Extension Mechanisms Each Handle

  • Plugins: A plugin is a directory containing a .codex-plugin/plugin.json manifest. The manifest declares which skills (directories), MCP servers, App connectors, and hooks the plugin provides. A plugin is a "capability container"; it does not execute itself, but registers what it contains with the runtime. Core roles: PluginsManager (orchestration), PluginStore (caching), loader (loading), marketplace (discovery).
  • Skills: A skill is a Markdown document (SKILL.md) plus optional scripts; essentially an "instruction manual for the model". System skills are embedded with the binary (the skills crate uses include_dir!) and are extracted to $CODEX_HOME/skills/.system on first launch. User skills come from the config layer, plugin skill roots, and extra roots. Core roles: SkillsManager, loader::load_skills_from_roots.
  • Tools: This is the Responses API tool primitive layer, defining how model-callable function tools are serialized into JSON and how they are executed. ToolSpec is the tool description for the API, and the ToolExecutor trait is the runtime contract. There is also tool_discovery for on-demand discovery of installable plugins/connectors, and request_plugin_install for the model to proactively request plugin installation.
  • Hooks: An event-hook engine that runs user-defined shell commands at 10 event points: before/after tool calls, session start/end, before/after compaction, etc. Hooks come from the config layer (hooks/hooks.json or TOML) and inline plugin declarations. ClaudeHooksEngine is the core, discovery finds handlers, and dispatcher selects and executes them.

Plugin Cache Layout and Version Selection (State Machine)

PluginStore stores each plugin under cache/<marketplace>/<plugin>/<version>. The same plugin may have multiple version directories. active_plugin_version is the selection logic: prefer local (development state), otherwise take the highest semver.

Skill Loading Flow

A skill root (SkillRoot) has four scopes: Repo (in-project .codex/skills), User (user-level, including plugin skill roots and $HOME/.agents/skills), System (embedded system skills), and Admin (/etc/codex/skills). Priority is Repo > User > System > Admin; for skills with the same name, the first one scanned wins.

System Skill Embedding and Extraction (Lifecycle)

The skills crate uses include_dir!("$CARGO_MANIFEST_DIR/src/assets/samples") to embed all system skill samples (skill-creator, plugin-creator, skill-installer, openai-docs, imagegen) into the binary at compile time. At startup, install_system_skills checks the .codex-system-skills.marker fingerprint; it only rewrites the entire .system directory when the fingerprint changes, avoiding repeated IO on every launch.

Hooks Engine Data Flow

Hook handler sources are threefold: managed requirements (lowest priority, run first), config layer (hooks.json or TOML), and inline plugin hook declarations. discover_handlers merges these three sources into an ordered ConfiguredHandler list. When an event fires, select_handlers filters by event name + matcher regex, then execute_handlers runs commands concurrently, collects results in completion order, and reorders them according to config order.

Hook Event Matrix (State Machine / Decision Table)

There are 10 event names, but the matcher field is meaningful for only 8 (Stop and UserPromptSubmit are not filtered by matcher; all handlers run). Events are divided into two categories: Thread scope (SessionStart/SubagentStart) and Turn scope (the rest). Stateless events (PreCompact/PostCompact) carry no session-level side effects.

Tool Primitive Layer (Responses API Contract)

The tools crate defines how model tools are serialized for the OpenAI Responses API and how they are executed. ToolSpec is an enum: Function (ordinary function tool), Namespace (namespace grouping), ToolSearch (lazy discovery), ImageGeneration, WebSearch, and Freeform (custom). The ToolExecutor trait binds "tool description" and "execution logic" together, and ToolExposure controls whether a tool is directly exposed to the model, lazily discovered, or hidden for internal dispatch only.

Startup Sync Flow (Lifecycle)

maybe_start_plugin_startup_tasks_for_config is the startup scheduler. It concurrently spawns four kinds of background tasks: curated marketplace git repo sync, marketplace auto-upgrade, remote installed-plugin cache refresh + bundle sync, and curated plugin ID cache warmup. These are all "best-effort"—failure only warns and does not block startup.

Communication Boundary Annotations

  • Cross-crate boundary: core-plugins depends on the generic LoadedPlugin<M> from codex-plugin via the type alias LoadedPlugin<McpServerConfig>; the MCP configuration type is provided by codex-config. This is the core decoupling point: the plugin model is not tied to a specific MCP configuration type.
  • Plugin → Skills: PluginLoadOutcome::effective_plugin_skill_roots() packages the plugin's skill roots into PluginSkillRoots (with plugin_id and plugin_root) and feeds them into SkillsLoadInput.
  • Plugin → Hooks: load_plugin_hooks parses the hook files declared in the plugin manifest into PluginHookSources and feeds them into ClaudeHooksEngine::new.
  • Plugin → MCP: effective_mcp_servers() merges the MCP server configurations of all active plugins; same name is first-wins.
  • Plugin → Model: capability_summaries() produces a "what skills/MCP/App does this plugin have" summary for the model, with descriptions truncated to 1024 characters.
  • Remote backend: remote.rs calls backend endpoints such as /installed and /catalog via chatgpt_base_url and requires CodexAuth authentication; when not authenticated it returns empty lists without error.

Key Findings

  • Caching and invalidation: PluginsManager uses LoadedPluginsCache (generation + key double-check) to avoid repeated loading. Any remote cache change, marketplace upgrade, install/uninstall triggers clear_loaded_plugins_cache, increments the generation, and forces a reload next time. The skill side also has dual caching: by cwd and by config-key, preventing cross-session / cross-role pollution.
  • Version selection policy: The local version always takes precedence over any semver version—this is a development convenience, but it also means that if a developer leaves a local subdirectory in the cache, the release version will never be selected.
  • Atomic installation: replace_plugin_root_atomically uses a staged directory + rename for atomic replacement. On failure it tries to roll back to a backup; if rollback also fails it keeps the backup path and reports the error—this is the last line of defense under double fault.
  • Plugin manifest path safety: resolve_manifest_path requires all relative paths to start with ./, contain no .., and only use Normal components—preventing plugin manifests from escaping their own root directory.
  • Hook deduplication: select_handlers runs only once per tool call, even if a matcher regex matches multiple aliases (e.g. apply_patch|Write|Edit), avoiding duplicate hook execution.
  • System skill fingerprint: It hashes all file paths + contents of the embedded directory with DefaultHasher, adds a v1 salt, and writes it into the marker file. If the fingerprint matches, extraction is skipped—accelerating startup.
