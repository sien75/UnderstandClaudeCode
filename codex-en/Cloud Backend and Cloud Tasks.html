Cloud Backend and Cloud Tasks

This module covers how the Codex client interacts with the "cloud": sending a user prompt to the OpenAI/ChatGPT backend to generate a response, dispatching a code task to Codex Cloud for remote execution and pulling the diff back to apply locally, and how to switch authentication and model sources when the user isn't using OpenAI but connecting to local Ollama / LM Studio or Amazon Bedrock. The core question is "how data leaves the local machine and how it returns from the cloud".

What Users Can Do

  • Send a chat/completion request to the OpenAI Responses API and stream the answer → codex-api's ResponsesClient + SSE stream
  • Query backend account, quota, rate limits, and token usage → backend-client
  • Submit code tasks to Codex Cloud for remote execution, check status, list tasks, view diff, and apply back to local → cloud-tasks + cloud-tasks-client
  • Use the TUI to browse/select cloud tasks and apply them → cloud-tasks's app/ui
  • Switch model providers: OpenAI / Azure / Amazon Bedrock / local Ollama / LM Studio → model-provider + model-provider-info + models-manager + ollama + lmstudio
  • Use a local proxy to forward Responses requests upstream (hiding the API Key) → responses-api-proxy

Key Concepts

Provider
A connectable model backend described by ModelProviderInfo: base_url, env_key, auth, wire_api (now only responses), retry/timeouts. codex-api's Provider is its runtime form.
AuthProvider
A trait responsible for attaching authentication headers to outgoing HTTP requests. Implementations include BearerAuthProvider (normal Bearer token + ChatGPT-Account-Id + FedRAMP headers), AgentIdentityAuthProvider (Agent Identity signature), Bedrock SigV4, and Unauthenticated (local OSS).
Path Style
backend-client decides whether to use /api/codex/... (CodexApi) or /wham/... (ChatGptApi, i.e. backend-api) based on base_url. cloud-tasks also reuses the same set.
Responses SSE Stream
codex-api parses the POST /responses text/event-stream into a sequence of ResponseEvents (Created, OutputTextDelta, ToolCallInputDelta, ReasoningSummaryDelta, Completed, RateLimits, ServerModel…), and pushes them to the upper layer via an mpsc channel.
Cloud Task Lifecycle
Pending (queued/running) → Ready (completed with diff) → Applied (already applied) or Error. With best-of-N, one turn has multiple sibling attempts.

Main Data Flow: Send a Responses Request and Receive a Streaming Response

The upper layer (core) receives the user prompt, assembles it into a ResponsesApiRequest (model, input, tools, reasoning, store…), and hands it to ResponsesClient. ResponsesClient JSON-encodes the request body (additionally attaches item ids for Azure scenarios), attaches session/thread/subagent headers, and passes it to EndpointSession. EndpointSession uses Provider.build_request to construct the final URL + default headers + query, then lets AuthProvider.apply_auth attach authentication headers, and finally sends a POST via HttpTransport (underlying reqwest), receiving text/event-stream. spawn_response_stream spawns a tokio task, parses SSE events one by one, maps them to ResponseEvents and pushes them into the channel. On errors (response.failed/incomplete), it classifies them by error code into ContextWindowExceeded / QuotaExceeded / CyberPolicy / ServerOverloaded / Retryable etc. ApiError, and then api_bridge translates them into core-layer CodexErr.

Subgraph: How Authentication Headers Are Produced (AuthProvider Selection)

codex-api only defines the AuthProvider trait; the actual implementation chosen is decided by model-provider's resolve_provider_auth / auth_provider_from_auth. Rules: BedrockApiKey on a non-Bedrock provider directly errors; if the provider has api_key or experimental_bearer_token configured, use BearerAuthProvider; otherwise map according to current CodexAuth type: Chatgpt/ApiKey/PAT → Bearer, AgentIdentity → AgentIdentityAuthProvider (signature), local OSS with no auth → Unauthenticated. Bedrock uses its own SigV4 signature (amazon_bedrock/auth.rs).

Subgraph: backend-client Account/Quota/Task API

backend-client is a thin HTTP client facing the ChatGPT/Codex backend (/api/codex or /wham); all cloud tasks, quotas, and account verification go through it. It holds base_url, reqwest Client, SharedAuthProvider, optional ChatGPT-Account-Id and FedRAMP flag. Each method constructs the URL (choosing one of two prefixes by PathStyle), attaches headers, sends reqwest, bails or parses JSON by status code. Rate-limit related logic maps the backend's RateLimitStatusPayload into the protocol-layer RateLimitSnapshot (primary/secondary windows, credits, spend control, plan_type).

Main Data Flow: Cloud Tasks — Submit, Query, Apply

cloud-tasks is the entry point for the codex cloud subcommand. The CLI has five subcommands (exec/status/list/apply/diff) plus a default TUI. init_backend reads base_url (default chatgpt.com/backend-api), loads the ChatGPT login state via codex-login, and constructs cloud-tasks-client's HttpClient (which wraps backend-client internally). HttpClient implements the CloudBackend trait: list_tasks/get_task_summary/get_task_diff/get_task_text/list_sibling_attempts/apply_task(_preflight)/create_task. create_task wraps the prompt into input_items (optionally attaching CODEX_STARTING_DIFF), POST /tasks to get the task id. Apply flow: first get_task_details to get the unified diff (or use diff_override), verify it's a unified diff, then call codex-git-utils's apply_git_patch in the local cwd to run git apply (preflight mode only validates without writing to disk), and classify by exit code into Success/Partial/Error.

Subgraph: Model Discovery — models-manager

models-manager is responsible for providing the upper layer with an "available model list + default model". It has three refresh strategies: Online (network only), Offline (cache only), and OnlineIfUncached (use cache if fresh, otherwise pull from network). OpenAiModelsManager holds OpenAiModelsEndpoint (injected by model-provider, wraps ModelsClient + auth), pulls the remote catalog via GET /models?client_version=, makes conditional requests with ETag, persists to models_cache.json (TTL 5 minutes). StaticModelsManager uses a preset catalog. Finally it sorts by priority, filters by auth (whether codex backend), marks the default, and produces a ModelPreset list for the selector.

Subgraph: Local OSS (Ollama / LM Studio)

When the user selects --oss or configures an ollama/lmstudio provider, ensure_oss_ready starts: get the provider from Config, construct OllamaClient/LMStudioClient, probe whether the server is online (Ollama hits /api/tags or /v1/models; LM Studio hits /models), list local models, pull/download if missing, and finally (LM Studio only) load the model in the background. Ollama also verifies version ≥ 0.13.4 to support the Responses API. These providers have requires_openai_auth=false, and auth resolves to UnauthenticatedAuthProvider.

Subgraph: responses-api-proxy (Local Reverse Proxy Hides API Key)

A minimal HTTP reverse proxy: read the auth header from stdin (only once, from_static + set_sensitive to avoid leaks), start tiny_http listening on 127.0.0.1, only allow POST /v1/responses, forward headers as-is (remove original Authorization/Host, replace with injected auth and upstream Host), forward to --upstream-url (default api.openai.com/v1/responses), and return the upstream response (including streaming) as-is. Optional --dump-dir persists request/response for debugging. Used to let codex hit OpenAI without touching the API Key.

Errors and Edge Cases

  • codex-api classifies backend error codes into specific ApiErrors (context_length_exceeded → ContextWindowExceeded; insufficient_quota → QuotaExceeded; cyber_policy → CyberPolicy; server_is_overloaded/slow_down → ServerOverloaded; rate_limit_exceeded with retry-after → Retryable{delay}). api_bridge then translates HTTP 4xx/5xx/429 (usage_limit_reached → UsageLimitReached) into core's CodexErr. This is a clear decision table + state mapping boundary.
  • Not logged in / non-codex backend auth in cloud-tasks directly calls process::exit(1) and does not enter the TUI.
  • Before apply, the diff must be verified as a unified git diff; non-unified diffs directly return Error without touching the working tree; preflight does not write to disk.
  • SSE idle timeout (stream_idle_timeout, default 300s) will break the stream and report a Stream error.
  • BedrockApiKey auth can only be configured on the Amazon Bedrock provider; other providers will report UnsupportedOperation.
  • Azure Responses endpoint needs to explicitly attach id to input items when store=true.

Notes

  • Path style: chatgpt.com/backend-api → /wham/...; otherwise /api/codex/.... cloud-tasks's default base_url is chatgpt.com/backend-api.
  • The realtime/responses websocket endpoint (v1/v2 protocol) is another channel for streaming calls; this analysis only mapped the overview, not deeply read.
  • cloud-tasks's TUI (app.rs/ui.rs/scrollable_diff.rs/new_task.rs) is the ratatui rendering layer, only mapped.
  • models.json / prompt.md are generated/static resources, ignored.
  • cloud-tasks-mock-client provides a MockClient for local TUI debugging (CODEX_CLOUD_TASKS_MODE=mock).
