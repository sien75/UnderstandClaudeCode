App Server Thread and Turn API

External clients send JSON-RPC via the app-server transport. This scenario threads the thread/start thread-creation request, the turn/start turn-initiation request, event notifications after subscribing, and the thread/read thread read into one data flow: client input is first deserialized by the protocol layer into strongly-typed requests, then dispatched by the message processor to the thread or turn handler, and finally the thread summary, turn state, and notifications are written back to the original connection or to connections subscribed to that thread.

Input
JSON-RPC request sent by external clients: thread/start carries thread startup parameters such as model, working directory, permissions/sandbox, instructions, and dynamic tools; turn/start carries threadId, user input, context, and turn-specific settings; thread/read carries threadId and whether to include turns.
Output
JSON-RPC response and server notification: thread/start returns the new thread and a configuration snapshot and broadcasts thread/started; turn/start returns an InProgress turn, and subsequent turn/started, turn/completed, etc. notifications are sent only to connections subscribed to that thread; thread/read returns a Thread view synthesized from persisted thread data and live thread state.

  • Communication boundary: transport only handles connections, read/write queues, and JSON-RPC message classification; business dispatch happens in MessageProcessor, and real thread/turn semantics are handled by request processors.
  • Lifecycle: thread/start immediately creates a listener for the initiating connection after thread creation. The listener continuously reads core thread events, so client subscription is not a separate “subscribe” method; it is automatically attached via thread entry points such as start/resume/fork, and thread/unsubscribe later removes the connection.
  • Permission/sandbox boundary: thread startup rejects passing both permissions and the legacy sandbox; turn startup also converts turn-specific permissions, sandbox, working directory, and environment selection into thread-setting overrides for the core thread to execute.
  • State machine: the turn is first returned as InProgress in the API response; the listener then advances the state to InProgress, Completed, Failed, or Interrupted based on core events and sends notifications.
  • Read path: thread/read prefers reusing the persisted ThreadStore; if the thread is loaded but persistence has not yet flushed, it is filled in from the live thread snapshot. When turns are requested, rollout history is replayed; ephemeral threads do not support reading full turns.
