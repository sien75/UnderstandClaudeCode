Agent Thread Protocol, State, History Persistence, and Recovery

An external client submits a user input, approval reply, or thread control action to Codex; the system turns it into an event stream with an associated ID, while writing recoverable session entries to rollout and queryable indexes and derived state to SQLite. When recovering a thread, the system first re-establishes a "live thread" handle, then loads history and rebuilds metadata; if initialization fails midway, it actively discards the half-baked result to avoid leaving a seemingly recoverable but incomplete thread.

Input
User input, thread settings, approval replies, thread rollback/compression/close actions, and inter-agent communication.
Output
Associated event streams consumable by the client, JSONL history replayable line by line, paginated SQLite thread index, and derived state such as sub-agent relationships, memory, and traces.

The protocol layer separates "what is submitted" from "what happened": Op is the command from client to thread, EventMsg is the status and output from thread to client. Event.id re-associates asynchronous events back to the original submission; although agent communication goes through the normal submission lifecycle, it is first encoded as model history items, so inter-agent context is preserved after recovery.

Local thread storage treats the rollout file as a compatible and replayable fact stream, and SQLite as a fast query and derived state layer. The writer receives append, persist, flush, and close commands serially through a dedicated task; file creation can be deferred when not yet materialized, and on failure the termination error is retained for subsequent API returns. List reading prefers the state store, falling back to scanning rollout when missing or needing repair, and backfills discovered metadata into the index.

  • State machine boundary: thread goes from initializing, active, archived to deleted; sub-agent goes from open to closed, descendant queries are provided by SQLite graph relationships.
  • Recovery boundary: rollout retains original order, SQLite can be backfilled from rollout; so index corruption does not equal history loss.
  • Context fragments are identified by explicit start/end markers, avoiding confusion between normal user text and system-injected content during recovery or compression.
  • Memory writes go through phased extraction, integration, and saving, with the read side attaching references and usage information; it is a derived result of thread history, not a replacement for rollout.
  • Rollout-trace reduces protocol events into thread, reasoning, tool, and compression traces, serving as a diagnostic view; write failures use best-effort strategy and do not block the main thread.
