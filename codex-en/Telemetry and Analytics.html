Telemetry and Analytics

This module manages all "observable data" produced while Codex runs—from token usage of a turn and latency of a tool call to replay tracing of an entire session. It consists of four crates, each responsible for one data channel: analytics aggregates app-server protocol events into business telemetry events and sends them to the backend; otel plugs into the OpenTelemetry standard (traces/logs/metrics) and exports via OTLP; rollout persists session content as JSONL files for later replay; rollout-trace is a finer-grained trace bundle that records the full lifecycle of inference/tool/code cells for diagnostic replay.

What this module manages

From the user's perspective, the following actions generate data once Codex is running:

  • Start a session/thread → analytics emits thread_initialized, rollout writes the SessionMeta first line, otel records conversation_starts
  • Run a turn (send a request to the model) → analytics tracks turn token usage/latency/profile, otel records API request latency and SSE events, rollout-trace records the inference-call lifecycle
  • Call a tool (shell/apply_patch/MCP) → analytics generates tool-item events, otel records tool-call counts and latency, rollout persists response items
  • Compaction occurs (context compression) → analytics emits compaction events, rollout-trace records compaction request/install checkpoints
  • Session ends → rollout file closes and is eventually compressed to .zst, rollout-trace writes RolloutEnded

Data eventually flows to: analytics → HTTP POST to the codex backend; otel → OTLP gRPC/HTTP to a remote collector; rollout → local ~/.codex/sessions/ JSONL files + SQLite metadata; rollout-trace → local trace bundle directory (enabled only when CODEX_ROLLOUT_TRACE_ROOT env var is set).

Module details

analytics — business telemetry event collection and reporting

Aggregates app-server protocol traffic (client requests/responses, server notifications, turn lifecycle, tool calls, guardian approvals, etc.) into structured events and HTTP POSTs them to the codex backend for product analysis.

Input
app-server protocol messages (ClientRequest/ServerNotification/ServerResponse, etc.) and custom facts produced by core (TurnProfile/TurnTokenUsage/CodexCompactionEvent, etc.)
Output
TrackEventsRequest JSON array, POSTed to {base_url}/codex/analytics-events/events
Core data structures
AnalyticsFact (facts.rs:445) — the raw-fact enum fed into the reducer, split into two categories: protocol-surface facts (Initialize/ClientRequest/ClientResponse/Notification/ServerRequest/...) and custom facts (CustomAnalyticsFact::SkillInvoked/Compaction/Goal/GuardianReview/TurnProfile/...).
AnalyticsReducer (reducer.rs:137) — the state-machine aggregator, maintaining six HashMaps (requests/turns/connections/threads/tool_items/pending_reviews) that stitch scattered protocol facts into complete turn events and tool-item events.
TrackEventRequest (events.rs:60) — the final reported event enum with 22 variants (SkillInvocation/ThreadInitialized/GuardianReview/Compaction/Goal/TurnEvent/TurnSteer/CommandExecution/FileChange/McpToolCall/.../PluginUsed/PluginInstalled/...).
Notes
Event deduplication: app_used and plugin_used events use a (turn_id, connector_id/plugin_id) set, cleared after 4096 keys. AcceptedLineFingerprints events get an isolated HTTP request (should_send_in_isolated_request). Debug builds can set CODEX_ANALYTICS_EVENTS_CAPTURE_FILE to write events locally instead of sending them over the network. When the queue is full, try_send failures are dropped with a warning.

otel — OpenTelemetry standard integration layer

Wraps the OpenTelemetry SDK to provide Codex with traces (distributed tracing), logs (structured log export), and metrics (metric collection) over three OTLP export paths. SessionTelemetry builds session-level business metrics on top.

Input
OtelSettings config (exporter endpoint/protocol/TLS/tracestate/span_attributes) and runtime tracing events
Output
OTLP gRPC or HTTP export to a remote collector; local metrics global instance for other crates to call
Core data structures
OtelProvider (provider.rs:56) — holds SdkTracerProvider + SdkLoggerProvider + MetricsClient, built once from(OtelSettings). Drop forces flush + shutdown.
OtelExporter (config.rs:88) — exporter config enum: None / Statsig (internal metrics, downgraded to None in debug builds) / OtlpGrpc{endpoint,headers,tls} / OtlpHttp{endpoint,headers,protocol,tls}.
MetricsClient (metrics/client.rs:260) — wraps the OpenTelemetry Meter, providing counter/histogram/gauge/record_duration/start_timer interfaces, internally caching an InstrumentKey→Counter/Gauge/Histogram mapping to avoid repeated creation.
SessionTelemetry (events/session_telemetry.rs:103) — session-level telemetry wrapper carrying SessionTelemetryMetadata (conversation_id/auth_mode/account_id/model/originator/...), providing record_api_request/log_sse_event/tool_result_with_tags/record_turn_ttft business methods, while writing metrics counter/histogram and tracing events (log_event!/trace_event! macros route to codex_otel.log_only and codex_otel.trace_safe targets respectively).
Notes
tracestate supports W3C Trace Context propagation: parent trace context can be injected from TRACEPARENT/TRACESTATE env vars, or configured tracestate entries can be merged and injected. span_attributes are attached to every span at on_start via SpanAttributesProcessor. The OTLP HTTP client distinguishes multi-threaded tokio runtime (async reqwest + TokioBatchSpanProcessor) from single-threaded (blocking reqwest). Target routing: codex_otel.log_only → logs only; codex_otel.trace_safe → safe to export trace; other codex_otel.* → both.

rollout — session JSONL persistence and discovery

Persists every protocol message, response item, and execution event of a Codex session to JSONL files under ~/.codex/sessions/, supporting later resume, list queries, and search. Works with the SQLite state store for fast metadata queries.

Input
core-produced RolloutItem (ResponseItem / EventMsg / InterAgentCommunication / Compacted / TurnContext / SessionMeta)
Output
~/.codex/sessions/YYYY/MM/DD/rollout-{timestamp}-{thread_id}.jsonl files; SQLite thread metadata DB; list/search query results
Core data structures
RolloutRecorder (recorder.rs:75) — clone-safe recorder holding mpsc::Sender<RolloutCmd> and a background writer task handle. new() has two modes: Create (precompute path + lazily create file) and Resume (immediately open existing file).
RolloutWriterState (recorder.rs:1459) — mutable state of the background writer: writer(Option<JsonlWriter>) / deferred_log_file_info / pending_items / meta. On I/O failure enter_recovery_mode drops the file handle but keeps pending_items; next barrier retries (write_pending_with_recovery retries twice).
policy (policy.rs) — decides which RolloutItem to persist: most ResponseItems are kept (except CompactionTrigger/Other); EventMsg is selectively kept (turn lifecycle/compaction/user message kept, delta/begin types dropped).
compression (compression.rs) — background compression worker: spawn_rollout_compression_worker scans sessions/ and archived_sessions/, compressing .jsonl files older than 7 days into .jsonl.zst. RolloutLineReader transparently reads plain or compressed formats. materialize_rollout_for_append decompresses compressed files back to plain before appending.
Notes
list_threads has filesystem and SQLite dual paths: DB is preferred; if DB is unavailable or errors, it falls back to filesystem scanning and records record_fallback. search (search_term) does full filesystem scan + reconcile to fix the DB. The resume path matches cwd with three-level fallback: cached cwd → latest TurnContext cwd → parsing rollout head metadata.

rollout-trace — diagnostic-level trace bundle recording and replay

A finer-grained diagnostic trace than rollout: records each inference call (request/response payload), each tool dispatch (dispatch/runtime/result), and the full lifecycle of each code-mode JavaScript cell. Raw events are written to trace.jsonl + payloads/ directory; offline replay_bundle() replays a structured RolloutTrace graph.

Input
hot-path trace context calls (ThreadTraceContext / InferenceTraceContext / ToolDispatchTraceContext / CodeCellTraceContext / CompactionTraceContext / McpCallTraceContext)
Output
{trace_root}/trace-{trace_id}-{thread_id}/ directory: manifest.json + trace.jsonl + payloads/*.json; replayed RolloutTrace (state.json)
Core data structures
TraceWriter (writer.rs:36) — thread-safe append-only writer protected by a Mutex. create() builds the bundle directory and writes the manifest. write_json_payload() writes a payload file and returns RawPayloadRef. append_with_context() writes RawTraceEvent to trace.jsonl and increments seq.
RawTraceEvent (raw_event.rs:33) — unified envelope: schema_version / seq / wall_time_unix_ms / rollout_id / thread_id / codex_turn_id / payload. RawTraceEventPayload has 25+ variants (RolloutStarted/ThreadStarted/CodexTurnStarted/InferenceStarted+Completed+Failed+Cancelled/ToolCallStarted+RuntimeStarted+RuntimeEnded+Ended/CodeCellStarted+InitialResponse+Ended/CompactionRequestStarted+Completed+Failed+Installed/AgentResultObserved/ProtocolEventObserved/Other).
ThreadTraceContext (thread.rs:76) — no-op-capable handle: Disabled or Enabled(EnabledThreadTraceContext{writer,root_thread_id,thread_id}). start_root_or_disabled() reads the env var to decide whether to enable. Child threads share the root bundle's writer via start_child_thread_trace_or_disabled.
TraceReducer (reducer/mod.rs:88) — deterministic replay state machine: reads trace.jsonl line by line, apply_event() updates the RolloutTrace graph by payload type. Maintains pending state (pending_code_cell_starts / pending_agent_interaction_edges / pending_compaction_replacement_item_ids) to handle out-of-order events.
RolloutTrace (model/mod.rs:56) — final replay graph: threads / codex_turns / conversation_items / inference_calls / code_cells / tool_calls / terminal_sessions / compactions / interaction_edges / raw_payloads. raw_payloads stores only RawPayloadRef references, not inline large payloads.
Notes
All trace contexts are no-op-capable: when CODEX_ROLLOUT_TRACE_ROOT is not set, disabled() is returned and hot-path calls return immediately without serialization cost. Payload files are written before the event that references them (so replay breakpoints never point to a missing payload). code-cell model-visible call_id and runtime cell_id have a bridge mapping (code_cell_ids_by_runtime), handling the out-of-order case where core starts executing a tool before the inference-response payload arrives. AgentResultObserved is the explicit graph edge for v2 multi-agent child→parent completion delivery, not inferred from inference requests.

Cross-crate data flow relationships

  • analytics ↔ otel: analytics events.rs calls codex_login::default_client::originator() to obtain product_client_id; AnalyticsReducer does not directly depend on otel, but TrackEventRequest carries CodexRuntimeMetadata (codex_rs_version/runtime_os/runtime_arch) collected by events.rs current_runtime_metadata().
  • rollout → otel: rollout's compression worker metrics module (compression.rs:802) calls codex_otel::global() to get MetricsClient and records compression run/file/materialization metrics (codex.rollout_compression.* prefix).
  • rollout → analytics: rollout does not directly call analytics, but the SessionMeta and TurnContext it records are among the data sources the analytics reducer uses to correlate thread_id/turn_id.
  • rollout-trace is independent: rollout-trace is a self-contained diagnostic system that does not depend on the other three crates' runtime data flows. Its TraceWriter writes local files directly, and replay_bundle() reads files offline for replay, completely decoupled from the online telemetry channels.
