Non-Interactive Task Execution

When the user runs codex exec <prompt>, the command-line arguments are first recognized by the main CLI as a non-interactive subcommand, then handed wholesale to the exec crate. The exec crate is responsible for packaging the prompt, images, structured-output schema, working directory, model, permissions, and sandbox configuration into an in-process app-server session, and then initiating a turn/start. When the user runs codex review, the main CLI constructs an equivalent exec CLI, placing the review parameters into exec's Review subcommand, and then follows the same non-interactive execution pipeline, only submitting review/start.

Input
Command-line arguments, optional stdin prompt, image paths, JSON Schema file, configuration overrides, working directory, and permission/sandbox options; the review scenario additionally takes one of uncommitted/base/commit/custom prompt.
Output
In default mode, process logs, tool calls, diffs, plans, and token usage are written to stderr, and the final message is written to stdout when suitable for automated consumption; in --json mode, stdout is a stream of per-line JSON events. Both modes can write the last agent message to the file specified by --output-last-message.

The key state boundary here is the switch between the "interactive CLI" and the "non-interactive exec". codex review does not implement its own execution engine; it is translated into exec's Review command, avoiding divergence between review and exec in configuration, authentication, sandbox, and output loops. The permission boundary is also determined here: headless default approval_policy is never, --full-auto is compatible with workspace-write, and --dangerously-bypass-approvals-and-sandbox switches to danger-full-access.

Session creation and recovery both go through the app-server protocol. New tasks use thread/start; resume first resolves the thread via thread/list, state store, and rollout metadata, then uses thread/resume; if resume has no target, it falls back to creating a new thread. SessionConfiguredEvent is no longer waits for subsequent streaming events but is constructed directly from the start/resume response, reducing non-interactive startup latency.

Non-interactive mode does not initiate approvals or input requests to the user. MCP elicitation from the app-server is automatically canceled; command execution approval, file modification approval, request_user_input, dynamic tools, authentication refresh, attestation, and permission approval are all rejected. This is part of the Permission/Sandbox boundary: tasks can run according to existing configuration but cannot request human confirmation while running.

In default output mode, stdout is strictly reserved for the final message, and other process information goes to stderr, allowing scripts to directly capture the answer body. JSON mode is completely different: stdout is an event stream, including thread.started, turn.started, item.started, item.updated, item.completed, turn.completed, turn.failed, and error; the last message itself is not repeated as a trailing stdout line, but appears in item.completed's agent_message, or is written to the --output-last-message file.

  • State Machine: A task moves from configuration initialization into session start/resume, then into submitted/running, and after receiving completed/failed/interrupted requests unsubscribe and closes the client.
  • Communication: the exec crate exchanges JSON-RPC-style requests and notifications with the agent runtime via InProcessAppServerClient; key methods are thread/start, thread/resume, turn/start, review/start, thread/read, turn/interrupt, thread/unsubscribe.
  • Permission/Sandbox boundaries: exec headless mode does not ask for approvals by default; runtime approval requests are rejected; sandbox/permission profile is passed to the app-server in thread/start or thread/resume parameters.
  • Lifecycle: the app-server client lifecycle is wrapped by the exec session; after starting, it submits the task, and after the terminal state it unsubscribes and shuts down; if the server reports a non-retryable error or the task fails/is interrupted, the process exits with a non-zero status.
