Network Proxy (network-proxy / uds / stdio-to-uds)

This is the "network sandbox" that Codex uses when running commands generated by the AI. When Codex asks a child process (compile, run tests, install dependencies, git operations) to access the network, it does not connect directly to the internet; instead, all environment variables are rewritten to point to a local loopback proxy, which then decides per policy whether the request may go out, where it may go, and how. The full chain is: child process → loopback HTTP/SOCKS5 proxy → policy decision → direct connection or upstream proxy or MITM decryption → target server. The companion uds crate provides cross-platform Unix domain socket send/receive, and stdio-to-uds is a small bridge that connects a process's standard input/output to a Unix socket, typically used to route git-ssh through SOCKS5 via the proxy.

One-sentence breakdown of core components: NetworkProxy (proxy.rs) is the main entry point and assembler, responsible for binding ports, starting HTTP and SOCKS5 listeners, and writing environment variables for child processes; NetworkProxyState (runtime.rs) is the live policy state, holding configuration, allow/deny glob sets, MITM state, and an interception recorder — all "can this host be reached" decisions live here; http_proxy.rs / socks5.rs are the protocol-level entry handlers for the two protocols, applying protocol-level policy and routing; mitm.rs terminates TLS with a self-signed CA when it needs to inspect HTTPS inner requests, then decides whether to allow the request based on method/hook policy after decryption; network_policy.rs is the unified policy evaluation + audit event emission layer; policy.rs handles low-level host normalization and glob matching; and config.rs / state.rs / certs.rs / mitm_hook.rs / upstream.rs / connect_policy.rs / responses.rs / reasons.rs manage configuration models, constraint validation, CA certificates, request hooks, upstream clients, local bind policy, response construction, and reason strings respectively.

Main Data Flow A: HTTP Proxy Path

Input: child process HTTP request (absolute-form) or CONNECT (HTTPS tunnel), routed to the loopback proxy via HTTP_PROXY/HTTPS_PROXY environment variables.
Output: after approval, forwarded to the target (direct, upstream proxy, or Unix socket), or a 403/503 interception response is returned.
Notes: the x-unix-socket header is a macOS-specific escape hatch with a strict allowlist; CONNECT in Limited mode or with a mitm_hook must enable MITM to be allowed.

Main Data Flow B: MITM Decryption Inner Layer

Input: CONNECT or SOCKS5 (443) traffic that needs MITM has its TLS terminated by the self-signed CA.
Output: after policy + hook evaluation on the inner HTTP request, forward it or return an interception response.
Notes: the CA is persisted under codex_home/proxy/; per-host ECDSA P-256 certificates are signed; a hooked host that does not match any hook is blocked (mitm_hook_denied) to prevent bypass.

Main Data Flow C: SOCKS5 Path

Input: child process SOCKS5 TCP/UDP request, routed via ALL_PROXY (socks5h://).
Output: after approval, direct connection or MITM, or a PermissionDenied error is returned.
Notes: under Limited mode all UDP is blocked and all non-443 TCP is blocked; 443 TCP goes through MITM in Limited or when a hook exists, otherwise direct.

Policy State Machine and Hot Reload

NetworkProxyState holds the currently effective configuration inside an RwLock<ConfigState>. Every external query (current_cfg / enabled / host_blocked / ...) first calls reload_if_needed, letting ConfigReloader check whether the configuration source (e.g., config.toml) has changed; if so, it rebuilds ConfigState (recompiles glob sets, recompiles MITM hooks, and rebuilds MitmState if necessary). When Codex itself changes policy, it goes through add_allowed_domain / add_denied_domain / set_network_mode / replace_config_state: these copy the current configuration, modify the copy, validate with validate_policy_against_constraints to ensure it stays within managed constraints (preventing child tasks from escalating themselves), then write it back while preserving blocked buffers.

State machine essentials: NetworkMode has Limited and Full states; Limited→Full is allowed (escalation is constrained), while Full→Limited is restricted. MITM state is constructed once when network.mitm=true during build_config_state (MitmState holds the CA + UpstreamClient), and becomes None on subsequent reloads if MITM is turned off. HostBlockDecision is a stateless computation result (Allowed / Blocked{Denied, NotAllowed, NotAllowedLocal}).

Communication Boundaries

  • Child process ↔ proxy: loopback TCP, HTTP/1.1 (absolute-form / CONNECT) or SOCKS5 (including UDP associate). Child processes are forced to the proxy via HTTP_PROXY/HTTPS_PROXY/ALL_PROXY/NO_PROXY environment variables; on macOS, GIT_SSH_COMMAND is additionally set so git-ssh goes through SOCKS5.
  • Proxy ↔ target: direct TCP+TLS, or via upstream HTTP proxy (read from HTTP_PROXY/HTTPS_PROXY env), or via Unix socket (x-unix-socket header, macOS + allowlist only).
  • Proxy ↔ Codex main process: NetworkProxyState is a shared Arc; the main process calls its methods directly to change policy, get blocked_snapshot / drain_blocked, and set BlockedRequestObserver callbacks; the ConfigReloader trait abstracts "where configuration comes from".
  • Audit boundary: every policy decision emits a tracing::event!(target: codex_otel.network_proxy) codex.network_proxy.policy_decision event, carrying conversation_id / app_version / decision / source / reason / protocol / etc.; BlockedRequest also goes into an in-memory ring buffer (max 200 entries) and is reported via observer callback.

Auxiliary Flow: uds / stdio-to-uds Bridge

Input: a process's stdin/stdout or a Unix socket path.
Output: bidirectionally bridge stdio to a Unix socket, or provide cross-platform UnixListener/UnixStream.
Notes: uds wraps tokio's native UnixListener/UnixStream on Unix, enforcing directory mode 0700; on Windows it wraps uds_windows + async_io. stdio-to-uds takes a socket path argument, connects, then uses tokio::io::copy in both directions, shutting down the write end after stdin→socket copy (tolerating NotConnected races). Typical scenario: on macOS, GIT_SSH_COMMAND injects nc -X 5 -x <socks_addr> %h %p, routing git-ssh through SOCKS5.

Key Decision Table (host_blocked decision order)

StepConditionResult
1deny glob set matchesBlocked(Denied)
2anot allow_local_binding and host is loopback/private IP literalBlocked(NotAllowedLocal), unless an explicit allowlist exact match
2bnot allow_local_binding and DNS resolves to private / failsBlocked(NotAllowedLocal)
3allowlist is empty or no matchBlocked(NotAllowed)
4none of the above blockAllowed
5baseline says NotAllowed and a NetworkPolicyDecider is configuredhanded to decider for final ruling (may override to Allow)

Note: the decider only intervenes when baseline says NotAllowed, and only an Allow counts as an override (audit source=decider, policy_override=true). Other baseline decisions Denied/NotAllowedLocal take effect directly and the decider cannot overturn them.

Entry
NetworkProxy::builder().build() → NetworkProxy::run() starts both HTTP and SOCKS5 (optional) tokio tasks, returning a NetworkProxyHandle that can wait/shutdown.
Environment injection
NetworkProxy::apply_to_env writes HTTP_PROXY/HTTPS_PROXY/WS_PROXY/ALL_PROXY/FTP_PROXY/NO_PROXY/NODE_USE_ENV_PROXY/ELECTRON_GET_USE_PROXY/GIT_SSH_COMMAND (macOS) so that child-process HTTP clients all go through the proxy; when MITM is enabled, the CA trust bundle path is also written to SSL_CERT_FILE/REQUESTS_CA_BUNDLE and 8 other keys.
Unix socket escape hatch
The x-unix-socket header is only allowed on macOS + allow_unix_sockets match + allowed method, relaying the request to a local daemon (e.g. docker.sock). Remote abuse is strictly prevented: Unix socket support forces loopback binding.
CA management
certs.rs persists ca.pem+ca.key under codex_home/proxy/ (0600/0644, rejects symlinks/group-readable), signs per-host ECDSA P-256 certificates, and merges the original root certificates + self-signed CA into ca-bundle-<sha>.pem for child-process trust.
MITM hooks
mitm_hook.rs compiles hooks per host, matching method/path_prefix/glob, query, and header; when hit, it can strip specified request headers or inject secret headers fetched from env/file. HookedHostNoMatch (hooks configured but request matches none) is blocked (mitm_hook_denied) to prevent bypass.
