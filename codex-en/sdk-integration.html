Developers embed Codex via TypeScript or Python SDK

Developers pass natural language, images, and run options to the SDK; the SDK is responsible for creating or resuming a session, sending a turn into Codex, and organizing events returned by the process or App Server into objects that can be synchronously awaited or consumed one by one. This document traces only up to the communication boundary between the SDK and the Codex process/App Server.

Input
Thread options, text or local images, optional structured output constraints, and cancellation signals.
Output
Individual turn events, or aggregated final response, completed items, and usage; communication, parsing, and turn failures become SDK exceptions.

The TypeScript SDK is a lightweight wrapper that "launches the CLI once per turn." The thread object first saves the session ID; the first run returns the ID from the CLI, and subsequent turns use that ID to resume. Synchronous results are also aggregated after consuming the same event stream.

The core difference between the Python SDK and TypeScript is the long-lived connection: constructing a high-level client starts codex app-server, using line-delimited JSON-RPC on the same process to carry multiple threads and turns. The reader thread returns responses to waiters by request ID and routes notifications by turn ID, so events that arrive before the caller registers are also buffered without loss.

  • TypeScript cancellation uses AbortSignal passed directly to the subprocess; Python exposes an explicit turn interrupt RPC, and also allows continuing to steer into an active turn.
  • TypeScript currently only accepts text and local images; Python also accepts remote images, skills, and resource mentions, with types validated by Pydantic-generated models.
  • Python's retry utility can wrap operations on the caller side, but the base turn chain does not replay silently; this prevents a single user input from being executed twice. The TypeScript chain similarly has no automatic retry.
  • The boundary stops here: how the CLI's exec implementation and the App Server enter Rust core after receiving thread/start and turn/start belongs to downstream modules.
