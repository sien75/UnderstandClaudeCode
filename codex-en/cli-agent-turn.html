A user initiates and completes an agent turn from the CLI

After the user enters a task on the command line, Codex first interprets the startup parameters and selects an interactive interface or one-shot execution path; it then creates or resumes a session, feeds the user input into the agent turn, continuously receives model events, executes approved tools, and streams progress and final results back to the terminal. This page traces this data flow along the real implementation and marks the boundaries between the process, model service, tools, and sandbox.

Input
The user's command-line arguments, text or image prompts, working directory, and options for model, approval, sandbox, and session resumption.
Output
Continuously refreshed agent explanations, tool execution progress, and final responses in the terminal; session history and turn state are saved according to configuration for later resumption.

The first data flow determines "how to start." Without a subcommand, it enters the TUI; with exec, it launches a non-interactive client; resume merges session targeting conditions into the TUI configuration. The TUI also decides whether to connect to an embedded app-server, a local daemon, or an explicit remote service. Startup or configuration failures terminate before submitting any user input.

  resume-thread --> ready-turn

The second data flow is the agent turn itself. The app-server converts API input into core input and delivers it to the thread; the core first injects permissions, skills, plugins, environment, and history, then initiates a streaming request to the model service. If the model requests a tool, the router dispatches according to parallel capability; command tools also go through approval policy and sandbox permission checks. Tool results are written back to the context and trigger the next model request, until the model gives an answer requiring no further action, or an interruption or non-retryable error occurs.

  • Communication boundary: The CLI and app-server exchange thread/turn requests through in-process channels or remote connections; the core communicates with the model provider via Responses streaming; MCP, dynamic tools, and system processes are further external boundaries.
  • State machine: A turn cycles through "prepare context — model sampling — tool execution — result feedback" and only exits the loop when no further action is needed, the user interrupts, or a fatal error occurs.
  • Non-interactive exec cannot pop up approval prompts or collect user input by default; such server-side requests are rejected. The interactive TUI can present approval requests to the user.
  • The resume path locates records through the thread/list and thread/resume APIs, rather than the exec reading rollout files directly.
