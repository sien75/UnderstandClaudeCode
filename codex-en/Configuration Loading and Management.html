Configuration Loading and Management

Codex configuration is not governed by a single file, but by a layered stack: from the bottom-layer MDM system policy, to system-level config.toml, to enterprise cloud-distributed policy packages, to user personal config, to project-local config, and finally to command-line temporary overrides—each layer is stacked on top by priority, with higher-priority layers overriding lower-priority ones. This mechanism must ensure administrators can enforce constraints (for example, "sandbox must be read-only") while leaving enough customization space for users and projects. In addition, there is a parallel "cloud configuration" path: pulling enterprise-managed configuration packages from the ChatGPT backend, with local caching and background refresh.

User Perspective: Where Configuration Comes From and What Overrides What

When a user runs codex, configuration is stacked from lowest to highest priority as follows (numbers from ConfigLayerSource::precedence()):

MDM managed preferences (precedence 0)
Preferences pushed by device-management tools on macOS, lowest priority. Read-only constraints.
System config.toml (precedence 10)
On Unix this is /etc/codex/config.toml; on Windows it is %ProgramData%\OpenAI\Codex\config.toml. Administrator-level local machine configuration.
Enterprise cloud-managed configuration (precedence 15)
Enterprise-managed configuration fragments pulled from the ChatGPT backend, containing both config and requirements parts.
User config.toml (precedence 20/21)
$CODEX_HOME/config.toml (default ~/.codex/config.toml). The main battlefield for daily user configuration. If --profile <name> selects profile-v2, an additional layer <name>.config.toml is stacked on top (precedence 21).
Project .codex/config.toml (precedence 25)
From cwd upward to the project root, every directory containing .codex/ counts as a layer. However, project configuration is trust-checked: projects not marked as trusted in user config have their project layers disabled (recorded but not effective). Project config also has a denylist: sensitive fields such as openai_base_url, model_provider, notify, etc. are stripped, preventing projects from secretly changing user credential destinations.
Command-line --config override (precedence 30)
Temporary overrides passed via -c key=value or --config key=value, effective only for this session, represented by the SessionFlags layer.
legacy managed_config.toml (precedence 40/50)
The old managed_config.toml mechanism is now reinterpreted as requirements constraints, but as a config layer it is also "best-effort" stacked on top. Being phased out.

Besides config layers, there is a parallel requirements (constraints) chain: system requirements.toml + cloud requirements + legacy managed_config + macOS MDM. These combine into ConfigRequirements, which is not "overridden" but applied as hard constraints at the end—for example, if an administrator requires the sandbox to be read-only, a user-configured workspace-write is rejected.

Command-line parameters such as --model, --sandbox-mode, --approval-policy take another path: ConfigOverrides (harness overrides). They do not enter the config-layer stack, but directly override the corresponding fields when the final Config object is assembled, and are subject to mutual-exclusion validation (for example, sandbox_mode and permission_profile cannot both be given).

System Internals: Main Configuration Loading Flow

The entry point is ConfigBuilder::build(). It collects all inputs, calls load_config_layers_state() to stack the layers, then calls Config::load_config_with_layer_stack() to parse the stacked TOML into a runtime Config object and apply constraints and overrides.

Cloud Configuration Path (Independent Sub-Flow)

Acquisition of the cloud configuration package (CloudConfigBundle) is an independent "startup fetch + cache + background refresh" path, initiated by cloud_config_bundle_loader() at startup. The result is stored as a CloudConfigBundleLoader (a shared Future) inside ConfigLoadOptions, which the main flow awaits once in load_config_layers_state to obtain it.

Key Concepts Explained

ConfigLayerStack (configuration layer stack)
The list of all config layers ordered from lowest to highest priority, plus requirements constraints and startup warnings. It is the intermediate product of configuration loading and can be serialized to show the user "where this configuration came from."
effective_config (effective configuration)
The merged TOML after stacking all non-disabled layers in the stack with merge_toml_values. During merging, key aliases are normalized (e.g., legacy codex field name mappings) and network domains are normalized.
ConfigRequirements (configuration requirements)
Hard constraints forcibly imposed by administrators; they are not overridden but validated: allowed_sandbox_modes, allowed_approval_policies, allowed_approvals_reviewers, network/filesystem constraints, mcp_server identity constraints, etc. Applied in load_config_with_layer_stack; user-configured values not in the allowed list are rejected.
ConfigOverrides (harness overrides)
Command-line parameters (--model, --sandbox-mode, --approval-policy, --permission-profile, etc.) and harness internal overrides (cwd, codex_self_exe, etc.). They do not enter the TOML layer stack but directly override fields when the Config object is assembled, with mutual-exclusion validation.
Project trust (TrustLevel)
Project-local configuration (.codex/config.toml, hooks, exec policy) is not effective by default; it only becomes effective after the project path is marked trust = "trusted" in the user config.toml [projects] section. This is a security mechanism to prevent cloned malicious repositories from automatically executing dangerous operations via their configuration.
strict_config (strict validation)
When enabled, serde_ignored + serde_path_to_error detect all unrecognized fields in TOML; unknown keys cause errors rather than being silently ignored. CLI overrides are also validated the same way.
CloudConfigBundleLoader
A wrapper around a shared Future, turning the time-consuming "fetch cloud config package at startup" operation into a lazy + multi-awaitable operation. The main flow calls .get().await once in load_config_layers_state to obtain the result.

Module in One Sentence

codex-rs/config
Turns multi-source TOML (files/CLI/cloud/MDM) into a stacked ConfigLayerStack + ConfigRequirements—the "merger + validator" of configuration.
codex-rs/cloud-config
Turns the enterprise-managed configuration package from the ChatGPT backend into an HMAC-signed cached CloudConfigBundle—the "cloud transport + cache layer" of configuration.
codex-rs/core/src/config
Parses + constrains + overrides the ConfigLayerStack into a runtime Config object, and manages configuration consumption for editing/permissions/network proxy/etc.—the "final assembly + consumption" of configuration.

Input: CLI arguments, environment variables, config.toml files (system/user/project), cloud config packages, MDM preferences.
Output: A globally effective Config object consumed by the agent runtime, permission system, MCP, hooks, and all other subsystems; plus a ConfigLayerStack for displaying provenance and editing user configuration.

Notes:

  • Project configuration has a denylist (PROJECT_LOCAL_CONFIG_DENYLIST): sensitive fields such as openai_base_url, model_provider, notify, otel are stripped, preventing the project layer from secretly changing credential destinations or notification behavior.
  • Legacy managed_config.toml is simultaneously interpreted as a requirements layer (hard constraint) and a config layer (best-effort override), existing on dual tracks and being phased out.
  • Cloud configuration cache has a 1-hour TTL, HMAC signature, and user_id+account_id-based auth; corruption is fail-closed (treated as a miss and refetched). Background refresh only updates the cache file, not the already-loaded runtime config (snapshotted once at startup).
  • Config lockfile mechanism: if the user configures [debug.config_lockfile], a replacement configuration layer is loaded from the lockfile path, used to lock a reproducible configuration snapshot.
  • profile-v2 mechanism: --profile foo loads foo.config.toml as a second user layer on top of the base user config; it only needs to contain overrides. If the base config still has old-style profile = "foo" or [profiles.foo], an error is returned requesting migration.
