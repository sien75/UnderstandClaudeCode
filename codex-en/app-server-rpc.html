Desktop or IDE manages threads and runs agent turns via App Server

The desktop or IDE encodes thread management and agent operations as JSON-RPC requests, delivered to the App Server via stdin/stdout, WebSocket, or other configured transports. The server completes connection initialization, capability and permission checks, thread and turn routing, and then returns lifecycle events and errors as notifications to the client.

Input
Client connection parameters, and JSON-RPC requests such as thread/start, thread/resume, turn/start, turn/steer, turn/interrupt.
Output
Corresponding JSON-RPC responses, and continuously returned thread, turn, approval, error, and completion notifications to the client.

The first data flow is process startup and connection handling. The command line first determines the transport, session source, authentication, and remote control policy; the runtime loads configuration and state libraries, then establishes connections via stdio, Unix Socket, WebSocket, or remote control. Each connection has its own initialization state, experimental capabilities, notification subscription set, and serialization gate.

The second stage is JSON-RPC decoding and connection handshake. Requests are first deserialized by protocol method; initialize is the only request allowed in the uninitialized state. It validates the client name, saves experimental capabilities and notification preferences, and returns platform information. Subsequent requests that are uninitialized, re-initialized, or use unenabled experimental fields receive JSON-RPC errors.

The third stage manages threads. thread/start loads and merges configuration in the background, checks that sandbox and permissions are not provided simultaneously, validates dynamic tools, and then passes the thread creation request to core. thread/resume preferentially reuses a still-running thread, otherwise rebuilds from explicit history, state library records, or rollout files; after success, it automatically attaches the current connection's event listener. The core's model and tool loop is the lower boundary; only the effective configuration, history, input, dynamic tools, and tracing context are handed over here.

The fourth stage runs turns. turn/start converts client input, additional context, and current turn settings into a core UserInput operation and immediately returns the in-progress turn; turn/steer only allows directing to the currently steerable turn; turn/interrupt validates the target turn, with normal interrupts waiting for the TurnAborted event before responding, while startup interrupts respond immediately after submission succeeds.

Finally, the thread listener converts core lifecycle events into App Server notifications or deferred responses. After notifications enter the outbound envelope, directed messages are only sent to the corresponding connection, broadcasts only to connections that are initialized and have not unsubscribed from that method; experimental fields are filtered by connection capability. When a slow WebSocket queue fills up, the connection is proactively disconnected; on disconnection, the RPC gate is closed, thread listeners are unbound, and command process resources are cleaned up; normal shutdown also waits for background tasks and closes threads within bounds.

  • Communication boundary: stdio, Unix Socket, WebSocket, and remote control only convert messages into unified TransportEvent; business routing does not depend on a specific transport.
  • State machine boundary: Connections transition one-way from "uninitialized" to "initialized"; after disconnection, the RPC gate closes; turns progress from InProgress after submission, driven by core events.
  • Cancellation semantics: The success response for a normal turn/interrupt is bound to the TurnAborted event, avoiding "interrupt submitted" being mistaken for "turn already terminated."
