Interactive Terminal Session

When the user runs codex in the terminal, they enter the TUI, type a prompt in the bottom input area, and submit it. This scenario turns a piece of user input into a core session turn: the TUI first handles terminal events, drafts, queuing, and local rendering, then passes the input to the core session via the app-server thread/turn interface; model output, tool calls, permission requests, and errors flow back through the app-server event stream to the currently visible thread, ultimately updating the chat history, status bar, and approval dialogs.

Input
The prompt, image/mention attachments, current working directory, model/reasoning strength, approval policy, and permission configuration entered by the user in the interactive terminal.
Output
A running or completed Codex conversation turn: includes the user message, streaming model response, tool execution status, permission/approval requests, error messages, token usage, and session resumption information.

  • Communication boundary: the TUI accesses the embedded or remote app-server through `AppServerSession`; the core session's `CodexThread` is held by the app-server, and the TUI does not consume core `EventMsg` directly.
  • State-machine boundary: the input area switches among "draft, queued, waiting for session configured, turn running, approval dialog, completed/error"; a turn already in progress causes subsequent ordinary prompts to be queued or become a steer.
  • Permission/sandbox boundary: approval policy, permission profile, workspace roots, and possible sandbox overrides are passed to the backend with the `turn/start` request; commands, file changes, and permission expansions that require human confirmation come back to the TUI as a `ServerRequest`.
  • Lifecycle boundary: at startup the app-server bootstrap, main thread channel, and TUI event stream are established; at shutdown the app-server is closed first, then the terminal display is cleaned up and token usage plus a resume hint is returned.
