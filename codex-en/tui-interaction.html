Terminal Interaction: Turning User Input into a Continuously Updated, Approvable Conversation

After the user starts Codex, they can resume an old task or start a new one, entering text, images, mentions, and slash commands in the bottom editor; during execution, the model's responses, reasoning, plans, and tool execution gradually appear in the conversation area. When encountering commands, file modifications, permissions, or MCP requests, normal input temporarily yields to an approval view; after the user decides, the result is sent back to the running task.

Input
Startup parameters, tasks to resume, keyboard and paste events, user messages, and task events and approval requests continuously sent from the app-server.
Output
Chronologically updated terminal conversation, current running status, approval interface requiring user action, and messages, interruptions, and approval decisions submitted to the app-server.

Startup and event convergence: The TUI first completes configuration, terminal, and app-server initialization; the session selector loads tasks page by page and can asynchronously preview history. Once in chat, all input sources converge into a single event loop, so rendering, network notifications, and user keystrokes do not independently modify the terminal.

Input data flow: Keystrokes are first handed to the current modal view; only when no modal is active do they enter the editor. Upon submission, image capability is checked, shell escapes and slash commands are parsed, mentions are encoded, and the current model, permissions, and collaboration mode are attached. Empty or non-submittable messages remain local; a new turn is created when the task is idle, or queued as a steer when the task is running.

Output and lifecycle: App-server notifications are first filtered by task, then update the turn status, tool units, or streaming controller based on the event type. Text deltas are not written character-by-character directly to the scroll buffer; instead, they are accumulated in source, and only complete lines that have stabilized are committed, while the mutable tail is preserved. When complete, canonical history units are generated from the full Markdown. This way, when the terminal width is adjusted, the committed scroll buffer and the active tail remain in order.

Approval state machine: When a request arrives, the active tail of the response is first flushed to avoid the approval box being inserted mid-text. If the user is typing, approval may be briefly delayed; the request then enters a unified queue. After the user chooses allow once, allow session, deny, or cancel, the view wraps the decision into a specified task operation, sends it back to the app-server via AppEvent, and then displays the next item in the queue.

  • Boundary: The TUI does not perform model inference or tool execution; it passes user intent and approval decisions to the app-server; the core execution chain is handled by the core module.
  • State machine emphasis: Idle, turn running, waiting for approval, interrupted, and finished are not independent pages, but orthogonal states jointly maintained by ChatWidget, BottomPane, and the streaming controller.
  • The resume picker does not filter local paths for remote workspaces; pagination, search tokens, and preview loading tokens prevent older asynchronous results from overwriting a new selection.
  • Ctrl+C first cancels the current modal or running task; only when there is nothing cancellable does it enter the double-tap exit window, avoiding accidental session loss from a single mis-press.
