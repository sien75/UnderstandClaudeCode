Sandbox and Command Execution Isolation

Codex needs to run model-generated commands and modify files on the user's machine. This code is responsible for three things: wrapping commands in a sandbox (restricting which directories they can read/write and whether they can access the network), deciding whether a command should be allowed automatically (policy engine + dangerous-command heuristics), and using a separate server to actually execute commands and file operations (exec-server, either local or remote). Overall goal: when the model says "run rm -rf /", the system can block it; when the model says "run npm test", the system can run it safely in a restricted environment.

Trigger timing: the model produces a shell command or file-edit action during a turn → codex-core first uses execpolicy / shell-command to make a policy decision (allow / prompt / forbidden) → after user approval → SandboxManager converts the command + permission profile into platform-native sandbox launch parameters → exec-server receives a SandboxExecRequest, spawns a process or reads/writes files inside the sandbox → the output stream is returned to codex-core.

File System Sandbox Execution Flow (inside exec-server)

exec-server doesn't just run commands; it also performs file reads and writes on behalf of the model. File operations also go through the sandbox, but the mechanism is different: it uses the codex binary itself as a helper, restarts itself in codex-fs-helper mode, applies the sandbox, and then exchanges JSON requests over stdin/stdout. Below is the data flow for the file system sandbox.

    fs-req["client sends fs/readFile / fs/writeFile etc. JSON-RPC"] --> fs-handler

    fs-handler -->|"no, read locally directly"| local-fs

    fs-spawn-helper["spawn codex itself as helper, apply seatbelt/bwrap/win-token"] --> fs-stdin
    fs-stdin["write FsHelperRequest JSON into helper stdin"] --> fs-helper-main
    fs-helper-main["fs_helper_main.rs: helper process entrypoint, read stdin, perform file operations, write JSON back"] --> fs-resp
    fs-resp["FsHelperPayload result returned to client"] --> flow-fs-end
    flow-fs-end[("file operation complete")]

Command Policy Decision State Machine (execpolicy)

execpolicy is a Starlark DSL engine. Users write rules like prefix_rule(pattern=["git","commit"], decision="allow"). Policy.check() feeds the command token list to the rules, and matched prefix rules produce a Decision (Allow / Prompt / Forbidden). When multiple rules match, the strictest one is chosen (Forbidden beats Prompt beats Allow). If no rule matches, a heuristic fallback is used (provided by shell-command's safety checks).

graph TD
    cmd-tokens["command token list, e.g. git commit -m"] --> exact-match
    exact-match{"look up rules_by_program by first token"}
    exact-match -->|"match"| prefix-match
    exact-match -->|"no match"| host-exe-match
    host-exe-match{"first token is an absolute path? look up host_executables"}
    host-exe-match -->|"matched basename rule"| prefix-match
    host-exe-match -->|"no match"| heuristic

    prefix-match -->|"no match"| heuristic
    heuristic["shell-command: is_known_safe / command_might_be_dangerous"] --> collect-decision
    collect-decision["take the maximum Decision from all matched rules"] --> eval-out
    eval-out[("Evaluation: decision + matchedRules")]

Module Inventory

sandboxing (cross-platform sandbox manager)
Turns the "command the model wants to run + permission profile" into "platform-native sandbox launch parameters". Input: SandboxCommand + PermissionProfile; output: SandboxExecRequest. macOS uses seatbelt (sandbox-exec + .sbpl policy), Linux uses codex-linux-sandbox (bwrap + seccomp), Windows uses restricted token. Also handles details such as merging additional permissions, MITM CA readable roots, and proxy network policies.
linux-sandbox (Linux sandbox helper)
Implementation of the codex-linux-sandbox binary. run_main parses CLI arguments (--permission-profile JSON, --sandbox-policy-cwd, etc.). By default it uses bubblewrap to create a mount namespace + file system view, then re-enters itself to add seccomp + no_new_privs, and finally execvp the user command. legacy landlock is an alternative path. Also includes proxy_routing (builds a proxy bridge in netns) and bwrap argument construction (bwrap.rs, 2708 lines).
windows-sandbox-rs (Windows sandbox)
Restricted-token sandbox for Windows. Implemented with de-elevated token + ACL (allow/deny read/write) + WFP network filtering + ConPTY + desktop isolation. Includes setup (create sandbox user, firewall rules), spawn_prep (command preparation), and unified_exec (unified execution backend: elevated / legacy). About 17k lines; the largest cross-platform difference.
execpolicy (command policy engine)
Starlark DSL parser + prefix rule matcher. Users define policies with prefix_rule / network_rule / host_executable; Policy.check() performs prefix matching on commands and produces Allow/Prompt/Forbidden. The amend module supports appending rules at runtime (with file locking and deduplication). parser.rs uses the starlark library to execute the DSL.
execpolicy-legacy (legacy command policy)
Predecessor to execpolicy, based on custom policy files + execv_checker for command validation. Kept for backward compatibility with old configurations.
shell-command (command parsing and safety checks)
Parses shell scripts (bash -lc) / PowerShell commands into token lists (ParsedCommand). The command_safety module uses heuristics to determine whether a command is "known safe" (is_known_safe_command, whitelist) or "potentially dangerous" (command_might_be_dangerous, blacklist + rm -rf detection). Used as a fallback when execpolicy does not match any rule.
shell-escalation (Unix exec interception)
Intercepts child-process exec() calls via a patched shell (execve_wrapper) and sends EscalateRequest (file/argv/env) to escalate_server. The server decides Run (execute directly), Escalate (re-run with elevated privileges or different sandbox config), or Deny based on policy. Used for secondary approval when a command running inside the sandbox tries to invoke an external program.
exec-server (command/file execution server)
Standalone JSON-RPC server responsible for actual process launching and file operations. protocol.rs defines all methods (process/start, process/read, fs/readFile, etc.). The server module (processor → handler → registry) routes requests; local_process.rs spawns processes with pty/pipe and streams output back; fs_sandbox.rs wraps file operations in a sandbox (restarts codex itself as a helper); client.rs is the client implementation. Supports both local (stdio) and remote (WebSocket + Noise encrypted relay) modes.
bwrap (bundled bubblewrap)
Bundled bubblewrap binary entrypoint; used on Linux when the system has no bwrap installed.
process-hardening (process hardening)
Hardening executed during the pre-main phase (ctor): disable core dumps (setrlimit RLIMIT_CORE=0), disable ptrace attach (Linux prctl PR_SET_DUMPABLE=0 / macOS PT_DENY_ATTACH), and clear dangerous environment variables such as LD_PRELOAD / DYLD_*. Prevents processes from being debugged or hijacked via library injection.

Key Boundaries and Communication

  • codex-core and exec-server: JSON-RPC over stdio (local) or WebSocket + Noise (remote). First initialize handshake, then initialized notification, then process/* and fs/* methods. Process output is streamed back via process/output notifications; process exit sends process/exited + process/closed.
  • SandboxManager and platform sandbox: transform() is a pure function that converts the permission profile into argv. macOS prepends /usr/bin/sandbox-exec -p policy to argv; Linux prepends codex-linux-sandbox --permission-profile json; Windows does not modify argv, relying instead on token/ACL settings at spawn time.
  • exec-server and sandbox helper: fs_sandbox uses codex itself as the helper; after spawning it writes FsHelperRequest JSON via stdin, and the helper writes FsHelperResponse JSON to stdout when done. Environment variables are whitelisted (PATH/TMPDIR) to prevent secret leakage into the sandbox.
  • shell-escalation client and server: Unix socket (FD passed via the CODEX_ESCALATE_SOCKET environment variable). The client sends EscalateRequest, and the server replies with EscalateAction (Run / Escalate / Deny). The patched shell's execve_wrapper intercepts exec() to trigger this communication.

Cross-Platform Differences

  • macOS: relies on sandbox-exec (seatbelt) to execute .sbpl policy files. seatbelt.rs translates PermissionProfile into readable/writable roots (subpath rules) + network (localhost ports / unix sockets) + glob deny. Only trusts /usr/bin/sandbox-exec to prevent PATH injection.
  • Linux: defaults to bubblewrap (creates namespace) + seccomp (limits syscalls). landlock is a legacy fallback (supports full-disk readable + restricted writes, but not read-only restrictions). WSL1 does not support bwrap (no user namespace) and will error. Proxy mode builds a bridge in netns, allowing connections only to the proxy port.
  • Windows: uses de-elevated token + ACL (path-level read/write allow/deny) + WFP (Windows Filtering Platform) for network filtering. Requires setup beforehand to create sandbox users and firewall rules. Supports ConPTY and desktop isolation.
