模型回合与工具执行

用户从 TUI、exec、app-server 或 SDK 发起一次请求后，最终都会进入同一条核心链路：请求先变成一个带 ID 的 submission，session loop 按顺序接收它，创建本回合的 TurnContext，把用户输入、附加上下文、技能和插件提示写入历史，然后向模型发起流式采样。模型如果只回复文本，回合结束；如果要求调用工具，系统先按当前权限和工具注册表执行工具，再把工具输出追加回模型输入，继续下一轮采样，直到模型明确结束或没有待处理输入。

Input
用户消息、附加上下文、线程设置、权限配置、当前工作区和可用工具
Output
面向客户端的事件流：推理/文本增量、工具开始和完成、token 用量、diff、最终 assistant 消息或错误
graph TD
  external-client["外部入口提交用户请求"] --> enqueue-submission["包装成带 ID 的 Submission"]
  enqueue-submission --> session-loop["Session loop 顺序分发 Op"]
  session-loop --> create-turn["创建本回合 TurnContext"]
  create-turn --> steer-or-start["判断是追加到活跃回合还是启动新任务"]
  steer-or-start --> record-input["记录用户输入和附加上下文"]
  record-input --> inject-skills["注入技能、插件和扩展上下文"]
  inject-skills --> build-history["从历史构造模型输入"]
  build-history --> build-tools["按本回合配置暴露工具"]
  build-tools --> stream-model["向模型发起流式采样"]
  stream-model --> handle-text["文本/推理增量转成客户端事件"]
  stream-model --> handle-tool["模型工具调用进入 ToolRouter"]
  handle-tool --> dispatch-tool["按工具注册表和权限执行工具"]
  dispatch-tool --> tool-output["工具输出写回模型输入"]
  tool-output --> build-history
  stream-model --> complete-response["response.completed 汇总用量和是否继续"]
  complete-response --> need-follow-up["判断是否还有工具、待输入或压缩续跑"]
  need-follow-up --> build-history
  need-follow-up --> finish-turn["无后续工作时运行 stop hook 并结束回合"]
  finish-turn --> emit-final["发送最终事件、token 和 diff"]
  • Codex facade 只暴露提交和接收事件的队列接口；具体行为由后台 session loop 消费 Op 后分派。
  • TurnContext 是一次回合的事实来源，包含模型、权限、sandbox、环境、技能、动态工具、输出 schema 和 telemetry。
  • 工具不是直接由模型执行；模型输出先被解析成 ToolCall，再由 ToolRouter 从注册表找到对应 executor。
  • 回合可能循环多次：工具输出、自动压缩、stop hook continuation 或用户中途输入都会让系统重新构造模型输入并继续采样。
