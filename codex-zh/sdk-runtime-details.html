SDK 公共能力与本地运行时打包

SDK 使用者从稳定的 TypeScript 导出或 Python 顶层对象进入 Codex。两种实现最终都要找到与当前平台匹配的 Codex 可执行文件，但 TypeScript 随 npm 可选依赖解析二进制并按回合启动进程，Python 则优先使用独立的 wheel 运行时包，并让一个 App Server 连接承载账户、线程、回合和目标操作。

输入
SDK 配置、账户与线程操作、用户输入、目标预算，以及当前操作系统和 CPU 架构。
输出
类型化的同步/异步 SDK 结果；或一个包含正确平台 Codex CLI 的可安装 wheel，供 SDK 在运行时启动。

TypeScript 的入口只导出经过选择的公共类型与类。客户端构造时解析 npm 平台包；运行时把嵌套配置转成 CLI 可理解的 TOML 覆盖项，并在 PATH 前置随包工具目录。

Python 的公开层同时提供同步和异步门面；异步版本把阻塞客户端工作放入线程。除普通回合外，目标操作会把多个物理回合折叠成一个逻辑流：目标暂停、阻塞、达限或完成后才真正结束，并在中断及回合滚动竞态中维护一致的逻辑回合编号。

Python 的 CLI 二进制独立发布为平台 wheel。构建钩子禁止源码包并固定非纯 Python 平台标签；开发/发布辅助链路从 SDK 的精确依赖 pin 推导版本，下载对应 GitHub Release 资产、暂存运行时包并强制安装，最后再次验证版本和二进制可用性。

  • TypeScript 的类型 schema 明确了事件与线程项目的判别联合；Python 的协议模型主要由生成代码提供，本报告将生成文件标为忽略，只深入手写的适配、校验和生命周期逻辑。
  • Python 的批准模式会同时配置 sandbox 与 approval policy，避免把危险的组合分散给调用方；初始化元数据和版本解析保证服务端能识别 SDK 来源。
  • 运行时下载包含多级回退：匿名直链、带令牌的 GitHub API、最后使用已认证的 gh。任何一步失败都会保留可诊断的上下文，而不会静默选择错误平台。
