登录与认证

这一组 crate 解决的核心问题是:用户怎么把"我是谁、我有什么权限"交给 Codex,以及 Codex 在运行过程中怎么把这份凭据安全地存下来、按需刷新、过期或 401 时自动恢复、登出时干净地撤销。入口是 codex-login,它对外暴露 AuthManager 作为整个程序唯一的凭据真相源;周边 crate 各管一段:agent-identity 管机器代理身份 JWT,keyring-store/secrets 管 OS keyring 和加密文件存储,aws-auth 管 Bedrock 的 SigV4 签名,chatgpt crate 是消费方——拿 AuthManager 给的 token 去打 ChatGPT 后端接口,install-context 则决定 Codex 自身是怎么装上的(影响 originator 头和捆绑资源路径)。

从用户视角看,这一场景能触发的动作有:① 装好 Codex 后第一次 codex login 走浏览器 OAuth 或设备码登录;② 直接 codex login --api-key / 拿到一个 at- 开头的 PAT / 拿到 AgentIdentity JWT 贴进去;③ 设了 CODEX_API_KEY 或 CODEX_ACCESS_TOKEN 环境变量免登录直接跑;④ 外部宿主(IDE/插件)通过 ExternalAuth trait 临时塞 ChatGPT token 或自定义 bearer 命令进来;⑤ 运行中 token 过期或被服务端拒掉时,AuthManager 自动刷新或走 401 恢复;⑥ codex logout 撤销并清掉本地凭据。

模块清单

codex-login (auth/manager.rs)
认证总中枢。把"持久化里的 auth.json + 环境变量 + 外部 provider"三种来源按固定优先级合并成一个 CodexAuth 快照缓存起来,对外提供取 token、刷新、登出、401 恢复。输入:codex_home、存储模式、环境变量、可选外部 provider。输出:当前有效的 CodexAuth(或 None=未登录)。
codex-login (auth/storage.rs)
凭据怎么落地。按存储模式在 文件 / OS keyring 直连 / 加密 age 文件 / 自动回退 / 进程内 ephemeral 之间选一个后端,统一 load/save/delete 接口。输入:AuthDotJson 序列化字节。输出:同一份字节读回。Note:keyring 账号名是 codex_home 路径的 sha256 截短,避免多实例撞车。
codex-login (server.rs + device_code_auth.rs + pkce.rs)
交互式登录的两套链路。浏览器链路起一个本地 1455 端口接 OAuth 回调;设备码链路无浏览器时让用户去另一个设备开链接输码。两条链最后都走 exchange_code_for_tokens 拿到 id/access/refresh 三件套再持久化。输入:issuer、client_id、PKCE。输出:写好的 auth.json。
codex-login (auth/personal_access_token.rs / agent_identity.rs / bedrock_api_key.rs / external_bearer.rs / access_token.rs)
非 OAuth 的几条登录支线。PAT 调 /whoami 补全账号元数据;AgentIdentity JWT 走 JWKS 验签后注册任务;Bedrock API key 直接落盘;外部 bearer 跑用户配置的命令拿 stdout 当 token。输入:各自的原始凭据串。输出:能塞进 CodexAuth 的结构。
codex-login (auth/revoke.rs)
登出时撤销 OAuth token。优先撤 refresh_token,没有再撤 access_token,撤失败不影响本地清理。输入:auth.json 里的 token。输出:向 auth.openai.com/oauth/revoke 发一个 POST。
codex-agent-identity
机器代理身份的密码学层。生成 ed25519 密钥对、解 JWT、抓 JWKS 验签、给任务签名出 AgentAssertion 头、注册任务拿 task_id(必要时 curve25519 解密)。输入:JWT 或 PKCS8 私钥。输出:验过的 claims 或签名头。Note:这是给自动化 agent 跑的,不是人登录。
codex-keyring-store
OS keyring 的薄封装。把 keyring crate 的 load/save/delete 包成 KeyringStore trait,带 Mock 实现供测试。输入:service+account。输出:明文串。Note:secrets crate 和 auth storage 都靠它做底层。
codex-secrets
本地加密 secrets 仓。用 age(scrypt)加密一个 JSON map 存到 codex_auth.age 等文件,加密密钥放 OS keyring;auth 的 SecretsKeyringAuthStorage 用它做"双保险"加密存储。输入:scope+name+value。输出:解密后的串。Note:CodexAuth 命名空间专门隔开 CLI 认证凭据。
codex-aws-auth
Bedrock 的 SigV4 签名。用 AWS SDK 默认链(profile/region/env)解析凭据,再对出站请求做 SigV4 签名。输入:AwsAuthConfig + 待签请求。输出:带 Authorization/x-amz-date 的请求头。Note:跟 OAuth 那套完全平行,不走 AuthManager。
codex-chatgpt
认证的消费方之一。用 AuthManager 拿 ChatGPT token 后调后端 API:列 connectors、拉 task diff、查 workspace 的 plugins 开关。输入:Config + CodexAuth。输出:反序列化的业务对象。Note:它只读不写认证状态。
codex-install-context
探测 Codex 自身的安装方式(standalone/npm/bun/brew/other)和捆绑资源路径。输入:current_exe、环境标志。输出:InstallMethod + 资源目录。Note:和认证无直接关系,但 originator 头和 CA 资源路径会影响登录请求怎么发。

数据流一:启动时加载凭据(优先级链)

这条链回答"程序起来时,到底用哪份凭据"。优先级是固定的:CODEX_API_KEY 环境变量 > 进程内 ephemeral 存储(外部塞进来的 ChatGPT token)> CODEX_ACCESS_TOKEN 环境变量(免登录 PAT/AgentIdentity)> 持久化存储(file/keyring/secrets)。这个顺序保证:外部宿主临时注入的凭据能盖过本地登录,而本地登录又盖过环境变量。ephemeral 排在 CODEX_ACCESS_TOKEN 前面,是因为外部宿主注入就是用 ephemeral 存的。

数据流二:浏览器 OAuth 登录

用户跑 codex login(不带额外参数)时走这条。Codex 在本地 1455 端口(被占则发取消请求重试,再不行退到 1457)起一个 tiny_http 服务,生成 PKCE 和 state,拼好授权 URL 用 webbrowser 打开。用户在浏览器里完成 ChatGPT 登录后,OpenAI 的授权服务器把 code 回调到 /auth/callback。本地服务拿 code 去 /oauth/token 换 id/access/refresh 三件套,再用 id_token 走一次 token-exchange 拿一个 API key,最后把整套存成 auth.json,并把浏览器重定向到 /success 页面收尾。任何一步失败都渲染一个错误页并报错退出;用户主动取消走 /cancel。

数据流三:设备码登录

无浏览器环境(SSH/容器)下走这条。Codex 先向 /deviceauth/usercode 申请一个用户码和轮询间隔,打印链接和码让用户去另一台有浏览器的设备上完成登录;同时开始轮询 /deviceauth/token,服务端在用户完成前回 403/404,Codex 按间隔重试,最长 15 分钟。用户确认后服务端返回 authorization_code 和 code_verifier,Codex 再用标准的 exchange_code_for_tokens 换 token、校验 workspace、持久化。和浏览器链相比,它少了本地回调服务器和 API key 兑换那一步。

数据流四:Token 刷新与 401 恢复状态机

这是认证运行时最关键的一段。UnauthorizedRecovery 是一个显式状态机,只在请求收到 401 时被驱动(每次重试调一次 next())。它分两大模式:Managed(本地托管的 ChatGPT 登录)走 Reload→RefreshToken 两步——先从盘重载(防止另一个 Codex 进程已经刷过 token),重载后凭据没变才去 auth.openai.com 刷;刷成功就持久化新 token 并重载缓存。External(外部宿主塞进来的 ChatGPT token 或自定义 bearer 命令)只走一步——回调外部 provider 的 refresh() 拿新 token 存 ephemeral 再重载。API key / PAT / Bedrock 这些不可刷新的凭据直接不进恢复,401 原样上抛。

另外,在每次 auth() 被调用时还有一条主动刷新支线:如果 access_token 的 exp 在 5 分钟内、或 last_refresh 已超过 8 天,就先发一轮 refresh,避免等到 401 才动手。永久刷新失败会被缓存,同一份凭据不再重试网络,直接 fail-fast。

数据流五:登出与撤销

登出有两条:普通 logout 只删本地存储;logout_with_revoke 先尝试向 OpenAI 撤销 OAuth token(优先 refresh_token,没有就撤 access_token)再删本地。撤销失败只打告警不阻断——本地凭据照删,因为"撤销"是尽力而为,不能因为网络问题让用户登不出。删的时候 ephemeral 和持久化两个存储都删,保证外部注入的临时凭据也被清掉。最后一定 reload 一次清空内存缓存。

    E2 -- 普通登出 --> E3[删 ephemeral 与持久化存储]
    E2 -- 带撤销登出 --> E4[读当前 auth.json]
    E4 --> E5{有可撤销 token?}
    E5 -- refresh_token 非空 --> E6[POST /oauth/revoke 撤 refresh_token]
    E5 -- 否 access_token 非空 --> E7[POST /oauth/revoke 撤 access_token]
    E5 -- 都没有 --> E8[跳过撤销]

    E7 --> E9
    E9 -- 否 --> E10[告警但仍继续]

数据流六:凭据存储后端选择

auth.json 到底存哪儿,由 AuthCredentialsStoreMode + AuthKeyringBackendKind 两个配置决定。File 模式就是 codex_home/auth.json(0600 权限);Keyring 模式分 Direct(直接进 OS keyring)和 Secrets(进加密 age 文件,密钥放 keyring);Auto 模式优先 keyring、失败回退文件;Ephemeral 模式进进程内 HashMap,进程退出即失(外部宿主临时凭据走这条)。keyring 的账号名是 codex_home 路径的 sha256 截短,保证同一台机器上不同 codex_home 互不干扰。

边角:外部 bearer 命令刷新

自定义 model provider 可以配一条 shell 命令当认证源(BearerTokenRefresher)。它实现了 ExternalAuth trait,resolve() 时按配置的 refresh_interval 缓存 stdout 当 bearer token;refresh() 时无视缓存直接重跑命令。这条链和 ChatGPT 的 OAuth 刷新完全独立,走的是 401 恢复里 External 模式的同一条通路,只是不碰盘、不存 ephemeral。

边角:AgentIdentity 任务签名

AgentIdentity 不是给人用的登录,是给自动化 agent 跑的。JWT 里带着 agent_runtime_id 和一个 PKCS8 私钥;加载时除了验签,还会用私钥对 agent_runtime_id:timestamp 签名,POST 到 /v1/agent/{id}/task/register 注册一个 task,服务端可能返回明文 task_id 也可能返回 curve25519 加密的(用 ed25519 私钥派生 curve25519 私钥解密)。之后每次请求带 AgentAssertion 头,内容是 agent_runtime_id:task_id:timestamp 的签名。这套和 OAuth token 完全平行,不参与刷新/撤销。

边角:AWS Bedrock SigV4 签名

Bedrock 走的是 AWS 标准 SigV4,跟 OpenAI 的 OAuth 体系毫无关系。AwsAuthContext::load 用 AWS SDK 默认凭据链(profile/region/env/IMDS)解析出 credentials provider 和 region,之后每个出站请求用 sign_request 加 Authorization、x-amz-date、x-amz-security-token 头。凭据 provider 超时/瞬时错误算可重试,配置错误不可重试。
