支撑服务：把配置与扩展变成可控、可通信、可观测的代理能力

这一组 crate 不直接决定代理如何推理，而是把本地配置、插件市场、技能、Hook、MCP、网络访问、云端任务和遥测接到代理运行时。对用户而言，触发点通常是启用插件、添加市场、启动一次任务、允许一次联网，或配置观测出口；系统随后完成发现、校验、物化、注册、执行和回传。

输入
分层配置、插件或技能声明、Hook 事件、MCP 调用、模型 API 请求、网络策略与云任务命令
输出
已注册的代理能力、受策略约束的外部通信、可恢复的云端任务结果，以及日志、指标和分析事件

插件数据流从配置层与远端已安装状态汇合开始。加载器按插件逐个读取 manifest，并把技能、MCP server、App connector 和 Hook 组合为统一结果；管理器串行化刷新、缓存和安装，安装时先物化来源，再原子写入插件仓库和用户配置。产品限制在“准入”阶段执行，运行期只消费已经准入的内容。

扩展 API 把不同能力收束为类型化 contributor 注册表：配置、上下文、工具、MCP、审批、线程和回合生命周期都有独立插槽。Hook 是另一条显式生命周期边界；它把事件序列化后交给已发现的命令，按匹配规则执行，并允许前置 Hook 拒绝工具或修改输入。权限请求因此既能由内建策略决定，也能被外部 Hook 审阅。

通信侧分成三层：codex-api 负责把内部请求编码为 Responses、WebSocket、搜索、图片等 HTTP 协议并解析流；rmcp-client 负责 stdio、进程内或 streamable HTTP 的 MCP 会话及 OAuth；network-proxy 在真正出网前统一应用域名、Unix socket、MITM 和动态审批策略。网络拒绝会被观察者记录，而不是静默失败。

其余支撑链条各自承担明确边界：config 将系统、用户、项目、云端要求与运行时覆盖合并，并保留来源和诊断；core-skills 发现、筛选、渲染并按提及注入技能；connectors 合并目录与缓存中的 App 元数据；cloud-tasks 提供云任务创建、环境检测、轮询和差异展示；external-agent 模块迁移其他代理的 MCP、Hook、命令与会话；analytics 把原始事实归并成产品事件；otel 安装日志、Trace 与指标 exporter，并在关闭时 flush。utils 下的小 crate 提供路径、PTY、流式文本解析、图片、模板、缓存、休眠抑制等无业务状态的基础能力。

  • 配置是权限与生命周期的共同根：插件准入、技能禁用、Hook 信任和网络约束都从可追踪的配置层派生。
  • 远端插件、MCP OAuth、模型 API、云任务和 OTLP exporter 都是外部通信边界，失败会转成领域错误或诊断信息。
  • 插件缓存与远端刷新使用串行化状态，避免并发刷新覆盖；遥测和分析事件通过队列与去重限制控制容量。
  • 测试、生成 schema/protobuf 和固定模板未作为运行时实现展开；小型工具 crate 已按公开 API 和归属完成映射。
