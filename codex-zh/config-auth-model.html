配置、认证与模型连接

用户启动 Codex 后，分层配置会被合并成一份受管理策略约束的运行时配置；系统据此选择模型服务、读取或刷新登录凭据、发现可用模型，最后把带认证信息的请求交给 HTTP/SSE 传输层。模型回合的内容构造属于上层边界，本模块停在“连接已具备发送推理请求的条件”。

输入
系统、用户、项目、会话与命令行配置，CODEX_HOME 中的凭据或系统钥匙串，以及可选的外部认证命令
输出
确定的模型提供方、当前有效身份、可用模型元数据，以及能够访问模型或 ChatGPT 后端的认证 HTTP 客户端

第一段数据流决定“去哪里请求”。高优先级配置覆盖低优先级配置，但 requirements 与权限约束仍可拒绝不被允许的最终值。

认证数据流刻意维护单一快照：运行中的各组件不会各自读取 auth.json。文件、系统钥匙串和临时外部凭据共享存储接口；自动模式会优先钥匙串，失败时回退文件。

请求遇到 401 时不会无限重试。受管 ChatGPT 身份先从存储重载一次，再走 OAuth 刷新；外部身份只通知宿主或重跑认证命令一次；API Key 与不可刷新的个人令牌直接把错误交还上层。

provider 把配置和身份组合成运行时能力。Bedrock 走专用实现；普通兼容服务按配置决定是否使用 OpenAI 身份、静态 token、环境变量或外部命令。模型目录可以由配置完全固定，也可以在 bundled、磁盘缓存与远程 /models 之间协调。

最终通信边界分为两类：ChatGPT 后端客户端负责路径风格、账号/FedRAMP 等业务头；通用 Codex 客户端负责代理、自定义 CA、压缩、trace 头、普通响应和 SSE 字节流。两者都从统一认证提供器注入 Authorization，避免业务调用点复制 token 逻辑。

  • 权限边界：managed requirements 在 Config 形成前约束用户选择；登录方式与 ChatGPT workspace 也可被组织配置强制限定。
  • 秘密边界：认证调试输出不暴露 token；钥匙串模式把长期凭据交给操作系统，外部 bearer 可只存在于进程内。
  • 缓存边界：模型缓存以客户端版本和 TTL 判定新鲜度，并保存 ETag；当前实现注明 provider 身份尚未进入缓存资格判定。
  • 失败策略：模型目录联网失败会保留已有目录；HTTP 非成功状态保留足够上下文供上层分类，401 恢复状态机严格有界。
