TUI 渲染与交互输入

这个模块讲的是 Codex 终端界面(TUI)里"用户怎么打字、怎么提交、怎么回答 agent 的问题、怎么恢复旧会话"这一整条交互链,以及这些动作触发后界面是怎么一笔一笔画出来的。核心容器是 BottomPane(底部面板):它同时持有常驻的 ChatComposer(聊天输入框)和一个临时的 view 栈(弹窗/模态)。所有键盘事件先到这里,由它决定"这键给谁"。渲染时它把自己组装成一个 flex 布局:状态行 + 待处理输入预览 + 输入框(或最顶层弹窗)。会话恢复则走另一条独立的 alt-screen 流程(resume_picker)。

主视角是数据流:从"按键"进,到"画到屏幕上"出;辅助视角是状态机(view 栈的入栈/出栈、RequestUserInputOverlay 的多题问答状态机)和通信边界(AppEvent 发往 app 层、TuiEvent/background 事件回灌)。

关键概念速览

BottomPane(底部面板)
底部那块交互区的总管。持有常驻的 ChatComposer + 一个 view 栈。键盘事件先到它,它决定路由到哪个 surface。它不管"退出/中断"这种进程级决策,那归上层 ChatWidget。
ChatComposer(聊天合成器)
真正的多行输入框,内含一个 TextArea(纯文本编辑)+ 弹窗(@提及、/斜杠命令、技能)+ 历史导航 + 底部 footer 提示。提交后产出 InputResult::Submitted。
BottomPaneView trait(view 栈接口)
所有能盖在 composer 之上的弹窗/模态都实现这个 trait:审批overlay、用户输入请求 overlay、MCP elicitation、列表选择等。trait 定义了键盘处理、完成语义、Ctrl+C 路由、paste-burst、定时 tick、以及"是否能被新请求消费"的钩子。
RequestUserInputOverlay(用户输入请求模态)
agent 主动问用户多选题/自由文本时弹出。复用 ChatComposer 当 notes 输入框,内部一个多题问答状态机:每题可选+可写备注,提交后跳下一题,最后一题提交全部答案发回 app。
TranscriptState(对话流状态)
ChatWidget 里管"当前活跃 cell(正在流式输出的那段)+ 历史 markdown 拷贝缓冲 + 计划增量缓冲"的状态体。渲染时 active_cell 被包成 TranscriptAreaRenderable 画出来。
SessionPicker(会话恢复选择器)
独立的 alt-screen 全屏界面,列出 app-server 的历史会话线程。游标分页拉取、本地搜索过滤、Ctrl+T 看 transcript 预览、回车恢复选中会话。

流程一:底部面板的键盘路由与渲染

用户每次按键,都先进入 BottomPane::handle_key_event。这里的核心决策是:现在有没有活动的 view(弹窗)?有,把键交给最顶层 view;没有,交给 composer。特殊键(interrupt_turn、Ctrl+C、Esc)有专门的优先级处理。渲染则走 BottomPane::as_renderable,把"状态行 / 待处理预览 / composer"按 flex 纵向堆叠,或当有 view 时直接画 view。

流程三:RequestUserInputOverlay 多题问答状态机

当 agent 调用工具需要用户输入(选择题+自由文本)时,app 层发来 ToolRequestUserInputParams。BottomPane 调 push_user_input_request 把 overlay 压入 view 栈、禁用 composer。overlay 内部是个多题状态机:焦点在 Options(选项列表)和 Notes(备注 composer)之间切换,提交当前题后跳下一题,最后一题或检测到未答完时进入确认子流程,最终把答案 HashMap 发回 app。

流程四:会话恢复选择器(resume_picker)

辅助:BottomPaneView trait(view 栈的统一接口)

所有能盖在 composer 上的弹窗都实现 BottomPaneView。这个 trait 是 view 栈能堆叠、能被新请求消费、能定时 tick 的关键。它把"完成语义(Accepted/Cancelled)"、"Ctrl+C 优先消费"、"paste-burst 协同"、"app-server 请求撤回"这些跨弹窗的公共行为统一了。BottomPane 只需要在 handle_key_event / push_* / dismiss_app_server_request 里调对应 trait 方法,不用关心具体 view 类型。

  • handle_key_event / on_ctrl_c / handle_paste:输入路由
  • is_complete + completion:完成语义,决定 pop 时是否级联 dismiss 子 view
  • try_consume_approval_request / try_consume_user_input_request / try_consume_mcp_server_elicitation_request:让已在场的同类 view 直接吞掉新请求,而不是再叠一层
  • dismiss_app_server_request:别的客户端已答了同一个请求,本地 view 同步撤回
  • pre_draw_tick + next_frame_delay:时间驱动 UI(倒计时、动画)的 tick
  • terminal_title_requires_action:告诉终端标题栏显示"需要操作"而不是普通 spinner

辅助:审批延迟(approval_prompt_typing_idle_delay)

当 agent 请求审批(执行命令/权限)时,如果用户正在打字,BottomPane 不会立刻弹审批框打断输入,而是把请求塞进 delayed_approval_requests 队列,等用户停止输入 1 秒后再提升成模态。每次可见字符键都会刷新 last_composer_activity_at,从而重置延迟。pre_draw_tick 里调 maybe_show_delayed_approval_requests_at 检查是否到点。

辅助:streaming 与 transcript 的衔接

渲染层的"上面那块"(transcript)由 ChatWidget 的 streaming 模块喂数据。agent 消息 delta 进 handle_streaming_delta,累积在 stream_controller;完成时 flush_answer_stream_with_separator 把流式 cell 合并成单个 AgentMarkdownCell,并发 AppEvent::ConsolidateAgentMessage 让历史可重渲染。TranscriptState 维护 active_cell_revision(缓存失效计数)、agent_turn_markdowns(可拷贝的逐轮 markdown)、计划增量缓冲等。

辅助:TextArea 的编辑能力

textarea.rs 是个自带 Vim 模式的多行编辑器。核心字段:raw text + cursor_pos + wrap_cache(软换行缓存)+ elements(用户文本元素,如 mention 标记的字节范围)+ kill_buffer(剪切板,区分字符级/行级)。输入分两条路:Emacs 风格(input_with_keymap,绑 EditorKeymap)和 Vim 风格(handle_vim_input → normal/insert/operator/text-object)。set_text_inner 是全量替换文本的入口,会重建 elements、clamp cursor、清 wrap_cache,但保留 kill_buffer(因为提交/清空后用户还指望 Ctrl+Y 找回)。

Notes(补充)

  • 分层路由:BottomPane 只管"键给谁",进程级决策(quit/interrupt 的最终语义)留给 ChatWidget。Ctrl+C 在 view 里先消费(关弹窗),再给 composer 的 history-search 取消,再落到清空 draft,最后才轮到 ChatWidget 当退出/中断。这个优先级在 on_ctrl_c 和 handle_key_event 里很关键。
  • composer 复用:RequestUserInputOverlay 不自己造输入框,而是 new 一个 ChatComposer(plain_text 配置,关掉 popup/slash/图片),这样 notes 输入的样式、粘贴行为、keymap 都和主输入框一致。
  • paste-burst:大段粘贴会被切成突发帧渐进 flush,期间 BottomPane 安排短延迟重绘给 flush 留窗口。view trait 也有 is_in_paste_burst / flush_paste_burst_if_due 让模态里的 composer 一起参与。
  • auto-resolution:RequestUserInputOverlay 支持 agent 设的 autoResolutionMs——60s 隐藏宽限 + 60s 可见倒计时,到点自动提交空答案。任何按键/粘贴都会 snooze。
  • 渲染缓存:TranscriptAreaRenderable 用 active_cell_revision 做缓存失效;BottomPane 的 set_* 方法都走"先比较,变了才 request_redraw"的模式,避免线程切换时无谓重画。
  • alt-screen 隔离:resume_picker 用 AltScreenGuard 进 alternate screen,Drop 时切回,保证主聊天界面不被 picker 的全屏列表污染。
