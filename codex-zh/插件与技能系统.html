插件与技能系统

这是 Codex 的四大扩展机制——插件(plugins)、技能(skills)、工具(tools)、钩子(hooks)——的数据流分析。这四套机制让 Codex从一个静态 CLI 变成一个可被用户、市场、远端后端持续注入新能力的运行时。入口在 codex-rs/core-plugins/src/lib.rs,它把插件管理器(PluginsManager)暴露给上层,而插件管理器又是连接技能、工具、钩子的枢纽:一个插件被加载后,会同时贡献技能根目录、MCP 服务器配置、App 连接器声明、钩子事件源,这些产物分别流向 skills/tools/hooks 三个子系统。

整体一句话
把"用户配置 + 市场清单 + 远端目录"这三路输入,变成"已加载的技能/MCP/App/钩子"四路输出,供模型和执行器消费。
Input(业务含义)
用户在 config.toml 里写的 [plugins] 段、~/.codex/plugins/cache 里已缓存的插件目录、远端后端返回的已装插件列表、市场 marketplace.json 里登记的插件源。
Output(业务含义)
PluginLoadOutcome——一组 LoadedPlugin,每个携带技能根、MCP 服务器表、App 声明、钩子源;以及给模型看的 PluginCapabilitySummary 摘要。
Notes
插件 ID 格式恒为 <plugin_name>@<marketplace_name>,缓存目录按 plugins/cache/<marketplace>/<plugin>/<version> 布局。OpenAI 官方精选市场(openai-curated / openai-api-curated)走特殊的 git 同步 + sha 截断版本号路径;非精选市场走 semver 比较。

四大扩展机制各管什么

  • 插件(plugins):一个插件是一个目录,内有 .codex-plugin/plugin.json 清单。清单声明这个插件提供哪些技能(目录)、哪些 MCP 服务器、哪些 App 连接器、哪些钩子。插件是"能力集装箱",自身不执行,只是把里面装的东西注册给运行时。核心角色:PluginsManager(编排)、PluginStore(缓存)、loader(加载)、marketplace(市场发现)。
  • 技能(skills):技能是 Markdown 文档(SKILL.md)+ 可选脚本,本质是"给模型看的操作手册"。系统技能随二进制嵌入(skills crate 用 include_dir!),首次启动时解压到 $CODEX_HOME/skills/.system;用户技能来自配置层、插件技能根、额外根。核心角色:SkillsManager、loader::load_skills_from_roots。
  • 工具(tools):这是 Responses API 的工具原语层,定义模型可调用的函数工具如何序列化成 JSON、如何执行。ToolSpec 是给 API 的工具描述,ToolExecutor trait 是运行时契约。还有 tool_discovery 让模型按需发现可装的插件/连接器,request_plugin_install 让模型主动请求装插件。
  • 钩子(hooks):事件钩子引擎,在工具调用前后、会话开始/结束、压缩前后等 10 个事件点上跑用户定义的 shell 命令。钩子来自配置层(hooks/hooks.json 或 toml)和插件内联声明。ClaudeHooksEngine 是核心,discovery 找处理器,dispatcher 选并执行。

钩子引擎数据流

钩子处理器来源有三:托管需求(managed requirements,最低优先级先跑)、配置层(hooks.json 或 toml)、插件内联钩子声明。discover_handlers 把三路汇成一个有序 ConfiguredHandler 列表。事件触发时,select_handlers 按事件名 + matcher 正则筛选,然后 execute_handlers 并发跑命令,按完成顺序收集结果再按配置顺序重排。

通信边界标注

  • 跨 crate 边界:core-plugins 通过 LoadedPlugin<McpServerConfig> 类型别名依赖 codex-plugin 的泛型 LoadedPlugin<M>,MCP 配置类型由 codex-config 提供。这是核心解耦点:插件模型不绑死 MCP 配置具体类型。
  • 插件 → 技能:PluginLoadOutcome::effective_plugin_skill_roots() 把插件的技能根打包成 PluginSkillRoot(带 plugin_id 和 plugin_root),喂给 SkillsLoadInput。
  • 插件 → 钩子:load_plugin_hooks 把插件清单里声明的钩子文件解析成 PluginHookSource,喂给 ClaudeHooksEngine::new。
  • 插件 → MCP:effective_mcp_servers() 合并所有活跃插件的 MCP 服务器配置,同名先到先得。
  • 插件 → 模型:capability_summaries() 生成给模型看的"这个插件有什么技能/MCP/App"摘要,描述长度截断 1024 字符。
  • 远端后端:remote.rs 通过 chatgpt_base_url 调用后端 /installed、/catalog 等端点,需要 CodexAuth 认证;未认证时只返回空列表不报错。

关键发现

  • 缓存与失效:PluginsManager 用 LoadedPluginsCache(generation + key 双重检查)避免重复加载,任何远端缓存变更、市场升级、安装/卸载都会 clear_loaded_plugins_cache 让 generation 自增,迫使下次重新加载。技能侧也有双缓存:按 cwd 和按 config-key,防止跨会话/跨角色污染。
  • 版本选择策略:local 版本总是优先于任何 semver 版本——这是开发态便利,但也意味着如果开发者在缓存目录留了 local 子目录,正式版本永远不会被选中。
  • 原子安装:replace_plugin_root_atomically 用 staged 目录 + rename 实现原子替换,失败时尝试回滚到 backup,回滚也失败时保留 backup 路径并报错——这是双重故障下的最后防线。
  • 插件清单路径安全:resolve_manifest_path 强制所有相对路径必须以 ./ 开头、不含 ..、只走 Normal 组件——防止插件清单逃逸自身根目录。
  • 钩子去重:select_handlers 对一个工具调用只跑一次,即使 matcher 正则匹配了多个别名(如 apply_patch|Write|Edit),避免同一钩子重复执行。
  • 系统技能指纹:用 DefaultHasher 对嵌入目录的所有文件路径+内容做哈希,加 v1 盐,写进 marker 文件。指纹匹配则跳过解压——启动加速。
