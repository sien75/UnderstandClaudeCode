非交互执行任务

用户运行 codex exec <prompt> 时，命令行参数先被主 CLI 识别为非交互子命令，然后整体交给 exec crate。exec crate 负责把提示词、图片、结构化输出 schema、工作目录、模型、权限和沙箱配置整理成一次 in-process app-server 会话，再发起 turn/start。用户运行 codex review 时，主 CLI 会构造一个等价的 exec CLI，把 review 参数放进 exec 的 Review 子命令，随后走同一条非交互执行管道，只是提交的是 review/start。

输入
命令行参数、可选 stdin 提示词、图片路径、JSON Schema 文件、配置覆盖项、工作目录和权限/沙箱选项；review 场景额外输入 uncommitted/base/commit/custom prompt 之一。
输出
默认模式下，过程日志、工具调用、diff、计划和 token 用量写到 stderr，最终消息在适合自动化消费时写到 stdout；--json 模式下，stdout 是逐行 JSON 事件。两种模式都可以把最后一条 agent 消息写入 --output-last-message 指定的文件。

这一段的关键状态边界是“交互 CLI”和“非交互 exec”之间的切换。codex review 没有单独实现一套执行引擎，而是被翻译成 exec 的 Review 命令，避免 review 和 exec 在配置、认证、沙箱、输出循环上分叉。权限边界也在这里确定：headless 默认 approval_policy 为 never，--full-auto 兼容为 workspace-write，--dangerously-bypass-approvals-and-sandbox 会切到 danger-full-access。

session 的创建和恢复都通过 app-server 协议完成。新任务使用 thread/start；resume 先通过 thread/list、状态库和 rollout 元数据解析 thread，再使用 thread/resume；如果 resume 没有目标，就退回创建新 thread。SessionConfiguredEvent 不再等待后续流式事件，而是直接由 start/resume 响应构造，减少非交互启动延迟。

非交互模式不会向用户发起审批或输入请求。来自 app-server 的 MCP elicitation 会自动取消；命令执行审批、文件修改审批、request_user_input、动态工具、认证刷新、attestation 和权限审批都会被拒绝。这是 Permission/Sandbox 边界的一部分：任务可以按已有配置运行，但不能在运行中要求人工确认。

默认输出模式的 stdout 被严格保留给最终消息，其他过程信息写 stderr，这让脚本可以直接捕获回答正文。JSON 模式则完全不同：stdout 是事件流，事件包括 thread.started、turn.started、item.started、item.updated、item.completed、turn.completed、turn.failed 和 error；最后消息本身不单独作为 stdout 尾行重复输出，只在 item.completed 的 agent_message 中出现，或写入 --output-last-message 文件。

  • State Machine：一次任务从配置初始化进入 session start/resume，再进入 submitted/running，收到 completed/failed/interrupted 后请求 unsubscribe 并关闭 client。
  • Communication：exec crate 与 agent 运行时通过 InProcessAppServerClient 交换 JSON-RPC 风格请求和通知，关键方法是 thread/start、thread/resume、turn/start、review/start、thread/read、turn/interrupt、thread/unsubscribe。
  • Permission/Sandbox boundaries：exec headless 默认不询问审批；运行时审批请求被拒绝；sandbox/permission profile 在 thread/start 或 thread/resume 参数中传入 app-server。
  • Lifecycle：app-server client 的生命周期由 exec session 包住，启动后提交任务，任务终态后取消订阅并 shutdown；如果服务端报告不可重试错误或任务失败/中断，进程以非零状态退出。
