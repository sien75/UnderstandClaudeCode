基础能力 crate：把配置与请求变成可执行、受约束的 Codex 行为

这些独立 crate 不直接拥有一次对话，却共同支撑每次对话：它们先把默认值、用户配置和托管配置折叠为稳定的功能集合，再据此选择工具与执行后端；命令进入策略匹配和权限收敛后，才会被包装进平台沙箱。旁路能力负责云端策略缓存、Git 工作区识别、技能安装、密钥保护、反馈采集、提示词组装和终端识别。

输入
本地与云端配置、模型工具调用、命令和工作目录、调用者授予的附加权限
输出
一组确定的功能开关、模型可见工具、策略判定，以及可以安全交给操作系统执行的沙箱请求
graph TD
  config-in["读取默认值、基础配置、配置档与临时覆盖"] --> resolve-features["按优先级合并功能开关"]
  resolve-features --> normalize-features{"是否有依赖功能需要补齐？"}
  normalize-features -->|是| enable-dependency["补齐协作或代码模式依赖"]
  normalize-features -->|否| choose-tools["直接选择模型工具"]
  enable-dependency --> choose-tools
  choose-tools --> shell-enabled{"Shell 与统一执行是否启用？"}
  shell-enabled -->|否| classic-shell["使用经典后端或禁用 Shell"]
  shell-enabled -->|是| composition-enabled{"是否允许统一执行与 zsh 组合？"}
  composition-enabled -->|是| zsh-unified["选择统一 PTY + zsh 桥接"]
  composition-enabled -->|否| direct-unified["选择直接统一 PTY，或安全回退"]

功能注册表同时记录生命周期：稳定能力可默认开启，实验能力进入菜单，开发中能力保持关闭，已移除键只负责兼容旧配置。这样上层不必在各处重复解释旧键，也不会由一个组合开关意外开启其依赖能力。

graph TD
  command-in["收到命令与可选策略文件"] --> parse-policy["解析 Starlark 策略并校验示例"]
  parse-policy --> policy-valid{"策略是否合法？"}
  policy-valid -->|否| reject-policy["返回带文件与行号的错误"]
  policy-valid -->|是| exact-match["按首个程序名匹配前缀规则"]
  exact-match --> found-rule{"找到明确规则？"}
  found-rule -->|是| rule-decision["汇总 allow、prompt 或 deny"]
  found-rule -->|否| host-match["尝试解析主机可执行文件别名"]
  host-match --> host-found{"主机规则命中？"}
  host-found -->|是| rule-decision
  host-found -->|否| heuristic["使用调用方启发式兜底"]
  rule-decision --> normalize-permissions["规范化临时网络与文件权限"]
  heuristic --> normalize-permissions
  normalize-permissions --> permission-valid{"权限路径与 glob 是否有效？"}
  permission-valid -->|否| reject-permission["拒绝不安全的权限扩张"]
  permission-valid -->|是| intersect-permissions["只保留请求与实际授权的交集"]
  intersect-permissions --> choose-sandbox{"平台沙箱是否必须？"}
  choose-sandbox -->|否| unsandboxed["保留原命令执行形态"]
  choose-sandbox -->|是| platform-sandbox["选择 Seatbelt、bubblewrap/Landlock 或 Windows 沙箱"]
  platform-sandbox --> transform-request["注入策略、工作目录、代理与证书后生成执行请求"]

权限链遵循“不能越授越多”：附加权限先去重并规范化路径，读权限的 glob 只允许表达拒绝；随后授权结果与原请求求交集，保留仍有约束作用的 deny 项。沙箱管理器还会依据平台、网络要求和显式偏好决定是否必须启用原生沙箱。

  • 云配置的启动快照不会被后台刷新中途替换，避免同一次运行的策略语义漂移。
  • 技能安装使用嵌入内容指纹与 marker，内容未变化时无需重复写入。
  • Git、文件搜索、反馈、代理和安装上下文 crate 都保持窄 API，让 core/app-server 只编排，不复制平台细节。
  • execpolicy-legacy 仅维持旧策略格式兼容；新命令策略由 execpolicy 的解析、规则和判定模型承接。
