状态与记忆

这一组 crate 负责 Codex 在本地进程里所有"持久化状态"的读写:会话线程的元数据、生成的记忆、全局消息历史、以及 agent 之间父子派生的拓扑图。你可以把它理解成 Codex CLI 跑完一次对话后,"留下来的、下次还能用上的东西"全部归这里管。技术上是 4 个 SQLite 数据库 + 一批 JSONL 文件 + 一个 git-baseline 工作区,用 Rust async + sqlx 实现。

涉及 5 个 crate:

  • codex-rs/state — 核心状态运行时 StateRuntime,持有 4 个 SQLite 连接池(state/logs/goals/memories),提供线程元数据、目标(goal)、记忆任务、回填(backfill)、日志落库等全部读写 API。
  • codex-rs/memories — 记忆读路径(read 子 crate:注入、引用解析、用量统计)与写路径(write 子 crate:启动时两阶段记忆提取流水线 phase1/phase2)。
  • codex-rs/message-history — 全局消息历史文件 history.jsonl 的追加与裁剪,跨进程并发安全。
  • codex-rs/thread-store — 存储中立的线程持久化抽象 ThreadStore trait + 本地文件/SQLite 实现 LocalThreadStore,管 rollout JSONL 写入与线程元数据同步。
  • codex-rs/agent-graph-store — 线程派生父子拓扑的存储中立 trait + 本地实现,底层直接复用 StateRuntime 的 spawn edge 表。

用户/系统能做什么

  • 启动一次新会话 → state 写 threads 表 + thread-store 创建 rollout JSONL + message-history 追加用户输入
  • 会话进行中持续追加输出 → thread-store append_items 写 rollout,state.touch_thread_updated_at 更新时间戳
  • 会话结束/恢复 → thread-store load_history 读回 rollout,state 读取线程元数据
  • 启动时自动跑记忆流水线 → memories/write phase1 从旧 rollout 提取 raw memory,phase2 合并成 MEMORY.md
  • 列出/搜索历史线程 → state.list_threads / thread-store.search_threads
  • 归档/删除/取消归档线程 → thread-store 对应操作 + state 同步
  • agent 派生子 agent → agent-graph-store 记录父子边,state.upsert_thread_spawn_edge
  • 记录用量并裁剪记忆 → state.memories.record_stage1_output_usage / prune_stage1_outputs_for_retention
  • tracing 日志落库 → state.log_db 作为 tracing Layer 后台批量写 logs DB

关键流程一:StateRuntime 初始化(打开 4 个 SQLite 库)

StateRuntime::init 是所有上层状态访问的入口。它按固定顺序打开 state、logs、goals、memories 四个 SQLite 库,任一失败都会回滚已打开的连接池。每个库都走相同的三步:create_if_missing + WAL 模式 → 连接池(max 5 连接)→ 跑迁移。打开后还会:确保 backfill_state 有一行初始记录、读取 threads 表的最大 updated_at/recency_at 时间戳缓存到 AtomicI64(后续热路径直接内存读)、跑一次 logs 库的启动维护(裁剪超限日志)。

    A2 -->|成功| A4[打开 logs DB]
    A4 -->|失败| A3

通信边界:四个库各自独立连接池,设计上把 logs 单独分库是为了降低与 state 主库的写锁竞争(WAL + busy_timeout 5s)。恢复路径(recovery.rs)负责识别 SQLite 损坏错误,支持备份后重建。

关键流程二:记忆两阶段流水线(phase1 + phase2)

这是整个场景最复杂的数据流。start_memories_startup_task 在根会话启动时被调用,跳过临时会话/未开 MemoryTool feature/子 agent 会话。通过后 spawn 一个 tokio 任务,顺序跑 phase1 和 phase2。

Phase 1(提取):从 state.memories 里 claim 过期线程的 stage-1 任务(基于 updated_at 时间窗口 + memory_mode=enabled + 排除当前线程),拿到任务后并发(最多 8 个)调用模型把 rollout 历史压成一段 raw_memory + rollout_summary,成功后写回 stage1_outputs 表。任务调度用 SQLite 的 INSERT...ON CONFLICT 做乐观锁 claim,带 lease/重试/退避,避免多进程重复跑。

Phase 2(合并):先 claim 全局唯一锁(memory_consolidate_global job),再 prepare_memory_workspace(给 memories 目录建 git baseline),从 DB 取 phase2 输入选集,重建 raw_memories.md 和 rollout_summaries/,生成工作区 diff 文件,然后 spawn 一个独立的合并 agent(走 ThreadManager 起新线程),让模型读 diff + raw memories 后编辑 MEMORY.md。agent 完成后 reset baseline、标记 job 成功。

状态机:stage1 job 有 running/done/failed 三态,带 lease_until/retry_at/retry_remaining 字段控制重试。phase2 是全局单例 job(memory_consolidate_global,job_key='global'),成功后有 6 小时冷却(PHASE2_SUCCESS_COOLDOWN_SECONDS)。memory_mode 有 enabled/polluted 两态,polluted 表示线程被标记需要重新提取。

关键流程三:线程持久化(thread-store + state 协作)

ThreadStore trait 是存储中立的线程持久化边界,定义了 create/resume/append/persist/flush/shutdown/discard/load_history/read/list/search/update_metadata/archive/unarchive/delete 等操作。LocalThreadStore 是本地实现:rollout JSONL 文件做 durable replay 格式(无 SQLite 也能读),SQLite state DB 做可查询的元数据索引。

LiveThread 是活跃线程的句柄,持有 Arc + ThreadMetadataSync。ThreadMetadataSync 从 canonical rollout items 里派生元数据更新(cwd、preview、first_user_message、title、git info),按节流间隔(5s)批量回写 state。LiveThreadInitGuard 用于会话初始化还未完成时的回滚:drop 时 discard 掉 live writer 而不强制刷盘。

graph TD
    C1[LiveThread.create] --> C2[thread-store create_thread]
    C2 --> C3[建 rollout JSONL 文件]
    C2 --> C4[ThreadMetadataSync.for_create 初始元数据]
    C4 --> C5[state upsert_thread 写 threads 表]
    C6[会话追加输出] --> C7[thread-store append_items]
    C7 --> C8[RolloutRecorder 写 JSONL]
    C7 --> C9[ThreadMetadataSync 观察 rollout item]
    C9 --> C10{节流间隔到?}
    C10 -->|否| C11[暂存 patch]
    C10 -->|是| C12[state update_thread_metadata]
    C13[会话结束] --> C14[thread-store shutdown_thread]
    C14 --> C15[flush 残留 + 关 writer]
    C16[恢复会话] --> C17[thread-store load_history]
    C17 --> C18[读 rollout JSONL 还原历史]

通信边界:append_items 只管写 rollout,不做元数据推断;元数据补丁由上层观察后通过 update_thread_metadata 显式下发。这样存储实现保持简单,策略留在 live 层。

关键流程四:消息历史文件(history.jsonl)

message-history 是一个独立的、全局的、追加式 JSONL 文件(~/.codex/history.jsonl),每行一条 {session_id, ts, text}。append_entry 用 O_APPEND + 单次 write 系统调用保证原子性,加 advisory file lock + 重试(10 次,100ms 间隔)避免多 TUI 进程交叉写。写完后在锁内 enforce_history_limit:超过 max_bytes 就从头丢旧行,裁到 soft cap(80%)避免下次立刻又裁。

graph TD
    D1[append_entry] --> D2{persistence=None?}
    D2 -->|是| D3[直接返回]
    D2 -->|否| D4[构造 JSON 行 + \n]
    D4 --> D5[O_APPEND 打开文件 0o600]
    D5 --> D6[spawn_blocking 拿锁]
    D6 --> D7{try_lock 成功?}
    D7 -->|WouldBlock| D8{重试 < 10?}
    D8 -->|是| D9[sleep 100ms 回 D7]
    D8 -->|否| D10[报错 WouldBlock]
    D7 -->|成功| D11[seek End + write_all]
    D11 --> D12[enforce_history_limit 裁剪]
    D12 --> D13[返回 Ok]

关键流程五:agent 派生拓扑(agent-graph-store)

AgentGraphStore trait 定义了线程派生的父子边操作:upsert 边、更新状态、列直接子节点、BFS 列所有后代。LocalAgentGraphStore 直接委托给 StateRuntime 的同名方法,底层就是 state DB 的 threads_spawn_edges 表。每个子线程最多一条入边;list_descendants 按深度优先再按 thread id 排序,保证稳定输出,方便上层合并 live 内存态与持久态。

关键流程六:tracing 日志落库(log_db Layer)

log_db 模块是一个 tracing_subscriber::Layer,捕获 tracing 事件格式化成 LogEntry,经 bounded mpsc 队列(容量 512)送到后台任务,批量(128 条/批或 2s 间隔)INSERT 到 logs DB。这样日志路径不阻塞业务热点。state 还会按 thread_id 分桶(每桶 10MiB / 1000 行上限)做日志裁剪,启动时跑一次维护。

模块清单

state (StateRuntime)
把 4 个 SQLite 库 + 回填 + 线程元数据 + 目标 + 记忆任务调度,统一成一个运行时入口。Input: codex_home 路径 + provider;Output: Arc<StateRuntime>,可查可写所有持久状态。
state.log_db
把 tracing 事件流变成 logs DB 里的批量行。Input: tracing Event;Output: logs 表 INSERT。
state.runtime.memories (MemoryStore)
把记忆提取/合并任务调度变成 jobs 表 + stage1_outputs 表的增删改。Input: 线程 updated_at 时间戳;Output: 任务 claim/成功/失败状态机。
memories/write
把旧 rollout 历史变成 MEMORY.md 文件。Input: rollout JSONL + 模型推理;Output: memories/ 目录下的 raw_memories.md、rollout_summaries/、MEMORY.md。
memories/read
把 agent 读取记忆文件的行为变成用量统计 + 引用解析。Input: bash 命令字符串 / 模型输出 citation;Output: MemoriesUsageKind 标签 + MemoryCitation 结构。
thread-store
把会话生命周期(create/append/flush/shutdown/resume)变成 rollout JSONL 文件 + threads 表元数据。Input: CreateThreadParams/RolloutItem;Output: 持久化 rollout + StoredThread 元数据。
message-history
把一条用户输入文本变成 history.jsonl 的一行。Input: text + conversation_id;Output: 追加到全局 JSONL 文件。
agent-graph-store
把 agent 派生关系变成 threads_spawn_edges 表的父子边。Input: parent/child ThreadId + status;Output: 持久化的拓扑边。

Notes

  • 四个 SQLite 库文件名固定:state_5.sqlite / logs_2.sqlite / goals_1.sqlite / memories_1.sqlite,放在 codex_home 下。可用 CODEX_SQLITE_HOME 环境变量覆盖 home 目录。
  • 所有库用 WAL + synchronous=Normal + busy_timeout=5s + incremental auto-vacuum,兼顾性能与崩溃安全。
  • stage1 任务 claim 用单条 INSERT...ON CONFLICT...WHERE 的 SQL 实现乐观锁,避免应用层加锁,跨进程安全。
  • memory workspace 用 git baseline 仓库做 diff,让 phase2 agent 能看到"删除了哪些记忆"的信号。
  • ThreadMetadataSync 的节流(5s)避免每次 append 都打 SQLite,降低写放大。
  • history.jsonl 用 O_APPEND + 单次 write + advisory lock,POSIX 保证 ≤ PIPE_BUF 字节写入原子,跨进程安全。
  • LocalThreadStore 的 rollout JSONL 是无 SQLite 也能读的兼容层,SQLite 只是加速查询的索引,设计上保留了 SQLite-less 修复能力。
