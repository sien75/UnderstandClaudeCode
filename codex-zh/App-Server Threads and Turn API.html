应用服务器线程与回合 API

外部客户端通过 app-server transport 发送 JSON-RPC。这个场景把 `thread/start` 的建线程请求、`turn/start` 的发起回合请求、订阅后的事件通知、以及 `thread/read` 的线程读取串成一条数据流：客户端输入先被协议层反序列化成强类型请求，再由 message processor 分派到线程或回合处理器，最终把线程摘要、回合状态和通知写回原连接或订阅了该线程的连接。

Input
外部客户端发送的 JSON-RPC request：`thread/start` 带模型、工作目录、权限/沙箱、指令、动态工具等线程启动参数；`turn/start` 带 threadId、用户输入、上下文和本回合设置；`thread/read` 带 threadId 和是否包含 turns。
Output
JSON-RPC response 和 server notification：`thread/start` 返回新线程与配置快照并广播 `thread/started`；`turn/start` 返回 InProgress turn，后续 `turn/started`、`turn/completed` 等通知只发给订阅该线程的连接；`thread/read` 返回由持久化线程数据和实时线程状态合成的 Thread 视图。

  • 通信边界：transport 只负责连接、读写队列和 JSON-RPC message 分类；业务分派发生在 `MessageProcessor`，真正的线程/回合语义在 request processors 中完成。
  • 生命周期：`thread/start` 创建线程后立即为发起连接建立 listener。listener 负责持续读取 core 线程事件，因此客户端订阅不是单独的 “subscribe” 方法，而是通过 start/resume/fork 等线程入口自动挂接，`thread/unsubscribe` 再移除连接。
  • 权限/沙箱边界：线程启动时拒绝同时传 `permissions` 与旧 `sandbox`；回合启动时也会把本回合权限、沙箱、工作目录、环境选择转成线程设置覆盖，交给 core 线程执行。
  • 状态机：回合在 API 响应中先以 `InProgress` 返回；之后 listener 根据 core 事件把状态推进到 `InProgress`、`Completed`、`Failed` 或 `Interrupted` 并发通知。
  • 读取路径：`thread/read` 优先复用持久化 ThreadStore；如果线程已加载但持久化还没落盘，则从 live thread snapshot 补齐。请求 turns 时会重放 rollout 历史，ephemeral 线程不支持读取完整 turns。
