代码模式(code-mode)

代码模式是 codex 给模型提供的一个"会写代码的子大脑"。普通聊天里,模型只能一次调一个工具(shell、读文件、MCP…),拿到结果再想下一步。代码模式下,模型改调一个叫 exec 的工具,把一段 JavaScript 源码丢进去;这段 JS 在一个独立的 V8 沙箱里跑,沙箱里全局挂着 tools 对象——里面就是模型本来能用的那些工具。于是模型可以在 JS 里循环、判断、拼装、并行调多个工具、存中间变量,像写脚本一样把活儿干完,再把累积的文本/图片输出吐回给模型。

从用户视角看:用户在 IDE/编辑器里用 codex 时,代码模式并不需要用户额外点按钮。它对用户透明——是给模型的一种"增强工具调用"能力。当模型判断"这事儿得组合好几个工具、有逻辑分支"时,它就调 exec 写段 JS;脚本可能很快跑完(一次性返回结果),也可能跑得久——这时 exec 会先"让出(yield)"一段中间输出,模型接着调 wait 工具继续拉取后续输出,直到脚本结束。用户看到的就是模型在干活、在汇报,只是背后是 JS 在编排。

这套能力由三个 crate 协作实现(本模块只讲这三个 crate 内部的数据流):

  • code-mode-protocol(协议层):定义跨边界的数据类型和 trait——ExecuteRequest/WaitRequest、RuntimeResponse、CodeModeSession/CodeModeSessionProvider/CodeModeSessionDelegate trait,以及工具描述生成(build_exec_tool_description 把 enabled_tools 渲染成给模型看的 JS 声明)。纯数据/契约,无运行时逻辑。
  • code-mode(核心实现):CodeModeService 实现 CodeModeSession,管 cell(一次 exec 调用 = 一个 cell)生命周期;内部 CellActor 是个 tokio 异步 actor,负责"V8 线程"和"宿主回调"之间的编排;runtime 子模块在独立 OS 线程上跑 V8 isolate,安装全局函数、执行 JS 模块、把 JS 里的工具调用通过 channel 传回 actor。
  • code-mode-host(宿主二进制):目前只有一个空 main.rs(fn main(){})。是为未来"远程宿主进程"预留的占位——协议里 CodeModeSessionProvider 的注释说实现可以跨进程共享一个远程 host。当前实际使用的 provider 是 InProcessCodeModeSessionProvider(在 code-mode crate 里),即 cell 就在 codex 本进程内跑。

边界说明(这些文件不在本模块 scope,但数据流要经过):codex 主进程(core crate)的 tools/code_mode/execute_handler.rs 是 exec 工具的入口,它调 parse_exec_source 解析源码、构造 ExecuteRequest、调 CodeModeSession::execute;tools/code_mode/wait_handler.rs 处理 wait 工具;tools/code_mode/delegate.rs 里的 CodeModeDispatchBroker 实现 CodeModeSessionDelegate,把 code-mode 运行时发来的嵌套工具调用(invoke_tool)派发给真实的 codex 工具(shell、MCP、读文件等)执行。也就是说:JS 沙箱里的 tools.foo() → 运行时 RuntimeEvent::ToolCall → cell actor → ServiceCellHost::invoke_tool → delegate.invoke_tool → core 的真实工具 → 结果原路返回 → resolve_tool_response 把结果塞回 JS 的 Promise。这就是代码模式最核心的通信边界。

模块拆解(一句话)

code-mode-protocol / description.rs
把"模型能用的工具表"变成"给模型看的 exec 工具描述 + TS 声明"。输入:enabled_tools 工具定义;输出:拼好的工具描述字符串(含 declare const tools: {...})。还负责 parse_exec_source 解析首行 // @exec: {...} pragma,以及把 JSON Schema 渲染成 TypeScript 类型供模型理解工具入参/出参。
code-mode-protocol / session.rs + runtime.rs + response.rs
定义跨边界的契约类型。输入/输出都是业务语义:ExecuteRequest=一次"跑这段JS"的请求;RuntimeResponse::Yielded/Terminated/Result=运行时让出/被终止/最终结果;CodeModeSession trait=会话能 execute/wait/terminate/shutdown;CodeModeSessionDelegate trait=宿主必须提供 invoke_tool/notify/cell_closed 三个回调。
code-mode / service.rs(CodeModeService)
把外部请求变成 cell 生命周期管理。输入:ExecuteRequest/WaitRequest;输出:StartedCell/WaitOutcome。维护 stored_values(跨 cell 共享的键值存储,JS 里 store/load)、cells 表(cell_id → CellHandle)、shutting_down 标志。ServiceCellHost 是 CellHost trait 实现,把 cell actor 的回调桥接到 CodeModeSessionDelegate。
code-mode / cell_actor(CellActor + run_cell)
一次 exec 的"导演"。输入:CellRequest + host;输出:CellHandle + CellEvent 流。它是个 tokio select 循环,同时盯着四路:命令通道(observe/terminate)、yield 超时、运行时事件(Started/Pending/ContentItem/YieldRequested/ToolCall/Notify/Result)、后台任务(notification/tool JoinSet)。按 ObserveMode 决定何时把累积输出吐给观察者:YieldAfter=超时就吐,PendingFrontier=运行时暂停在 pending 边界时吐。这是整条链的状态机核心。
code-mode / runtime(mod.rs + globals + callbacks + module_loader + timers + value)
在独立 OS 线程里跑 V8 isolate 执行 JS。输入:JS 源码 + 工具表 + stored_values;输出:RuntimeEvent 流。globals 装全局函数(删掉 console/Atomics/WebAssembly 等危险全局);callbacks 是每个全局函数的 V8 回调(tool/text/image/store/load/notify/setTimeout/yield_control/exit);module_loader 编译 ES 模块、检查 Promise 完成状态、resolve 工具 Promise;timers 用独立线程实现 setTimeout;value 做 V8↔JSON 序列化和图片输出归一化(禁止 http(s) 远程图,只收 data: URI)。
code-mode-host / main.rs
占位空 main,为未来远程宿主进程预留。

关键状态机与边界标注

  • cell 生命周期状态机(cell_actor/mod.rs 的 run_cell):一个 cell 依次经历:Running(有 observer 在等)→ 要么 Yielded(让出后等下次 wait)→ ... → Completed/Terminated。observer 同一时刻只能有一个(第二个 observe 直接 Busy);terminate 优先级最高,会 cancel token + terminate_execution 强拆 V8。completed_event 会被 buffer,如果当下没 observer 就存着,等下一个 observe 直接发。
  • 运行时暂停模式(runtime/mod.rs next_runtime_command):PauseUntilResumed 模式下,JS 跑到 pending(没结果也没工具调用挂起)时会发 RuntimeEvent::Pending 并阻塞等控制命令——Continue=放行到下一条 RuntimeCommand,Resume=继续转圈看有没有新事件,Terminate=强退。这是给"模型想看 JS 跑到哪了"的 PendingFrontier 观察模式用的。
  • 通信边界 1:V8 线程 ↔ cell actor(runtime/mod.rs):用 std::mpsc(RuntimeCommand/RuntimeControlCommand 进,V8 线程收)和 tokio mpsc(RuntimeEvent 出,actor 收)。V8 线程是同步阻塞线程,actor 是 tokio task,两边靠 channel 解耦。
  • 通信边界 2:cell actor ↔ CodeModeService(cell_actor/types.rs CellHandle):CellHandle 持 mpsc::UnboundedSender<CellCommand>,observe/terminate/shutdown 都是通过发命令给 actor,actor 处理后通过 oneshot 回送 CellEvent。
  • 通信边界 3:code-mode ↔ core 宿主(service.rs ServiceCellHost + protocol CodeModeSessionDelegate):JS 里每调一次 tools.foo,链路最终走到 delegate.invoke_tool,delegate 在 core 里实现(CodeModeDispatchBroker),把调用派发给真实工具执行器,结果原路返回。这是"沙箱内 JS"和"真实系统能力"之间唯一的桥。
  • 安全边界(runtime/globals.rs + value.rs):删掉 console/Atomics/SharedArrayBuffer/WebAssembly 全局;禁止远程图片 URL(只收 data: URI);动态 import 直接抛 "Unsupported import";store 只接受可 JSON 序列化的值。沙箱无 Node、无 fs、无网络。
