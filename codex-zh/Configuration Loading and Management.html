配置加载与管理

Codex 的配置不是单一文件说了算,而是一套分层叠加(layered stack)体系:从最底层的 MDM 系统策略,到系统级 config.toml,到企业云端下发的策略包,到用户个人配置,再到项目本地配置,最后到命令行临时覆盖——每一层都按优先级往上叠,高优先级覆盖低优先级。这套机制既要保证管理员能强制约束(比如"只允许 read-only 沙箱"),又要给用户和项目留足自定义空间。除此之外还有一条并行的"云端配置"链路:从 ChatGPT 后端拉取企业托管配置包,带本地缓存和后台刷新。

用户视角:配置从哪里来,谁能覆盖谁

一个用户跑 codex 时,配置会按下面这个优先级从低到高叠加(数字来自 ConfigLayerSource::precedence()):

MDM 管理偏好 (precedence 0)
macOS 上由设备管理工具下发的偏好,最低优先级。只读约束。
系统 config.toml (precedence 10)
Unix 上是 /etc/codex/config.toml,Windows 上是 %ProgramData%\OpenAI\Codex\config.toml。管理员级别的本机配置。
企业云端托管配置 (precedence 15)
从 ChatGPT 后端拉取的 enterprise-managed 配置片段,可含 config 和 requirements 两部分。
用户 config.toml (precedence 20/21)
$CODEX_HOME/config.toml(默认 ~/.codex/config.toml)。用户日常配置主战场。若用 --profile <name> 选了 profile-v2,会再叠一层 <name>.config.toml(precedence 21)。
项目 .codex/config.toml (precedence 25)
从 cwd 往上找到项目根,每个含 .codex/ 的目录都算一层。但项目配置受信任检查:未在用户配置里标记 trusted 的项目,其项目层会被 disable(只记录不生效)。项目配置还有黑名单:openai_base_url、model_provider、notify 等敏感字段会被剥掉,防止项目偷改用户凭证去向。
命令行 --config 覆盖 (precedence 30)
-c key=value 或 --config key=value 传入的临时覆盖,只对本次会话生效,用 SessionFlags 层表示。
legacy managed_config.toml (precedence 40/50)
老的 managed_config.toml 机制,现在重新解释成 requirements 约束,但作为 config 层也"尽力而为"地叠在最上面。正在淘汰中。

除了 config 层,还有一条平行的 requirements(约束)链:system requirements.toml + 云端 requirements + legacy managed_config + macOS MDM,这些组合成 ConfigRequirements,它不是被"覆盖"的,而是作为硬性约束在最后施加——比如管理员要求 sandbox 只能是 read-only,那用户配的 workspace-write 就会被拦下来。

命令行的 --model、--sandbox-mode、--approval-policy 等参数走的是另一条路:ConfigOverrides(harness overrides),它们不进 config 层叠加,而是在最终 Config 对象组装时直接覆盖对应字段,且会做互斥校验(比如 sandbox_mode 和 permission_profile 不能同时给)。

关键概念解释

ConfigLayerStack(配置层栈)
所有 config 层按优先级从低到高排好的列表,加上 requirements 约束和 startup 警告。是配置加载的中间产物,可序列化展示给用户"这条配置来自哪一层"。
effective_config(生效配置)
把栈里所有未 disable 的层用 merge_toml_values 逐层叠加后的合并 TOML。合并时还会做 key 别名归一化(比如 codex 老版本的字段名映射)和网络域名归一化。
ConfigRequirements(配置约束)
管理员强制施加的硬约束,不是被覆盖而是被校验:allowed_sandbox_modes、allowed_approval_policies、allowed_approvals_reviewers、network/filesystem 约束、mcp_server 身份约束等。在 load_config_with_layer_stack 里施加,用户配的值若不在 allowed 列表里就会被拒。
ConfigOverrides(harness 覆盖)
命令行参数(--model、--sandbox-mode、--approval-policy、--permission-profile 等)和 harness 内部覆盖(cwd、codex_self_exe 等)。不进 TOML 层叠加,直接在 Config 对象组装时覆盖字段,做互斥校验。
项目信任(TrustLevel)
项目本地配置(.codex/config.toml、hooks、exec policy)默认不生效,必须在用户 config.toml 的 [projects] 里把项目路径标记为 trust = "trusted" 才生效。这是防止 clone 一个恶意仓库后其配置自动执行危险操作的安全机制。
strict_config(严格校验)
开启后用 serde_ignored + serde_path_to_error 检测 TOML 里所有未识别字段,碰到未知 key 直接报错(而非静默忽略)。CLI 覆盖也会做同样校验。
CloudConfigBundleLoader
一个共享 Future 的包装器,让"启动时拉云端配置包"这个耗时操作变成 lazy + 可被多处 await 共享。主流程在 load_config_layers_state 里 .get().await 一次拿结果。

模块一句话

codex-rs/config
把多源 TOML(文件/CLI/云/MDM)变成一个分层叠加好的 ConfigLayerStack + ConfigRequirements——配置的"合并器+校验器"。
codex-rs/cloud-config
把 ChatGPT 后端的企业托管配置包变成带 HMAC 签名缓存的 CloudConfigBundle——配置的"云端运输+缓存层"。
codex-rs/core/src/config
把 ConfigLayerStack 解析+约束+覆盖成运行时 Config 对象,并管理编辑/权限/网络代理等配置消费——配置的"最终装配+消费"。

Input: CLI 参数、环境变量、config.toml 文件(系统/用户/项目)、云端配置包、MDM 偏好。
Output: 一个全局生效的 Config 对象,供 agent 运行时、权限系统、MCP、hooks 等所有子系统消费;外加一个 ConfigLayerStack 用于展示来源和编辑用户配置。

Notes:

  • 项目配置有黑名单(PROJECT_LOCAL_CONFIG_DENYLIST):openai_base_url、model_provider、notify、otel 等敏感字段会被剥掉,防止项目层偷改凭证去向或通知行为。
  • legacy managed_config.toml 同时被解释成 requirements 层(硬约束)和 config 层(尽力覆盖),双轨存在,正在淘汰。
  • 云端配置缓存有 1 小时 TTL、HMAC 签名、按 user_id+account_id 鉴权,损坏时 fail-closed(当作 miss 重新拉)。后台刷新只更新缓存文件,不改变已加载的运行时配置(启动时快照一次)。
  • config lockfile 机制:若用户配了 [debug.config_lockfile],会从 lockfile 路径加载一层替换配置,用于锁定可复现的配置快照。
  • profile-v2 机制:--profile foo 会加载 foo.config.toml 作为用户层的第二层叠加在 base user config 之上,只放 override 即可。若 base config 里还有老式 profile = "foo" 或 [profiles.foo],会直接报错要求迁移。
