代理线程的协议、状态、历史持久化与恢复

外部客户端把一次用户输入、审批答复或线程控制动作提交给 Codex；系统把它变成带关联 ID 的事件流，同时把可恢复的会话条目写入 rollout，把可查询的索引与派生状态写入 SQLite。线程恢复时，系统先重新建立“活线程”句柄，再加载历史并重建元数据；初始化中途失败则主动丢弃半成品，避免留下看似可恢复但内容不完整的线程。

输入
用户输入、线程设置、审批答复、线程回滚/压缩/关闭动作，以及代理之间的通信。
输出
可供客户端消费的关联事件流、可逐行重放的 JSONL 历史、可分页检索的 SQLite 线程索引，以及子代理关系、记忆和追踪等派生状态。

  classify-op -->|新输入或设置| emit-event["产生带关联 ID 的事件"]

  emit-event --> canonical-item["筛出可持久化的规范历史条目"]
  canonical-item --> append-live["活线程追加条目并观察元数据"]
  append-live --> metadata-change{"标题、预览或时间等元数据变化？"}
  metadata-change -->|是| sync-metadata["同步线程元数据"]
  metadata-change -->|否| continue-run["继续当前线程"]
  sync-metadata --> continue-run

协议层把“提交什么”和“发生了什么”分开：Op 是客户端到线程的命令，EventMsg 是线程到客户端的状态与输出。Event.id 将异步事件重新关联到原提交；代理通信虽然走正常提交生命周期，但会先编码成模型历史条目，因此恢复后仍保留代理间上下文。

本地线程存储把 rollout 文件视为兼容且可重放的事实流，把 SQLite 视为快速查询与派生状态层。写入器通过单独任务串行接收追加、持久化、刷盘和关闭命令；尚未物化时可以延迟建文件，失败后保留终止错误供后续 API 返回。列表读取优先使用状态库，缺失或需要修复时回退扫描 rollout，并把发现的元数据补回索引。

  derived-state --> searchable["线程可分页检索、恢复和审计"]
  • 状态机边界：线程从初始化中、活跃、归档到删除；子代理边从 open 变为 closed，后代查询由 SQLite 图关系提供。
  • 恢复边界：rollout 保留原始顺序，SQLite 可由 rollout 回填；这让索引损坏不会等同于历史丢失。
  • 上下文片段通过显式起止标记识别，避免恢复或压缩时把普通用户文本误当成系统注入内容。
  • 记忆写入分阶段提取、整合并保存，读取侧再附带引用与使用信息；它是线程历史的派生结果，不替代 rollout。
  • rollout-trace 把协议事件归约成线程、推理、工具和压缩轨迹，属于诊断视图；写入失败采用尽力而为策略，不阻断主线程。
