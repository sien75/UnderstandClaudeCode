用户从 CLI 发起并完成一次代理回合

用户在命令行输入任务后，Codex 会先解释启动参数并选择交互式界面或一次性执行路径；随后创建或恢复会话，把用户输入送入代理回合，持续接收模型事件、执行获批工具，并把进度与最终结果流式返回终端。本页沿真实实现追踪这条数据流，并标出进程、模型服务、工具与沙箱之间的边界。

输入
用户的命令行参数、文字或图片提示、工作目录，以及模型、审批、沙箱和恢复会话等选项。
输出
终端中持续刷新的代理解释、工具执行进度和最终答复；会话历史与回合状态按配置保存，以便后续恢复。

第一段数据流决定“以什么方式开始”。不带子命令时进入 TUI；使用 exec 时启动无交互客户端；resume 则把会话定位条件合并进 TUI 配置。TUI 还会判断连接嵌入式 app-server、本机 daemon 或显式远端服务。启动或配置失败会在真正提交用户输入前终止。

  resume-thread --> ready-turn

第二段数据流是代理回合本身。app-server 把 API 输入转换成核心输入并投递给线程；核心先注入权限、技能、插件、环境和历史，再向模型服务发起流式请求。模型若请求工具，路由器按并行能力分派；命令工具还要经过审批策略和沙箱权限校验。工具结果会写回上下文并触发下一次模型请求，直到模型给出不再需要后续动作的答复，或发生中断、不可重试错误。

  • 通信边界：CLI 与 app-server 通过进程内通道或远端连接交换 thread/turn 请求；核心与模型供应商通过 Responses 流通信；MCP、动态工具与系统进程属于进一步的外部边界。
  • 状态机：一个回合在“准备上下文—模型采样—工具执行—结果回填”之间循环，只有无需后续动作、用户中断或致命错误才离开循环。
  • 非交互 exec 默认不能弹出审批或收集用户输入；收到这类服务端请求会拒绝。交互 TUI 则能把审批请求呈现给用户。
  • 恢复路径通过 thread/list 与 thread/resume API 定位记录，而不是由 exec 直接读取 rollout 文件。
