终端交互：把用户输入变成持续更新、可审批的对话

用户启动 Codex 后，可以恢复旧任务或开始新任务，在底部编辑器输入文字、图片、提及与斜杠命令；运行期间，模型的回答、推理、计划和工具执行会逐步出现在对话区。遇到命令、文件修改、权限或 MCP 请求时，普通输入会暂时让位给审批视图，用户决定后结果再送回正在运行的任务。

输入
启动参数、要恢复的任务、键盘和粘贴事件、用户消息，以及 app-server 持续发来的任务事件和审批请求。
输出
按时间顺序更新的终端对话、当前运行状态、需要用户处理的审批界面，以及提交给 app-server 的消息、打断和审批决定。

启动与事件汇合：TUI 先完成配置、终端和 app-server 初始化；会话选择器按页加载任务并可异步预览历史。进入聊天后，所有输入源汇合到同一个事件循环，因此绘制、网络通知和用户按键不会各自直接修改终端。

输入数据流：按键先交给当前模态视图；没有模态时才进入编辑器。提交时会检查图片能力、解析 shell 转义和斜杠命令、编码提及并附上当前模型、权限与协作模式。空消息或无法提交的消息留在本地；任务空闲时创建新 turn，任务运行中则作为 steer 排队。

输出与生命周期：app-server 通知先按任务过滤，再根据事件类型更新 turn 状态、工具单元或流控制器。文本增量不会直接逐字符写入滚屏，而是先积累源码、只提交已经稳定的完整行，并保留可变化的尾部；结束时再从完整 Markdown 生成规范历史单元。这样调整终端宽度时，已提交滚屏与活动尾部仍保持顺序。

审批状态机：请求到达时先结束回答的活动尾部，避免审批框插入半段文字。若用户正在输入，审批可短暂延迟；随后请求进入统一队列。用户选择一次允许、会话允许、拒绝或取消后，视图把决定包装成指定任务的操作，经 AppEvent 回到 app-server，再显示队列中的下一项。

  • 边界：TUI 不执行模型推理或工具，它把用户意图和审批决定交给 app-server；核心执行链由 core 模块负责。
  • 状态机重点：空闲、turn 运行、等待审批、被中断、结束并不是独立页面，而是 ChatWidget、BottomPane、流控制器共同维护的正交状态。
  • 恢复选择器对远程工作区不过滤本地路径；分页、搜索 token 与预览加载 token 防止较旧的异步结果覆盖新选择。
  • Ctrl+C 优先取消当前模态或运行任务；在没有可取消工作时才进入双击退出窗口，避免一次误触直接丢失会话。
