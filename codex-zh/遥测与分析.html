遥测与分析

这个模块管的是 Codex 运行时产生的所有"可观测数据"——从一次 turn 的 token 用量、一次工具调用的延迟，到整个会话的回放追踪。它由四个 crate 组成，各管一条数据通道：analytics 把 app-server 协议事件聚合成业务遥测事件发往后端；otel 接入 OpenTelemetry 标准（traces/logs/metrics）通过 OTLP 协议导出；rollout 把会话内容持久化成 JSONL 文件供日后回放；rollout-trace 是更细粒度的 trace bundle，记录推理/工具/代码单元的完整生命周期用于诊断回放。

这个模块管什么

从用户视角看，当 Codex 跑起来后，以下动作都会产生数据：

  • 启动一个会话/线程 → analytics 发 thread_initialized 事件，rollout 写 SessionMeta 首行，otel 记录 conversation_starts
  • 跑一个 turn（向模型发请求）→ analytics 追踪 turn 的 token 用量/延迟/profile，otel 记录 API 请求延迟和 SSE 事件，rollout-trace 记录推理调用生命周期
  • 调用一个工具（shell/apply_patch/MCP）→ analytics 生成工具项事件，otel 记录工具调用计数和延迟，rollout 持久化响应项
  • 发生 compaction（上下文压缩）→ analytics 发 compaction 事件，rollout-trace 记录压缩请求/安装检查点
  • 会话结束 → rollout 文件关闭并 eventual 压缩为 .zst，rollout-trace 写 RolloutEnded

数据最终流向：analytics → HTTP POST 到 codex 后端；otel → OTLP gRPC/HTTP 到远端收集器；rollout → 本地 ~/.codex/sessions/ JSONL 文件 + SQLite 元数据；rollout-trace → 本地 trace bundle 目录（需设 CODEX_ROLLOUT_TRACE_ROOT 环境变量才启用）。

模块详解

analytics — 业务遥测事件采集与上报

把 app-server 协议流量（客户端请求/响应、服务端通知、turn 生命周期、工具调用、guardian 审批等）聚合成结构化事件，HTTP POST 到 codex 后端做产品分析。

Input
app-server 协议消息（ClientRequest/ServerNotification/ServerResponse 等）和 core 产生的自定义事实（TurnProfile/TurnTokenUsage/CodexCompactionEvent 等）
Output
TrackEventsRequest JSON 数组，POST 到 {base_url}/codex/analytics-events/events
核心数据结构
AnalyticsFact（facts.rs:445）—— 所有进入 reducer 的原始事实枚举，分两大类：协议表面事实（Initialize/ClientRequest/ClientResponse/Notification/ServerRequest/...）和自定义事实（CustomAnalyticsFact::SkillInvoked/Compaction/Goal/GuardianReview/TurnProfile/...）。
AnalyticsReducer（reducer.rs:137）—— 状态机聚合器，维护 requests/turns/connections/threads/tool_items/pending_reviews 六张 HashMap，把零散的协议事实拼成完整的 turn 事件和工具项事件。
TrackEventRequest（events.rs:60）—— 最终上报的事件枚举，22 种变体（SkillInvocation/ThreadInitialized/GuardianReview/Compaction/Goal/TurnEvent/TurnSteer/CommandExecution/FileChange/McpToolCall/.../PluginUsed/PluginInstalled/...）。
Notes
事件去重：app_used 和 plugin_used 事件用 (turn_id, connector_id/plugin_id) 集合去重，上限 4096 key 后清空。AcceptedLineFingerprints 事件独占一个 HTTP 请求（should_send_in_isolated_request）。debug 构建可设 CODEX_ANALYTICS_EVENTS_CAPTURE_FILE 把事件写本地文件而不发网络。队列满时 try_send 失败直接丢弃并告警。

otel — OpenTelemetry 标准接入层

封装 OpenTelemetry SDK，为 Codex 提供 traces（分布式追踪）、logs（结构化日志导出）、metrics（指标采集）三路 OTLP 导出能力。SessionTelemetry 在其上封装会话级业务指标。

Input
OtelSettings 配置（exporter 端点/协议/TLS/tracestate/span_attributes）和运行时 tracing 事件
Output
OTLP gRPC 或 HTTP 导出到远端收集器；本地 metrics 全局实例供其他 crate 调用
核心数据结构
OtelProvider（provider.rs:56）—— 持有 SdkTracerProvider + SdkLoggerProvider + MetricsClient，from(OtelSettings) 一次性构建。Drop 时 force_flush + shutdown。
OtelExporter（config.rs:88）—— 导出器配置枚举：None / Statsig（内部 metrics 用，debug 构建降级为 None）/ OtlpGrpc{endpoint,headers,tls} / OtlpHttp{endpoint,headers,protocol,tls}。
MetricsClient（metrics/client.rs:260）—— 封装 OpenTelemetry Meter，提供 counter/histogram/gauge/record_duration/start_timer 接口，内部缓存 InstrumentKey→Counter/Gauge/Histogram 映射避免重复创建。
SessionTelemetry（events/session_telemetry.rs:103）—— 会话级遥测封装，携带 SessionTelemetryMetadata（conversation_id/auth_mode/account_id/model/originator/...），提供 record_api_request/log_sse_event/tool_result_with_tags/record_turn_ttft 等业务方法，同时写 metrics counter/histogram 和 tracing 事件（log_event!/trace_event! 宏分别路由到 codex_otel.log_only 和 codex_otel.trace_safe target）。
Notes
tracestate 支持 W3C Trace Context 传播：可从环境变量 TRACEPARENT/TRACESTATE 注入父 trace context，也可配置 tracestate 条目合并注入。span_attributes 通过 SpanAttributesProcessor 在 on_start 时附加到每个 span。OTLP HTTP 客户端区分多线程 tokio runtime（用 async reqwest + TokioBatchSpanProcessor）和单线程（用 blocking reqwest）。target 路由：codex_otel.log_only → 只导出日志；codex_otel.trace_safe → 可安全导出 trace；其余 codex_otel.* → 两者都导出。

rollout — 会话 JSONL 持久化与发现

把 Codex 会话的每一条协议消息、响应项、执行事件持久化到 ~/.codex/sessions/ 下的 JSONL 文件，支持后续 resume、列表查询、搜索。配合 SQLite 状态库做快速元数据查询。

Input
core 产生的 RolloutItem（ResponseItem / EventMsg / InterAgentCommunication / Compacted / TurnContext / SessionMeta）
Output
~/.codex/sessions/YYYY/MM/DD/rollout-{timestamp}-{thread_id}.jsonl 文件；SQLite 线程元数据库；列表/搜索查询结果
核心数据结构
RolloutRecorder（recorder.rs:75）—— 克隆安全的 recorder，持有 mpsc::Sender<RolloutCmd> 和后台 writer 任务句柄。new() 分 Create（预计算路径+延迟创建文件）和 Resume（立即打开已有文件）两种模式。
RolloutWriterState（recorder.rs:1459）—— 后台 writer 的可变状态：writer(Option<JsonlWriter>) / deferred_log_file_info / pending_items / meta。I/O 失败时 enter_recovery_mode 丢弃文件句柄但保留 pending_items，下次 barrier 重试（write_pending_with_recovery 两次重试）。
policy（policy.rs）—— 决定哪些 RolloutItem 该持久化：ResponseItem 大部分保留（CompactionTrigger/Other 除外）；EventMsg 选择性保留（turn 生命周期/compaction/user message 等保留，delta/begin 类丢弃）。
compression（compression.rs）—— 后台压缩 worker：spawn_rollout_compression_worker 扫描 sessions/ 和 archived_sessions/，7天以上未修改的 .jsonl 压缩为 .jsonl.zst。RolloutLineReader 透明读取 plain 或压缩格式。materialize_rollout_for_append 在追加前把压缩文件解压回 plain。
Notes
list_threads 有文件系统和 SQLite 双路径：优先查 DB，DB 不可用或出错时 fallback 到文件扫描并 record_fallback 记录。搜索（search_term）走文件系统全量扫描 + reconcile 修复 DB。resume 路径匹配 cwd 时三级回退：缓存的 cwd → 最新 TurnContext 的 cwd → 解析 rollout head metadata。

rollout-trace — 诊断级 trace bundle 记录与回放

比 rollout 更细粒度的诊断 trace：记录每次推理调用（请求/响应 payload）、每次工具分派（dispatch/runtime/结果）、每个代码单元（code-mode JavaScript cell）的完整生命周期。原始事件写 trace.jsonl + payloads/ 目录，离线 replay_bundle() 回放出结构化 RolloutTrace 图。

Input
hot-path 各 trace context 调用（ThreadTraceContext / InferenceTraceContext / ToolDispatchTraceContext / CodeCellTraceContext / CompactionTraceContext / McpCallTraceContext）
Output
{trace_root}/trace-{trace_id}-{thread_id}/ 目录：manifest.json + trace.jsonl + payloads/*.json；replay 后的 RolloutTrace（state.json）
核心数据结构
TraceWriter（writer.rs:36）—— 线程安全的 append-only writer，Mutex 保护。create() 建 bundle 目录写 manifest。write_json_payload() 写 payload 文件返回 RawPayloadRef。append_with_context() 写 RawTraceEvent 到 trace.jsonl，自增 seq。
RawTraceEvent（raw_event.rs:33）—— 统一信封：schema_version / seq / wall_time_unix_ms / rollout_id / thread_id / codex_turn_id / payload。RawTraceEventPayload 有 25+ 变体（RolloutStarted/ThreadStarted/CodexTurnStarted/InferenceStarted+Completed+Failed+Cancelled/ToolCallStarted+RuntimeStarted+RuntimeEnded+Ended/CodeCellStarted+InitialResponse+Ended/CompactionRequestStarted+Completed+Failed+Installed/AgentResultObserved/ProtocolEventObserved/Other）。
ThreadTraceContext（thread.rs:76）—— no-op-capable 句柄：Disabled 或 Enabled(EnabledThreadTraceContext{writer,root_thread_id,thread_id})。start_root_or_disabled() 读环境变量决定是否启用。子线程通过 start_child_thread_trace_or_disabled 共享 root bundle 的 writer。
TraceReducer（reducer/mod.rs:88）—— 确定性回放状态机：逐行读 trace.jsonl，apply_event() 按 payload 类型更新 RolloutTrace 图。维护 pending 状态（pending_code_cell_starts / pending_agent_interaction_edges / pending_compaction_replacement_item_ids）处理乱序事件。
RolloutTrace（model/mod.rs:56）—— 最终回放图：threads / codex_turns / conversation_items / inference_calls / code_cells / tool_calls / terminal_sessions / compactions / interaction_edges / raw_payloads。raw_payloads 只存 RawPayloadRef 引用，不内联大 payload。
Notes
所有 trace context 都是 no-op-capable：未设 CODEX_ROLLOUT_TRACE_ROOT 时返回 disabled()，hot-path 调用直接返回不付序列化代价。payload 文件先于引用它的 event 写入（防 replay 断点指向不存在的 payload）。code-cell 的 model-visible call_id 和 runtime cell_id 之间有 bridge 映射（code_cell_ids_by_runtime），处理 core 开始执行工具早于推理响应 payload 到达的乱序情况。AgentResultObserved 是 v2 多智能体子→父完成投递的显式图边，不等推理请求隐式推断。

跨 crate 数据流关联

  • analytics ↔ otel：analytics 的 events.rs 调用 codex_login::default_client::originator() 获取 product_client_id；AnalyticsReducer 不直接依赖 otel，但 reducer 产生的 TrackEventRequest 里携带 CodexRuntimeMetadata（codex_rs_version/runtime_os/runtime_arch），这些在 events.rs 的 current_runtime_metadata() 中采集。
  • rollout → otel：rollout 的 compression worker 的 metrics 模块（compression.rs:802）调用 codex_otel::global() 获取 MetricsClient，记录压缩运行/文件/物化指标（codex.rollout_compression.* 前缀）。
  • rollout → analytics：rollout 不直接调用 analytics，但 rollout 记录的 SessionMeta 和 TurnContext 是 analytics reducer 关联 thread_id/turn_id 的数据来源之一。
  • rollout-trace 独立：rollout-trace 是自包含的诊断系统，不依赖其他三个 crate 的运行时数据流。它的 TraceWriter 直接写本地文件，replay_bundle() 离线读文件回放，与在线遥测通道完全解耦。
