MCP 与外部工具接入

这一块讲 Codex 如何跟"外面的世界"打交道。Model Context Protocol(MCP)是一套 JSON-RPC 协议,Codex 在这里同时扮演两个角色:一方面它是个 MCP 客户端,去连接用户在 config.toml 里声明的各种外部 MCP 服务器(比如 GitHub、文件系统、自定义工具),把那些服务器提供的工具、资源拉回来,喂给模型用;另一方面它又是个 MCP 服务器,把自己的"编码代理"能力打包成 codex / codex-reply 两个工具,暴露给 Claude、IDE 这类外部 MCP 客户端来调用。除此之外,还有一套 插件/扩展 机制,让第三方包声明自己的 MCP 服务器、贡献工具、挂进对话生命周期。

用户和外部系统能做什么

  • 在 config.toml 里声明 MCP 服务器(stdio 子进程或 streamable HTTP),Codex 启动时自动拉起、握手、列工具 → MCP 客户端接入
  • 对外部 MCP 服务器做 OAuth 登录(codex mcp login <server>),令牌存进 keyring 或文件 → MCP 客户端接入
  • 模型在对话里调用 mcp__<server>__<tool> 工具,Codex 路由到对应服务器执行 → MCP 客户端接入
  • 外部 MCP 客户端(Claude/IDE)通过 stdio 连到 codex-mcp 进程,调用 codex 工具发起一轮编码任务,收到 codex/event 事件流和最终结果 → MCP 服务器暴露
  • 外部客户端用 codex-reply 工具带着 threadId 续接已有会话 → MCP 服务器暴露
  • 外部客户端在执行/打补丁需要批准时收到 elicitation/create 请求,回复同意/拒绝 → MCP 服务器暴露
  • 插件包在自己的 mcp_servers 文件里声明 MCP 服务器,Codex 加载并合并进运行时目录 → 插件/扩展接入
  • 扩展通过 Extension API 贡献工具、挂载到对话/工具生命周期 → 插件/扩展接入

整体架构:三个角色

整个 scope 跨六个 crate,职责清晰分层:codex-mcp 是"大脑",负责把配置变成运行时服务器目录、聚合并发的客户端连接、统一对外暴露工具/资源 API;rmcp-client 是"传输层",封装单个 MCP 服务器的连接生命周期(stdio 进程、HTTP、OAuth);mcp-server 是 Codex 反过来当服务器的实现,处理外部 JSON-RPC、驱动 codex-core 跑一轮任务;connectors 负责把多来源工具合并、过滤、做策略裁决;plugin 定义插件清单和加载;ext 是内置扩展(skills、memories、web-search、goal、image-generation、mcp 执行器插件、guardian)和扩展 API 本身。

流程一:Codex 作为 MCP 客户端 —— 从配置到工具可调用

这条主线回答:"我在 config.toml 写了一个 MCP 服务器,然后模型就能调它的工具,中间发生了什么?"入口在 codex-core 的会话初始化(不在本 scope),它会调用 codex-mcp 暴露的 effective_mcp_servers,把配置里的服务器目录变成"待启动"的运行时视图,再用 McpConnectionManager::new 并发拉起所有客户端。

关键的"状态机"在 RmcpClient 内部:ClientState 有三个态——Connecting(握手中,持有 pending transport)→ Ready(持有 RunningService 和可选的 OAuthPersistor)→ Closed。每次操作走 run_service_operation 管道:先刷新 OAuth 令牌,带瞬态重试(250ms/1000ms 退避),遇 HTTP 404 会话过期就在 session_recovery_lock 信号量下用保存的 TransportRecipe 重建传输并重放一次操作。Elicitation 暂停态(ElicitationPauseState)会让操作超时时钟在服务器向用户征询输入时暂停,避免用户思考时被超时打断。

OAuth 登录子流程:HTTP 服务器在握手前会先做鉴权状态判定(determine_streamable_http_auth_status):有 bearer token 环境变量 → BearerToken;有可用存储 OAuth 令牌 → OAuth;存储令牌需授权 → NotLoggedIn;否则做 RFC 8414 /.well-known/oauth-authorization-server 发现,能发现 → NotLoggedIn,不能 → Unsupported。用户跑 codex mcp login 时走 perform_oauth_login:起一个本地回调 HTTP 服务,构建授权 URL,开浏览器/打印链接,等回调 code,换令牌,存进 keyring(带文件兜底 ~/.codex/.credentials.json)。

流程二:Codex 作为 MCP 服务器 —— 外部客户端驱动编码任务

这条主线回答:"Claude 或 IDE 通过 MCP 调用 codex 工具,让它写代码,中间发生了什么?"入口是 mcp-server 的 run_main(main.rs 只是 arg0 分发壳)。它起三个 tokio 任务:stdin 读、消息处理、stdout 写,用有界 mpsc 通道串起来。MessageProcessor 是核心调度器,持有 codex-core 的 ThreadManager,把外部 JSON-RPC 请求翻译成对 codex-core 会话的操作。

这是一个典型的 会话生命周期状态机:未初始化 → 收到 initialize → 已初始化(只允许一次,第二次返 invalid_request)。每次 codex 工具调用创建一个新 Codex 线程,进入"运行中事件循环";TurnComplete/Error/Err 是终态,但线程本身不销毁,留着等 codex-reply 续接。多线程可并发,所以每条 codex/event 都带 threadId 多路复用。running_requests_id_to_codex_uuid 这张映射表是取消和终态清理的唯一真相源。

审批的双向通信是这条流程的精华:codex-core 想执行命令或打补丁时,不是自己问用户,而是通过 OutgoingMessageSender::send_request 发一个 elicitation/create JSON-RPC 请求给外部客户端,注册一个 oneshot 回调,事件循环 continue 继续抽事件(不阻塞);外部客户端回复后走 process_response 唤醒回调,回调任务再把 Op::ExecApproval/Op::PatchApproval 提交回 codex-core。补丁审批在通道断开时主动 Deny,比命令审批更保守。

流程三:插件与扩展接入 —— 第三方能力如何挂进 Codex

这条主线回答:"一个插件包声明了自己的 MCP 服务器、技能、应用连接器、钩子,这些怎么流进 Codex 的运行时?"分为两条并行路径:插件清单路径(plugin crate,负责静态加载 MCP 服务器/技能/应用声明)和扩展 API 路径(ext/extension-api,负责运行时把扩展挂进对话/工具生命周期)。

这里有三个值得标注的边界:通信边界在 ExtensionRegistry——host 和扩展之间只通过 trait 对象(Arc<dyn ...Contributor>)通信,扩展私有状态用 ExtensionData(TypeId 键的 Any map)按 session/thread/turn 三层作用域隔离,扩展拿不到 host 内部结构。状态机边界在生命周期 contributor——线程有 start/resume/idle/stop,轮次有 start/stop/abort/error,工具调用有 start/finish(Completed/Blocked/Failed/Aborted),构成嵌套的状态层次。裁决表在 AppToolPolicyEvaluator——每个应用工具的"启用+审批模式"按固定优先级链裁决:管理要求(不可改)> 工具级配置 > 应用默认 > 破坏性/开放世界提示,输出 Auto/OnFailure/Always/OnRequest 四档。

各模块一句话

codex-mcp(大脑)
把 McpConfig 里的服务器目录变成并发的 AsyncManagedClient 集合,聚合工具/资源,统一暴露给 codex-core。Input: 配置+鉴权;Output: 可调用工具列表+资源读取+工具调用路由。
codex-mcp/rmcp_client(单客户端生命周期)
把一个 EffectiveMcpServer 变成一个握手完成、工具已列、可路由调用的 ManagedClient。Input: 服务器配置+运行时上下文;Output: ManagedClient 或 StartupOutcomeError。
codex-mcp/connection_manager(聚合器)
把多个 ManagedClient 聚合成一个对外接口,路由 (server,tool) 调用,管理启动状态事件。Input: 服务器 map;Output: list_all_tools/call_tool/list_resources/read_resource。
codex-mcp/mcp(配置层)
把 ResolvedMcpCatalog 和鉴权策略变成 EffectiveMcpServer 运行时视图,处理 OAuth 状态计算和工具名前缀化。Input: McpConfig;Output: effective_mcp_servers + 状态快照。
codex-mcp/elicitation + auth_elicitation
把服务器发起的 CreateElicitation 请求变成 Codex 协议事件,按审批策略自动接受/拒绝或交用户。Input: 服务器 elicitation;Output: ElicitationResponse 或 Event。
codex-mcp/plugin_config
把插件 mcp_servers 文件(两种格式)解析标准化成 McpServerConfig,绑定环境变量来源。Input: 插件根+文件内容;Output: PluginMcpConfigParseOutcome。
rmcp-client(传输层)
把 TransportRecipe 变成一个可重连的 RmcpClient,封装 stdio/HTTP/OAuth 三种传输。Input: 传输配方;Output: initialize/list_tools/call_tool/shutdown。
rmcp-client/stdio_server_launcher
把 StdioServerCommand 变成 rmcp Transport,本地用 tokio 子进程,远程用 ExecBackend。Input: 命令行;Output: Transport 字节流+进程句柄。
rmcp-client/http_client_adapter
把 rmcp JSON-RPC 操作变成 HTTP 请求(SSE/单消息),处理 401/403/404。Input: JSON-RPC 消息;Output: HTTP 请求+响应流。
rmcp-client/oauth + perform_oauth_login
把服务器 URL 变成浏览器 OAuth 授权码流程,令牌存 keyring(带文件兜底)。Input: URL+scope;Output: StoredOAuthTokens。
mcp-server(Codex 当服务器)
把外部 JSON-RPC 请求变成对 codex-core 线程的操作,把 Event 流变成 codex/event 通知。Input: stdio JSON-RPC;Output: 工具结果+事件流+elicitation。
connectors(应用连接器发现)
把 ChatGPT 应用目录+可达工具+插件声明按 ID 合并,裁决每个工具的启用和审批。Input: 目录 API+工具元数据;Output: 合并后的 AppInfo+AppToolPolicy。
plugin(插件模型)
把插件清单变成 LoadedPlugin,聚合成 effective 资源并集。Input: 清单+磁盘根;Output: MCP 服务器/技能/应用/钩子并集。
ext/extension-api(扩展契约)
定义十一类 Contributor trait 和三层 ExtensionData 状态,host 通过 ExtensionRegistry 查询。Input: 扩展 install 注册;Output: 工具/提示/MCP 叠加/生命周期回调。
ext/mcp(MCP 执行器插件)
当线程选定执行器插件时,解析其 stdio MCP 服务器并贡献为 SelectedPlugin 叠加。Input: 选定插件根;Output: McpServerContribution::SelectedPlugin。
ext/skills
把用户提到的技能主提示读出来注入轮次输入,贡献技能 list/read 工具。Input: 用户输入+技能根;Output: 提示片段+技能工具。
ext/memories
贡献记忆库 developer 指令和 list/read/search/add_note 工具(本地 codex_home 存储)。Input: 配置;Output: 提示片段+记忆工具。
ext/web-search
贡献 web.run 工具(OpenAI 模型提供商后端)。Input: 配置+鉴权;Output: WebSearchTool。
ext/goal
跨线程/轮次管理持久化目标状态,做预算核算和进度计数,贡献 goal 工具。Input: 状态 DB;Output: 目标工具+steering+核算。
ext/image-generation
贡献 image_gen.imagegen 工具(Codex images 后端)。Input: 配置+鉴权;Output: ImageGenerationTool。
ext/guardian
捕获线程 id 和 host 注入的 AgentSpawner,用于 fork 审查子代理。Input: 线程 id;Output: 线程本地 forked-from 上下文。

关键发现

  • 双向身份:Codex 在 MCP 协议里既是 client(codex-mcp+rmcp-client)又是 server(mcp-server),两边共用 rmcp 的 JSON-RPC 模型但职责完全不同——client 侧聚合外部工具喂给模型,server 侧把 codex 编码能力暴露给外部客户端。
  • 三级缓存:codex_apps 工具有启动快照缓存(启动时立即可用)+ 运行时缓存(listed_tools 命中)+ 硬刷新(hard_refresh_codex_apps_tools_cache 绕过缓存),解决远程应用工具列表慢的问题。
  • 不可信元数据剥离:非 codex_apps 服务器的 connector_* 元数据会被 sanitize_tool_connector_metadata 剥离,防止外部服务器伪造连接器身份;只有受信的 codex_apps 服务器保留。
  • 会话恢复:RmcpClient 遇 HTTP 404 用保存的 TransportRecipe 在 session_recovery_lock 下重建传输并重放操作,对调用方透明。
  • elicitation 暂停时钟:服务器向用户征询输入时,ElicitationPauseState 让操作超时时钟暂停,避免用户思考时被超时打断。
  • 审批的双向通道:mcp-server 不直接问用户,而是发 elicitation/create 给外部客户端,事件循环不阻塞,客户端回复经 process_response 唤醒 oneshot 回调;补丁审批在通道断开时主动 Deny。
  • 插件 MCP 名字冲突:effective_mcp_servers 用首个定义胜出策略,有 RegistrationPrecedence 优先级层(Plugin > SelectedPlugin > Config > Compatibility > Extension)。
  • 扩展状态三层隔离:ExtensionData 按 session/thread/turn 三个作用域用 TypeId 键 Any map 隔离扩展私有状态,扩展拿不到 host 内部结构。
  • mcp-server 是 MVP:resources/prompts/logging/completion 都是 stub 只记日志,只实现了 tools 相关方法和 elicitation;tasks/* 直接 METHOD_NOT_FOUND。
