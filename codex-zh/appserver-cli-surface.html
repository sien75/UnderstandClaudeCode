App Server 与 CLI 管理面

这一模块说明桌面端、IDE、脚本或管理员如何通过 Codex 的公开 JSON-RPC API 和命令行控制会话、回合、账号、插件、MCP、远程控制与隔离执行。外部输入先被协议层还原成有类型的请求，再经过连接初始化、实验能力和权限边界，最后按资源作用域串行送到对应处理器；结果和持续事件沿原连接返回。

输入
CLI 子命令，或来自 stdio、WebSocket、Unix socket、远程控制通道的 JSON-RPC 请求与通知
输出
类型化响应、错误与持续通知；或者一个受会话和文件系统边界约束的远程执行环境

公开 API 的主干不是一组彼此独立的 handler。协议宏先给每个方法绑定参数、响应、实验标记和串行化作用域；消息处理器再执行连接级检查，并把调用分派到 thread、turn、account、plugin、MCP、文件系统或进程处理器。这样同一 thread 或进程上的写操作不会互相越过，而无关资源仍可并发。

通信层把相同的消息语义放到不同载体上：本地 CLI 常用 stdio，桌面应用和守护进程可使用 Unix socket 或 WebSocket，远程控制则在认证、配对和期望状态同步后转发连接。传输层只负责连接、帧和出站背压；请求语义仍由统一消息处理器掌握。

命令行管理面既能直接启动交互式 Codex，也能管理长期运行的 App Server。`app-server` 子命令负责服务生命周期、远控开关、代理和协议产物；独立的 `exec-server` 面向执行器，把进程、文件系统和 HTTP 请求统一包装成 JSON-RPC，并用初始化会话、进程登记表及沙箱上下文约束资源。

  • 类型边界：v2 请求字段在 wire 上使用 camelCase，配置 RPC 例外地保持与 config.toml 一致的 snake_case；方法参数、响应和通知可共同生成 TypeScript 与 JSON Schema。
  • 权限边界：初始化能力决定实验 API 是否可见，连接权限门限制高风险方法；exec-server 另外校验初始化顺序、会话归属和文件系统沙箱上下文。
  • 兼容边界：协议仍保留 v1 映射，但新 API 集中在 v2；客户端既可嵌入同进程服务，也可通过 Unix socket 或 WebSocket 连接远端。
  • 执行结果可能不是一次性响应：长命令输出、回合 item、审批请求、账号状态和远控状态均可通过服务器通知继续推送。
