云端后端与云任务

这套模块讲的是 Codex 客户端怎么和"云端"打交道:把一个用户提示送到 OpenAI/ChatGPT 后端跑出回答、把一段代码任务丢到 Codex Cloud 远程执行再把 diff 拉回来 apply、以及当用户不用 OpenAI 而是接本地 Ollama / LM Studio 或 Amazon Bedrock 时怎么切换认证和模型来源。核心是"数据怎么从本地出去、怎么从云端回来"。

用户能做什么

  • 发一个对话/补全请求到 OpenAI Responses API 并流式收回答 → codex-api 的 ResponsesClient + SSE 流
  • 查后端账号、配额、速率限制、token 用量 → backend-client
  • 把代码任务提交到 Codex Cloud 远程跑、查状态、列任务、看 diff、apply 回本地 → cloud-tasks + cloud-tasks-client
  • 用 TUI 浏览/选择云端任务并 apply → cloud-tasks 的 app/ui
  • 切换模型提供商:OpenAI / Azure / Amazon Bedrock / 本地 Ollama / LM Studio → model-provider + model-provider-info + models-manager + ollama + lmstudio
  • 用一个本地代理把 Responses 请求转发到上游(隐藏 API Key)→ responses-api-proxy

关键概念

Provider(提供商)
一个可对接的模型后端,由 ModelProviderInfo 描述:base_url、env_key、auth、wire_api(现在只剩 responses)、retry/超时。codex-api 的 Provider 是它运行时形态。
AuthProvider(认证提供者)
trait,负责给出去的 HTTP 请求贴认证头。实现有 BearerAuthProvider(普通 Bearer token + ChatGPT-Account-Id + Fedramp 头)、AgentIdentityAuthProvider(Agent Identity 签名)、Bedrock SigV4、Unauthenticated(本地 OSS)。
路径风格 PathStyle
backend-client 按 base_url 决定走 /api/codex/... (CodexApi) 还是 /wham/... (ChatGptApi,即 backend-api)。cloud-tasks 也复用同一套。
Responses SSE 流
codex-api 把 POST /responses 的 text/event-stream 解析成 ResponseEvent 序列(Created、OutputTextDelta、ToolCallInputDelta、ReasoningSummaryDelta、Completed、RateLimits、ServerModel…),并通过 mpsc channel 推给上层。
Cloud Task 生命周期
Pending(排队/跑着)→ Ready(完成有 diff)→ Applied(已 apply)或 Error。best-of-N 时一个 turn 有多个 sibling attempts。

主数据流:发一次 Responses 请求并收流式回答

上层(core)拿到用户 prompt,组装成 ResponsesApiRequest(model、input、tools、reasoning、store…),交给 ResponsesClient。ResponsesClient 把请求体 JSON 编码(Azure 场景额外 attach item id),贴上 session/thread/subagent 头,交给 EndpointSession。EndpointSession 用 Provider.build_request 拼出最终 URL+默认头+query,再让 AuthProvider.apply_auth 贴认证头,最后通过 HttpTransport(底层 reqwest)发出 POST,以 text/event-stream 接收。spawn_response_stream 起一个 tokio task,逐条解析 SSE event,映射成 ResponseEvent 推进 channel。出错(response.failed/incomplete)时按错误码分类成 ContextWindowExceeded / QuotaExceeded / CyberPolicy / ServerOverloaded / Retryable 等 ApiError,再由 api_bridge 翻译成 core 层的 CodexErr。

主数据流:云任务——提交、查、apply

cloud-tasks 是 codex cloud 子命令的入口。CLI 有五个子命令(exec/status/list/apply/diff)+ 默认的 TUI。init_backend 读 base_url(默认 chatgpt.com/backend-api),用 codex-login 加载 ChatGPT 登录态,构造 cloud-tasks-client 的 HttpClient(内部包了 backend-client)。HttpClient 实现 CloudBackend trait:list_tasks/get_task_summary/get_task_diff/get_task_text/list_sibling_attempts/apply_task(_preflight)/create_task。create_task 把 prompt 包成 input_items(可选附带 CODEX_STARTING_DIFF),POST /tasks 拿回 task id。apply 流程:先 get_task_details 拿 unified diff(或用 diff_override),校验是 unified diff,再调 codex-git-utils 的 apply_git_patch 在本地 cwd 跑 git apply(preflight 模式只校验不落盘),按退出码分 Success/Partial/Error。

子图:responses-api-proxy(本地反代隐藏 API Key)

一个极简 HTTP 反代:stdin 读 auth header(只读一次,from_static + set_sensitive 避免泄漏),起 tiny_http 监听 127.0.0.1,只放行 POST /v1/responses,把请求头原样转发(去掉原 Authorization/Host,换成注入的 auth 和上游 Host),转发到 --upstream-url(默认 api.openai.com/v1/responses),把上游响应(含流式)原样回吐。可选 --dump-dir 把请求/响应落盘做调试。用于让 codex 在不接触 API Key 的前提下打 OpenAI。

错误与边界

  • codex-api 把后端错误码分类成具体 ApiError(context_length_exceeded → ContextWindowExceeded;insufficient_quota → QuotaExceeded;cyber_policy → CyberPolicy;server_is_overloaded/slow_down → ServerOverloaded;rate_limit_exceeded 带 retry-after → Retryable{delay})。api_bridge 再把 HTTP 4xx/5xx/429(usage_limit_reached → UsageLimitReached)翻译成 core 的 CodexErr。这是一个明显的决策表 + 状态映射边界。
  • 未登录 / 非 codex backend auth 在 cloud-tasks 直接 process::exit(1),不进 TUI。
  • apply 前必须校验是 unified git diff,非 unified 直接 Error 不动工作树;preflight 不落盘。
  • SSE 空闲超时(stream_idle_timeout,默认 300s)会断流并报 Stream error。
  • BedrockApiKey auth 只能配 Amazon Bedrock provider,别的 provider 会报 UnsupportedOperation。
  • Azure Responses 端点需要给 input 项显式 attach id(store=true 时)。

Notes

  • 路径风格:chatgpt.com/backend-api → /wham/...;否则 /api/codex/...。cloud-tasks 默认 base_url 就是 chatgpt.com/backend-api。
  • realtime/responses websocket 端点(v1/v2 协议)是流式通话的另一条通道,本分析只 mapped 概览,未深读。
  • cloud-tasks 的 TUI(app.rs/ui.rs/scrollable_diff.rs/new_task.rs)是 ratatui 渲染层,只 mapped。
  • models.json / prompt.md 是生成物/静态资源,ignored。
  • cloud-tasks-mock-client 提供一个 MockClient 用于本地 TUI 调试(CODEX_CLOUD_TASKS_MODE=mock)。
