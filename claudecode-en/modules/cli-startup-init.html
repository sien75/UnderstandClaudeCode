CLI Startup and Session Initialization

This module covers what happens inside the program between the moment the user types claude and hits Enter in the terminal, and the moment an interactive REPL input box appears on screen. The core goal of the entire chain is to "enter the conversation state quickly and reliably" — speed comes from lots of parallel prefetching and fast-path early returns that avoid loading the whole CLI; reliability comes from a series of gates like trust dialogs, permission sandbox checks, and configuration error handling. From a data-flow perspective, the input is the command line typed by the user (argv), and the output is either a running interactive REPL process or a one-off headless result printed before exit.

Input
The claude command the user typed in the terminal plus any trailing arguments (subcommands, flags, prompt text, stdin piped content)
Output
A ready interactive REPL (screen rendered, context assembled, permission mode determined), or a one-off print-and-exit result (--print mode)

The whole chain is roughly divided into four segments, each expanded in a diagram below. The first segment is the "fast-path dispatch" in the entry file cli.tsx — it acts like a gatekeeper, recognizing a set of special invocations that don't need the full CLI loaded and handling them directly.

In the diagram above, every "Yes" branch is an "early return," meaning the corresponding scenario never loads the bulky main.tsx; that is the key design for startup speed — for example, --version doesn't even require a dynamic import. Only when all fast paths miss does execution fall through to the bottom, loading the full CLI and calling main(). Note that --update/--upgrade gets rewritten into the update subcommand and continues downward (rather than returning directly).

The second segment enters main() and run() in main.tsx: main() performs a batch of security and state initialization that must happen before any code runs, and strips special invocation forms (cc:// URLs, ssh, assistant) from argv and stashes them in a global pending object for the default action handler to take over later; run() builds the commander program and registers a preAction hook — this hook runs before any command actually executes, and is responsible for centralizing config system setup, MDM/keychain prefetching, init(), log sinks, migrations, remote settings, and other "common initialization."

Note that init() (in entrypoints/init.ts, memoized to run once) is where the config system is actually "opened": it calls enableConfigs(), applies "safe" environment variables (the subset before trust dialog), applies extra CA certificates, hooks graceful shutdown, initializes 1P event logging and GrowthBook, configures mTLS/global HTTP agent, preconnects to Anthropic API, CCR upstream proxy, JetBrains/GitHub repo detection, etc. Config parse errors are handled differently for interactive vs non-interactive modes (dialog or stderr exit). After preAction, execution enters the specific command's action handler.

The third segment is the first half of the default action handler — this segment deconstructs user options, performs a pile of validations, kicks off several prefetches in parallel, and then calls setup() to lay the runtime foundation for working directory/session memory/git/worktree. This segment is the densest area of startup optimization: lots of IO is parallelized or deferred until after the first frame.

There is a key parallel design here: setup() (src/setup.ts) and getCommands()/getAgentDefinitionsWithOverrides() are kicked off at the same time — because setup() is mostly UDS socket binding (~20ms) and doesn't compete for disk IO, while command reading is file IO, so they naturally overlap. However, --worktree makes setup() call process.chdir(), so in that case commands must wait until chdir completes; they are gated to null and re-fetched after setup. Inside setup() there are a few gates worth knowing: bypassPermissions mode validates "must not run as root/sudo, must be inside an offline Docker/sandbox" (otherwise exit); the tengu_started event is emitted immediately after sinks are attached (the earliest reliable "process has started" signal for publishing health monitoring).

The fourth segment is the second half of the action handler — it decides which final branch the process takes. In interactive mode it first runs showSetupScreens() (trust dialog/OAuth login/onboarding/resume picker); only after trust is established does it initialize LSP, run startup prefetches, parse MCP config, and initialize versioned plugins. Then, based on --init-only, --print, options.continue, _pendingConnect/_pendingSSH/_pendingAssistantChat, options.resume, etc., it branches to different exits. All interactive exits eventually converge on launchRepl().

A few details worth knowing for newcomers: --init-only is a hidden switch that "only runs Setup + SessionStart:startup hooks then exits," used for environment initialization scripts; --print mode trust is implicit (the help text states "use only in trusted directories"), so it skips the trust screen but runs applyConfigEnvironmentVariables to apply the full (potentially dangerous) environment variables; in interactive mode, once the trust screen is rejected, process.exitCode is set and all subsequent steps that might trigger code execution (including apiKeyHelper) are skipped, to prevent plugin code from running in an untrusted directory. launchRepl (src/replLauncher.tsx) itself is very thin — it dynamically imports App and REPL components and hands them to renderAndRun for rendering; the real REPL logic lives in screens/REPL, outside the scope of this module.

Natural observations from auxiliary perspectives: State machine: the most critical state split is "interactive vs non-interactive" (isNonInteractiveSession, determined by -p/--init-only/--sdk-url/!stdout.isTTY), which decides whether trust goes through a dialog or is implicit, and whether MCP/plugin initialization is fire-and-forget or awaited; beneath that are sub-states for session source: continue/resume/connect/ssh/assistant/new. Permissions: permissionMode is parsed by initialPermissionModeFromCLI from --permission-mode/--dangerously-skip-permissions; bypass mode has a dedicated sandbox validation gate inside setup() (root/docker/internet), and auto mode has an async GrowthBook-gated validation. Lifecycle: the tengu_started event is the earliest reliable "process has started" signal (emitted immediately after sinks are attached), tengu_timer{event:'startup'} records startup latency before the first blocking dialog renders, and at session end setup() reads the previous session's tengu_exit metrics from projectConfig and emits them.

  • The entire startup chain relies heavily on "dynamic import + parallelization/deferral" to compress time-to-first-frame: almost every branch in cli.tsx is await import(...), --version requires zero imports; in main.tsx the action handler parallelizes setup/commands/agents/MCP/git/user/model prefetches.
  • --bare (equivalent to CLAUDE_CODE_SIMPLE=1) is a "minimal mode" master switch: it skips UDS messaging, teammate snapshots, iTerm/Terminal backup/restore, plugin prefetching, attribution hooks, release notes, startup prefetches, etc., slimming down the CLI for scripted invocations.
  • Trust is a hard boundary: in interactive mode, code before showSetupScreens deliberately avoids executing plugin LSP / apiKeyHelper in untrusted directories; non-interactive mode trust is implicit but requires "only use -p in directories you trust."
  • The feature('XXX') calls in cli.tsx are build-time dead-code-elimination gates; external builds will cut out those ant-only/experimental branches entirely, so the same source behaves differently internally vs externally.
  • Coverage note: this module focuses on the main data flow of "type claude to REPL ready." Internal details of cli/print.ts (5,596 lines) runHeadless, bridgeMain/daemonMain/templatesMain and other fast paths, and screens/REPL post-render interaction logic were not deeply read; they are marked only as exit nodes in the main flow. Pure option-registration lines in main.tsx (968-1005, 3811-3884 subcommand/option chains) and the 4395-4684 tail subcommand actions are only mapped at the level of responsibilities.
