Tool System Framework and File/Shell Tools

When Claude (the LLM) decides "read a file, modify some code, or run a shell command," it emits a tool_use request block. This module covers the full lifecycle of that request from landing to being returned to the LLM: how the right tool is found, how inputs are validated, how the permission gate is passed, how it is actually executed, and how the result is assembled back into the message stream so the model can continue reasoning. Scope is limited to built-in file tools (Read/Edit/Write), the Shell tool (Bash), and two search tools (Glob/Grep, the latter wrapping ripgrep).

Input
A tool_use block emitted by the model in an assistant message: tool name + arguments (e.g. {file_path, old_string, new_string} or {command}), plus current session context (permission rule set, file-read cache, abort signal, etc.).
Output
A user message containing a tool_result block (success returns tool output/summary, failure returns an error description with is_error), which the model sees in the next round and uses to decide.
  • Tools are not loose functions but instances of a unified contract (the Tool interface): each tool carries validateInput, checkPermissions, call, mapToolResultToToolResultBlockParam, and other methods that the framework calls in fixed order.
  • Concurrency safety is the key watershed: read-only tools marked isConcurrencySafe=true (Read/Glob/Grep, and read-only Bash commands) are run in parallel batches, while writes and uncertain commands run serially to avoid overwriting each other.
  • Permission is not a single gate but multiple layers — tool self-check + rule matching + PreToolUse hooks + canUseTool decision — and any layer denying produces an error result returned directly to the model.
  • File-edit tools have a "staleness check": the file must have been Read first and its disk mtime must not have changed before writing is allowed, preventing the model from overwriting the user's latest changes based on stale content.

I. Tool Registration and Tool Pool Assembly

All built-in tools are centrally registered in tools.ts. Whether a tool is "visible" is decided by several filters: environment/feature flags determine whether to load, isEnabled() determines whether it is enabled, and deny rules in the permission context determine whether it is hidden from the model. The final tool list handed to the model is the built-in tools merged with MCP tools, deduplicated by name, with deny rules applied.

graph TD
  base-list["getAllBaseTools list all built-in tools"] --> feat-gate{"feature/env flag enabled?"}
  feat-gate -->|No| skip["Skip this tool (e.g. SleepTool/WorkflowTool)"]
  feat-gate -->|Yes| enabled-check["Call tool.isEnabled()"]
  enabled-check --> deny-check{"Hit by deny rule by whole name?"}
  deny-check -->|Yes| hide["Remove from tool pool: model never sees it"]
  deny-check -->|No| keep["Keep in allowedTools"]
  keep --> merge-mcp["Merge with MCP tools"]
  merge-mcp --> dedup["Deduplicate by name, built-ins win"]
  dedup --> final-pool["Final tool pool handed to model"]
  simple-mode{"CLAUDE_CODE_SIMPLE?"} -->|Yes| simple-only["Keep only Bash/Read/Edit(+coordinator mode tools)"]
  simple-only --> final-pool

Key points: getAllBaseTools is the single source of "theoretically all tools," and comments emphasize it must stay in sync with the Statsig system cache configuration, otherwise cross-user prompt caching breaks. filterToolsByDenyRules uses the same matcher as runtime permission decisions, so a prefix rule like mcp__server hides the entire group from the model before it ever sees the tools, not just at call time.

II. Tool Contract: Tool Interface and buildTool

src/Tool.ts defines the uniform shape for all tools. A tool is an object with a name, input schema (Zod), prompt template, and a set of methods. Several key methods form the fixed pipeline the framework calls:

  • validateInput: pure logic validation (path format, parameter range, deny-path prediction), no permission-touching I/O. Failure returns an error directly, without entering the permission flow.
  • checkPermissions: the tool's own permission opinion (allow/ask/deny), called only after validateInput passes. It is combined with the generic permission system (hasPermissionsToUseTool); a tool saying ask does not mean the user will definitely be asked.
  • isConcurrencySafe: decides whether this tool can be batched and run in parallel with other read-only tools. Defaults to false (conservative); Read/Glob/Grep return true, and Bash returns true only when the command is judged read-only.
  • isReadOnly / isDestructive: semantic tags affecting UI collapsing, auto-classifier input, etc.
  • call: the actual work function, receiving permission-approved input and context, returning a ToolResult.
  • mapToolResultToToolResultBlockParam: serializes call output into the tool_result text/image block sent to the model.
  • renderToolUseMessage / renderToolResultMessage: pure UI rendering, does not affect what is sent to the model.

buildTool(def) is a factory: it merges the tool definition with a set of "safe defaults" (isEnabled defaults to true, isConcurrencySafe defaults to false, checkPermissions defaults to allow), so individual tools can omit common stub methods and callers always receive a complete object. The defaults are fail-closed — uncertain means unsafe, unsafe means permission required.

III. Complete Data Flow for a Single Tool Call

This is the core of the module. The model may emit several tool_use blocks in one turn. runTools first splits them by "concurrency safety": consecutive read-only tools merge into one concurrent batch, each write/uncertain tool becomes its own serial batch. Then each block follows the runToolUse → checkPermissionsAndCallTool main pipeline. The diagram below shows the key phases, branches, and error exits for one call.

graph TD
  find-tool["Find tool by name findToolByName"] --> tool-found{"Found?"}
  tool-found -->|No| unknown-tool["Return tool_result: No such tool available"]
  tool-found -->|Yes| aborted{"Session already aborted?"}
  aborted -->|Yes| cancel-msg["Return tool_result: cancelled"]
  aborted -->|No| parse-input["Zod parse inputSchema.safeParse"]
  parse-input --> parse-ok{"Type valid?"}
  parse-ok -->|No| zod-err["Return InputValidationError + schema hint"]
  parse-ok -->|Yes| validate-input["Call tool.validateInput business validation"]
  validate-input --> validate-ok{"Passed?"}
  validate-ok -->|No| validate-err["Return tool custom error message"]
  validate-ok -->|Yes| backfill["backfillObservableInput: expand ~/relative paths for hooks"]
  backfill --> pre-hooks["Run PreToolUse hooks"]
  pre-hooks --> hook-stop{"Hook requested stop?"}
  hook-stop -->|Yes| stop-msg["Return Execution stopped by hook"]
  hook-stop -->|No| resolve-perm["resolveHookPermissionDecision + canUseTool"]
  resolve-perm --> perm-decision{"Decision?"}
  perm-decision -->|deny / ask rejected| deny-msg["Return is_error permission-denial message + trigger PermissionDenied hook"]
  perm-decision -->|allow| call-tool["Call tool.call to perform actual operation"]
  call-tool --> call-ok{"Execution succeeded?"}
  call-ok -->|No| classify-err["classifyToolError + run PostToolUseFailure hook"]
  classify-err --> err-msg["Return is_error exception message"]
  call-ok -->|Yes| map-result["mapToolResultToToolResultBlockParam serialize"]
  map-result --> post-hooks["Run PostToolUse hooks (can rewrite MCP output)"]
  post-hooks --> assemble["Assemble tool_result + acceptFeedback + extra content blocks"]
  assemble --> yield-msg["Produce user message, flow back into model context"]

A few easy-to-overlook details:

  • Input goes through Zod first, then business validation. The model often generates malformed arguments (arrays written as strings, etc.). The Zod gate returns an error with a "schema not sent" hint, guiding the model to use ToolSearch to load the deferred tool and retry.
  • backfill happens before permissions/hooks. backfillObservableInput expands ~ / relative paths into absolute paths so hook allowlists cannot be bypassed by a tilde. But it only changes the "copy for hooks"; call() still receives the model's original fields — keeping embedded path strings in tool results ("File created at: X") and transcript/VCR hashes stable.
  • Permission decisions draw from three stacked sources: PreToolUse hooks can directly give allow/deny/modify input; resolveHookPermissionDecision prioritizes hook conclusions; only when there is no hook conclusion does it fall through to canUseTool (i.e. hasPermissionsToUseTool).
  • Permission denials also run hooks: in auto mode, after the classifier denies, the PermissionDenied hook runs; if the hook says retry, an extra "this command is now approved, retry" meta-message is inserted for the model.
  • Errors also run hooks: runPostToolUseFailureHooks lets hooks intervene when a tool throws (e.g. auto-fix). AbortError and Shell errors have dedicated branches and are not blindly logErrored.
  • Results larger than 30K/100K characters: exceeding maxResultSizeChars is persisted to a disk file, and the model receives a persisted-output wrapper with preview; the Read tool is an exception (set to Infinity) to avoid a "read file → triggers another read" loop.

IV. Concurrent Batching: Why Read-Only Can Run in Parallel

partitionToolCalls scans the model's tool_use sequence for this turn and calls tool.isConcurrencySafe(input) for each block. Consecutive blocks returning true are merged into one batch handled by runToolsConcurrently (tools are launched with all, capped at 10 concurrent); a false breaks the run, and that write operation goes alone through runToolsSerially. In serial batches, each tool updates currentContext after it finishes (because writes may change shared state like readFileState); concurrent batches defer contextModifier application until the end of the batch — reads do not affect each other, so they can safely run in parallel.

Note: isConcurrencySafe parse failures (e.g. shell-quote parse throws) are caught and treated as false, conservatively downgrading to serial. Bash concurrency safety equals "whether the command is read-only": isReadOnly calls checkReadOnlyConstraints to decide; compound commands containing cd are handled specially.

V. Permission Gate Internals: Rules + Hooks + Classifier

hasPermissionsToUseTool (src/utils/permissions/permissions.ts) is the generic permission layer. It combines the tool's self-reported checkPermissions, the user's configured allow/ask/deny rules, PreToolUse hook decisions, and (in auto mode) an AI classifier conclusion, ultimately giving allow/ask/deny. Mode switches affect the exit: dontAsk mode turns ask directly into deny (no user prompt); auto mode hands ask to the classifier instead of popping an interactive dialog.

graph TD
  tool-self["Tool self-check checkPermissions"] --> rule-check["Match allow/ask/deny rules"]
  rule-check --> deny-hit{"Hit deny?"}
  deny-hit -->|Yes| deny-out["behavior: deny"]
  deny-hit -->|No| allow-hit{"Hit allow?"}
  allow-hit -->|Yes| allow-out["behavior: allow"]
  allow-hit -->|No| ask-out["behavior: ask"]
  ask-out --> mode{"Permission mode?"}
  mode -->|dontAsk| ask-to-deny["Convert to deny: DONT_ASK_REJECT_MESSAGE"]
  mode -->|auto| classifier["Call AI classifier (bash uses Haiku side-query)"]
  classifier --> cls-verdict{"Classifier result?"}
  cls-verdict -->|safe| allow-out
  cls-verdict -->|unsafe| deny-out
  cls-verdict -->|error/unavailable| fallback{"fail-closed?"}
  fallback -->|Yes| deny-out
  mode -->|default| prompt-user["Show interactive dialog asking user"]
  prompt-user --> user-say{"User choice?"}
  user-say -->|Accept| allow-out
  user-say -->|Reject| deny-out

Bash permissions are more complex, handled in bashToolHasPermission: first it uses tree-sitter for AST-level safe parsing (detecting command injection, eval, zsh builtins, etc.), parses into SimpleCommand[], then goes through "exact match → sandbox auto-allow → prefix/wildcard rules → deny/ask classifier." AST judged too complex (command substitution / control flow) respects deny rules and then downgrades to ask, never silently allowing. When sandbox is enabled and the command is suitable, it auto-allows (still respecting explicit deny/ask rules).

VI. FileReadTool: Read Deduplication and Multiple Types

From the user's perspective: "Claude wants to look at a file." The Read tool's call first checks the readFileState cache: if the same file, same offset/limit range has already been read and the disk mtime hasn't changed, it returns a file_unchanged stub ("file unchanged," not resending the full text, saving cache tokens). Otherwise it branches by extension: text (with line numbers + cyber-risk reminder), image (base64 image block), PDF (page-extracted images or document blocks), notebook (mapped cells). On ENOENT it tries to find similar filenames for a "Did you mean" hint.

graph TD
  read-start["Read.call(file_path, offset, limit)"] --> dedup-lookup["Check readFileState cache"]
  dedup-lookup --> dedup-hit{"Same range and mtime unchanged?"}
  dedup-hit -->|Yes| unchanged-stub["Return file_unchanged stub"]
  dedup-hit -->|No| skill-discover["Background discover/activate skills for this path"]
  skill-discover --> call-inner["Enter callInner, branch by extension"]
  call-inner --> ext-switch{"File type?"}
  ext-switch -->|Text| text-branch["Read bytes → detect encoding → normalize line endings → add line numbers → attach cyber-risk reminder"]
  ext-switch -->|Image| image-branch["Base64 encode as image block"]
  ext-switch -->|PDF| pdf-branch["Extract pages by pages param → Document/image blocks"]
  ext-switch -->|Notebook| nb-branch["Parse cells → map into tool_result text"]
  ext-switch -->|Other binary| bin-err["Error: cannot Read binary files"]
  text-branch --> enoent-check
  image-branch --> enoent-check
  pdf-branch --> enoent-check
  nb-branch --> enoent-check
  enoent-check{"ENOENT?"}
  enoent-check -->|Yes| suggest["Find similar filenames / cwd suggestions for Did you mean"]
  enoent-check -->|No| update-state["Update readFileState (mtime/content/range)"]
  suggest --> throw-err["Throw friendly error"]
  update-state --> read-map-result["mapToolResultToToolResultBlockParam serialize by type"]

Key points: validateInput blocks deny paths, UNC paths (to prevent NTLM credential leakage), binary extensions, and blocking device files before permissions — these can be decided without needing permission. checkPermissions delegates to the generic checkReadPermissionForTool. maxResultSizeChars: Infinity is intentional, preventing a "read large file → persist → model reads the persisted file" loop.

VII. FileEditTool / FileWriteTool: Staleness Checks and Atomic Writes

The core defense of the two write tools is the "staleness check": the model must have Read the file first, and disk mtime must not have changed since the last Read, before writing is allowed. This prevents the model from overwriting code the user just changed based on content from several turns ago. Edit does read-modify-write: read original content → use findActualString to handle quote styles → generate patch → write to disk → update readFileState → notify LSP (didChange/didSave) and VSCode diff view. Write is whole-file replacement, additionally requiring the file to have been Read and not be a partial view.

graph TD
  write-start["Edit/Write.call"] --> mkdir["Ensure parent directory exists (outside critical section)"]
  mkdir --> backup["fileHistory backs up old content (idempotent by content hash)"]
  backup --> load-current["Read current file content + encoding + line endings"]
  load-current --> stale-check{"Has been Read and mtime unchanged?"}
  stale-check -->|No| stale-err["Throw FILE_UNEXPECTEDLY_MODIFIED_ERROR"]
  stale-check -->|Yes| compute-new["Edit: getPatchForEdit generates new content / Write: use input.content directly"]
  compute-new --> write-disk["writeTextContent atomic write"]
  write-disk --> lsp-notify["Notify LSP didChange/didSave + VSCode"]
  lsp-notify --> update-state["Update readFileState (post-write mtime)"]
  update-state --> return-diff["Return filePath/patch/userModified etc."]

Important constraints:

  • No await inside the critical section: from "read current content" to "write to disk" the event loop must not yield, otherwise concurrent edits could interleave. mkdir and backup are explicitly placed outside the critical section.
  • Write does not change line endings: whatever line endings the model provides in content are used, avoiding writing CRLF bash scripts on Linux.
  • UNC paths skip I/O validation: validateInput returns true directly for UNC paths, leaving the decision to the permission layer — because fs.existsSync on Windows UNC paths can trigger SMB authentication and potentially leak credentials.
  • Team memory leak prevention: before writing team memory files, secrets are scanned; a hit causes rejection.

VIII. BashTool: From Command to Shell Process

From the user's perspective: "Claude runs a shell command." BashTool's call first handles special "simulated sed edit" (writes directly to disk so the user preview matches what was written); otherwise it enters runShellCommand, an async generator: it uses exec to spawn a shell process, and as it runs, feeds the latest output back through the onProgress callback, yielding bash_progress events from the generator. Timeout, explicit run_in_background, or assistant-mode blocking-budget overrun all move the command to background, returning a backgroundTaskId for the model to later fetch results with TaskOutput.

graph TD
  bash-start["Bash.call(input)"] --> sed-check{"_simulatedSedEdit?"}
  sed-check -->|Yes| apply-sed["applySedEdit writes directly to disk"]
  sed-check -->|No| start-exec["exec spawns shell process (timeout/sandbox/onProgress)"]
  start-exec --> bg-req{"run_in_background?"}
  bg-req -->|Yes| spawn-bg["spawnShellTask moves to background, return backgroundTaskId"]
  spawn-bg --> bg-return["Return empty stdout + backgroundTaskId"]
  bg-req -->|No| race["Promise.race(result vs progress threshold)"]
  race --> early-done{"Command finishes quickly?"}
  early-done -->|Yes| return-result["Return ExecResult directly"]
  early-done -->|No| auto-bg{"Timeout / assistant budget reached?"}
  auto-bg -->|Yes and allowed| to-bg["Move to background to continue, notify on completion"]
  auto-bg -->|No| stream-progress["Loop yield bash_progress events"]
  stream-progress --> final-result["Get final stdout/code/interrupted"]
  return-result --> final-result
  to-bg --> final-result
  final-result --> interpret["interpretCommandResult semantic exit-code interpretation"]
  interpret --> persist-check{"Output file > threshold?"}
  persist-check -->|Yes| copy-dest["Copy to tool-results directory, model reads via Read"]
  persist-check -->|No| use-stdout["Use stdout directly"]
  copy-dest --> assemble-out["Assemble Out(stdout/isImage/persistedOutputPath...)"]
  use-stdout --> assemble-out
  assemble-out --> bash-map-result["mapToolResult...: large output wrapped persisted-output, images as image blocks"]

Bash concurrency safety depends on whether the command is read-only: isReadOnly calls checkReadOnlyConstraints, and isConcurrencySafe directly reuses the isReadOnly result. So ls / cat run in parallel with other read-only tools, while git push runs serially. preparePermissionMatcher splits compound commands into subcommand lists so hook rules like Bash(git *) can hit the git subcommand inside ls && git push — if it can't split, it fails safely and lets hooks run.

IX. GlobTool / GrepTool: ripgrep Wrappers

Both search tools are read-only, concurrency-safe, and go through checkReadPermissionForTool. Glob calls the internal glob function to find filenames by pattern; Grep actually invokes ripgrep (rg) for content search. Grep's call translates model parameters into rg command-line flags: --hidden, excludes VCS directories, --max-columns 500 (prevents base64/compressed files from flooding), -l/-c/-n/-i/-C/-A/-B/-U, etc., plus ignore patterns from the permission context and plugin-cache exclusions. Paths are passed as absolute paths to ripgrep, then results are relativized back to cwd-relative paths to save tokens.

graph TD
  grep-start["Grep.call(pattern, path, output_mode, ...)"] --> build-args["Assemble rg args: --hidden / exclude VCS / --max-columns / pattern flags"]
  build-args --> add-ignore["Add readFile ignore pattern + plugin cache exclusions"]
  add-ignore --> call-rg["ripgrep(args, absolutePath, abortSignal)"]
  call-rg --> rg-done{"rg returned?"}
  rg-done --> timeout{"RipgrepTimeoutError?"}
  timeout -->|Yes| propagate["Propagate upward (let model know search didn't finish, rather than assuming no matches)"]
  timeout -->|No| mode-switch{"output_mode?"}
  mode-switch -->|content| content-branch["Apply head_limit/offset → relativize paths → assemble content lines"]
  mode-switch -->|count| count-branch["Output filename:count + total summary"]
  mode-switch -->|files_with_matches| files-branch["Found N files + filename list"]
  content-branch --> map-out
  count-branch --> map-out
  files-branch --> map-out["mapToolResultToToolResultBlockParam serialize"]

Grep's validateInput only checks path existence (UNC skipped). head_limit is applied before relativizing paths, avoiding useless per-line processing on a thousand-line result that will be truncated to a few dozen wide lines. Glob's call is similar but simpler: call glob for the file list, relativize, return filenames + whether truncated.
