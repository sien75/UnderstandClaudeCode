Tool System Framework and File/Shell Tools

When Claude (the LLM) decides "read a file, modify some code, run a shell command," it emits a tool_use request block. This module covers the full lifecycle of that request from landing to being returned to the LLM: how the right tool is found, how inputs are validated, how permissions are gated, how it's actually executed, and how the result is assembled back into the message stream so the model can continue reasoning. Scope is limited to built-in file tools (Read/Edit/Write), the Shell tool (Bash), and two search tools (Glob/Grep, the latter wrapping ripgrep).

Input
A tool_use block emitted by the model in an assistant message: tool name + parameters (e.g., {file_path, old_string, new_string} or {command}), plus current session context (permission rule set, file-read cache, abort signal, etc.).
Output
A user message containing a tool_result block (on success: tool return content/summary; on failure: error description with is_error), which the model sees and acts on in the next turn.
  • Tools are not loose functions but instances of a unified contract (the Tool interface): each tool carries validateInput, checkPermissions, call, mapToolResultToToolResultBlockParam, and other methods the framework calls in a fixed order.
  • Concurrency safety is the key watershed: read-only tools marked isConcurrencySafe=true (Read/Glob/Grep, plus read-only Bash commands) are run in parallel batches, while writes and uncertain commands run serially to avoid overwriting each other.
  • Permissions are not a single gate but multiple stacked layers: "tool self-check + rule matching + PreToolUse hooks + canUseTool decision"—any layer denying produces an error result returned directly to the model.
  • File-edit tools have a "staleness check": the file must have been Read first and its disk mtime unchanged before writing is allowed, preventing the model from overwriting a user's latest changes based on stale content.

1. Tool Registration and Tool Pool Assembly

All built-in tools are centrally registered in tools.ts. Whether a tool is "visible" is decided by several filters: environment/feature flags determine whether to load it, isEnabled() determines whether it's enabled, and deny rules in the permission context determine whether to hide it from the model. The final tool list handed to the model is the merged, deduplicated-by-name, deny-filtered result of built-in tools and MCP tools.

graph TD
  base-list["getAllBaseTools list all built-in tools"] --> feat-gate{"feature/env flag enabled?"}
  feat-gate -->|No| skip["Skip this tool (e.g., SleepTool/WorkflowTool)"]
  feat-gate -->|Yes| enabled-check["Call tool.isEnabled()"]
  enabled-check --> deny-check{"Hit by deny rule on full name?"}
  deny-check -->|Yes| hide["Remove from tool pool: model never sees it"]
  deny-check -->|No| keep["Keep in allowedTools"]
  keep --> merge-mcp["Merge with MCP tools"]
  merge-mcp --> dedup["Dedup by name, built-in wins"]
  dedup --> final-pool["Final tool pool handed to model"]
  simple-mode{"CLAUDE_CODE_SIMPLE?"} -->|Yes| simple-only["Keep only Bash/Read/Edit(+coordination mode tools)"]
  simple-only --> final-pool

Key point: getAllBaseTools is the single source of "theoretically all tools"; comments emphasize it must stay in sync with Statsig's system cache configuration, otherwise cross-user prompt caching breaks. filterToolsByDenyRules uses the same matcher as runtime permission decisions, so a prefix rule like mcp__server masks the whole group from the model before it even sees the tools, not just at call time.

2. Tool Contract: Tool Interface and buildTool

src/Tool.ts defines the uniform shape for all tools. A tool is an object with a name, input schema (Zod), prompt template, and a set of methods. Several key methods form the fixed pipeline the framework calls:

  • validateInput: pure logic validation (path format, parameter range, deny-path pre-check), no permission-sensitive I/O. On failure it returns an error directly without entering the permission flow.
  • checkPermissions: the tool's own permission opinion (allow/ask/deny), called only after validateInput passes. It stacks with the generic permission system (hasPermissionsToUseTool); a tool saying ask doesn't guarantee the user will finally be asked.
  • isConcurrencySafe: decides whether this tool can run in parallel with other read-only tools. Defaults to false (conservative); Read/Glob/Grep return true; Bash returns true only when the command is judged read-only.
  • isReadOnly / isDestructive: semantic tags affecting UI folding, auto-classifier input, etc.
  • call: the actual work function, receiving permission-cleared input and context, returning a ToolResult.
  • mapToolResultToToolResultBlockParam: serializes call output into the tool_result text/image block sent to the model.
  • renderToolUseMessage / renderToolResultMessage: pure UI rendering, doesn't affect content sent to the model.

buildTool(def) is a factory: it merges the tool definition with a set of "safe defaults" (isEnabled defaults to true, isConcurrencySafe defaults to false, checkPermissions defaults to allow), so individual tools can omit common stub methods and callers always get a complete object. Defaults are fail-closed—uncertain means unsafe/needs permission.

3. Complete Data Flow of a Single Tool Call

This is the core of the module. A model may emit several tool_use blocks in one turn. runTools first splits them by "concurrency safety": consecutive read-only tools merge into one concurrent batch, each write/uncertain tool becomes its own serial batch. Then each block goes through the main pipeline of runToolUse → checkPermissionsAndCallTool. The diagram below shows the key stages, branches, and error exits of one call.

graph TD
  find-tool["Find tool by name findToolByName"] --> tool-found{"Found?"}
  tool-found -->|No| unknown-tool["Return tool_result: No such tool available"]
  tool-found -->|Yes| aborted{"Session already aborted?"}
  aborted -->|Yes| cancel-msg["Return tool_result: cancelled"]
  aborted -->|No| parse-input["Zod parse input inputSchema.safeParse"]
  parse-input --> parse-ok{"Type valid?"}
  parse-ok -->|No| zod-err["Return InputValidationError + schema hint"]
  parse-ok -->|Yes| validate-input["Call tool.validateInput business validation"]
  validate-input --> validate-ok{"Pass?"}
  validate-ok -->|No| validate-err["Return tool custom error message"]
  validate-ok -->|Yes| backfill["backfillObservableInput: expand ~/relative paths for hooks"]
  backfill --> pre-hooks["Run PreToolUse hooks"]
  pre-hooks --> hook-stop{"Hook requests stop?"}
  hook-stop -->|Yes| stop-msg["Return Execution stopped by hook"]
  hook-stop -->|No| resolve-perm["resolveHookPermissionDecision + canUseTool"]
  resolve-perm --> perm-decision{"Decision?"}
  perm-decision -->|deny / ask rejected| deny-msg["Return is_error permission denial + trigger PermissionDenied hook"]
  perm-decision -->|allow| call-tool["Call tool.call to perform actual operation"]
  call-tool --> call-ok{"Execution succeeded?"}
  call-ok -->|No| classify-err["classifyToolError + run PostToolUseFailure hook"]
  classify-err --> err-msg["Return is_error exception message"]
  call-ok -->|Yes| map-result["mapToolResultToToolResultBlockParam serialize"]
  map-result --> post-hooks["Run PostToolUse hooks (can rewrite MCP output)"]
  post-hooks --> assemble["Assemble tool_result + acceptFeedback + extra content blocks"]
  assemble --> yield-msg["Produce user message, flow back to model context"]

A few easily overlooked details:

  • Input passes Zod then business validation. Models often generate mistyped parameters (arrays written as strings, etc.); Zod returns an error with a "schema not sent" hint, guiding the model to load the deferred tool via ToolSearch and retry.
  • backfill happens before permissions/hooks. backfillObservableInput expands ~/relative paths into absolute paths so hook allowlists can't be bypassed by tildes. But it only modifies the "copy for hooks"; call() still receives the model's original fields—so path strings embedded in tool results ("File created at: X") and transcript/VCR hashes remain stable.
  • Permission decisions come from three stacked sources: PreToolUse hooks can directly give allow/deny/modify input; resolveHookPermissionDecision prioritizes hook conclusions; only with no hook conclusion does it fall back to canUseTool (i.e., hasPermissionsToUseTool).
  • Permission denials also run hooks: in auto mode, after the classifier denies, the PermissionDenied hook runs; if the hook says retry, an extra meta-message "this command is now approved, you may retry" is inserted for the model.
  • Errors also run hooks: runPostToolUseFailureHooks lets hooks intervene when a tool throws (e.g., auto-fix). AbortError and Shell errors have dedicated branches and don't blindly logError.
  • Results may exceed 30K/100K characters: beyond maxResultSizeChars they are persisted to a disk file, and the model receives a persisted-output wrapper with preview; Read tool is the exception (set to Infinity, avoiding a loop of "read file → trigger read of persisted file").

4. Concurrent Batching: Why Read-Only Can Run in Parallel

partitionToolCalls scans the model's tool_use sequence for this turn and calls tool.isConcurrencySafe(input) for each block. Consecutive blocks returning true merge into one batch and go through runToolsConcurrently (all tools launched concurrently up to a limit of 10); encountering false breaks the batch, and that write/uncertain operation goes through runToolsSerially alone. Within a serial batch each tool updates currentContext after finishing (because writes may modify shared state like readFileState); concurrent batches defer contextModifier until the end of the batch—reads don't affect each other, so parallel execution is safe.

Note: if isConcurrencySafe parsing fails (e.g., shell-quote parse throws), it's caught and downgraded to false, conservatively becoming serial. Bash concurrency safety equals "whether the command is read-only": isReadOnly calls checkReadOnlyConstraints; compound commands containing cd are handled specially.

5. Permission Gate Internals: Rules + Hooks + Classifier

hasPermissionsToUseTool (src/utils/permissions/permissions.ts) is the generic permission layer. It combines the tool's self-reported checkPermissions, user-configured allow/ask/deny rules, PreToolUse hook decisions, and (in auto mode) an AI classifier conclusion, ultimately producing allow/ask/deny. Mode switches affect the exit: dontAsk turns ask directly into deny (don't ask the user); auto hands ask to the classifier instead of popping an interactive dialog.

graph TD
  tool-self["Tool self-check checkPermissions"] --> rule-check["Match allow/ask/deny rules"]
  rule-check --> deny-hit{"Hit deny?"}
  deny-hit -->|Yes| deny-out["behavior: deny"]
  deny-hit -->|No| allow-hit{"Hit allow?"}
  allow-hit -->|Yes| allow-out["behavior: allow"]
  allow-hit -->|No| ask-out["behavior: ask"]
  ask-out --> mode{"Permission mode?"}
  mode -->|dontAsk| ask-to-deny["Convert to deny: DONT_ASK_REJECT_MESSAGE"]
  mode -->|auto| classifier["Call AI classifier (bash uses Haiku side-query)"]
  classifier --> cls-verdict{"Classifier result?"}
  cls-verdict -->|safe| allow-out
  cls-verdict -->|unsafe| deny-out
  cls-verdict -->|error/unavailable| fallback{"fail-closed?"}
  fallback -->|Yes| deny-out
  mode -->|default| prompt-user["Pop interactive dialog to ask user"]
  prompt-user --> user-say{"User choice?"}
  user-say -->|Accept| allow-out
  user-say -->|Reject| deny-out

Bash permissions are more complex, in bashToolHasPermission: first tree-sitter AST-level safe parsing (detects command injection, eval, zsh builtins, etc.); after parsing into SimpleCommand[], it goes through "exact match → sandbox auto-allow → prefix/wildcard rules → deny/ask classifier." AST judged too-complex (containing command substitution/control flow) respects deny rules and downgrades to ask, never silently allowing. When sandbox is enabled and the command is sandbox-suitable, it automatically allows (still respecting explicit deny/ask rules).

6. FileReadTool: Read Deduplication and Multi-Type Support

User perspective: "Claude wants to look at a file." The Read tool's call first checks the readFileState cache: if the same file with the same offset/limit range has already been read and disk mtime hasn't changed, it returns a file_unchanged stub ("file unchanged," not resending the full text, saving cache tokens). Otherwise it branches by extension: text (with line numbers + cyber-risk reminder), image (base64 image block), PDF (page-extracted images or document blocks), notebook (cell mapping). On ENOENT it tries to find similar filenames for a "Did you mean" hint.

graph TD
  read-start["Read.call(file_path, offset, limit)"] --> dedup-lookup["Check readFileState cache"]
  dedup-lookup --> dedup-hit{"Same range and mtime unchanged?"}
  dedup-hit -->|Yes| unchanged-stub["Return file_unchanged stub"]
  dedup-hit -->|No| skill-discover["Background discover/activate skills under this path"]
  skill-discover --> call-inner["Enter callInner branch by extension"]
  call-inner --> ext-switch{"File type?"}
  ext-switch -->|Text| text-branch["Read bytes→detect encoding→normalize newlines→add line numbers→attach cyber-risk reminder"]
  ext-switch -->|Image| image-branch["Encode as base64 image block"]
  ext-switch -->|PDF| pdf-branch["Extract pages by pages arg→Document/image blocks"]
  ext-switch -->|Notebook| nb-branch["Parse cells→map into tool_result text"]
  ext-switch -->|Other binary| bin-err["Error: cannot Read binary"]
  text-branch --> enoent-check
  image-branch --> enoent-check
  pdf-branch --> enoent-check
  nb-branch --> enoent-check
  enoent-check{"ENOENT?"}

  enoent-check -->|No| update-state["Update readFileState (mtime/content/range)"]

  update-state --> read-map-result["mapToolResultToToolResultBlockParam serialize by type"]

Key points: validateInput blocks deny paths, UNC paths (prevents NTLM credential leak), binary extensions, and blocking device files before permissions—no permission needed to judge these. checkPermissions delegates to the generic checkReadPermissionForTool. maxResultSizeChars: Infinity is intentional, avoiding the loop of "read large file → persist → model reads the persisted file."

7. FileEditTool / FileWriteTool: Staleness Check and Atomic Write

The core defense of the two write tools is the "staleness check": the model must have Read the file first, and disk mtime must not have changed since the last Read, before writing is allowed. This prevents the model from overwriting code the user just changed based on content from several turns ago. Edit does read-modify-write: read original content → process quote style with findActualString → generate patch → write disk → update readFileState → notify LSP (didChange/didSave) and VS Code diff view. Write is whole-file replacement, additionally requiring the file was Read first and is not a partial view.

graph TD
  write-start["Edit/Write.call"] --> mkdir["Ensure parent directory exists (outside critical section)"]
  mkdir --> backup["fileHistory backup old content (idempotent by content hash)"]
  backup --> load-current["Read current file content + encoding + newlines"]
  load-current --> stale-check{"Already Read and mtime unchanged?"}
  stale-check -->|No| stale-err["Throw FILE_UNEXPECTEDLY_MODIFIED_ERROR"]
  stale-check -->|Yes| compute-new["Edit: getPatchForEdit generates new content / Write: use input.content directly"]
  compute-new --> write-disk["writeTextContent atomic write to disk"]
  write-disk --> lsp-notify["Notify LSP didChange/didSave + VSCode"]
  lsp-notify --> update-state["Update readFileState (post-write mtime)"]
  update-state --> return-diff["Return filePath/patch/userModified etc."]

Important constraints:

  • No await inside the critical section: from "read current content" to "write disk" the event loop must not yield, otherwise concurrent edits could interleave. mkdir and backup are explicitly placed outside the critical section.
  • Write does not rewrite line endings: whatever line endings the model gives in content are used as-is, avoiding writing bash scripts with CRLF on Linux.
  • UNC paths skip I/O validation: validateInput returns true directly for UNC paths, deferring judgment to the permission layer—because fs.existsSync on Windows UNC paths can trigger SMB authentication and leak credentials.
  • Team memory leak prevention: before writing team memory files, secrets are scanned; hits are rejected.

8. BashTool: From Command to Shell Process

User perspective: "Claude runs a shell command." BashTool's call first handles the special "simulated sed edit" (writes directly to disk so the user preview matches what was written), otherwise enters runShellCommand, an async generator: it spawns a shell process with exec, and while running feeds the latest output to the generator via the onProgress callback, yielding bash_progress events. Timeout, explicit run_in_background, or assistant-mode blocking-budget overrun all move the command to background, returning a backgroundTaskId for the model to later fetch results via TaskOutput.

graph TD
  bash-start["Bash.call(input)"] --> sed-check{"_simulatedSedEdit?"}
  sed-check -->|Yes| apply-sed["applySedEdit write to disk directly"]
  sed-check -->|No| start-exec["exec spawn shell process (timeout/sandbox/onProgress)"]
  start-exec --> bg-req{"run_in_background?"}
  bg-req -->|Yes| spawn-bg["spawnShellTask move to background, return backgroundTaskId"]
  spawn-bg --> bg-return["Return empty stdout + backgroundTaskId"]
  bg-req -->|No| race["Promise.race(result vs progress threshold)"]
  race --> early-done{"Command ends quickly?"}
  early-done -->|Yes| return-result["Return ExecResult directly"]
  early-done -->|No| auto-bg{"Timeout / assistant budget reached?"}
  auto-bg -->|Yes and allowed| to-bg["Move to background continue, notify on completion later"]
  auto-bg -->|No| stream-progress["Loop yield bash_progress events"]
  stream-progress --> final-result["Get final stdout/code/interrupted"]
  return-result --> final-result
  to-bg --> final-result
  final-result --> interpret["interpretCommandResult semantic exit-code interpretation"]

  persist-check -->|No| use-stdout["Use stdout directly"]

  use-stdout --> assemble-out
  assemble-out --> bash-map-result["mapToolResult...: large output wraps persisted-output, images go as image block"]

Bash concurrency safety depends on whether the command is read-only: isReadOnly calls checkReadOnlyConstraints, and isConcurrencySafe directly reuses the isReadOnly result. So ls/cat run in parallel with other read-only tools, while git push runs serially. preparePermissionMatcher splits compound commands into subcommand lists so hook rules like Bash(git *) can hit the git subcommand inside ls && git push—if it can't split, it fails safe and lets hooks run.

9. GlobTool / GrepTool: ripgrep Wrappers

Both search tools are read-only, concurrency-safe, and use checkReadPermissionForTool. Glob calls the internal glob function to find filenames by pattern; Grep actually invokes ripgrep (rg) for content search. Grep's call translates model parameters into rg command-line flags: --hidden, exclude VCS dirs, --max-columns 500 (prevents base64/compressed files from flooding output), -l/-c/-n/-i/-C/-A/-B/-U, etc., plus ignore patterns from the permission context and plugin-cache exclusions. Paths are passed as absolute paths to ripgrep, and results are relativized back to cwd-relative paths to save tokens.

graph TD
  grep-start["Grep.call(pattern, path, output_mode, ...)"] --> build-args["Assemble rg args: --hidden/exclude VCS/--max-columns/pattern flags"]
  build-args --> add-ignore["Add readFile ignore patterns + plugin-cache exclusions"]
  add-ignore --> call-rg["ripgrep(args, absolutePath, abortSignal)"]
  call-rg --> rg-done{"rg returned?"}
  rg-done --> timeout{"RipgrepTimeoutError?"}
  timeout -->|Yes| propagate["Throw upward (let model know search didn't finish, rather than falsely thinking no matches)"]
  timeout -->|No| mode-switch{"output_mode?"}
  mode-switch -->|content| content-branch["Apply head_limit/offset→relativize paths→assemble content lines"]
  mode-switch -->|count| count-branch["Output filename:count + total summary"]
  mode-switch -->|files_with_matches| files-branch["Found N files + filename list"]
  content-branch --> map-out
  count-branch --> map-out
  files-branch --> map-out["mapToolResultToToolResultBlockParam serialize"]

Grep's validateInput only checks path existence (skipping UNC). head_limit is applied before relativizing paths, avoiding useless per-line processing for wide patterns that would be truncated from thousands of lines down to dozens. Glob's call is similar but simpler: call glob for file list, relativize, return filenames + whether truncated.
