Agents, Tasks, and Multi-Agent Coordination

This is Claude Code's "parallel execution brain." When the Claude in the main conversation wants to hand some work off to others — dispatch a sub-agent to search code, start a background shell to run tests, or spin up a multi-agent team to divide labor — all those requests converge here and are routed to the appropriate execution channel. A request from entry to final delivery back to the main conversation passes through: choosing who does it → running in what isolated environment → how task status is registered and transitioned → how completion or failure notifies the main loop → and how to stop or resume mid-flight.

From the user / external caller's perspective, there are four kinds of operations: (1) Dispatch a sub-agent (wait synchronously for the result, or run in the background and be notified later) — via the Agent tool; (2) Start a background shell task (bash monitoring, long-running commands) — via Bash tool's run_in_background or the Task tool family; (3) Form a multi-agent team (one leader with several teammates, each in an independent process/session, messaging each other) — via TeamCreate + Agent with a name parameter; (4) Check progress / fetch output / stop a running task — via TaskOutput, TaskList, TaskStop. This module covers the source implementation of these four chains.

Input
A tool call produced by the main conversation: a natural-language task description (prompt), optional sub-agent type/model/isolation mode/background flag; or a shell command; or a message sent to a teammate.
Output
A structured tool result returns to the main conversation: the sub-agent's final text reply + token/tool-call statistics; or an async_launched handle for a background task (agentId + output file path); or the shell task's exit code + output; or a "dispatched" confirmation for a teammate. Terminal notifications are injected back into the main loop message queue as <task-notification> XML.

I. Agent Tool: The Main Entry Point for Dispatching Sub-Agents

User perspective: "I want Claude to handle a subtask in parallel" — the Claude in the main conversation calls the Agent tool, passing prompt and optional subagent_type/model/run_in_background/name. The Agent tool is the scheduling hub for all of this; it must answer several questions: is this a normal sub-agent, a teammate pulled into a team, or a fork worker sharing the parent's context? Synchronous or background? Does it need worktree isolation?

  • The "should it run in the background?" step is the watershed: synchronous sub-agents share the parent's abortController (pressing ESC cancels both), while background sub-agents get an independent abortController (ESC canceling the main loop won't accidentally kill the background task).
  • The foreground path has a unique "race loop": each round Promise.races "next message" against "signal to switch to background" — once a background signal is received, it immediately switches to background without blocking for the next round.
  • Worktree isolation: when isolation: "worktree" is chosen, a temporary git worktree is opened for the sub-agent; it is auto-cleaned if unchanged after completion, or kept and returned with worktreePath if there are changes.

II. Sub-Agent Execution Loop: runAgent

Once the Agent tool has decided "who to dispatch and in what isolation", real execution is handed to runAgent(). It is an async generator that yields each message produced by the sub-agent. It arms the sub-agent: parsing which tools it should use, installing the system prompt, connecting its declared MCP servers, registering its frontmatter hooks, preloading skills it needs, and then feeding it into the main query() inference loop.

  • Tool resolution resolveAgentTools (in agentToolUtils.ts) filters by permission mode: async sub-agents can only use tools on the ASYNC_AGENT_ALLOWED_TOOLS whitelist; in-process teammates additionally get Agent and task-coordination tools so they can dispatch synchronous sub-agents.
  • The fork path is special: it uses useExactTools=true, taking the parent's tool array and system prompt directly so the API request prefix is byte-for-byte identical to hit the prompt cache — therefore fork workers cannot nest forks (recursive protection in AgentTool.call uses querySource + message scanning double checks).
  • Each agent's transcript is written to a sidechain JSONL (recordSidechainTranscript), which is the data foundation for SendMessage resume, --resume recovery, and UI panel progress viewing.
  • The finally cleanup is the key to "leaving no tails": kill shell tasks spawned by this agent, unregister session hooks, release perfetto tracing, clear todo keys — otherwise hundreds of agents in a long session would slowly leak memory.

III. Task State Machine: LocalAgentTask

Every dispatched sub-agent has a corresponding LocalAgentTaskState in AppState.tasks. The status field comes from the TaskStatus union type, and transitions have strict constraints — this is the most typical state-machine view in the module.

graph TD
  pending["pending<br/>Initial state from createTaskStateBase"] -->|"Set directly at spawn"| running["running<br/>registerAsyncAgent / registerAgentForeground"]
  running -->|"Normal completion<br/>completeAgentTask"| completed["completed<br/>With result + endTime"]
  running -->|"Exception<br/>failAsyncAgent"| failed["failed<br/>With error"]
  running -->|"User/TaskStop kill<br/>killAsyncAgent"| killed["killed<br/>abort + cleanup"]
  running -->|"Foreground→background<br/>backgroundAgentTask"| runningBg["running (isBackgrounded=true)<br/>Foreground loop exits, background continues"]
  runningBg -->|"Background completes"| completed
  runningBg -->|"Background fails"| failed
  runningBg -->|"Background killed"| killed
  completed -->|"evictAfter expires<br/>not retain"| evicted["Removed from AppState"]
  failed --> evicted
  killed --> evicted
  running -->|"SendMessage resume<br/>resumeAgentBackground"| running
  killed -->|"SendMessage resurrect<br/>resumeAgentBackground"| running
  completed -.->|"retain=true<br/>UI holds"| retained["Kept in AppState<br/>No evictAfter"]
  • The state machine is defined in src/Task.ts: TaskStatus = pending | running | completed | failed | killed; isTerminalTaskStatus() marks the last three as terminal — preventing messages from being injected into a dead teammate and preventing cleanup paths from re-entering.
  • The notified flag is the deduplication key: after a task reaches a terminal state, enqueueAgentNotification injects a <task-notification> XML into the main loop. But TaskStop also proactively marks notified to suppress redundant notifications. All terminal transition functions (complete/fail/kill) first updateTaskState and then evictTaskOutput — state flips first, disk cleanup second, ensuring TaskOutput block waits are released immediately.
  • The retain field controls whether the UI "holds" the task: retained tasks have no evictAfter, are not cleaned up by background metrics, and trigger disk bootstrap to merge the sidechain transcript UUID into messages for panel viewing.
  • The pendingMessages queue: when SendMessage sends a message to a running agent, the message enters task.pendingMessages and is drained into the agent's API input at the tool-turn boundary by drainPendingMessages — without interrupting the current round.
  • Panel visibility predicate isPanelAgentTask: all pill/panel filters must share this single predicate, otherwise they become inconsistent — agents with agentType 'main-session' are not panel agents.

IV. Background Lifecycle Driver: runAsyncAgentLifecycle

Whether dispatched directly to background from the Agent tool, switched from foreground to background, or resurrected by SendMessage, all paths eventually converge on runAsyncAgentLifecycle() (in agentToolUtils.ts). It wraps "run agent + track progress + terminal transition + notification" into a shared driver.

graph TD
  start["runAsyncAgentLifecycle<br/>taskId + makeStream"] --> makeTracker["Create ProgressTracker<br/>+ ActivityDescriptionResolver"]
  makeTracker --> onCache["onCacheSafeParams callback<br/>(enable summary service)"]
  onCache --> loop["for await makeStream() messages"]
  loop --> pushMsg["Push message into agentMessages"]
  pushMsg --> retainCheck{"task.retain?<br/>UI holds"}
  retainCheck -->|"Yes"| appendLive["Append messages to state<br/>Display live"]
  retainCheck -->|"No"| progress["Update progress<br/>Emit task_progress SDK event"]
  appendLive --> progress
  progress --> loop
  loop -->|"Stream ends"| finalize["finalizeAgentTool<br/>Extract last assistant text + stats"]
  finalize --> complete["completeAsyncAgent<br/>Status flipped to completed (before notification)"]
  complete --> handoff{"Auto mode<br/>Needs handoff classification?"}
  handoff -->|"Yes"| classify["classifyHandoffIfNeeded<br/>Safety classifier reviews sub-agent output"]
  handoff -->|"No"| worktree["Clean up worktree"]
  classify --> worktree
  worktree --> notify["enqueueAgentNotification<br/>status=completed + finalMessage"]
  notify --> doneLoop["finally: clear skills / dumpState"]
  loop -->|"AbortError"| killBranch["killAsyncAgent + extractPartialResult<br/>Notify status=killed + partial result"]
  loop -->|"Other exception"| failBranch["failAsyncAgent<br/>Notify status=failed + error info"]
  killBranch --> doneLoop
  failBranch --> worktree
  • Flip state before sending notification is a hard constraint (gh-20236): completeAsyncAgent/failAsyncAgent/killAsyncAgent immediately flip the status to terminal so that TaskOutput(block=true)'s waitForTaskCompletion unblocks right away; classifier calls, worktree cleanup, and other potentially slow steps happen after the state flip — they must not block the state machine.
  • extractPartialResult: when killed, it walks collected assistant messages in reverse and extracts the first one with text, using it as the finalMessage notification — even if canceled mid-flight, the parent agent can see how far the sub-agent got.
  • classifyHandoffIfNeeded: triggered only in auto permission mode + TRANSCRIPT_CLASSIFIER feature on; it uses an independent classifier model to review whether the sub-agent's actions violate safety policies — when unavailable it allows but warns; when blocking it returns a SECURITY WARNING to the parent agent.

V. Stopping Tasks: TaskStop → stopTask

When the user or main conversation wants to stop a background task (shell or agent) mid-flight, they call the TaskStop tool. It is a thin wrapper; the real logic lives in stopTask() in tasks/stopTask.ts.

  • Polymorphic dispatch: getTaskByType (in tasks.ts) returns concrete implementations such as LocalShellTask / LocalAgentTask / RemoteAgentTask / DreamTask based on the type field; each implements kill(taskId, setAppState). Note the Task interface now only keeps kill — spawn/render are no longer polymorphic (removed in commit 22546).
  • When a shell task is killed, it proactively marks notified to suppress the noisy "exit code 137" notification, but emits an SDK task_terminated event so SDK consumers (e.g., VS Code) still see the shutdown; agent tasks are not suppressed — the AbortError catch uses extractPartialResult to produce a meaningful payload.

VI. Multi-Agent Collaboration: Teams, Teammates, Messages

The above describes the binary relationship between "main conversation and one sub-agent." Multi-agent teams are a separate mechanism: a leader creates a team via TeamCreate, then dispatches teammates via Agent with a name parameter; teammates message each other via SendMessage. This is implemented collaboratively by TeamCreateTool / TeamDeleteTool / SendMessageTool / spawnMultiAgent.

  • Team information persists on disk in a TeamFile (utils/swarm/teamHelpers.ts), not in AppState — this lets tmux independent-process teammates read the same roster. Session-level cleanup is registered in registerTeamForSessionCleanup to avoid leftover team files (gh-32730).
  • in-process teammate uses AsyncLocalStorage isolation, not an independent process, and its lifecycle is bound to the leader process — therefore it cannot dispatch background agents or spawn teammates (the team roster is flat; nesting would break). These constraints are hard-checked at the AgentTool.call entry.
  • SendMessage routing priority: first check appState.agentNameRegistry (name→agentId mapping, registered at spawn); if it hits a LocalAgentTask, use the pendingMessages queue; if the task is stopped, resumeAgentBackground resurrects it from the sidechain transcript; otherwise fall back to teammate mailbox (tmux/inter-process).
  • Structured messages are used for team governance: shutdown_request/response implement graceful shutdown handshake (in-process directly aborts self, tmux uses gracefulShutdown); plan_approval_response lets the leader approve a teammate's plan mode and return the permissionMode to inherit.
  • UDS/bridge across sessions: SendMessage also supports uds:<socket> and bridge:<session> addresses; the former writes to a local Unix socket, the latter delivers across machines via Remote Control — bridge must force human confirmation (bypass-immune safety check) and only allows plain text.

VII. Coordinator Mode: Leader Does Not Do the Work

In normal mode, the Claude in the main conversation uses tools itself; in Coordinator mode (COORDINATOR_MODE feature flag + CLAUDE_CODE_COORDINATOR_MODE environment variable), Claude becomes a pure dispatcher — using only Agent/SendMessage/TaskStop, and delegating all real work to worker sub-agents. This is configured by coordinator/coordinatorMode.ts.

graph TD
  env["Environment variable<br/>CLAUDE_CODE_COORDINATOR_MODE"] --> gate{"COORDINATOR_MODE flag on?"}
  gate -->|"No"| off["isCoordinatorMode()=false<br/>Normal mode"]
  gate -->|"Yes"| on["isCoordinatorMode()=true"]
  on --> sysPrompt["getCoordinatorSystemPrompt()<br/>Inject leader role prompt"]
  on --> userCtx["getCoordinatorUserContext()<br/>Tell worker available toolset + scratchpad"]
  on --> forceAsync["In Agent tool shouldRunAsync=true<br/>All spawns forced background"]
  on --> escKill["On ESC cancel<br/>killAllRunningAgentTasks kills all"]
  on --> summary["Enable AgentSummary<br/>Periodic summary of worker progress"]
  match["matchSessionMode<br/>On --resume"] --> switch{"Session mode ≠ current?"}
  switch -->|"Yes"| flip["Flip env var<br/>Return prompt message"]
  switch -->|"No"| noop["No-op"]
  • The Coordinator system prompt (coordinatorMode.ts lines 111-369) is a complete playbook: it defines the leader role, Agent/SendMessage/TaskStop tool usage, <task-notification> format, research → synthesize → implement → verify phases, concurrency strategy (read-only in parallel, writes serial), continue vs spawn decision table, and worker prompt writing guidelines.
  • Key constraints: leader does not set a model parameter (workers use the default model to do real work), does not inspect one worker inside another (workers auto-notify on completion), and does not fabricate results (results arrive as separate messages).
  • matchSessionMode handles mode mismatch on resume: if the old session stored coordinator mode but it is currently off, it flips the env so isCoordinatorMode() returns the correct value — because isCoordinatorMode() reads env live without caching.
  • The scratchpad directory (tengu_scratch gate) gives workers a permission-prompt-free shared knowledge base — the leader tells workers the path in userContext, and workers read and write handoff information there.

VIII. Task Tool Family: Create/Get/List/Output/Update/Stop

Besides Agent (dispatch agents) and SendMessage (inter-agent communication), there is a set of "task board" tools that let multi-agent teams share a todo list to coordinate work. These tools operate on the disk-based task list (utils/tasks.ts; team = project = task list), which is a separate concept from the "background execution tasks" above — here it is a Kanban-style TODO board.

  • TaskCreate: creates a pending task in the current task list (getTaskListId(); leader uses team name, normal session uses sessionId), triggers TaskCreated hooks; if a hook returns blockingError, the creation is rolled back. Auto-expands the task panel.
  • TaskUpdate: changes task status (pending→in_progress→completed), owner, dependencies. Used by teammates to claim work and mark completion.
  • TaskList/TaskGet: list all / query one, for teammate board coordination.
  • TaskOutput: reads the output of a background execution task (LocalAgent/LocalShell/Remote). With block=true, waitForTaskCompletion blocks until terminal state or timeout — this is the main conversation's synchronization primitive for "wait for a background agent to finish", relying on the prompt state flip in Section III to unblock.
  • TaskStop: see Section V, polymorphic kill.

IX. Remote and Dream: Two Other Task Types

Besides LocalAgent (local sub-agent) and LocalShell (local bash), the task system has two more members:

  • RemoteAgentTask (tasks/RemoteAgentTask/): dispatches an agent to run in a remote CCR environment (ant-only, isolation: "remote" or ultraplan/review scenarios). registerRemoteAgentTask creates task state, remoteTaskMetadata persists to sidecar for --resume reconnection to still-alive remote sessions; restoreRemoteAgentTasks pulls the latest state from CCR at startup. Kill is dispatched polymorphically via RemoteAgentTask.kill.
  • DreamTask (tasks/DreamTask/): a special multi-round iterative task (DreamPhase: starting/updating), addDreamTurn accumulates turns, completeDreamTask/failDreamTask handle terminal transitions, and it also implements the Task interface to participate in polymorphic kill.
  • All concrete Task implementations are registered in getAllTasks() (tasks.ts): LocalShellTask, LocalAgentTask, RemoteAgentTask, DreamTask are always present; LocalWorkflowTask/MonitorMcpTask are feature-flagged and require'd on demand (to avoid circular dependencies + dead-code elimination).
  • Task ID prefixes are allocated by type (local_bash→b, local_agent→a, remote_agent→r, in_process_teammate→t, dream→d), followed by an 8-digit base36 random — 36^8 ≈ 2.8 trillion combinations, resistant to symlink brute-force attacks.

Key Design Points Recap

  • Flip state first, notify later: all terminal transition functions first updateTaskState then perform potentially slow cleanup/classification/notification, ensuring TaskOutput block waits unblock immediately (gh-20236).
  • notified deduplication: enqueueAgentNotification atomically uses the notified flag to prevent duplicate notifications; TaskStop also marks notified to suppress shell exit noise.
  • Foreground↔background reversible: a foreground agent can switch to background mid-flight (backgroundSignal race), and a background agent can be resumed or resurrected by SendMessage — both driven by the same runAsyncAgentLifecycle.
  • Isolation layers: synchronous sub-agents share the parent's abortController; background sub-agents have independent abortControllers (ESC doesn't accidentally kill); in-process teammates use AsyncLocalStorage same-process isolation; tmux teammates are independent processes; remote agents run in independent CCR environments.
  • Permissions narrow down the inheritance chain: the worker's tool pool is reassembled with the worker's own permissionMode (assembleToolPool), not inheriting the parent's tool restrictions; when allowedTools is provided it replaces all session-level rules to prevent leakage, but keeps cliArg (SDK --allowedTools).
  • Fork cache alignment: fork workers inherit the parent's system prompt + parent tool array + parent thinkingConfig, so the API request prefix is byte-for-byte identical to hit the prompt cache — therefore nested forks are prohibited, and recursive protection relies on querySource (autocompact does not change context.options) + message scanning double insurance.
  • Cleanup leaves no leftovers: runAgent's finally kills shell tasks spawned by the agent, unregisters hooks, clears todo keys, releases perfetto — so hundreds of agents in a long session don't slowly leak memory.
