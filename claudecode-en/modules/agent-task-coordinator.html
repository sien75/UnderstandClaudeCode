Agents, Tasks, and Multi-Agent Orchestration

This is Claude Code's "parallel execution brain." When the Claude in the main conversation wants to hand off some work—dispatch a subagent to search code, start a background shell to run tests, or spin up a multi-agent team to divide and conquer—all those requests converge here and are routed to the appropriate execution channel. From the moment a request enters until its result is returned to the main conversation, it passes through: selecting who does the work → what isolation environment to run in → how task state is registered and transitioned → how the main loop is notified on completion or failure → how to stop or resume midway.

From the user/external caller's perspective, there are four categories of things you can do: (1) dispatch a subagent (wait synchronously for the result, or run in the background and be notified later)—via the Agent tool; (2) start a background shell task (bash monitoring, long-running commands)—via the Bash tool's run_in_background or the Task tool family; (3) form a multi-agent team (a leader with several teammates, each in independent processes/sessions, messaging each other)—via TeamCreate + Agent with the name parameter; (4) check progress/fetch output/stop running tasks—via TaskOutput, TaskList, and TaskStop. This module covers the source implementation of these four paths.

Input
A tool call produced by the main conversation: a natural-language task description (prompt), optional subagent type/model/isolation mode/background flag; or a shell command; or a message sent to a teammate.
Output
A structured tool result returned to the main conversation: the subagent's final text reply plus token/tool-call statistics; or a background task async_launched handle (agentId + output file path); or the shell task's exit code + output; or a teammate "dispatched" confirmation. Final-state notifications are injected back into the main loop message queue as <task-notification> XML.

1. The Agent Tool: Main Entry for Dispatching Subagents

From the user's point of view: "I want Claude to handle a subtask in parallel"—the main conversation's Claude calls the Agent tool, passing prompt and optional subagent_type/model/run_in_background/name. The Agent tool is the dispatch hub for all of this. It must answer several questions: Is this dispatching a normal subagent, pulling a teammate into a team, or forking a worker that shares the parent's context? Should it run synchronously or in the background? Does it need worktree isolation?

  • The entry point is AgentTool.tsx's call(); the entire dispatch logic is about 1,000 lines and is the most critical section of this module.
  • The "should it run in background?" step is the watershed: synchronous subagents share the parent conversation's abortController (user pressing ESC cancels both), while background subagents get an independent abortController (ESC canceling the main loop does not accidentally kill the background).
  • The foreground path has a unique "race loop": each round Promise.races "next message" against "signal to move to background"—once a background signal is received it immediately transitions to background without blocking for the next round.
  • Worktree isolation: when isolation: "worktree" is selected, a temporary git worktree is opened for the subagent; it is auto-cleaned after completion if there are no changes, otherwise retained and the result includes worktreePath.

2. Subagent Execution Loop: runAgent

Once the Agent tool has decided "who to dispatch and in what isolation," actual execution is handed to runAgent(). It is an async generator that yields each message produced by the subagent. It is responsible for arming the subagent: resolving which tools it should use, installing the system prompt, connecting its declared MCP servers, registering its frontmatter hooks, preloading the skills it needs, and then feeding it into the main query() inference loop.

  • Tool resolution resolveAgentTools (in agentToolUtils.ts) filters by permission mode: async subagents may only use tools on the ASYNC_AGENT_ALLOWED_TOOLS allowlist; in-process teammates additionally get Agent and task-coordination tools so they can dispatch synchronous subagents.
  • The fork path is special: it uses useExactTools=true, taking the parent's tool array and system prompt directly so the API request prefix is byte-for-byte identical to hit the prompt cache—therefore a fork worker cannot fork again (recursive protection in AgentTool.call uses querySource + message scan).
  • Each agent's transcript is written to a sidechain JSONL (recordSidechainTranscript); this is the data foundation for SendMessage resume, --resume recovery, and UI panel progress viewing.
  • The finally cleanup is key to "leaving no tail": kill shell tasks spawned by this agent, unregister session hooks, release perfetto tracing, clear todos keys—otherwise hundreds of agents in a long session would slowly leak memory.

3. Task State Machine: LocalAgentTask

Every dispatched subagent has a corresponding LocalAgentTaskState in AppState.tasks. The status field comes from the TaskStatus union type, and transitions are strictly constrained—this is the most typical state-machine view in this module.

graph TD
  pending["pending<br/>createTaskStateBase initial state"] -->|"set directly at spawn"| running["running<br/>registerAsyncAgent / registerAgentForeground"]
  running -->|"normal completion<br/>completeAgentTask"| completed["completed<br/>with result + endTime"]
  running -->|"throws exception<br/>failAsyncAgent"| failed["failed<br/>with error"]
  running -->|"user/TaskStop kill<br/>killAsyncAgent"| killed["killed<br/>abort + cleanup"]
  running -->|"foreground to background<br/>backgroundAgentTask"| runningBg["running (isBackgrounded=true)<br/>foreground loop exits, background continues"]
  runningBg -->|"background completes"| completed
  runningBg -->|"background fails"| failed
  runningBg -->|"background killed"| killed
  completed -->|"evictAfter expires<br/>non-retain"| evicted["removed from AppState"]
  failed --> evicted
  killed --> evicted
  running -->|"SendMessage resume<br/>resumeAgentBackground"| running
  killed -->|"SendMessage revive<br/>resumeAgentBackground"| running
  completed -.->|"retain=true<br/>UI holds"| retained["kept in AppState<br/>no evictAfter"]
  • The state machine is defined in src/Task.ts: TaskStatus = pending | running | completed | failed | killed; isTerminalTaskStatus() marks the last three as terminal—preventing messages from being injected into a dead teammate and preventing cleanup paths from re-entering.
  • The notified flag is the deduplication key: after a task reaches a terminal state, enqueueAgentNotification injects <task-notification> XML into the main loop. But TaskStop also marks notified to suppress redundant notifications. All terminal transition functions (complete/fail/kill) call updateTaskState before evictTaskOutput—state flips first, disk cleanup second—so TaskOutput's block wait is released immediately.
  • The retain field controls whether the UI "holds" this task: retained tasks do not set evictAfter, are not cleaned up by background metrics, and trigger disk bootstrap to merge the sidechain transcript UUID into messages for panel viewing.
  • The pendingMessages queue: when SendMessage sends a message to a running agent, the message enters task.pendingMessages and is drained into the agent's API input at the tool-turn boundary by drainPendingMessages—without interrupting the current turn.
  • The panel visibility predicate isPanelAgentTask: all pill/panel filters must share this single predicate, otherwise they will be inconsistent—agentType 'main-session' does not count as a panel agent.

4. Background Lifecycle Driver: runAsyncAgentLifecycle

Whether dispatched directly in the background from the Agent tool, transitioned from foreground to background, or revived after a SendMessage stop, all paths eventually converge on runAsyncAgentLifecycle() (in agentToolUtils.ts). It wraps "run agent + track progress + terminal transition + notification" into a shared driver.

graph TD
  start["runAsyncAgentLifecycle<br/>taskId + makeStream"] --> makeTracker["create ProgressTracker<br/>+ ActivityDescriptionResolver"]
  makeTracker --> onCache["onCacheSafeParams callback<br/>(enables summary service)"]
  onCache --> loop["for await makeStream() messages"]
  loop --> pushMsg["push message into agentMessages"]
  pushMsg --> retainCheck{"task.retain?<br/>UI holds"}
  retainCheck -->|"yes"| appendLive["append messages to state<br/>display live"]
  retainCheck -->|"no"| progress["update progress<br/>emit task_progress SDK event"]
  appendLive --> progress
  progress --> loop
  loop -->|"stream ends"| finalize["finalizeAgentTool<br/>extract last assistant text + stats"]
  finalize --> complete["completeAsyncAgent<br/>status flips to completed (before notification)"]
  complete --> handoff{"auto mode<br/>needs handoff classification?"}
  handoff -->|"yes"| classify["classifyHandoffIfNeeded<br/>safety classifier reviews subagent output"]
  handoff -->|"no"| worktree["clean up worktree"]
  classify --> worktree
  worktree --> notify["enqueueAgentNotification<br/>status=completed + finalMessage"]
  notify --> doneLoop["finally: clean skills / dumpState"]
  loop -->|"AbortError"| killBranch["killAsyncAgent + extractPartialResult<br/>notify status=killed + partial result"]
  loop -->|"other exception"| failBranch["failAsyncAgent<br/>notify status=failed + error"]
  killBranch --> doneLoop
  failBranch --> worktree
  • Flip state before sending notification is a hard constraint (gh-20236): completeAsyncAgent/failAsyncAgent/killAsyncAgent immediately flip status to terminal so TaskOutput(block=true)'s waitForTaskCompletion unblocks immediately; classifier calls, worktree cleanup, and other potentially hanging steps happen after the state flip—they must not block state-machine progress.
  • extractPartialResult: when killed, it walks backward through collected assistant messages and extracts the first one with text as finalMessage for the notification—even if canceled midway, the parent agent can see how far the subagent got.
  • classifyHandoffIfNeeded: triggered only in auto permission mode when the TRANSCRIPT_CLASSIFIER feature is on; it uses an independent classifier model to review whether the subagent's actions violate safety policies—if unavailable it allows through but warns; if blocked it returns a SECURITY WARNING to the parent agent.

5. Stopping Tasks: TaskStop → stopTask

When the user or main conversation wants to stop a background task (shell or agent) midway, they call the TaskStop tool. It is a thin wrapper; the real logic is in tasks/stopTask.ts's stopTask().

  • Polymorphic dispatch: getTaskByType (in tasks.ts) returns concrete implementations such as LocalShellTask / LocalAgentTask / RemoteAgentTask / DreamTask based on the type field, each implementing kill(taskId, setAppState). Note that the Task interface now only keeps kill—spawn/render are no longer polymorphic (removed in commit 22546).
  • When a shell task is killed it proactively marks notified to suppress the "exit code 137" noise notification, but emits an SDK task_terminated event so SDK consumers (e.g., VS Code) still see the shutdown; agent tasks are not suppressed—because the AbortError catch uses extractPartialResult to produce a meaningful payload.

6. Multi-Agent Collaboration: Teams, Teammates, and Messages

What we've discussed so far is the binary relationship between the main conversation and a single subagent. Multi-agent teams are a separate mechanism: a leader creates a team via TeamCreate, then dispatches teammates via Agent with the name parameter; teammates message each other via SendMessage. This is implemented collaboratively by TeamCreateTool / TeamDeleteTool / SendMessageTool / spawnMultiAgent.

  mode -->|"splitpane/separate"| pane["handleSpawnSplitPane<br/>tmux/iTerm2 opens new pane for independent process"]

  delete --> activeCheck{"active members?"}
  activeCheck -->|"yes"| refuse["refuse cleanup"]
  activeCheck -->|"no"| cleanup["cleanupTeamDirectories<br/>clear AppState.teamContext + inbox"]
  • Team information is persisted on disk in TeamFile (utils/swarm/teamHelpers.ts), not in AppState—so tmux-independent-process teammates can read the same roster. Session-level cleanup is registered in registerTeamForSessionCleanup to avoid leftover team files (gh-32730).
  • In-process teammates use AsyncLocalStorage isolation, not independent processes, and their lifecycle is bound to the leader process—so they cannot dispatch background agents or spawn teammates (the team roster is flat; nesting would break). These constraints are hard-validated at the AgentTool.call entry.
  • SendMessage routing priority: first check appState.agentNameRegistry (name→agentId mapping, registered at spawn); if it hits a LocalAgentTask, use the pendingMessages queue; if the task is stopped, resumeAgentBackground revives it from the sidechain transcript; otherwise fall back to teammate mailboxes (tmux/inter-process).
  • Structured messages are used for team governance: shutdown_request/response implements graceful shutdown handshake (in-process aborts itself directly, tmux uses gracefulShutdown), plan_approval_response lets the leader approve a teammate's plan mode and return the permissionMode to inherit.
  • UDS/bridge across sessions: SendMessage also supports uds:<socket> and bridge:<session> addresses; the former writes to a local Unix socket, the latter delivers across machines via Remote Control—bridge requires mandatory user confirmation (bypass-immune security check) and only allows plain text.

7. Coordinator Mode: Leader Does Not Do the Work Directly

In normal mode, the Claude in the main conversation uses tools itself; in Coordinator mode (COORDINATOR_MODE feature flag + CLAUDE_CODE_COORDINATOR_MODE environment variable), Claude becomes a pure dispatcher—using only Agent/SendMessage/TaskStop, dispatching all real work to worker subagents. This is configured in coordinator/coordinatorMode.ts.

graph TD
  env["environment variable<br/>CLAUDE_CODE_COORDINATOR_MODE"] --> gate{"COORDINATOR_MODE flag on?"}
  gate -->|"no"| off["isCoordinatorMode()=false<br/>normal mode"]
  gate -->|"yes"| on["isCoordinatorMode()=true"]
  on --> sysPrompt["getCoordinatorSystemPrompt()<br/>inject leader role prompt"]
  on --> userCtx["getCoordinatorUserContext()<br/>tell workers available toolset + scratchpad"]
  on --> forceAsync["Agent tool shouldRunAsync=true<br/>force all spawns to background"]
  on --> escKill["on ESC cancel<br/>killAllRunningAgentTasks kills all"]
  on --> summary["enable AgentSummary<br/>periodic summary of worker progress"]
  match["matchSessionMode<br/>on --resume"] --> switch{"session mode != current?"}
  switch -->|"yes"| flip["flip env var<br/>return prompt"]
  switch -->|"no"| noop["no-op"]
  • The Coordinator system prompt (coordinatorMode.ts lines 111-369) is a complete playbook: it defines the leader role, how to use Agent/SendMessage/TaskStop tools, the <task-notification> format, four phases (research → synthesize → implement → verify), concurrency strategy (read-only in parallel, writes serialized), continue vs spawn decision table, and worker prompt writing guidelines.
  • Key constraints: the leader does not set a model parameter (workers do the real work with the default model), does not inspect one worker inside another (workers notify automatically when done), and does not fabricate results (results arrive as separate messages).
  • matchSessionMode handles mode mismatch on resume: if the old session stored coordinator mode but it is not currently enabled, it flips the env so isCoordinatorMode() returns the correct value—because isCoordinatorMode() reads env live and does not cache.
  • The scratchpad directory (tengu_scratch gate) gives workers a permission-prompt-free shared knowledge base—the leader tells workers the path in userContext, and workers read and write handoff information there.

8. Task Tool Family: Create/Get/List/Output/Update/Stop

Besides Agent (dispatch agents) and SendMessage (inter-agent communication), there is a set of "task list" tools that let multi-agent teams share a todo board to coordinate work. These tools operate on a task list on disk (utils/tasks.ts; team = project = task list), which is a separate concept from the "background execution tasks" above—this is a TODO board.

  • TaskCreate: creates a pending task in the current task list (getTaskListId(); leader uses team name, normal session uses sessionId), triggers TaskCreated hooks; if a hook returns a blockingError it rolls back and deletes. Auto-expands the task panel.
  • TaskUpdate: changes task status (pending→in_progress→completed), owner, dependencies. Used by teammates to claim work and mark completion.
  • TaskList/TaskGet: list all / query one, for teammate board coordination.
  • TaskOutput: reads output of a background execution task (LocalAgent/LocalShell/Remote). block=true uses waitForTaskCompletion to block until terminal state or timeout—this is the main conversation's synchronization primitive for "wait for a background agent to finish," relying on the timely state flips described in Section 3.
  • TaskStop: see Section 5, polymorphic kill.

9. Remote and Dream: Two Other Task Types

Besides LocalAgent (local subagent) and LocalShell (local bash), the task system has two more members:

  • RemoteAgentTask (tasks/RemoteAgentTask/): dispatches an agent to a remote CCR environment (ant-only, isolation: "remote" or ultraplan/review scenarios). registerRemoteAgentTask creates the task state, remoteTaskMetadata persists to sidecar for --resume reconnection to still-alive remote sessions; restoreRemoteAgentTasks fetches the latest state from CCR on startup to restore tasks. Kill is dispatched via RemoteAgentTask.kill polymorphically.
  • DreamTask (tasks/DreamTask/): a special multi-turn iterative task (DreamPhase: starting/updating), addDreamTurn accumulates turns, completeDreamTask/failDreamTask perform terminal transitions, also implementing the Task interface to participate in polymorphic kill.
  • All concrete Task implementations are registered in getAllTasks() (tasks.ts): LocalShellTask, LocalAgentTask, RemoteAgentTask, and DreamTask are always present; LocalWorkflowTask/MonitorMcpTask are require'd on demand under feature flags (avoids circular deps + dead code elimination).
  • Task ID prefixes are assigned by type (local_bash→b, local_agent→a, remote_agent→r, in_process_teammate→t, dream→d), followed by an 8-digit base-36 random value—36^8 ≈ 2.8 trillion combinations, resistant to symlink brute-force attacks.

Key Design Takeaways

  • Flip state before sending notification: all terminal transition functions first updateTaskState, then perform potentially hanging cleanup/classification/notification, ensuring TaskOutput's blocking wait is released immediately (gh-20236).
  • notified deduplication: enqueueAgentNotification uses the notified flag atomically to prevent duplicate notifications; TaskStop also marks notified to suppress shell exit noise.
  • Foreground↔background reversible: a foreground agent can transition to background midway (backgroundSignal race), and a background agent can be resumed or revived by SendMessage—both driven by the same runAsyncAgentLifecycle.
  • Isolation layers: synchronous subagents share the parent's abortController; background subagents have independent abortControllers (ESC does not accidentally kill); in-process teammates use AsyncLocalStorage same-process isolation; tmux teammates are independent processes; remote agents run in independent CCR environments.
  • Permissions narrow down the inheritance chain: a worker's tool pool is reassembled with the worker's own permissionMode (assembleToolPool), not inheriting the parent's tool restrictions; allowedTools replaces all session-level rules when provided to prevent leakage, but preserves cliArg (SDK --allowedTools).
  • Fork cache alignment: a fork worker inherits the parent's system prompt + parent's tool array + parent's thinkingConfig so the API request prefix is byte-for-byte identical to hit the prompt cache—therefore nested forks are forbidden, and recursive protection uses querySource (autocompact does not change context.options) + message scan as dual safeguards.
  • Cleanup leaves nothing behind: runAgent's finally kills shell tasks spawned by the agent, unhooks hooks, clears todos keys, and releases perfetto—so hundreds of agents in a long session do not slowly leak memory.
