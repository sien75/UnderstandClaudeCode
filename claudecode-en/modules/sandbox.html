Sandbox Mechanism (OS-Level Execution Isolation)

When Claude is about to run a bash command, read/write files, or make a network request, the system must decide one thing: should this command be executed inside an OS-level sandbox? The sandbox is a hard security boundary — implemented at the OS layer by @anthropic-ai/sandbox-runtime (sandbox-exec on macOS, bubblewrap+socat on Linux/WSL2), restricting which files the process can write and which hosts it can connect to. Violations don't let Claude go where it shouldn't; they are blocked, logged, and the user is prompted if necessary.

The entry point for this module is not a UI button, but the moment the agent autonomously executes a tool: before BashTool actually spawns the child process, it asks shouldUseSandbox(input); if the answer is yes, the original command is handed to SandboxManager.wrapWithSandbox to wrap it with a sandbox launcher before spawn. After the command finishes, cleanupAfterCommand cleans up temporary directories and scrubs any bare-git files implanted by an attacker. The whole lifecycle is exposed uniformly by SandboxManager (adapter), which forwards internally to the external package's BaseSandboxManager.

Input
A bash command triggered by the user/agent (including the dangerouslyDisableSandbox flag) + the currently merged settings (permissions allow/deny rules, sandbox.* config) + platform/dependency detection results.
Output
A command string that is either unchanged or wrapped by the sandbox launcher; stderr annotated with sandbox failures; possible violation events, permission popups, and (after user consent) WebFetch(domain) rules persisted to local settings.
  • The sandbox is not an "optional enhancement": when autoAllowBashIfSandboxed is enabled, sandbox hit = automatic bash allow (bypassing ask rules). So "whether to enter the sandbox" directly determines "whether to show a permission dialog" — the two are the same decision point.
  • The external package @anthropic-ai/sandbox-runtime is the real OS-level implementation; this module only translates CLI settings → runtime config and pipes events back, without reading its source.
  • There are three independent gates: platform/dependency support, settings enabled, and single-command exclusion / dangerouslyDisableSandbox exemption. If any gate fails, the command does not enter the sandbox.
  • Violations and permissions flow back along two independent paths: network violations go through the ask callback popup; FS/network violations go through the violation store and are surfaced in the UI expansion.

Figure 1: Sandbox Decision and Execution Flow for a Bash Command

From the user perspective: Claude decides to run npm install. Before spawn, BashTool asks "should this go into the sandbox?" in the order "global switch → explicit exemption → command on exclusion list → default into sandbox." Once sandboxing is decided, the command string is wrapped by the sandbox-runtime package into something like bwrap ... -- sh -c 'original command' (macOS equivalent), and after spawn the OS layer forcibly restricts reads/writes and network. After the command ends, cleanup removes the sandbox, and stderr lines tagged by sandbox-runtime are folded back into the output shown to Claude.

Figure 2: How Sandbox Restrictions Are Assembled from Settings

This step happens during initialize() and whenever settings change. convertToSandboxRuntimeConfig translates the scattered permissions rules and sandbox.* fields from multiple settings sources (user/project/local/policy/flag) into a SandboxRuntimeConfig the sandbox-runtime can consume (network + filesystem + ignoreViolations + ripgrep). The key point is that path semantics are two different systems — in permission rules /foo is "relative to the settings file directory", while in sandbox.filesystem.* /foo is an absolute path, and they must not be confused. Additionally, there is a set of "unconditionally hardcoded" write-deny paths: all settings.json files, .claude/skills, and bare-git files — these are the moat against escape.

Figure 3: Violation Event Handling and Permission Feedback

After the sandbox blocks something at the OS layer, there are two independent feedback paths. Network path: while the sandbox-runtime runs a command and encounters a host not in allowedDomains, it calls back to SandboxAskCallback — this callback is wired in the REPL as "queue a SandboxPermissionRequest popup", where the user chooses Yes / Don't ask again / No. Choosing "Don't ask again" persists WebFetch(domain:host) as an allow rule in localSettings and immediately calls refreshConfig so in-flight requests also benefit from the new rule. FS/general path: violations are written to SandboxViolationStore; annotateStderrWithSandboxFailures stitches them into the command output shown to Claude; two UI components subscribe to the store to display the count and the latest 10. There is also a startup hard gate: if failIfUnavailable=true and the sandbox fails to start, the REPL exits directly with gracefulShutdownSync(1) and refuses to run without a sandbox — this is the hard security gate for enterprise deployments.

  • autoAllowBashIfSandboxed welds "sandbox hit" to "automatic permission allow": in hasPermissionsToUseToolInner, as long as shouldUseSandbox returns true and auto-allow is on, it skips Bash's ask rule and directly allows, but explicit deny/ask rules still take priority (including subcommand-level checks to prevent echo ok && rm -rf / escape).
  • excludedCommands is a convenience feature, not a security boundary — code comments explicitly state that the real security control is the permission system that shows popups. It matches by prefix/exact/wildcard, and for compound commands splits on && and checks each segment, preventing the first segment from letting the whole command escape.
  • The dual path semantics are a frequent pitfall: in Edit(/foo), /foo is "relative to the settings file directory", while in sandbox.filesystem.allowWrite: ["/foo"], /foo is an absolute path. The latter was the fix for #30067 — when users write sandbox config they reasonably expect absolute paths. //foo is the historical absolute escape syntax, and both systems treat it as absolute.
  • At startup, if failIfUnavailable=true and the sandbox fails to start (missing deps/unsupported platform/not in enabledPlatforms), the REPL exits directly with gracefulShutdownSync(1) and refuses to run without a sandbox — the hard gate for deployments like NVIDIA. Otherwise it only sends a sandbox disabled notification pointing to /sandbox.
  • Bare git repo escape (#29316): an attacker implants HEAD/objects/refs in cwd so that subsequent unsandboxed git by Claude treats cwd as a bare repo and triggers core.fsmonitor. The adapter stats these files during config assembly: if they exist, denyWrite (read-only bind in sandbox); if they don't exist, add them to bareGitRepoScrubPaths and delete them in cleanupAfterCommand, so unsandboxed git won't see them.
  • Settings changes dynamically refresh via settingsChangeDetector subscription; the permission popup "Don't ask again" also manually calls refreshConfig once more to prevent race conditions (before the settings file flush event arrives, pending requests might slip through with the old config).
  • In swarm scenarios, a worker's network permission request is forwarded via mailbox to the leader for approval; on the leader side, useInboxPoller receives a SandboxPermissionRequestMessage and pops the same SandboxPermissionRequest component.
