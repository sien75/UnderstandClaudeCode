MCP Subsystem

MCP (Model Context Protocol) is Claude Code's core channel for connecting to external capabilities. This subsystem has two symmetric directions: ① Claude Code acts as a client, connecting to the various MCP servers configured by the user (local stdio processes, remote SSE/HTTP/WebSocket, claude.ai proxy connectors, IDE-embedded services), discovering the tools/resources/prompts they expose, and forwarding calls during conversations; ② Claude Code itself also serves as an MCP server, exposing its built-in tools to other AI clients via stdio (claude mcp serve). The entire flow revolves around the data pipeline of "connect → discover → call → process results → failure recovery," with additional state-machine branches for reconnection, session expiration, OAuth authorization, and URL-guided authorization (elicitation).

Input
The user's configured MCP server list (.mcp.json / config files / claude.ai connectors / IDE lockfiles / SDK dynamic injection), plus the MCP tool name and arguments that the model decides to call during a conversation; in server mode, it is the list_tools / call_tool JSON-RPC request sent by an external MCP client.
Output
Client mode: the tools/resources/prompts of external MCP servers are integrated into Claude's toolset, and the content returned after a model call (text/images/audio/binary) is normalized into message blocks and sent back to the model; server mode: Claude Code's built-in tools are exposed to external clients in MCP protocol form and the call results are returned.

Direction 1: Claude Code as an MCP Client

At startup (or when configuration changes), useManageMCPConnections collects all active server configurations and calls getMcpToolsCommandsAndResources to batch-establish connections and discover capabilities. Each server establishes a long-lived connection through connectToServer (memoized by a serialized config key). Based on the type, it selects StdioClientTransport / SSEClientTransport / StreamableHTTPClientTransport / WebSocketTransport / SdkControlClientTransport / or an in-process InProcessTransport (heavy servers such as Chrome and Computer Use run inside the main process to avoid spawning ~325 MB child processes). The connection carries a 30-second timeout (MCP_TIMEOUT); if it times out, the transport is closed and a timeout error is thrown.

For remote servers (SSE/HTTP/claude.ai-proxy), before connecting the system checks the "needs-auth cache" (15-minute TTL) and a "probed but no token" flag—if either matches, the connection is skipped and an McpAuthTool pseudo-tool is attached to that server, avoiding repeated probes every 15 minutes against servers that are destined to return 401. Remote server fetches are wrapped in three layers: innermost wrapFetchWithStepUpDetection (catches 403 step-up requests), middle wrapFetchWithTimeout (60-second fresh timeout per POST plus forced Streamable HTTP Accept header), and outermost createClaudeAiProxyFetch for claude.ai proxies (carries OAuth bearer, force-refreshes once on 401 and retries).

After a successful connection, fetchToolsForClient / fetchResourcesForClient / fetchCommandsForClient (all LRU-cached by server name) request tools/list, resources/list, and prompts/list respectively. The discovered MCP tools are wrapped into the internal Tool interface (names like mcp__<server>__<tool>), whose call method lazily executes via callMCPToolWithUrlElicitationRetry → callMCPTool. If a server declares resource support and no resource tools exist yet, ListMcpResourcesTool and ReadMcpResourceTool are attached. Any server that enters the needs-auth state gets an additional McpAuthTool so the model can trigger the OAuth flow.

When calling an MCP tool, callMCPTool initiates a JSON-RPC call with client.callTool, with its own timeout (default MCP_TOOL_TIMEOUT, about 27.8 hours—effectively infinite, because many MCP tools are long-running tasks) and 30-second progress logs. The returned result is first normalized by processMCPResult: text is preserved directly, images are compressed to API size limits, binary blobs are written to disk and only the path is returned (avoiding dumping base64 into context), and oversized output goes through persistToolResult to disk with a read guide. If the server returns isError: true, it throws McpToolCallError (carrying _meta so SDK consumers can still get metadata). If it returns 401/UnauthorizedError, it throws McpAuthError so the upper layer flips the server state to needs-auth; if it returns 404 + JSON-RPC -32001 (session expired) or derived -32000 Connection closed, the connection cache is cleared and McpSessionExpiredError is thrown; the call loop inside fetchToolsForClient automatically retries once (reconnecting with a new client).

Connection drops are detected via client.onerror / client.onclose. onclose clears the connectToServer / fetch* memoize caches so the next call reconnects automatically. onerror distinguishes terminal errors (ECONNRESET/ETIMEDOUT/EPIPE/…) from transient errors: terminal errors accumulate to 3 consecutive occurrences before actively closing the transport, letting pending callTools fail and triggering reconnection; HTTP classes also recognize two server-initiated abandonment signals, "SSE reconnection exhausted" and "session expired." stdio server cleanup escalates SIGINT → 100 ms → SIGTERM → 400 ms → SIGKILL, ensuring servers such as Docker containers that need explicit signals can exit gracefully.

  • Caching strategy: connectToServer is memoized by ${name}-${JSON(config)}; fetchTools/Resources/Commands are LRU-cached by server name (capacity 20). onclose / clearServerCache / reconnectMcpServerImpl all clear these three caches together, ensuring old data is not returned after reconnection.
  • needs-auth cache: a single 401 from a remote server writes to mcp-needs-auth-cache.json; within 15 minutes the system will not try to connect again; McpAuthTool clears the cache after OAuth completes or after /mcp manual auth via clearMcpAuthCache.
  • Session-expiration self-healing: under the Streamable HTTP protocol, a server returning 404 + -32001 means the session is invalid; the client clears the cache and automatically reconnects with a new session on the next call, retrying at most once from the caller's perspective.
  • URL elicitation: an MCP tool can return -32042 asking the user to open a URL (OAuth authorization, external confirmation, etc.). The system first runs the elicitation hook; if the hook does not handle it, the request is queued in the REPL for user action, or in print/SDK mode sent through the structuredIO callback; it retries up to 3 times.
  • Result normalization: images are compressed and downsampled, binary blobs are written to disk and only the path is returned, and oversized output is persisted to disk with a read guide—the goal is to keep context volume minimal.
  • Connection health monitoring: the useMcpConnectivityStatus hook checks failed/needs-auth counts when mcpClients changes and sends one notification each to local servers and claude.ai connectors ("/mcp"), letting the user know when a server is offline or needs authorization.
  • Project-level server approval: handleMcpjsonServerApprovals checks project servers with status=pending in .mcp.json at startup; a single server uses a single-choice dialog, multiple servers use a multi-choice dialog. Only after user approval does the server enter the connection flow—preventing project repositories from quietly launching untrusted servers.

Direction 2: Claude Code as an MCP Server

When an external AI client wants to use Claude Code's capabilities as MCP tools, run claude mcp serve (src/cli/handlers/mcp.tsx dynamically imports startMCPServer from src/entrypoints/mcp.ts). This path is straightforward: create a stdio MCP server, declare tools capability, and register ListToolsRequest and CallToolRequest handlers.

ListTools converts each built-in tool returned by getTools into an MCP-protocol Tool description: it calls tool.prompt() for the description, zodToJsonSchema for the inputSchema and outputSchema (outputSchema is only included when the root is type: object, avoiding anyOf/oneOf rejections on the SDK side). CallTool receives {name, arguments} and: finds the matching tool → checks isEnabled → calls validateInput → uses hasPermissionsToUseTool for permission validation → calls tool.call. Success serializes the result into text content; any error goes through getErrorParts to extract error text and returns a structured error with isError: true (preventing exceptions from breaking the stdio session).

The ToolUseContext constructed in server mode is a "stateless shell": mcpClients: [], messages: [], isNonInteractiveSession: true, thinkingConfig: disabled—MCP clients do not read message history; parameters are everything. File read state uses an LRU cache with a 100-file / 25 MB limit to prevent unbounded memory growth during long-running server sessions. There are two TODOs in the comments: "Also re-expose any MCP tools"—currently server mode only exposes Claude Code's own built-in tools and does not forward MCP tools connected in client mode (to avoid circular dependencies).

  • stdio transport: server mode only uses StdioServerTransport; stdin/stdout carry JSON-RPC and stderr is reserved for logs—this is the most universal way to access an MCP server; any MCP client can directly launch claude mcp serve.
  • Stateless context: in ToolUseContext, mcpClients/messages are empty and setAppState is a no-op—server mode does not maintain session state; each call_tool is an independent call.
  • Permission validation: before calling, the request passes through hasPermissionsToUseTool, reusing the same permission rules as client mode; requests from external clients are subject to the same approval/rules.
  • Errors do not kill the session: CallTool's try/catch converts any exception into a structured return with isError: true, preventing a single tool failure from collapsing the stdio session.
  • MCP tools are not forwarded yet: code comments explicitly mark this as a TODO; in the future MCP tools connected by the client may be passed through, but currently only built-in tools are exposed to avoid circular/performance issues.

OAuth Authorization Flow (Client-Mode Authentication)

Remote servers use performMCPOAuthFlow on first connection or when a token expires. If oauth.xaa (cross-app authorization) is configured, it goes through performMCPXaaAuth—as long as the IdP's id_token is still cached, the entire chain is zero-interaction: exchange the id_token for the MCP server's access/refresh token, store it in the same keychain slot, and subsequent ClaudeAuthProvider.tokens() calls are completely transparent.

The standard OAuth flow: find an available port, start a localhost callback server → use PKCE to generate an authorization URL (state + code_verifier + code_challenge) → open the browser for the user to authorize on the AS → receive the code in the callback → exchange for tokens → store in keychain. ClaudeAuthProvider implements the MCP SDK's OAuthClientProvider interface: tokens() reads the keychain and refreshes with refresh_token when expired (with in-process deduplication so concurrent requests share the same refresh promise); saveTokens writes to the keychain and clears the step-up flag; invalidateCredentials('tokens') clears tokens but keeps discovery information when refresh fails, so the next connection re-authorizes. When encountering 403 insufficient_scope, the fetch wrapper calls markStepUpPending(scope), causing tokens() to deliberately omit the refresh_token—RFC 6749 §6 forbids using refresh for privilege escalation, so authorization must be rerun to obtain a token with a larger scope.

  • XAA silent authorization: after one organization-level IdP login, all XAA servers share credentials, avoiding a browser popup per server.
  • Cross-process token consistency: keychain reads have a cache TTL; reconnectMcpServerImpl first calls clearKeychainCache, ensuring this process sees tokens changed by other processes such as the VS Code extension.
  • State/nonce protection: OAuth-sensitive parameters such as state/nonce/code_verifier/code are redacted in logs to prevent CSRF/session fixation.
