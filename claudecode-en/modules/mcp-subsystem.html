MCP Subsystem

MCP (Model Context Protocol) is the core channel through which Claude Code connects to external capabilities. This subsystem has two symmetrical directions: ① Claude Code acts as a client, connecting to the various MCP servers configured by the user (local stdio processes, remote SSE/HTTP/WebSocket, claude.ai proxy connectors, IDE-embedded services), discovering the tools/resources/prompts they expose, and forwarding invocations during conversations; ② Claude Code itself also acts as an MCP server, exposing its built-in tools to other AI clients via stdio (claude mcp serve). The entire flow centers on the data path of "connect → discover → invoke → process results → failure recovery," with additional state-machine branches for reconnection, session expiration, OAuth authorization, and URL-based elicitation.

Inputs
The user-configured list of MCP servers (.mcp.json / config files / claude.ai connectors / IDE lock files / SDK dynamic injection), plus the specific MCP tool name and parameters that the model decides to call during a conversation; in server mode, these are the list_tools / call_tool JSON-RPC requests sent by an external MCP client.
Outputs
Client mode: the tools/resources/prompts from external MCP servers are integrated into Claude's toolset; after the model invokes one, the returned content (text/image/audio/binary) is normalized into message chunks and sent back to the model. Server mode: Claude Code's built-in tools are exposed to external clients in MCP protocol form, and the invocation results are returned.

Direction 1: Claude Code as an MCP Client

At startup (or when configuration changes), useManageMCPConnections gathers all active server configurations and calls getMcpToolsCommandsAndResources to batch-establish connections and discover capabilities. Each server establishes a long-lived connection through connectToServer (memoized by a serialized config key): depending on the type, it selects StdioClientTransport / SSEClientTransport / StreamableHTTPClientTransport / WebSocketTransport / SdkControlClientTransport / or an in-process InProcessTransport (heavy servers like Chrome and Computer Use run directly in the main process to avoid spawning ~325MB child processes). The connection carries a 30-second timeout (MCP_TIMEOUT); if it times out, the transport is closed and a timeout error is thrown.

For remote servers (SSE/HTTP/claude.ai-proxy), before connecting the system checks the "needs-auth cache" (15-minute TTL) and the "probed but no token" flag — if either matches, the connection is skipped and an McpAuthTool pseudo-tool is attached to that server, avoiding repeated probes every 15 minutes against servers that are destined to return 401. Remote server fetches are wrapped in three layers: innermost wrapFetchWithStepUpDetection (captures 403 step-up requests), middle wrapFetchWithTimeout (fresh 60-second timeout per POST plus mandatory Streamable HTTP Accept header), and outermost createClaudeAiProxyFetch for claude.ai proxy connections (carries OAuth bearer, forces one refresh and retry on 401).

After a successful connection, fetchToolsForClient / fetchResourcesForClient / fetchCommandsForClient (all LRU-cached by server name) request tools/list, resources/list, and prompts/list respectively. Discovered MCP tools are wrapped into the internal Tool interface (names like mcp__<server>__<tool>), whose call method actually executes callMCPToolWithUrlElicitationRetry → callMCPTool at invocation time. If a server declares resource support and no resource tools exist yet, ListMcpResourcesTool and ReadMcpResourceTool are attached. Any server that enters the needs-auth state gets an additional McpAuthTool so the model can trigger the OAuth flow.

When invoking an MCP tool, callMCPTool uses client.callTool to initiate the JSON-RPC call, with its own timeout (default MCP_TOOL_TIMEOUT, about 27.8 hours — effectively infinite, because many MCP tools are long-running tasks) and 30-second progress logging. The returned result is first normalized by processMCPResult: text is kept as-is, images are compressed to API size limits, binary blobs are written to disk and only the path is returned (to avoid pumping base64 into context), and oversized output goes through persistToolResult to disk and returns a reading guide. If the server returns isError: true, a McpToolCallError is thrown (carrying _meta so SDK consumers can still obtain metadata). If it returns 401/UnauthorizedError, an McpAuthError is thrown so the upper layer flips the server state to needs-auth; if it returns 404 plus JSON-RPC -32001 (session expired) or the derived -32000 Connection closed, the connection cache is cleared and a McpSessionExpiredError is thrown; the invocation loop inside fetchToolsForClient automatically retries once (obtaining a new client and reconnecting).

Disconnections are detected via client.onerror / client.onclose. onclose clears the memoize caches for connectToServer / fetch*, so the next call automatically reconnects. onerror distinguishes terminal errors (ECONNRESET/ETIMEDOUT/EPIPE/…) from transient errors: terminal errors accumulate up to 3 consecutive times before actively closing the transport, causing pending callTool to fail and triggering reconnection; HTTP-style transports also recognize two server-initiated abandonment signals: "SSE reconnect exhausted" and "session expired." stdio server cleanup escalates through SIGINT → 100ms → SIGTERM → 400ms → SIGKILL, ensuring servers like Docker containers that require explicit signals can exit gracefully.

  • Caching strategy: connectToServer is memoized by ${name}-${JSON(config)}; fetchTools/Resources/Commands are LRU-cached by server name (capacity 20). onclose / clearServerCache / reconnectMcpServerImpl all clear these three caches together, ensuring old data is not returned after reconnection.
  • needs-auth cache: A single 401 from a remote server writes to mcp-needs-auth-cache.json; within 15 minutes no further connection attempts are made; McpAuthTool calls clearMcpAuthCache after completing OAuth or /mcp manual authentication.
  • Session expiration self-healing: Under the Streamable HTTP protocol, a server returning 404 + -32001 indicates session invalidation; the client clears the cache and automatically reconnects to obtain a new session on the next call, retrying at most once from the caller's perspective.
  • URL elicitation: MCP tools can return -32042 to ask the user to open a URL (OAuth authorization, external confirmation, etc.). The system first runs the elicitation hook; if the hook does not handle it, the request is queued in the REPL for user action, or delivered via structuredIO callback in print/SDK mode; it retries up to 3 times.
  • Result normalization: Images are compressed and downsampled, binary blobs are written to disk with only the path returned, and oversized output is persisted to disk with a reading guide — all aimed at minimizing context volume.
  • Connection health monitoring: The useMcpConnectivityStatus hook checks failed/needs-auth counts when mcpClients changes, sending one notification each for local servers and claude.ai connectors ("/mcp") so users know a server is down or needs authorization.
  • Project-level server approval: handleMcpjsonServerApprovals checks project servers with status=pending in .mcp.json at startup; a single server uses a single-select dialog, multiple servers use a multi-select dialog, and only approved servers enter the connection flow — preventing project repositories from silently launching untrusted servers.

Direction 2: Claude Code as an MCP Server

When an external AI client wants to use Claude Code's capabilities as MCP tools, run claude mcp serve (src/cli/handlers/mcp.tsx dynamically imports startMCPServer from src/entrypoints/mcp.ts). This path is straightforward: create a stdio MCP server, declare tools capability, and register ListToolsRequest and CallToolRequest handlers.

ListTools converts each built-in tool returned by getTools into an MCP-protocol Tool description: call tool.prompt() for the description, use zodToJsonSchema to convert inputSchema and outputSchema (outputSchema is only included when the root is type: object, to avoid anyOf/oneOf being rejected by SDK clients). CallTool receives {name, arguments}, then: find the matching tool → check isEnabled → call validateInput → use hasPermissionsToUseTool for permission checks → call tool.call. Success serializes the result as text content; any thrown error is broken into error text via getErrorParts and returned as a structured error with isError: true (so exceptions do not break the stdio session).

The ToolUseContext constructed in server mode is a "stateless shell": mcpClients: [], messages: [], isNonInteractiveSession: true, thinkingConfig: disabled — MCP clients do not read message history; parameters are everything. File-read state uses an LRU cache capped at 100 files / 25MB to prevent unbounded memory growth during long-running server sessions. Comments include two TODOs: "Also re-expose any MCP tools" — currently server mode only exposes Claude Code's own built-in tools and does not forward MCP tools connected in client mode (to avoid circular dependencies).

  • stdio transport: Server mode uses only StdioServerTransport; stdin/stdout carry JSON-RPC and stderr is reserved for logs — this is the most universal MCP server access method; any MCP client can directly launch claude mcp serve.
  • Stateless context: ToolUseContext has empty mcpClients/messages, and setAppState is a no-op — server mode does not maintain session state; each call_tool is an independent invocation.
  • Permission checks: Before invocation, hasPermissionsToUseTool applies the same permission rules used in client mode; requests from external clients are subject to the same approvals and rules.
  • Errors do not kill the session: CallTool's try/catch converts any exception into an isError: true structured response, preventing a single tool failure from crashing the stdio session.
  • MCP tools are not forwarded yet: Code comments explicitly mark this as a TODO; future versions may forward MCP tools connected in client mode, but currently only built-in tools are exposed to avoid circular/performance issues.

OAuth Authorization Flow (Client Mode Authentication)

Remote servers use performMCPOAuthFlow on first connection or when a token expires. If oauth.xaa (cross-app authorization) is configured, performMCPXaaAuth is used — as long as the IdP's id_token is still cached, the whole chain is zero-interaction: exchange the id_token for the MCP server's access/refresh token, store it in the same keychain slot, and subsequent ClaudeAuthProvider.tokens() calls are completely transparent.

The standard OAuth flow: find an available port and create a localhost callback server → use PKCE to generate an authorization URL (state + code_verifier + code_challenge) → open the browser for the user to authorize on the AS → receive the code in the callback → exchange for tokens → store in keychain. ClaudeAuthProvider implements the MCP SDK's OAuthClientProvider interface: tokens() reads from the keychain and refreshes with refresh_token when expired (with in-process deduplication so concurrent requests share the same refresh promise); saveTokens writes to the keychain and clears the step-up flag; invalidateCredentials('tokens') clears tokens but keeps discovery information when refresh fails, so the next connection re-runs authorization. When encountering 403 insufficient_scope, the fetch wrapper calls markStepUpPending(scope), causing tokens() to intentionally omit the refresh_token — RFC 6749 §6 forbids using refresh to escalate scope, so authorization must be rerun to obtain a token with broader scope.

  • XAA silent authorization: After a single organization-level IdP login, credentials are shared across all XAA servers, avoiding a browser popup for each server.
  • Cross-process token consistency: Keychain reads are cached with a TTL; reconnectMcpServerImpl first calls clearKeychainCache to ensure this process sees token changes made by other processes such as the VS Code extension.
  • State/nonce protection: Sensitive OAuth parameters such as state/nonce/code_verifier/code are redacted in logs to prevent CSRF/session fixation attacks.
