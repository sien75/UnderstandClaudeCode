Conversation Engine and LLM Streaming

This is the core channel in Claude Code from "one question" to "final reply." When the user types a question in the REPL (or an SDK call invokes ask), the engine's job is not simply to forward it to the model—it must first digest slash commands, attachments, and skill discovery in the input, then call the Claude API with full context; as the model streams tokens, the engine parses them incrementally, decides whether tools need to be called, feeds tool results back for another round, and loops until the model gives a final reply; along the way it also handles thinking, rate-limit retries, model fallback, prompt-too-long/overlong-output recovery, context compression, budget and turn limits. No step on this chain is allowed to "freeze" the conversation; every failure must have a clear exit and report.

Input
A single user question (text or content blocks with attachments) + current conversation history + tool/MCP/skill/permission context
Output
A result message (success / over budget / over turns / execution error), carrying final text, cumulative token usage, cost, model usage, and permission-denial records
  • The outer QueryEngine.submitMessage handles "session-level" affairs: assemble system prompt, run user-input preprocessing, convert messages yielded by query() into SDK messages, write transcript, and monitor budget/turn/structured-output retry limits.
  • The inner queryLoop in query.ts is the real "model-tool" loop: a while(true) that compresses context each round, then streams from the model, then decides "wrap up" or "run tools and go to next round."
  • The bottom layer, claude.ts:queryModel, is responsible for sending requests into the Anthropic SDK's streaming interface, parsing events incrementally and accumulating them into an assistant message; withRetry wraps it outside with retry/fallback/rate-limit backoff.

1. Overall Data Flow: From User Question to result Message

First, a full-chain overview. After the user hits Enter, the message enters QueryEngine.submitMessage for session-level preparation (system prompt, user-input processing, transcript persistence), then enters the query() loop. Each loop iteration either ends with "model didn't ask for tools → wrap up and return," or "model asked for tools → execute tools → back to top of loop." After the loop, submitMessage does a final result validation (isResultSuccessful): on failure it yields error_during_execution, on success success.

2. QueryEngine.submitMessage: Session-Level Orchestration

submitMessage is the entry point for SDK/REPL calls (ask is just a one-shot wrapper around it). It doesn't call the model directly; instead it owns "all preparation before model call" and "all cleanup after model call." Key design: each QueryEngine instance corresponds to one session, with mutableMessages, totalUsage, and readFileState persisting across turns; submitMessage starts a new turn on each call.

A few points worth noting: ① canUseTool is wrapped here to silently record permission denials (finally stuffed into the result for SDK reporting). ② The user message is written to transcript before entering the query loop—if the user clicks Stop a few seconds after sending and the process is killed, resume can still recover from this message instead of "session not found." ③ After entering the for await of query(), every assistant/user/compact_boundary message is pushed to mutableMessages and written to transcript; stream_event accumulates token usage and stop_reason. ④ Each step of the loop checks budget limits and structured-output retry limits; if exceeded it immediately yields the corresponding error result and returns.

3. query() Main Loop: The Model-Tool while(true)

This is the heart of the engine. query() is just a thin shell around queryLoop, responsible for marking consumed command UUIDs as completed on normal return. queryLoop is a while(true) that does three things each round: ① compress context (snip→microcompact→context-collapse→autocompact, in order, none exclusive); ② stream from the model to get an assistant message; ③ see if the model wants tools—if not, take the wrap-up path (including prompt-too-long/overlong-output recovery, stop hooks, token-budget continuation); if yes, run tools, collect attachments/memory prefetch/skill prefetch, assemble them into messages for the next round, and continue.

State is passed between iterations via a state: State object (messages, turnCount, compression tracking, recovery counts, previous transition reason). Every continue point explicitly writes a new State to avoid scattered assignments. A few recovery paths worth remembering: ① Model fallback: if streaming throws a FallbackTriggeredError, switch to fallbackModel and retry the same request, clearing already-generated assistant messages (using tombstones so the UI removes them). ② prompt-too-long: during streaming the error is "held back" instead of surfacing; at wrap-up time it first tries context-collapse drain, then reactive compact; if neither works it surfaces the error and early returns (without entering stop hooks, to avoid infinite loops). ③ max_output_tokens: first try raising max_tokens from the default 8k to 64k and retry once, then try "continuation message" recovery up to 3 times; only if exhausted does it surface the held-back error. ④ streaming empty response: if streaming ends without message_start or any content block, throw and fall back to non-streaming.

4. Stream Parsing: claude.ts queryModel

queryModelWithStreaming is wrapped by withStreamingVCR before calling queryModel. queryModel does two things: ① construct request parameters (model parsing, beta headers, tool schema, tool search filtering, prompt caching, cached microcompact, advisor, task_budget, etc.), and ② use withRetry to get a Stream, then for await parse it event by event.

Stream event handling is state-machine-like: message_start creates partialMessage and initial usage; content_block_start initializes a mutable block by type (text/thinking/tool_use/server_tool_use/advisor_tool_result); content_block_delta accumulates text/thinking/input_json/signature deltas into the corresponding block; content_block_stop wraps the block into an AssistantMessage and yields it (so one assistant message may be split into multiple yields, one per content block); message_delta writes final usage and stop_reason back into the last yielded message (mutating directly, not replacing the object, because the transcript write queue still holds the reference), and here it calculates cost, detects max_tokens/context_window_exceeded, and yields the corresponding errors; message_stop wraps up.

Robustness has two watchdogs: ① stream idle watchdog—if no chunk is received for N milliseconds (default 90s), actively abort the stream and fall back to non-streaming; ② stall detection—if chunk interval exceeds 30s, log but don't interrupt. After the stream ends it also validates "did we actually get message_start and at least one content block or stop_reason"; otherwise it treats it as the proxy returning an empty response, throws, and falls back to non-streaming.

5. Retry and Fallback: withRetry

withRetry is the request-level retry shell wrapped around anthropic.beta.messages.create(stream:true). Default max 10 retries, 529 (capacity overload) max 3 consecutive. Retry decisions: ① 529 on non-foreground sources is abandoned directly (avoid cascading amplification); ② 529 three times in a row with fallbackModel configured throws FallbackTriggeredError for the upper layer to switch models; ③ shouldRetry decides whether the error is retryable (rate limits, gateway 5xx, etc.); ④ max_tokens context-overflow errors call parseMaxTokensContextOverflowError to compute available space, lower next round's max_tokens, and continue; ⑤ persistent retry mode for 429/529 with long backoff (up to 5 minutes, capped at 6 hours), yielding an api_retry system message every 30s so the upper layer/user knows it's waiting.

Backoff duration prefers the server's retry-after header, otherwise exponential backoff (500ms × 2^n, capped at 32s). Before each retry it yields a SystemAPIErrorMessage, which QueryEngine.submitMessage converts into an api_retry SDK message to the frontend. The client instance is re-fetched after auth errors; stale connections (ECONNRESET/EPIPE) close keep-alive and retry.

6. Tool Execution Loop

When the model produces a tool_use block during the streaming response, queryLoop marks needsFollowUp=true. Tool execution has two paths: ① StreamingToolExecutor—while the model is still streaming, already-determined tool_use blocks are thrown in and run in parallel (only concurrency-safe read-only tools); getCompletedResults yields results as they complete during the streaming loop; after the stream, getRemainingResults cleans up. ② runTools—serial/concurrent batch execution when the streaming executor closes: partitionToolCalls splits tools into batches, concurrency-safe batches use runToolsConcurrently, write batches use runToolsSerially. Each tool result is pushed into toolResults as a user message (with tool_result block) and fed back to the model as messages in the next round.

Permissions are enforced here for real: canUseTool (wrapped in submitMessage to record denials) is called to decide allow/deny/ask; denied tools produce a tool_result with tool_use_error, which the model sees next round. Interruption handling: if the abort signal fires during streaming/tool execution, the executor generates synthetic tool_results for queued but unfinished tools (avoiding API 400 from unpaired tool_use without tool_result).

7. Context Compression Chain

Compression is not a single mechanism but four sequential steps run at the top of every loop round, none exclusive (tokens freed by an earlier step may make later steps no-ops): ① snip (HISTORY_SNIP feature gate)—removes "zombie" messages and expired markers; used by long SDK sessions to limit memory growth. ② microcompact—replaces large tool_result content (images/long outputs) with placeholders by tool_use_id, editable by time or cache; cached MC delays boundary messages until after the API response, using real cache_deleted_input_tokens. ③ context-collapse—folds history segments into granular summaries (read-time projection, original messages not deleted); when overflowing, recoverFromOverflow drains already-staged folds. ④ autocompact—threshold-triggered (when context nears the window limit) runs a full forked-agent summary, compressing the whole history into one summary message + attachments, producing a compact_boundary. autocompact has a circuit breaker: after N consecutive failures it stops retrying.

Besides these four "proactive" compressions, there's reactive compact—triggered only when the model stream returns prompt-too-long or media-too-large errors (errors are held during streaming, then one compression retry is attempted at wrap-up). After every successful compression, a post-compact message is yielded (summary + attachments + hook results), and messagesForQuery is replaced with the compressed array; the next round continues from the compressed context. task_budget deducts the final pre-compression context token count at compression time.

8. Token / Cost Counting

Token and cost accumulation happens in claude.ts:queryModel's message_delta branch: each time a message_delta with usage arrives, updateUsage accumulates this message's usage, then calculateUSDCost computes the USD cost for this message, and addToTotalSessionCost adds it to the session ledger. QueryEngine.submitMessage calls updateUsage / accumulateUsage on stream_event message_start/message_delta/message_stop, accumulating each message's usage into totalUsage and finally stuffing it into the result message. cost-tracker.ts maintains per-model usage (input/output/cacheRead/cacheCreation/webSearch/costUSD) and exposes getTotalCost / getModelUsage for result and budget checking.

Token estimation (tokenEstimation.ts) is used for compression threshold decisions and blocking-limit checks: without calling the API, roughTokenCountEstimationForMessages gives a rough estimate (by bytes/type conversion); autocompact's shouldAutoCompact and the blocking limit check at the top of queryLoop rely on it. Note: tokens freed by snip must be explicitly subtracted from the estimate (snipTokensFreed); otherwise snip may have already pushed context below the threshold, but stale usage still shows above threshold, falsely triggering blocking.

  • State-machine perspective: queryLoop's state.transition.reason (next_turn / collapse_drain_retry / reactive_compact_retry / max_output_tokens_escalate / max_output_tokens_recovery / stop_hook_blocking / token_budget_continuation) is an implicit state machine—every continue point carries a reason, making test assertions like "which recovery path was triggered" easy.
  • Permission perspective: canUseTool is wrapped in submitMessage to record denials; it's called during tool execution; deny doesn't throw but produces tool_use_error for the model to see; permission context (allowedTools, additionalWorkingDirectories) is updated into AppState after user-input processing.
  • Lifecycle perspective: one QueryEngine instance per session, state persists across turns; submitMessage once per turn; queryLoop one state per round; withRetry multiple attempts per request; streaming one stream per request. The abort signal threads through all layers, each with its own abort handling (pad synthetic results for streaming, pad synthetic results for tools, skip stop hooks).
  • Transcript persistence: user message is written to transcript before entering query (preventing resume failure after kill); assistant messages are written fire-and-forget (avoid blocking message_delta); compact_boundary flushes the retained segment tail first; result yields force flush in cowork/eager modes.
  • Non-streaming fallback: streaming empty response, idle timeout, proxy returning non-SSE body, etc. all trigger executeNonStreamingRequest—using messages.create(stream:false) with default 300s timeout (120s remote), then feeding the result through the same usage/cost accumulation path.
