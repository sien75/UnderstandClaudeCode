Conversation Engine and LLM Streaming

This is the core channel of Claude Code from "one user question" to "final response." When the user types a question in the REPL (or the SDK calls ask once), the engine's job is not simply to forward it to the model — it must first digest slash commands, attachments, and skill discovery in the user input, then call the Claude API with complete context; while the model streams tokens, the engine parses them on the fly, decides whether to call tools, feeds tool results back for another round, and loops until the model gives a final response; along the way it also handles thinking, rate-limit retries, model fallback, prompt-too-long / overlong-output recovery, context compression, budget caps, and turn limits. No step in this chain is allowed to "freeze" the conversation; there must be a clear exit and report for any failure.

Input
A user question (text or content blocks with attachments) + current conversation history + tool/MCP/skill/permission context
Output
A result message (success / over budget / over turn limit / execution error), with final text, cumulative token usage, cost, model usage, and permission-denial records
  • The outer QueryEngine.submitMessage handles "session-level" transactions: assembling the system prompt, running user-input preprocessing, converting messages yielded by query() into SDK messages, writing transcript, and watching budget/turn/structured-output retry limits.
  • The inner query.ts queryLoop is the real "model-tool" loop: while(true) compresses context each round, then calls the model in streaming mode, then decides whether to "finish" or "run tools and go to next round."
  • The bottom layer claude.ts:queryModel is responsible for sending the request into Anthropic SDK's streaming interface, parsing events one by one and accumulating them into an assistant message; withRetry wraps it with retry / fallback / rate-limit backoff.

I. Overall Data Flow: From User Question to Result Message

First, the full chain in one pass. After the user hits Enter, the message enters QueryEngine.submitMessage for session-level preparation (system prompt, user input processing, transcript persistence), then enters the query() loop. Each iteration either ends with "model didn't ask for tools → wrap up and return," or "model asked for tools → execute them → return to the top of the loop with results." After the loop, submitMessage performs a final result check (isResultSuccessful): failure yields error_during_execution, success yields success.

II. QueryEngine.submitMessage: Session-Level Orchestration

submitMessage is the entry point called by the SDK/REPL (ask is just a one-shot wrapper around it). It does not call the model directly; instead it owns "all preparation before calling the model" and "all cleanup after." Key design: each QueryEngine instance corresponds to one session, with mutableMessages, totalUsage, and readFileState persisting across turns; submitMessage starts a new turn each time it is called.

A few points worth noting: ① canUseTool is wrapped here to secretly record permission denials (later packed into the result for SDK reporting). ② User messages are written to transcript before entering the query loop — if the user hits Stop seconds after sending a message and the process is killed, resume can still recover from this message instead of "session not found." ③ After entering the query() for await, every assistant/user/compact_boundary message is pushed to mutableMessages and written to transcript; stream_event is used to accumulate token usage and stop_reason. ④ Each loop step checks budget limits and structured-output retry limits; exceeding them immediately yields the corresponding error result and returns.

III. query() Main Loop: Model-Tool while(true)

This is the heart of the engine. query() is just a thin shell around queryLoop, responsible for marking consumed command UUIDs as completed on normal return. queryLoop is a while(true) that does three things each round: ① compress context (snip → microcompact → context-collapse → autocompact, in order, not mutually exclusive); ② call the model in streaming mode and receive an assistant message; ③ see whether the model asked for tools — if not, take the wrap-up path (including prompt-too-long / overlong-output recovery, stop hooks, token-budget continuation); if yes, run the tools, collect attachments/memory prefetch/skill prefetch, assemble them into messages for the next round, and continue.

State is passed between iterations via the state: State object (messages, turnCount, compression tracking, recovery counts, previous round's transition reason). Every continue point explicitly constructs a new State to avoid scattered mutations. A few recovery paths worth remembering: ① Model fallback: if a FallbackTriggeredError is thrown during streaming, switch to fallbackModel and retry the same request, clearing already-generated assistant messages (using tombstones so the UI removes them). ② prompt-too-long: during streaming, the error is "held back" instead of being yielded, and at wrap-up time context-collapse is tried first to drain, then reactive compact; if neither works, the held error is yielded and early-returned (without stop hooks, to avoid infinite loops). ③ max_output_tokens: first try raising max_tokens from the default 8k to 64k for one retry, then try "continuation message" recovery up to 3 times; only if all are exhausted is the held error surfaced. ④ Empty streaming response: if streaming ends without message_start or any content block, throw and fall back to non-streaming.

IV. Streaming Parsing: claude.ts queryModel

queryModelWithStreaming calls queryModel after being wrapped by withStreamingVCR. queryModel does two things: ① construct request parameters (model resolution, beta headers, tool schema, tool search filtering, prompt caching, cached microcompact, advisor, task_budget, etc.), and ② use withRetry to obtain a Stream, then for await over it to parse events one by one.

Streaming event handling is state-machine-like: message_start creates partialMessage and initial usage; content_block_start initializes a mutable block by type (text/thinking/tool_use/server_tool_use/advisor_tool_result); content_block_delta increments text/thinking/input_json/signature into the corresponding block; content_block_stop wraps the block into an AssistantMessage and yields it (so one assistant message may be split into multiple yields, one per content block); message_delta writes final usage and stop_reason back into the last yielded message (mutating in place rather than replacing the object, because the transcript write queue still holds the reference), and here it also calculates cost, detects max_tokens/context_window_exceeded, and yields the corresponding errors; message_stop is a no-op.

Robustness has two watchdogs: ① stream idle watchdog — if no chunk is received for N milliseconds (default 90s), proactively abort the stream and fall back to non-streaming; ② stall detection — gaps between chunks over 30s are logged but do not interrupt. After the stream ends, it also validates that "message_start was actually received and at least one content block or stop_reason exists"; otherwise it is treated as the proxy returning an empty response, and an error is thrown to fall back to non-streaming.

V. Retry and Fallback: withRetry

withRetry is a request-level retry shell wrapping anthropic.beta.messages.create(stream:true). Default max 10 retries, with 529 (capacity overload) capped at 3 consecutive. Retry decisions: ① 529 on non-foreground sources is abandoned directly (avoid cascading amplification); ② 529 three times in a row and a fallbackModel is configured throws FallbackTriggeredError for the upper layer to switch models; ③ shouldRetry decides whether the error is retryable (rate limit, gateway 5xx, etc.); ④ max_tokens context-overflow errors call parsetMaxTokensContextOverflowError to compute available space, lowering the next round's max_tokens to continue; ⑤ persistent retry mode uses long backoff for 429/529 (up to 5 minutes, capped at 6 hours), yielding an api_retry system message every 30s so the upper layer/user knows it is waiting.

Backoff duration prefers the server's retry-after header, otherwise exponential backoff (500ms × 2^n, capped at 32s). Before each retry it yields a SystemAPIErrorMessage, which in QueryEngine.submitMessage is converted into an api_retry SDK message for the frontend. The client instance is re-fetched after auth errors, and stale connections (ECONNRESET/EPIPE) close keep-alive and retry.

VI. Tool Execution Loop

When the model produces tool_use blocks in the streaming response, queryLoop marks needsFollowUp=true. Tool execution has two paths: ① StreamingToolExecutor — while the model is still streaming, already-determined tool_use blocks are thrown in and run in parallel (only concurrency-safe read-only tools), getCompletedResults yields results as they arrive; after the stream, getRemainingResults cleans up. ② runTools — serial/concurrent batch execution when the streaming executor is closed: partitionToolCalls splits tool calls into batches, concurrency-safe batches use runToolsConcurrently, write batches use runToolsSerially. Each tool result is pushed into toolResults as a user message (with tool_result blocks) and fed back to the model as messages in the next round.

Permissions are enforced here for real: canUseTool (wrapped in submitMessage to record denials) is called and decides allow/deny/ask; denied tools produce a tool_result with tool_use_error, which the model sees next round. Interruption handling: if the abort signal fires during streaming/tool execution, the executor generates synthetic tool_results for queued-but-not-finished tools (avoiding unpaired tool_use without tool_result that would cause API 400).

VII. Context Compression Chain

Compression is not a single mechanism but four processes run in fixed order at the top of each loop, not mutually exclusive (tokens freed by an earlier step may make later steps no-ops): ① snip (HISTORY_SNIP feature gate) — removes "zombie" messages and expired markers, used by long SDK sessions to limit memory growth. ② microcompact — replaces large tool_result content (images / long output) by tool_use_id with placeholders, based on time or cached edits; cached MC delays boundary messages until after the API response, using real cache_deleted_input_tokens. ③ context-collapse — folds history paragraphs into granular summaries (read-time projection, originals not deleted), and when overflowing recoverFromOverflow drains already-staged folds. ④ autocompact — threshold-triggered (when context nears the window limit) complete forked-agent summary, compressing a whole segment of history into one summary message + attachments, producing a compact_boundary. autocompact has a circuit breaker: after N consecutive failures it stops retrying.

In addition to these four "active" compression methods, there is reactive compact — triggered only when the model's streaming response returns prompt-too-long or media-too-large errors (errors are held during streaming and one reactive compact is attempted at wrap-up). After any successful compression, a post-compact message is yielded (summary + attachments + hook results), and messagesForQuery is replaced with the compressed array; the next round continues from the compressed context. task_budget deducts the final pre-compression context token count during compression.

VIII. Token / Cost Counting

Token and cost accumulation happens in the message_delta branch of claude.ts:queryModel: each time a message_delta with usage is received, first updateUsage accumulates this message's usage, then calculateUSDCost computes this message's USD cost, then addToTotalSessionCost adds it to the session ledger. QueryEngine.submitMessage calls updateUsage / accumulateUsage on message_start/message_delta/message_stop stream_events, accumulating each message's usage into totalUsage, which ends up in the result message. cost-tracker.ts maintains per-model usage (input/output/cacheRead/cacheCreation/webSearch/costUSD) and exposes getTotalCost / getModelUsage for results and budget checks.

Token estimation (tokenEstimation.ts) is used for compression threshold decisions and blocking-limit checks: without calling the API, roughTokenCountEstimationForMessages produces a rough estimate (by bytes/type conversion), and both autocompact's shouldAutoCompact and the blocking limit check at the top of queryLoop rely on it. Note: tokens freed by snip must be explicitly subtracted from the estimate (snipTokensFreed); otherwise snip may have already compressed context below the threshold, but stale usage still shows above threshold, causing a false blocking trigger.

  • State machine view: queryLoop's state.transition.reason (next_turn / collapse_drain_retry / reactive_compact_retry / max_output_tokens_escalate / max_output_tokens_recovery / stop_hook_blocking / token_budget_continuation) is an implicit state machine — every continue point carries a reason, making it easy for tests to assert "which recovery path was triggered."
  • Permission view: canUseTool is wrapped in submitMessage to record denials; it is called during tool execution; deny does not throw but produces a tool_use_error for the model to see; permission context (allowedTools, additionalWorkingDirectories) is updated into AppState after user input processing.
  • Lifecycle view: QueryEngine is one instance per session, state persists across turns; submitMessage is once per turn; queryLoop is one state per round; withRetry is multiple attempts per request; streaming is one stream per request. The abort signal (abortController) runs through every layer, each with its own abort handling (stream fills synthetic results, tools fill synthetic results, stop hooks are skipped).
  • Transcript persistence: user messages are written to transcript before entering query (so kill-resume doesn't fail); assistant messages are written fire-and-forget (to avoid blocking message_delta); compact_boundary flushes first to preserve the tail of the retained segment; result is force-flushed before yield in cowork/eager modes.
  • Non-streaming fallback: streaming empty response, idle timeout, proxy returning non-SSE body, etc., all trigger executeNonStreamingRequest — using messages.create(stream:false) to retry, default timeout 300s (120s for remote), then the result goes through the same usage/cost accumulation path.
