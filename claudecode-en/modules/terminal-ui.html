Terminal UI Layer (React + Ink)

Everything the user sees in the terminal—the input box, streaming replies, tool-call cards, permission popups, diff views—is rendered by this layer. It is a single-page application running inside the terminal, built with React + Ink: the user types → the input component updates local state → on submit the user message is appended to the message array → a query is started → streaming events continuously append assistant messages and tool progress → every frame Ink renders the React tree into character art and writes it to stdout. The main axis of the data flow is the "unidirectional flow of the message array": almost every state change eventually lands on the messages array, and the rendering layer is simply a projection of it.

Input
User keyboard input (text, paste, slash commands, vim keys) + streaming events from the query engine (text deltas, tool-call blocks, thinking blocks, progress ticks)
Output
A continuously refreshed character interface on the terminal screen: a conversation transcript + an input box at the bottom, interspersed with permission popups, tool-result cards, and spinner animations

1. From Keystroke to Submitted Query

The user types in the bottom PromptInput. The input box maintains its own buffer and cursor state (including a vim-mode state machine). When Enter is pressed, PromptInput invokes the onSubmit passed down from the REPL. onSubmit does not send the request directly; it does a few things first: if it is a slash command marked immediate, it bypasses the queue and executes immediately (e.g. /btw, /config, and other local popup commands); otherwise the input is handed to handlePromptSubmit for unified processing—it expands pasted references, determines whether it is a normal conversation or a slash command, and queues when necessary (when Claude is busy the input is held and submitted after the current turn ends). Finally, for inputs that need to send a request, handlePromptSubmit calls onQuery, which appends the user message to messages and starts the query engine.

2. Rendering Streaming Replies

Once the query engine starts returning streaming events, each delta goes through onQueryEvent into handleMessageFromStream. This function translates raw events into mutations on the messages array: a normal delta is appended; a compact boundary message replaces the whole array (keeping a small amount of history for scrollback); a progress tick replaces the previous tick of the same tool in place (so sleep/bash ticks do not bloat the array to tens of thousands of entries per second). setMessages is a special wrapper: it synchronously updates messagesRef (so downstream logic reads the latest value immediately, without waiting for React batching), then hands the computed value to React's real setState. The Messages component receives the new array and uses VirtualMessageList to virtual-scroll and render only the rows in the viewport, then Ink reconciler → yoga layout → paint to the Screen buffer → write to stdout.

3. Tool Permission Popups

When the model wants to invoke a tool that requires confirmation, the query engine calls canUseTool (provided by useCanUseTool). It first runs hasPermissionsToUseTool to see whether the configuration already allows/denies it; if the result is allow it is let through directly, if deny it is rejected directly. Only when the result is ask does a popup become necessary: first the coordinator auto-check, swarm worker agent, and bash classifier speculative decision (2-second race) are tried; if none of those yield a conclusion, it falls through to handleInteractivePermission, which pushes a ToolUseConfirm into toolUseConfirmQueue. The REPL rendering layer sees the non-empty queue and mounts the PermissionRequest component in the focusedInputDialog === 'tool-permission' branch. After the user chooses, onDone dequeues the front item, resolves the pending Promise, and the query engine continues or aborts.

4. Turn Lifecycle and State Machine

A turn's loading state is driven by queryGuard (a tiny state machine: idle → running). isLoading is no longer a simple React state; it is derived from the queryGuard external store so that the "currently processing" fact is not delayed by React batching. When onQuery enters, tryStart() atomically flips idle to running and returns a generation number; if it is already running, it rejects and enqueues the input. In the finally block, queryGuard.end(thisGeneration) atomically checks the generation and flips back to idle, then resetLoadingState cleans up temporary state and the onTurnComplete callback notifies the outside. If the user cancels midway with Esc, forceEnd forcibly ends the turn and, when conditions are met, automatically rolls back the user message and restores the input box content.

5. Global State and Re-rendering

Global state lives in a minimal Zustand-style store (createStore, under 35 lines): getState, setState, subscribe. setState uses Object.is to skip no-op changes, then calls onChangeAppState for side effects (syncing the permission mode to CCR, persisting verbose/expandedView to global config, clearing auth cache), and finally notifies all subscribers. Components subscribe to slices via useAppState(selector) and only re-render when the selector's return value changes. This prevents the giant REPL component from re-computing on unrelated field changes. The messages array itself is local React state in REPL (not the global store), but it is synchronously mirrored via messagesRef so closures like onQuery and getToolUseContext can read the latest value immediately.

  • setMessages is the source of truth: it first writes messagesRef synchronously, then passes the computed value to React. This bypasses React batching latency—after setMessages in onQuery, reading messagesRef.current immediately returns the freshly appended user message without waiting for commit. Comments explicitly call this the Zustand pattern: the ref is the truth, React state is the projection.
  • Progress ticks are replaced in place: sleep/bash emit one tick per second; blindly appending would make the messages array grow to 13k entries and the transcript file 120MB. onQueryEvent detects a tick of the same kind and replaces the last one, and useLogMessages also skips disk writes when the length has not changed.
  • Virtual scrolling is the watershed: in transcript mode, when virtual scrolling is enabled (default fullscreen), FullscreenLayout + AlternateScreen wraps the UI and VirtualMessageList only mounts rows inside the viewport; when disabled it falls back to a 30-row cap + native terminal scrollback. Both modes share the same scrollRef and are mutually exclusive, so two ScrollBoxes are never mounted at the same time.
  • Immediate commands bypass the queue: slash commands marked immediate: true (/btw, /model, /mcp) execute immediately even while Claude is busy, rendered as a fixed overlay at the bottom rather than content in the scrollback—so they do not get pushed around by the main conversation's streaming refreshes.
  • Permission mode sync single throat: onChangeAppState is the only synchronization point for permission mode changes. Previously more than 8 places changed the mode but only 2 notified CCR; now the hook is on the diff, so any path that changes the mode automatically syncs to CCR and the SDK state stream.
  • vim is an independent state machine: useVimInput hands keystrokes to transition() in vim/transitions.ts, which decides the next state and action based on the current state (idle/count/operator/find, etc.) and the input—a small independent machine beside the main data-flow axis.
  • Rendering pipeline: Ink rendering goes through React reconciler → yoga layout computes dimensions → renderNodeToOutput draws nodes into the Screen character grid → diffs against the previous frame's output and writes the delta to stdout. renderToScreen also provides an isolated rendering entry point for single messages, used by the search feature—rendering one message and scanning character positions takes about 1–3ms.
