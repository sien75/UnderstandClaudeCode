MCP Exploration Server (lets external AI clients browse this repo)

This is a stand-alone MCP (Model Context Protocol) server with a straightforward purpose: any MCP-compatible AI client (Claude Desktop, Cursor, VS Code Copilot, etc.) that connects to it can browse this repository's source code like flipping through a book—listing directories, reading files, searching by regex, viewing architecture overviews, and even listing all built-in "tools" and "slash commands" of Claude Code and reading their implementations. In other words, it turns a local source tree into a set of AI-callable tools, so a remote client can "look at the code" without cloning the repository.

It is not coupled to the main Claude Code project: the main project is the object it "displays", while the server itself only exposes capabilities externally. There are two deployment forms—local STDIO (the client spawns a node process directly) and remote HTTP (deployed on Railway/Vercel/Docker, shared by multiple clients via Bearer Token auth). Three transport protocols coexist: STDIO, Streamable HTTP (modern MCP), and Legacy SSE (for older clients).

Input
MCP requests from an external AI client: invoking a tool (e.g. read_source_file with a relative path), reading a resource URI, or rendering a prompt template. In HTTP mode a Bearer Token is also attached for auth.
Output
The corresponding tool execution result: file content snippets, directory listings, search hit lines, architecture JSON, tool/command lists, etc., packaged in MCP protocol format and returned to the client for display or for feeding into its own LLM.

Entry Points and Transport Layer

The user (external AI client) can trigger three entry paths, all converging on the same "shared MCP server". The local scenario is simplest: the client writes a command in its config that spawns node dist/index.js, and the process exchanges JSON-RPC messages with the client over STDIO. The remote scenario is a persistent Express process; the client sends HTTP requests to /mcp (modern) or /sse + /messages (legacy protocol).

  client["External AI client initiates request"] -->|"STDIO spawns node process"| stdio-entry["index.ts main function"]
  client -->|"POST/GET /mcp"| http-entry["http.ts main function"]
  client -->|"GET /sse + POST /messages"| http-entry
  stdio-entry --> validate-stdio["validateSrcRoot validates source root"]
  http-entry --> validate-http["validateSrcRoot validates source root"]
  http-entry --> auth-mw{"MCP_API_KEY set?"}
  auth-mw -->|No| skip-auth["Allow directly"]
  auth-mw -->|Yes| check-bearer{"Authorization: Bearer matches?"}
  check-bearer -->|No| unauth["Return 401 Unauthorized"]
  check-bearer -->|Yes| skip-auth
  validate-stdio --> create-server-stdio["createServer() creates shared MCP instance"]
  skip-auth --> create-server-http["createServer() creates shared MCP instance"]
  create-server-stdio --> connect-stdio["StdioServerTransport connects"]
  create-server-http --> route-mcp{"Route dispatch"}
  route-mcp -->|"/mcp POST"| stream-http["StreamableHTTP transport handles"]
  route-mcp -->|"/mcp GET"| stream-http
  route-mcp -->|"/sse"| legacy-sse["SSE transport establishes/reuses session"]
  route-mcp -->|"/messages POST"| legacy-sse
  route-mcp -->|"/health"| health["Return health status JSON"]
  route-mcp -->|"/mcp DELETE"| delete-sess["Close and clean up session"]

The STDIO entry point is only 25 lines and does three things before exiting: validate the source root, create the shared MCP instance, and attach StdioServerTransport. The HTTP entry point is more complex; its core is three middleware/route pieces: auth, health check, and session management for the two transport protocols.

  • Auth is optional: it only takes effect when MCP_API_KEY is set, and /health is always exempt (convenient for deployment platform health probes).
  • Streamable HTTP uses a Map<sessionId, transport> to maintain multiple sessions in memory; POST /mcp creates/reuses a session, GET /mcp pulls the SSE stream, and DELETE /mcp actively cleans up.
  • Legacy SSE follows the old protocol: the client first GET /sse to establish a long-lived connection and obtain a sessionId, then sends subsequent commands via POST /messages?sessionId=xxx.
  • Every HTTP session calls createServer() once—meaning each session gets its own independent MCP instance, isolated from the others; STDIO has only one instance for the whole run.

Shared MCP Server: How Requests Arrive, Read Source, and Return

All transport protocols eventually hand JSON-RPC requests to the Server instance produced by the same factory function createServer(). At construction time this instance registers handlers for three categories of capabilities: resources (passively readable data, fetched by URI), tools (callable functions with input schemas), and prompts (prebuilt prompt templates, injected with source context and handed to the client's LLM). The client first calls list-style interfaces to get the menu, then calls the specific call/read/get to retrieve actual content.

There are only three underlying source-reading primitives: fs.readFile (read a single file), fs.readdir (list a directory), and walkFiles (recursively walk the whole tree for search). Every user-supplied path first passes through safePath()—it uses path.resolve to join the relative path under SRC_ROOT, then verifies the result still starts with SRC_ROOT; otherwise it returns null. This blocks directory-traversal attacks like ../../etc/passwd and is the service's sole permission boundary.

Input
Three list requests (ListTools / ListResources / ListPrompts) only return static menus; three call requests (CallTool / ReadResource / GetPrompt) carry tool name/URI/prompt name plus parameters, where parameters are usually a relative path or tool name.
Output
Responses in MCP protocol format: tools return content: [{type:"text", text:...}]; resources return contents: [{uri, mimeType, text}]; prompts return messages: [{role:"user", content:{type:"text", text}}]. Text content is the original file, directory listing, search hit lines, or assembled prompt.

Main Data Flow: Tool Calls (CallTool)

When a client invokes a tool, the request carries name and arguments; the server dispatches via a switch to one of eight tools. The core of each tool is "locate file → read content → assemble text return"; only the locate logic differs.

Resource Reading Flow (ReadResource)

Resources are read-only data "fetched directly by URI". The server registers 3 fixed URIs and 1 template URI claude-code://source/{path}. The template URI also goes through safePath, sharing the same secure path as the read_source_file tool, only at the protocol layer differently.

  tools -->|No| cmds?{"uri = commands?"}

Prompt Rendering Flow (GetPrompt)

The 5 prompt templates serve the purpose of "pre-feeding source context for the client's LLM". After the server receives the request, it locates the relevant source files by prompt name, reads them, and stuffs them into a preset question text, assembling the whole thing into a single user message returned to the client. What the client gets is a prompt ready to feed to its LLM.

graph TD
  getprompt["Client GetPrompt(name, args)"] --> pswitch{"Dispatch by prompt name"}
  pswitch -->|"explain_tool"| p-tool["Locate tools/toolName main file"]
  pswitch -->|"explain_command"| p-cmd["Locate commands/commandName"]
  pswitch -->|"architecture_overview"| p-arch["Read README + scan top level + tool/command lists"]
  pswitch -->|"how_does_it_work"| p-how["Look up featureMap for related paths"]
  pswitch -->|"compare_tools"| p-cmp["Read main source of two tools"]
  p-tool --> p-tool-read["Read main file source"]
  p-tool-read --> p-tool-out["Assemble 6-dimension analysis question"]
  p-cmd --> p-cmd-find["Try three candidates: directory/single file"]
  p-cmd-find --> p-cmd-out["Assemble command explanation question"]
  p-arch --> p-arch-out["Assemble architecture overview question"]
  p-how --> p-how-map{"feature in featureMap?"}
  p-how-map -->|Yes| p-how-read["Read files/list dirs by path (first 200 lines)"]
  p-how-map -->|No| p-how-empty["Prompt to use search/read to find it yourself"]
  p-how-read --> p-how-out["Assemble subsystem explanation question"]
  p-how-empty --> p-how-out
  p-cmp --> p-cmp-read["Read main source of both tools"]
  p-cmp-read --> p-cmp-out["Assemble comparison analysis question"]
  • how_does_it_work has a built-in featureMap that maps human colloquialisms ("permission system", "mcp client", "bridge") to a set of source paths—this is the only place in the service that carries "business knowledge"; all other tools are pure filesystem operations.
  • All prompts return a single user message with no system message; the client can overlay its own system prompt.
  • When reading source, how_does_it_work only takes the first 200 lines to avoid overly long prompts; the read_source_file tool supports precise startLine/endLine slicing so the client can page through itself.

Auxiliary View: Permission Boundary

The service's permission model is very thin, with only two layers: HTTP-layer Bearer Token (optional), and filesystem-layer safePath (mandatory).

graph TD
  req["Incoming request"] --> http-layer{"HTTP mode & MCP_API_KEY set?"}
  http-layer -->|Yes| bearer{"Authorization: Bearer matches?"}
  http-layer -->|No| pass-http["Allow"]
  bearer -->|No| deny-401["401 Unauthorized"]
  bearer -->|Yes| pass-http
  pass-http --> tool-call["Enter tool/resource handler"]
  tool-call --> path-arg{"Parameter contains relative path?"}
  path-arg -->|Yes| safepath["safePath: resolve + startsWith validation"]
  path-arg -->|No| direct["Scan fixed directories directly"]
  safepath -->|Escapes SRC_ROOT| deny-null["Return null → throw"]
  safepath -->|Inside SRC_ROOT| allow-read["Allow fs.readFile/readdir"]
  • /health (and Vercel's /api) is always exempt from auth for deployment platform health probes.
  • STDIO mode has no HTTP auth—the client that can spawn this process is already considered trusted.
  • safePath uses resolved.startsWith(SRC_ROOT); note that SRC_ROOT itself is an absolute path from path.resolve with no trailing slash, so paths exactly equal to SRC_ROOT also pass.
  • There is no per-tool permission granularity—any authenticated client can call all 8 tools and read any file under SRC_ROOT. This is a read-only exploration service, so the model is sufficient.

Auxiliary View: Session Lifecycle and State Machine

In HTTP mode each session has its own independent lifecycle. Streamable HTTP identifies sessions via the mcp-session-id header, Legacy SSE uses the query parameter sessionId. Both maintain transports in an in-memory Map; they are lost when the process restarts.

graph TD
  new-req["New request (no sessionId)"] --> create-sess["createServer + new transport"]
  create-sess --> gen-id["Generate sessionId (UUID)"]
  gen-id --> store-map["Store in transports Map"]
  store-map --> state-active["State: active"]
  reuse-req["Request with sessionId"] --> lookup{"Exists in Map?"}
  lookup -->|Yes| reuse-sess["Reuse transport to handle"]
  lookup -->|No| reject-400["400 Invalid/Missing session"]
  reuse-sess --> state-active
  state-active -->|"Client DELETE /mcp"| close-sess["transport.close()"]
  state-active -->|"transport.onclose fires"| del-map["Remove from Map"]
  close-sess --> del-map
  del-map --> state-gone["State: destroyed"]
  state-gone -->|"Client comes again"| reject-400
  • Each session independently calls createServer(), meaning each session has its own MCP Server instance, but all read the same on-disk source underneath—stateless business, naturally concurrent.
  • Vercel deployment is serverless: the Map does not persist across invocations, so Streamable HTTP session reuse basically fails; legacy SSE and stateless tool calls still work. The README explicitly recommends Railway/Docker for production.
  • STDIO mode has no "session" concept; there is one transport for the whole run, and it ends when the process exits.

Deployment Forms

The same createServer() core wrapped in different shells becomes different deployment options: local STDIO (src/index.ts), remote HTTP (src/http.ts), and Vercel serverless (api/vercelApp.ts exports an Express app). The Dockerfile copies the whole repository into the image, builds dist/, then starts with node dist/http.js, with CLAUDE_CODE_SRC_ROOT=/app/src pointing to the source copy inside the image. Railway uses the same Dockerfile with healthcheckPath=/health for automatic health probes.

  • The Docker image bundles both mcp-server/dist and the main project's src/—the "source" this server explores is baked into the image at deployment time and does not depend on the host filesystem.
  • src/index.ts.new is a same-name copy of index.ts (content identical), likely leftover from development and not part of the build.
