MCP Exploration Server (Let external AI clients browse this repository's source code)

This is a standalone MCP (Model Context Protocol) server with a straightforward purpose: after any MCP-compatible AI client (Claude Desktop, Cursor, VS Code Copilot, etc.) connects to it, the client can browse this repository's source code like flipping through a book — listing directories, reading files, searching by regex, viewing architecture overviews, and even listing all built-in Claude Code "tools" and "slash commands" and reading their implementations. In other words, it turns a local source tree into a set of AI-callable tools, so a remote client can "see the code" without cloning the repository.

It is not coupled to the main Claude Code project: the main project is the object it "displays," while the server itself only exposes capabilities externally. It has two deployment forms — local STDIO (the client directly spawns a node process to communicate) and remote HTTP (deployed on Railway/Vercel/Docker, shared by multiple clients authenticated via Bearer Token). Three transport protocols coexist: STDIO, Streamable HTTP (modern MCP), and Legacy SSE (for older clients).

Input
MCP requests from an external AI client: calling a tool (e.g., read_source_file with a relative path), reading a resource URI, or rendering a prompt template. In HTTP mode, a Bearer Token is also included for authentication.
Output
The execution result of the corresponding tool: file content snippets, directory listings, search hit lines, architecture JSON, tool/command lists, etc., wrapped in MCP protocol format and returned to the client, where it is shown to the user or fed to the client's own LLM.

Entry points and transport layer

The user (external AI client) triggers one of three paths, all converging on the same "shared MCP server." The local scenario is simplest: the client writes a command in its config to spawn node dist/index.js, and the process exchanges JSON-RPC messages with the client over STDIO. The remote scenario is a long-running Express process; clients send HTTP requests to /mcp (modern) or /sse + /messages (legacy protocol).

The STDIO entry point is only 25 lines and does three things before exiting: validate the source root directory, create the shared MCP instance, and attach StdioServerTransport. The HTTP entry point is more complex, with three core middleware/routing parts: authentication, health check, and session management for both transport protocols.

  • Authentication is optional: it only takes effect when MCP_API_KEY is set, and /health is always exempt (convenient for deployment platform liveness probes).
  • Streamable HTTP maintains multiple sessions in memory using a Map<sessionId, transport>; POST /mcp creates/reuses a session, GET /mcp pulls the SSE stream, and DELETE /mcp actively cleans up.
  • Legacy SSE follows the old protocol: the client first calls GET /sse to establish a long connection and obtain a sessionId, then sends commands via POST /messages?sessionId=xxx.
  • Each HTTP session calls createServer() once — meaning each session gets an independent MCP instance, so they don't interfere with each other; STDIO has only one instance throughout.

Shared MCP server: how requests arrive, how source code is read, how results are returned

All transport protocols ultimately hand JSON-RPC requests to the Server instance produced by the same factory function createServer(). At construction, this instance registers handlers for three categories of capabilities at once: resources (passively readable data fetched by URI), tools (callable functions with input schemas), and prompts (prebuilt prompt templates, injected with source context and handed to the client's LLM). The client first calls the list family of methods to get the menu, then calls the specific call/read/get to fetch actual content.

All source code reading boils down to only three low-level actions: fs.readFile (read a single file), fs.readdir (list a directory), and walkFiles (recursively walk the tree for search). Every user-supplied path must pass through safePath() — it uses path.resolve to join the relative path under SRC_ROOT, then checks that the result still starts with SRC_ROOT; otherwise it returns null. This blocks directory traversal attacks such as ../../etc/passwd and is the service's only permission boundary.

Input
Three list requests (ListTools / ListResources / ListPrompts) only return static menus; three call requests (CallTool / ReadResource / GetPrompt) carry a tool name/URI/prompt name plus parameters, where the parameter is usually a relative path or tool name.
Output
Response in MCP protocol format: tools return content: [{type:"text", text:...}]; resources return contents: [{uri, mimeType, text}]; prompts return messages: [{role:"user", content:{type:"text", text}}]. The text content is the raw file, directory listing, search hit lines, or assembled prompt.

Main data flow: tool call (CallTool)

When a client calls a tool, the request carries name and arguments, and the server dispatches via a switch to one of 8 tools. The core of each tool is "locate file → read content → assemble into text and return," differing only in the location logic.

Resource read flow (ReadResource)

Resources are read-only data fetched directly by URI. The server registers 3 fixed URIs and 1 template URI claude-code://source/{path}. The template URI also goes through safePath, sharing the same secure path as the read_source_file tool, just at the protocol layer.

Prompt rendering flow (GetPrompt)

The 5 prompt templates serve the purpose of "preparing source context for the client's LLM." After the server receives the request, it locates the relevant source files by prompt name, reads them, and stuffs them into a preset question text, assembling the whole thing into a user message to return. What the client gets is a prompt ready to feed directly to an LLM.

graph TD
  getprompt["Client GetPrompt(name, args)"] --> pswitch{"Dispatch by prompt name"}
  pswitch -->|"explain_tool"| p-tool["Locate tools/toolName main file"]
  pswitch -->|"explain_command"| p-cmd["Locate commands/commandName"]
  pswitch -->|"architecture_overview"| p-arch["Read README + scan top-level + tool/command lists"]
  pswitch -->|"how_does_it_work"| p-how["Look up featureMap for relevant paths"]
  pswitch -->|"compare_tools"| p-cmp["Read main source of two tools"]
  p-tool --> p-tool-read["Read main file source"]
  p-tool-read --> p-tool-out["Assemble 6-dimension analysis question"]
  p-cmd --> p-cmd-find["Try three candidates: directory/single file"]
  p-cmd-find --> p-cmd-out["Assemble command explanation question"]
  p-arch --> p-arch-out["Assemble architecture overview question"]
  p-how --> p-how-map{"feature in featureMap?"}
  p-how-map -->|"Yes"| p-how-read["Read files/list dirs per path (first 200 lines)"]
  p-how-map -->|"No"| p-how-empty["Prompt to use search/read to find it yourself"]
  p-how-read --> p-how-out["Assemble subsystem explanation question"]
  p-how-empty --> p-how-out
  p-cmp --> p-cmp-read["Read main source of both tools"]
  p-cmp-read --> p-cmp-out["Assemble comparison analysis question"]
  • how_does_it_work has a built-in featureMap that maps human phrases ("permission system", "mcp client", "bridge") to a set of source paths — this is the only place in the service that carries "business knowledge"; all other tools are pure filesystem operations.
  • All prompts return a single user message with no system message; the client can layer its own system prompt on top.
  • When reading source, how_does_it_work only takes the first 200 lines to avoid overly long prompts; the read_source_file tool supports precise slicing by startLine/endLine, so clients can paginate themselves.

Supporting view: permission boundary

The service's permission model is very thin, with only two layers: HTTP-layer Bearer Token (optional) and file-layer safePath (mandatory).

graph TD
  req["Incoming request"] --> http-layer{"HTTP mode & MCP_API_KEY set?"}
  http-layer -->|"Yes"| bearer{"Authorization: Bearer matches?"}
  http-layer -->|"No"| pass-http["Allow"]
  bearer -->|"No"| deny-401["401 Unauthorized"]
  bearer -->|"Yes"| pass-http
  pass-http --> tool-call["Enter tool/resource handler"]
  tool-call --> path-arg{"Parameter contains relative path?"}
  path-arg -->|"Yes"| safepath["safePath: resolve + startsWith validation"]
  path-arg -->|"No"| direct["Scan fixed directories directly"]
  safepath -->|"Traverses out of SRC_ROOT"| deny-null["Return null → throw"]
  safepath -->|"Inside SRC_ROOT"| allow-read["Allow fs.readFile/readdir"]
  • /health (and Vercel's /api) is always exempt from authentication for deployment platform liveness probes.
  • STDIO mode has no HTTP authentication — any client that can spawn this process is already considered trusted.
  • safePath uses resolved.startsWith(SRC_ROOT) for the check; note that SRC_ROOT itself is an absolute path produced by path.resolve, with no trailing slash, so paths exactly equal to SRC_ROOT also pass.
  • There is no per-tool permission differentiation — any authenticated client can call all 8 tools and read any file under SRC_ROOT. This is a read-only exploration service that doesn't write or modify, so the model is sufficient.

Supporting view: session lifecycle and state machine

In HTTP mode, each session has an independent lifecycle. Streamable HTTP uses the mcp-session-id header to identify sessions; Legacy SSE uses the query parameter sessionId. Both maintain transports in an in-memory Map, lost when the process restarts.

graph TD
  new-req["New request (no sessionId)"] --> create-sess["createServer + new transport"]
  create-sess --> gen-id["Generate sessionId (UUID)"]
  gen-id --> store-map["Store in transports Map"]
  store-map --> state-active["State: active"]
  reuse-req["Request with sessionId"] --> lookup{"Exists in Map?"}
  lookup -->|"Yes"| reuse-sess["Reuse transport to handle"]
  lookup -->|"No"| reject-400["400 Invalid/Missing session"]
  reuse-sess --> state-active
  state-active -->|"Client DELETE /mcp"| close-sess["transport.close()"]
  state-active -->|"transport.onclose fires"| del-map["Remove from Map"]
  close-sess --> del-map
  del-map --> state-gone["State: destroyed"]
  state-gone -->|"Client returns"| reject-400
  • Each session independently calls createServer(), meaning each session has its own MCP Server instance, but they all read the same disk source underneath — stateless business logic, naturally supporting concurrency.
  • Vercel deployment is serverless: the Map doesn't persist across invocations, so Streamable HTTP session reuse essentially fails; legacy SSE and stateless tool calls still work. The README explicitly recommends Railway/Docker for production.
  • STDIO mode has no "session" concept; there is one transport throughout, ending when the process exits.

Deployment forms overview

The same createServer() core, wrapped in different shells, becomes different deployment methods: local STDIO (src/index.ts), remote HTTP (src/http.ts), Vercel serverless (api/vercelApp.ts exports an Express app). The Dockerfile copies the entire repository into the image, builds dist/, then starts with node dist/http.js, with CLAUDE_CODE_SRC_ROOT=/app/src pointing to the source copy inside the image. Railway uses the same Dockerfile, configured with healthcheckPath=/health for automatic liveness probes.

  • The Docker image packages both mcp-server/dist and the main project's src/ — the "source code" explored by this server is baked into the image at deployment time and does not depend on the host filesystem.
  • src/index.ts.new is a same-name copy of index.ts (content identical), suspected to be a leftover from development and not involved in the build.
