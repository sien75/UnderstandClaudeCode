Background Forked Sub-Agents and Remote Agent Execution

This module covers two things. First, after each round of the Claude Code main conversation, the system "forks" several sub-agents in the background — they reuse the main thread's prompt cache and quietly do non-intrusive work, such as consolidating memory, generating next-sentence suggestions, maintaining Magic Doc, writing progress summaries for coordinator sub-agents. Second, when the session runs inside a remote CCR container, the local CLI subscribes to remote session events over WebSocket, forwards permission requests, tunnels outbound curl/gh traffic via CONNECT-over-WebSocket to the CCR ingress, where the server performs MITM TLS and injects credentials. The common thread of both parts is "data flow": the core of forked sub-agents is the cache-hit data flow, while the core of remote execution is the bidirectional flow of messages and bytes among the CLI, WebSocket, and CCR container.

Input
After each round of the main conversation, the REPLHookContext (containing systemPrompt, tools, messages, toolUseContext) — all background fork sub-agents draw from this; on the remote side, it is the SDK message stream pushed by the CCR server, plus local CONNECT requests from curl/gh etc. to the 127.0.0.1 tunnel port.
Output
Background side: consolidated memory files, Magic Doc updates, session memory markdown, prompt-suggestion text, coordinator summaries — all written back to disk or AppState without blocking the main conversation. Remote side: local REPL renders the remote agent's streaming messages; permission requests are decided locally and sent back to CCR; child-process traffic is transparently proxied to the real upstream.
  • The essence of "forked": initiate a new query with a system/tools/messages prefix identical to the main thread's, so Anthropic API's prompt cache hits, making background work almost a "pay only for output tokens" cheap operation.
  • All background fork sub-agents share a common pattern: tool definitions remain in the request (to keep the cache key consistent), but are allowed or denied on demand via the canUseTool callback — read-only tools are all denied, and only tools that need to modify specific files are allowed.
  • The remote side is fail-open across the board: if the WebSocket disconnects there is a reconnection budget, and if reconnection fails it degrades to "session ended"; if any step of upstreamproxy fails, the proxy is disabled and the session is never dragged down.

I. Fork Mechanism Core: runForkedAgent + CacheSafeParams

After each round of the main conversation, handleStopHooks stores the current round's CacheSafeParams into a module-level lastCacheSafeParams slot. Background fork sub-agents (promptSuggestion, postTurnSummary, /btw, etc.) read from this slot directly, without each penetrating parameters — this is the first guarantee of cache hit. CacheSafeParams carries five fields that affect the cache key: systemPrompt, userContext, systemContext, toolUseContext (containing tools/model/options), and forkContextMessages (message prefix).

thinking config is also part of the cache key, so the maxOutputTokens parameter is specially warned: in claude.ts it clamps budget_tokens, which changes thinking config and breaks the cache. It is only allowed in scenarios that "do not need to share the parent's cache" (such as compact summaries).

Several business-critical details: createSubagentContext by default isolates all mutable state — readFileState is cloned, abortController is newly created (but chained to the parent so parent abort propagates), and all mutation callbacks are set to no-op. This way fork file writes do not pollute the parent thread's file-state view. The exception is setAppStateForTasks: it must penetrate to the root store, otherwise background bash tasks would never register or be killable, becoming PPID=1 zombies. contentReplacementState is cloned rather than newly created by default — when the fork processes tool_use_ids carried in parent messages, a fresh state would treat them as unseen, make different replacement decisions, and the wire prefix would differ, causing a cache miss; cloning makes the same decisions and hits the cache.

II. autoDream: Time + Session Count Dual Gate for Memory Consolidation

User perspective: nothing to do. After each round of conversation, autoDream secretly checks "how long since the last memory consolidation" and "how many sessions have accumulated during that time." When both thresholds are crossed, it forks a sub-agent running the /dream prompt to read the session transcript and merge/revise/trim memory files. This lets Claude orient quickly in new sessions by precipitating recent learning into persistent memory.

Gate order is "cheapest check first": first a stat reads the lock file mtime (=lastConsolidatedAt); after passing the time gate it scans the session directory, and after passing the session-count gate it finally tries to acquire the lock. Each gate failure returns directly, avoiding wasted downstream cost.

  fork --> fork-ok{"Fork succeeded?"}
  fork-ok -->|Yes| complete["completeDreamTask + appendSystemMessage 'Improved'"]
  fork-ok -->|No/abort| fail-check{"Was it a user kill?"}

  fail-check -->|No| fail["failDreamTask + rollbackConsolidationLock revert mtime"]

The lock design is careful: the lock file is lastConsolidatedAt — its mtime is the last consolidation time, and its body is the holder PID. Acquiring the lock means overwriting it with your own PID, then re-reading to verify "what I wrote is still my PID" (when two reclaims race, the last writer wins and the loser bails). On failure the mtime is rolled back to the pre-lock value so the time gate can pass next time. HOLDER_STALE_MS=1h guards against PID reuse: even if the PID is alive, after 1h it is considered stale and can be preempted. Crash scenario: mtime is stuck and the PID is dead; the next process sees a dead PID and reclaims directly.

III. Background Fork Service Cluster (AgentSummary / PromptSuggestion / MagicDocs / SessionMemory)

These four services share the same runForkedAgent pattern, but each does different work. Commonalities: all use CacheSafeParams to share the parent's cache, all use canUseTool callbacks to control permissions, all skipTranscript or record as needed. Differences lie in "when to fork, what the fork does, and to what extent tools are allowed."

PromptSuggestion is the heaviest logic in this set. It first performs a bunch of suppression checks (aborted, early session with fewer than 2 rounds, last message was an API error, parent cache cold start with more than 10k uncached tokens, permission pending, elicitation queue, plan mode, external user throttling); any hit suppresses and logs. After the fork, it goes through 12 content filters — done/meta_text/parentheses-wrapped/API error prefix/single-character too short/too long/multiple sentences/formatted/evaluative phrasing/Claude-voice ("Let me...") — filtering out "things the model might say wrong" so only text that genuinely looks like what the user would type is delivered.

speculation is the advanced version of PromptSuggestion: the fork doesn't just generate text, it pre-runs the likely next user input — using an overlay filesystem for copy-on-write isolation of writes, allowing only read-only tools and read-only bash, and aborting + recording boundary when it hits file edits / non-read-only bash / unknown tools. When the user accepts the speculation, changes in the overlay are copied back to the main directory and the speculated messages are injected into the main conversation. Key constraints: the fork uses createChildAbortController (chained to parent), so parent abort propagates; onMessage accumulates toolUseCount as messages arrive, and forces abort after more than 100 messages or 20 rounds.

AgentSummary is the progress summary under coordinator mode: every 30s it forks once using the main thread's CacheSafeParams, pulls the latest messages from the transcript, denies all tools, and generates a 3-5 word present-continuous summary like "Reading runAgent.ts". It deliberately does not pass forkContextMessages into the closure — each tick re-reads from getAgentTranscript, preventing the original fork messages from being pinned forever by the timer.

MagicDocs is the only one that does not use runForkedAgent — it uses runAgent (because it needs its own agent definition and restricted toolset). In the FileReadTool listener it detects the # MAGIC DOC: [title] header and the following italic instruction line, registers tracking; when the conversation is idle (no tool call in the previous round) it sequentially updates each tracked doc. canUseTool only allows Edit and file_path must exactly equal that doc's path.

SessionMemory maintains a session-level markdown note. The trigger condition is "token threshold and tool-call threshold" or "token threshold and no tool call in the previous round" — the token threshold is hard to prevent over-extraction. The fork is only allowed to Edit that one memory file.

IV. Remote Agent Execution: Message Flow and Permission Bridging

User perspective: connect to a remote CCR session in the local claude CLI, and what you see is the remote agent's conversation — typing, streaming output, and y/n on permission requests. Behind the scenes, RemoteSessionManager coordinates one WebSocket subscription plus one HTTP POST upstream channel.

Data flow has three message categories: SDKMessage (assistant/user/stream_event/system/result/tool_progress and other rendering messages), control_request (CCR asks "can this tool be used"), and control_response (CLI answers "yes/no"). SDKMessages are converted by sdkMessageAdapter into REPL internal Message types for rendering; control_request is routed by RemoteSessionManager to the onPermissionRequest callback and ultimately fed to the local permission UI as a synthetic AssistantMessage by remotePermissionBridge.

The SDK message adapter handles a subtlety: remote user messages may be tool results (content contains tool_result blocks). In direct-connect mode (convertToolResults=true) these must be converted to UserMessage to fold-render like local tool results; in CCR live mode they are ignored — because the local REPL has already added the user message itself. parent_tool_use_id is unreliable (the agent side normalizeMessage hardcodes it to null), so judgment relies on content shape. Success-type result messages are treated as noise and ignored (isLoading=false is enough), only error ones are displayed.

The key to permission bridging: the tool is not running locally at all (it runs in the CCR container), so the AssistantMessage required by ToolUseConfirm is synthetic — the request's tool_use_id/name/input are used to forge an assistant message. If CCR has a tool that the local CLI doesn't know (such as an MCP tool), createToolStub builds a minimal Tool stub and routes it to FallbackPermissionRequest.

V. SessionsWebSocket: Reconnection State Machine

This WebSocket is the lifeline of the remote session. The close code determines the next behavior — permanently give up, reconnect with budget, or reconnect instantly.

Key points: authentication uses HTTP header (Authorization: Bearer + anthropic-version), not an in-flight auth message — so onopen immediately means connected state. 4001 (session not found) is handled separately because CCR may briefly consider a session stale during compaction; 3 retry budgets are given to avoid false death. isSessionsMessage deliberately only checks "has a string type field" rather than whitelisting — when the backend sends a new message type the client won't silently drop it; downstream decides how to handle it. Bun uses globalThis.WebSocket, Node uses the ws package, and the two event styles are unified (addEventListener vs .on).

VI. upstreamproxy: CONNECT-over-WebSocket Relay

User perspective: when the remote CCR container's agent runs curl/gh/kubectl, the HTTPS traffic of these child processes is transparently proxied — through a local 127.0.0.1 CONNECT relay, over the WebSocket tunnel to the CCR server, where the server MITM's TLS and injects organizational credentials (such as DD-API-KEY), then forwards to the real upstream. The agent in the container sees an HTTPS_PROXY env var pointing to this local relay.

Why WebSocket instead of raw CONNECT: the CCR ingress is GKE L7 path-prefix routing, with no connect_matcher; sessions/tunnel already uses this pattern. Byte streams are encapsulated in UpstreamProxyChunk protobuf (single bytes field), compatible with the server's gateway.NewWebSocketStreamAdapter.

  tunnel --> ws-err{"ws.onerror?"}
  ws-err -->|Yes not established| send-502["Return 502 Bad Gateway"]

  tunnel --> ws-close{"ws.onclose?"}
  ws-close --> cleanup["cleanupConn: clear pinger, close ws"]

There is a pile of engineering detail for adversarial edge cases: Bun's sock.write() is partial-write; when the kernel buffer is full it silently drops bytes, so the Bun relay maintains a writeBuf tail queue itself and drains callbacks to flush. Node's net.Socket buffers unconditionally and needs no tail queue. TCP may combine the CONNECT header and ClientHello into one packet, so the pending buffer must catch "bytes after CRLFCRLF"; if socket.data fires again before ws.onopen, those bytes also go to pending. Once the established flag is set true (server's first byte received), plaintext 502 can never be written again — it would pollute the client's TLS stream, so only direct close is allowed. The closed flag prevents ws.onerror followed by ws.onclose from sock.end() twice.

Security design is also worth highlighting: prctl(PR_SET_DUMPABLE, 0) is called via libc FFI to prevent same-UID ptrace heap capture — this blocks prompt-injected gdb -p $PPID from stealing the session token. The token file is unlinked after the relay is up, so if CA download or listen fails the supervisor restart can still retry. The NO_PROXY list excludes Anthropic API, GitHub, and various package registries — MITM would break non-Bun runtimes (Python httpx/certifi doesn't trust the forged CA). Child process env inheritance has a branch: if the parent already set HTTPS_PROXY+SSL_CERT_FILE (meaning the parent is running a relay), the child CLI inherits them directly, because the token file has already been unlinked by the parent and the child cannot re-init a relay.

The entire upstreamproxy is a fail-open exemplar: env not set → disabled, token gone → disabled, CA download failed → disabled, relay listen failed → disabled; each step only logs at logForDebugging warn level, never throws, never drags down the session.

VII. Remote Session History Pagination

sessionHistory.ts provides paginated event history for remote sessions — for "view older messages." It uses HTTP GET /v1/sessions/{id}/events, axios with OAuth header + anthropic-beta: ccr-byoc-2025-07-29 + x-organization-uuid. Two cursor modes: fetchLatestEvents uses anchor_to_latest=true to get the latest page; fetchOlderEvents uses before_id to get the older page before a specified cursor. Each page has 100 items; has_more indicates older items exist. validateStatus: () => true + .catch(() => null) downgrades any HTTP error to a null return, leaving handling to the caller — also fail-open style. createHistoryAuthCtx prepares auth headers once and reuses them, avoiding prepareApiRequest per page.
