Skills, Plugins, and Memory System

This module covers three "extension capabilities" in Claude Code: ① a user or Claude invokes a skill—a reusable workflow prepackaged with prompts and tool configuration; ② installing a plugin to extend capabilities—how plugins are discovered, installed, loaded, and how they contribute tools and commands; ③ how Claude "remembers" things—persistent memory based on CLAUDE.md, split into project-level, user-level, auto-extraction, and team-sync tracks. The three mechanisms are independent, but all revolve around the same theme: letting Claude reuse experience, capabilities, and context beyond the current session.

1. Skill Loading and Invocation

From the user's perspective: typing /skills in a session lists all currently available skills; Claude can also proactively invoke a skill via SkillTool. A skill is essentially a directory containing a SKILL.md file (with YAML frontmatter describing metadata: name, description, allowed tools, parameters, etc.; the body is the prompt shown to Claude). At startup, the system scans these directories from four sources: managed (policy-level), user-level (~/.claude/skills), project-level (.claude/skills walked upward from the project directory to the home directory), and additional --add-dir directories. There are also bundled skills (compiled into the CLI) and skills contributed by MCP servers.

Input
cwd at startup + .claude/skills directories at each level; the skill name chosen by the user/Claude at runtime + optional arguments
Output
An expanded prompt (with $ARGUMENTS, ${CLAUDE_SKILL_DIR}, etc. replaced), injected back into the conversation; at the same time, the tool allowlist, model, and effort for this invocation are adjusted as needed

  • Deduplication uses realpath to resolve symlinks so the same file isn't loaded twice via different paths (order-sensitive: managed > user > project)
  • A skill with a paths frontmatter is a "conditional skill"; it isn't loaded normally and only activates when a user action matches a path pattern (gitignore-style matching), saving tokens
  • Dynamic discovery: when reading/writing files in subdirectories of the cwd at runtime, it walks upward to find .claude/skills and loads them; gitignored directories are skipped
  • MCP skills come from remote servers and are untrusted—embedded !`...` shell commands in their bodies are never executed (security boundary)
  • Bundled skills may carry reference files; on first invocation they are lazily extracted to a private nonce directory (O_NOFOLLOW + 0o600 to prevent symlink attacks)
  • Fork mode: the skill runs in an independent sub-agent with its own token budget and result text is returned; inline mode: the expanded prompt is inserted directly back into the main conversation
  • Permissions: skills containing only "safety attributes" are auto-approved; skills with sensitive fields such as hooks or allowed-tools prompt the user; users can also configure Skill(name) allow/deny rules

2. Plugin Lifecycle

From the user's perspective: a plugin is a larger-granularity extension package than a skill—a single plugin can contribute skills, commands, hooks, MCP servers, LSP servers, agent definitions, and more. Users open the management UI via /plugin to browse/install/enable/disable, or use the command line with claude plugin install foo@marketplace. Plugins come from three sources: ① marketplace (the plugin marketplace, itself a git repository listing plugin entries); ② built-in plugins (compiled with the CLI, user-toggleable); ③ --plugin-dir temporary session plugins (pointing to a local directory, valid only for this session). Installation scopes are user/project/local/managed; managed is locked by enterprise administrators through policy.

Input
Plugin identifier chosen by the user (name@marketplace) + scope; or the marketplace repository declared in the config file
Output
A LoadedPlugin in memory—its skills/commands/hooks/MCP/LSP/agents are registered with the corresponding subsystems so Claude can use them next time

  • Loading priority: --plugin-dir session plugins > marketplace-installed plugins > built-in plugins; but managed policy overrides --plugin-dir (administrator intent takes precedence over local development convenience)
  • At startup there are two loading paths: loadAllPlugins (will clone missing marketplaces from the network) and loadAllPluginsCacheOnly (reads only cache, no network), the latter used by consumers that must not block startup; the former used for explicit refreshes such as /plugins and /reload-plugins
  • Installation is non-in-place update: download to temp directory → compute version (git commit SHA or manifest version) → copy to versioned cache directory → update installed_plugins.json; the old version in memory isn't replaced until restart
  • During uninstall, if other enabled plugins still depend on it, only warn without blocking (to avoid a delisted plugin freezing the entire dependency graph)
  • Uninstalling a project-scoped plugin guides the user to disable it in local scope, because .claude/settings.json is shared by the team
  • Plugins can contribute MCP/LSP servers; refreshActivePlugins bumps pluginReconnectKey to trigger MCP reconnect and reinitializes the LSP manager

3. Memory Read/Write and Sync

From the user's perspective: Claude can "remember" things thanks to a persistent memory system based on CLAUDE.md. Memory is split into four tracks: ① Project-level CLAUDE.md—.claude/CLAUDE.md at the project root, shared by the team, loaded when entering the project; ② User-level CLAUDE.md—~/.claude/CLAUDE.md, effective across projects; ③ Auto memory—Claude itself decides during the conversation what is worth remembering and writes it to ~/.claude/projects/<slug>/memory/, auto-injected next time; ④ Team memory sync—push local memory to a remote server so teammates can pull it down, achieving shared memory. Users can run the /memory command to open an editor and manually edit memory files.

Input
User messages, corrections, preferences, and project context during a conversation; or memory content handwritten by the user in the editor
Output
.md files written to the memory directory (with frontmatter: type/description) + a MEMORY.md index; in team scenarios, also synced to the remote

  • Memories are divided into four types (user/feedback/project/reference); anything derivable from current project state (code patterns, architecture, git history) is explicitly excluded—these should not be remembered
  • MEMORY.md is an index, not the memories themselves: each entry is one line - [Title](file.md) — one-sentence hook; it is truncated and flagged with a warning if it exceeds 200 lines or 25KB
  • Recall mechanism: scan directory, read frontmatter to build a list → use Sonnet for relevance selection (independent sideQuery, doesn't pollute main conversation) → inject selected files; previously injected files are filtered out to avoid duplication
  • Auto extraction (extractMemories) triggers at the end of each conversation turn (when the model produces a final reply with no tool calls), using a forked agent that shares the main conversation cache; it is mutually exclusive with direct writes by the main agent—if the main agent wrote memory, this turn's extraction is skipped
  • There is an extraction frequency throttle (tengu_bramble_lintel, extracts every turn by default) and a cursor (lastMemoryMessageUuid) to ensure only new messages are examined
  • Team memory sync: watcher listens for team directory writes → debounce 2s → push; push uses content hash to compute delta, uploading only changed entries; on conflict (HTTP 412), pull refreshes serverChecksums and retry
  • Secret safety: before push, scan with gitleaks rules; files matching secrets are excluded from upload, and only the rule id is recorded, not the value
  • Session memory is another track: used to compress context during compaction, activated only when auto-compact is enabled, written to dedicated session memory files
  • KAIROS mode: long sessions switch to append-only logs (one file per day), distilled into MEMORY.md + topic files by the nightly /dream skill
  • /memory command: pops a selector for the user to choose which memory file to edit, opened with $EDITOR/$VISUAL
