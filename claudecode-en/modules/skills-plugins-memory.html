Skills, Plugins, and Memory System

This module covers three "extensibility" topics in Claude Code: ① a user or Claude invokes a skill—a pre-packaged, reusable workflow bundling prompts and tool configuration; ② installing a plugin to extend capabilities—how plugins are discovered, installed, loaded, and how they contribute tools and commands; ③ how Claude "remembers" things—persistent memory based on CLAUDE.md, split across project-level, user-level, auto-extracted, and team-synced lines. The three mechanisms are independent, but all revolve around the same theme: letting Claude reuse experience, capabilities, and context beyond the current session.

1. Skill Loading and Invocation

From the user's perspective: typing /skills in a session lists all currently available skills; Claude can also proactively invoke a skill through SkillTool. A skill is essentially a directory containing a SKILL.md (with YAML frontmatter describing metadata: name, description, allowed tools, parameters, etc.; the body is the prompt Claude sees). At startup the system scans skill directories from four sources: policy-level (managed), user-level (~/.claude/skills), project-level (.claude/skills walking up from the project directory to the home directory), and additional --add-dir directories. There are also built-in skills (compiled into the CLI) and skills contributed by MCP servers.

Input
Startup cwd + .claude/skills directories at each level; at runtime the skill name selected by the user/Claude + optional arguments
Output
An expanded prompt (with $ARGUMENTS, ${CLAUDE_SKILL_DIR}, etc. replaced) injected back into the conversation; at the same time the tool allowlist, model, and effort for this invocation are adjusted as needed

  • Deduplication uses realpath to resolve symlinks, preventing the same file from being loaded twice through different paths (order-sensitive: policy > user > project)
  • Skills with a paths frontmatter are "conditional skills"; they are not loaded normally but activate when a user operation hits a path pattern (gitignore-style matching), saving tokens
  • Dynamic discovery: when reading/writing files under cwd subdirectories at runtime, the system looks upward for .claude/skills and loads them; gitignored directories are skipped
  • MCP skills come from remote servers and are untrusted—embedded shell commands !`...` in their bodies are never executed (security boundary)
  • Built-in skills may carry reference files; they are lazily extracted on first call to a private directory with nonce (O_NOFOLLOW + 0o600 to prevent symlink attacks)
  • fork mode: the skill runs in an independent sub-agent with its own token budget and the result text is passed back; inline mode: the expanded prompt is inserted directly into the main conversation
  • Permissions: skills containing only "safety properties" are auto-allowed; skills with hooks, allowed-tools, or other sensitive fields ask the user; users can also configure Skill(name) allow/deny rules

2. Plugin Lifecycle

From the user's perspective: a plugin is a larger-granularity extension package than a skill—a plugin can simultaneously contribute skills, commands, hooks, MCP servers, LSP servers, agent definitions, and more. Users browse/install/enable/disable plugins through /plugin, or on the command line with claude plugin install foo@marketplace. Plugins come from three sources: ① marketplace (the plugin marketplace itself is a git repository listing plugin entries); ② built-in plugins (compiled with the CLI, user-toggleable); ③ --plugin-dir temporary session plugins (pointing to a local directory, valid only for this session). Installation scopes are user/project/local/managed; managed is locked by enterprise administrators through policy.

Input
User-selected plugin identifier (name@marketplace) + scope; or a marketplace repository declared in config files
Output
A LoadedPlugin loaded into memory—its skills/commands/hooks/MCP/LSP/agents are all registered into the corresponding subsystems, ready for Claude next time

  • Loading priority: --plugin-dir session plugins > marketplace-installed plugins > built-in plugins; but managed policy overrides --plugin-dir (admin intent takes priority over local-dev convenience)
  • There are two startup load paths: loadAllPlugins (may clone missing marketplace plugins over the network) and loadAllPluginsCacheOnly (reads cache only, no network); the latter is for consumers that must not block startup; the former is for explicit refreshes like /plugins and /reload-plugins
  • Installation is non-in-place update: download to temp directory → compute version (git commit SHA or manifest version) → copy to versioned cache directory → update installed_plugins.json; the old version in memory is not swapped until restart
  • During uninstall, if other enabled plugins still depend on it, only warn without blocking (avoiding delisted plugins freezing the entire dependency graph)
  • Project-scope uninstall guides the user to disable via local scope because .claude/settings.json is team-shared
  • Plugins can contribute MCP/LSP servers; refreshActivePlugins bumps pluginReconnectKey to trigger MCP reconnect and reinitializes the LSP manager

3. Memory Read/Write and Sync

From the user's perspective: Claude can "remember" things through a persistent memory system based on CLAUDE.md. Memory is split across four lines: ① Project-level CLAUDE.md—.claude/CLAUDE.md at the project root, team-shared, loaded when entering the project; ② User-level CLAUDE.md—~/.claude/CLAUDE.md, effective across projects; ③ Auto memory—Claude decides during conversations what is worth remembering and writes to ~/.claude/projects/<slug>/memory/, automatically injected next time; ④ Team memory sync—push local memory to a remote server so teammates can pull it down, achieving shared memory. Users can open the editor with the /memory command to manually edit memory files.

Input
User messages, corrections, preferences, and project background during conversations; or memory content hand-written by the user in the editor
Output
.md files landed in the memory directory (with frontmatter: type/description) + a MEMORY.md index; in team scenarios they are also synced to a remote server

  etag -->|no| fetch["GET remote entries"]
  fetch --> write-local["writeRemoteEntriesToLocal: write back to local team directory"]
  • Memory is split into four types (user/feedback/project/reference); content derivable from current project state (code patterns, architecture, git history) is explicitly excluded—these should not be memorized
  • MEMORY.md is an index, not the memory itself: each line is - [Title](file.md) — one-sentence hook; if it exceeds 200 lines or 25 KB it is truncated and a warning is attached
  • Recall mechanism: scan directory and read frontmatter to build a list → use Sonnet for relevance selection (independent sideQuery, does not pollute main conversation) → inject selected files; already-injected files are filtered to avoid duplication
  • Auto extraction (extractMemories) triggers at the end of each conversation turn (when the model produces a final reply with no tool calls), using a forked agent that shares the main conversation cache; it is mutually exclusive with main-agent direct writes—if the main agent wrote memory this turn, extraction is skipped
  • Extraction frequency throttling (tengu_bramble_lintel, default every turn) and cursor (lastMemoryMessageUuid) ensure only new messages are examined
  • Team memory sync: watcher listens for writes in the team directory → debounce 2s → push; push computes delta using content hash, sending only changed entries; on conflict (HTTP 412) pull to refresh serverChecksums and retry
  • Secret safety: before push, scan with gitleaks rules; files hitting secrets are excluded from upload, only the rule id is recorded, not the value
  • Session memory is a separate line: used to compress context during compaction, activated only when auto-compact is on, written to dedicated session memory files
  • KAIROS mode: long sessions switch to append-only logs (one file per day), distilled by the nightly /dream skill into MEMORY.md + topic files
  • /memory command: pops a selector for the user to choose which memory file to edit, opened with $EDITOR/$VISUAL
