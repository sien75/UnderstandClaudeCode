IDE Bridge (IDE Integration)

When a user installs the Claude Code extension in VS Code or JetBrains, the extension needs to connect to the local CLI backend to do the real work. This module is the entire communication skeleton after the CLI starts in "bridge mode": it turns the CLI into a long-running process that establishes a bidirectional WebSocket channel with the IDE, authenticates with JWT, routes various IDE-initiated actions (editing files, viewing diffs, permission confirmations, starting sessions) to the CLI core, and streams the results back to the IDE panel.

From the user's perspective: you open the Claude Code panel in the IDE, type a sentence, and click an "Apply edits" button—these actions are not executed by the IDE extension itself, but handed over to the local CLI through this bridge channel. After the CLI finishes, it pushes changes, diffs, and pending permissions back to the IDE. This module explains how that channel is established, how authentication works, how messages travel back and forth, and how sessions start and end.

Input
Connection request from the IDE extension (including JWT token) + subsequent operation commands (start session, send user message, confirm/deny permission, view diff, etc.)
Output
A persistent bidirectional channel: every action submitted by the IDE is executed by the CLI, and results (streaming text, file changes, permission requests, errors) flow back to the IDE panel in real time

1. Bridge Mode Startup: From Command to Ready Channel

When the user types claude remote-control in the terminal (or clicks "Connect Remote Control" in the IDE extension), the CLI does not enter an interactive REPL but takes the bridgeMain branch. Its work can be broken into a few steps: confirm this machine is allowed to run bridge (logged in, workspace trusted, multi-session rollout enabled), register an "environment" with the server, get back an environmentId + secret pair, and then enter a long-polling loop waiting for the server to dispatch "work items."

Note that BRIDGE_MODE is a feature flag (GrowthBook rollout); when off the entire bridge command exits immediately, without even registering—so this capability is gradually rolled out per account/flag.

  • The trust check (checkHasTrustDialogAccepted) exists because bridge bypasses the main init flow and no one pops the trust dialog for it, so the user must have run claude normally at least once before.
  • Pre-creating a session (createBridgeSession) is non-blocking—failure does not block startup, it just means the user has to wait for the server to dispatch the first work item.

2. Authentication: Two Tokens, Each for Its Own Segment

This is the easiest part of the module to get confused about, but it has a clear layering. Bridge involves two credentials:

  • OAuth access token: the one you get from /login, used by bridge itself to call server-side management APIs directly (register environment, poll for work, stop work, archive session, unregister environment). This goes through withOAuthRetry: on 401 it first tries to refresh OAuth once; if refresh succeeds it retries the request once, and only a second 401 is treated as fatal.
  • Session ingress JWT (session_ingress_token): when the server dispatches each work item, it packs a short-lived JWT into the work secret (base64url-encoded JSON) and sends it down. This JWT is what proves "I am the legitimate worker for this session" on the session communication channel (WebSocket / SSE read stream + POST write stream). It expires (about 5 hours), after which the server dispatches a new work item and bridge gets a new JWT to reconnect.

There is also a layer of trusted device token: when the tengu_sessions_elevated_auth_enforcement rollout is on, bridge includes it in the X-Trusted-Device-Token header when calling registration APIs, allowing the server to require "high-privilege sessions must come from a registered trusted device." This token is enrolled during /login via POST /auth/trusted_devices, stored in the system keychain, and rolls over with 90-day validity.

  • The JWT is not verified for signature; only the payload is base64url-decoded to read exp and schedule refresh timers (jwtUtils.ts: decodeJwtExpiry). Signature verification is the server's job.
  • v1 (Session-Ingress) channels can accept either OAuth or JWT; bridge prefers OAuth because it has a standard refresh flow; v2 (CCR /worker/*) must use JWT because register_worker validates the session_id claim inside the JWT, which OAuth does not have.
  • On 401/403 (and not a suppressible cosmetic 403) bridge exits fatally without retry—authentication will not fix itself. 404/410 are treated as environment/session expired.
  • URL safety check: non-localhost base URLs must be HTTPS to prevent credentials from leaking in plaintext.

3. Bidirectional Message Flow: How IDE Clicks Reach the CLI and CLI Output Returns to the IDE

This is the core of the module. The channel is not actually a direct IDE-to-CLI connection; both sides connect through the server relay:

Uplink (CLI → server → IDE): the child CLI process runs with --print --output-format stream-json; every NDJSON line it produces (assistant text, tool_use, result, etc.) is read by the bridge parent process from its stdout, parsed into an activity summary ("Editing src/foo.ts") to update the status line; at the same time these SDK messages are POSTed to session-ingress via the transport's writeBatch, and the server distributes them to IDE/Web clients subscribed to that session.

Downlink (IDE → server → CLI): when the user types a sentence in the IDE, clicks "Apply edits," or approves a permission, those actions are sent by the IDE extension to the server, which packages "there is work" into a work item and drops it into this environment's queue. The bridge polling loop pollForWork pulls the work, decodes the JWT, acks it, and connects to the ingress channel (v1 WebSocket or v2 SSE). The server then pushes user messages and control_request messages (interrupt, set_model, can_use_tool permission asks) down this channel; bridge forwards them to the child process or local permission callbacks.

  • Echo deduplication is critical: messages POSTed by bridge itself are echoed back by the server. BoundedUUIDSet (ring buffer, capacity 2000) records recently sent UUIDs; echoed ones are dropped.
  • Redelivery deduplication: when the server switches transports or sequence-number negotiation fails, it may redeliver historical prompts; recentInboundUUIDs provides a last-resort filter.
  • Permissions form a bidirectional loop: child CLI wants to use a tool → sends control_request(can_use_tool) → bridge forwards to server → IDE pops a dialog → user chooses → control_response returns → bridge forwards back to child CLI stdin.
  • Attachment handling: files uploaded from the Web get a file_uuid and are sent with the message; bridge downloads them via OAuth call /api/oauth/files/{uuid}/content, writes to ~/.claude/uploads/{sessionId}/, prefixes @"path" to the message content, and the child CLI's Read tool can read it.
  • v1/v2 selection is decided by the server per session (work secret's use_code_sessions), not by bridge itself; CLAUDE_BRIDGE_USE_CCR_V2 is an internal Anthropic force switch.

4. Disconnections, Session Expiration, and System Sleep: Recovery Paths

Long-lived connections sometimes break. This module's recovery strategy has several layers; the key goals are not to lose user messages, not to retry forever, and to distinguish "brief network jitter" from "the environment is dead."

graph TD
  pollErr["pollForWork polling throws"] --> isAborted{"signal aborted?"}
  isAborted -->|yes| break["Exit loop"]
  isAborted -->|no| isFatal{"BridgeFatalError?"}
  isFatal -->|yes| classifyFatal{"401/403 auth?"}
  classifyFatal -->|yes| fatalExit["Fatal exit<br/>no retry"]
  classifyFatal -->|404/410 expired| fatalExit
  isFatal -->|no| isConn{"Connection error?<br/>(ECONNRESET/ETIMEDOUT...)"}
  isConn -->|yes| sleepDetect{"Interval far greater than<br/>max backoff?<br/>(system sleep)"}
  sleepDetect -->|yes| resetBudget["Reset error budget<br/>fresh retry"]
  sleepDetect -->|no| connBackoff["Exponential backoff 2s→120s<br/>give up after 10 min"]
  connBackoff --> heartbeatDuring["Keep heartbeating during backoff<br/>maintain 300s lease"]
  isConn -->|no| generalBackoff["Generic error backoff<br/>give up after 10 min"]
  connBackoff --> reconnectedOk{"Recovered?"}
  reconnectedOk -->|yes| logReconnect["Print Reconnected"]
  reconnectedOk -->|no| giveUp["Give up after 10 min<br/>fatal exit"]

  wsClose["ingress channel onClose"] --> closeCode{"Close code"}
  closeCode -->|normal 1000| teardown["Run teardown"]
  closeCode -->|abnormal| wakePoll["Wake polling loop<br/>quickly re-pull work"]
  wakePoll --> freshWork["Server redispatches work<br/>with new JWT"]
  freshWork --> reconnect["onWorkReceived<br/>reconnect channel with new token"]

  env404["poll returns 404<br/>(environment deleted)"] --> envLost["onEnvironmentLost"]
  envLost --> strategy1{"Strategy 1:<br/>re-register same envId<br/>+ reconnectSession"}
  strategy1 -->|success| sameSession["Keep original session<br/>URL still valid"]
  strategy1 -->|failure| strategy2["Strategy 2:<br/>archive old session<br/>create new session on new env"]
  strategy2 --> freshSession["currentSessionId updated<br/>reset sequence numbers/dedup sets"]
  strategy2 --> retryLimit{"Rebuilt > 3 times?"}
  retryLimit -->|yes| failed["State → failed<br/>trigger onFatalError"]
  retryLimit -->|no| continue["Continue polling"]
  • System sleep detection: if the interval between two polling errors is far greater than the max backoff (default 240s), the machine is considered to have slept, and the error budget is reset; otherwise it would immediately be abandoned as "already over 10 min" upon waking.
  • Heartbeat keeps the lease: work items have a 300s lease TTL; even while the channel is backing off and reconnecting, heartbeatActiveWorkItems keeps sending heartbeats to prevent the lease from expiring and being reclaimed by the server.
  • JWT expiration → heartbeats receive 401/403 → bridge does not treat this as fatal, but calls reconnectSession to requeue the work on the server side; the next poll gets the same work with a new JWT, and onWorkReceived reconnects the channel with the new token (existingHandle path).
  • Environment reconstruction has a 3-attempt cap to prevent infinite loops; after success the counter resets, so a failure hours later is not mistakenly killed by historical accumulation.
  • Perpetual mode (daemon) teardown is "local exit, server keeps"—no result, no stopWork; it relies on the server's 300s lease to naturally return to pending, and the next startup uses reconnectSession to resume from the pointer.

5. Session Lifecycle: From Dispatch to Cleanup

From the moment a session is dispatched until it ends, bridge manages its directory, timeout, title, worktree cleanup, and server-side stop notification. This is a lifecycle-oriented supplement.

graph TD
  workReceived["work item arrives<br/>type=session"] --> existing{"This sessionId<br/>already running?"}
  existing -->|yes| updateToken["Update ingress token<br/>reschedule refresh timer<br/>push new token to child stdin"]
  existing -->|no| atCap{"At capacity limit?"}
  atCap -->|yes| skipSpawn["Do not spawn<br/>wait for capacity"]
  atCap -->|no| spawn["safeSpawn child CLI<br/>--print --sdk-url ..."]
  spawn --> spawnFail{"Spawn throws?"}
  spawnFail -->|yes| stopWorkFail["stopWork notify failure<br/>clean worktree"]
  spawnFail -->|no| register["Record in activeSessions<br/>set start time/timeout/refresh timer"]
  register --> timeout["Set 24h timeout watchdog"]
  register --> title["Fetch session title async<br/>derive title on first user message"]
  register --> status["Start live status line refresh<br/>(tool trace/elapsed time)"]
  status --> running["Session running"]
  running -->|"child process close"| done["onSessionDone"]
  done --> status2{"Exit status"}
  status2 -->|completed| logComplete["Print completed"]
  status2 -->|failed| logFail["Print failed<br/>(not caused by shutdown)"]
  status2 -->|interrupted| skipLog["Skip stop notification<br/>(server-initiated or shutdown)"]
  logComplete --> stopWork["stopWorkWithRetry notify stop"]
  logFail --> stopWork
  stopWork --> worktree["Clean worktree<br/>(worktree mode)"]
  worktree --> mode{"Spawn mode?"}
  mode -->|single-session| archiveExit["Archive session<br/>abort loop → exit"]
  mode -->|multi-session| archiveIdle["Archive session<br/>return to idle for new work"]
  • Spawn mode is one of three: single-session (exit after one session), same-dir (multiple sessions share cwd), worktree (each session has its own git worktree, no file collisions). Pressing w while running toggles between same-dir and worktree.
  • Title strategy: server-set name (--name/Web rename) takes priority; otherwise wait for the first real user message, deriveSessionTitle truncates to 80 characters, and PATCHes back to the server so the IDE list can display it.
  • 24h timeout watchdog: when triggered, SIGKILL the child process, mark timedOut, and onSessionDone reclassifies interrupted as failed, then stops and cleans up normally.
  • Graceful shutdown order: SIGTERM → wait 30s grace → SIGKILL if still alive → stop all work → clean worktree → archive session → unregister environment → clear crash pointer. For single-session non-fatal exits, the session and environment are deliberately left alive and claude remote-control --continue is printed so the user can resume.
  • Crash recovery pointer (bridgePointer): immediately after spawn, write to .claude/bridge_pointer and refresh mtime hourly; normal teardown removes it, a crash leaves it, and next --continue reads it to recover.

6. REPL Bridge: Another Way to Connect Inside an Interactive Session

The above covers the standalone claude remote-control process mode. There is another form: the user is already running a normal interactive claude REPL and wants to also control the same session from Web/mobile—this goes through initReplBridge / initBridgeCore. It shares bridgeApi, workSecret, bridgeMessaging, and jwtUtils with standalone mode, but differs in that: the transport is in-process (HybridTransport/SSETransport) and no child process is spawned; environment/session lifecycle is bound to the REPL process; and there is additional in-place reconstruction logic after environment loss.

  • SSE sequence number carry-over (lastTransportSequenceNum) is a critical detail: when swapping transports, bring the old high-water mark so the new SSE first connection can send from_sequence_num; otherwise the server replays the entire session history from 0, treating each historical prompt as a new message.
  • Initial history flush has a flushGate: while history is being flushed, new messages are queued to avoid interleaving with historical POSTs and causing out-of-order arrival at the server.
  • Perpetual mode (daemon) teardown does not send result, stop work, or close transport—local exit, server keeps, relying on the 300s lease to return to pending; next --continue uses reconnectSession to resume from the pointer.
  • Outbound-only mode: only POST uplink, does not receive downlink control_request, and returns errors for mutable requests (interrupt/set_model)—used for "you may only watch my local results, not control me from outside" scenarios.
