IDE Bridge (IDE Integration)

When a user installs the Claude Code extension in VS Code or JetBrains, the extension needs to connect to the local CLI backend to do the actual work. This module is the complete communication framework after the CLI is started in "bridge mode": it turns the CLI into a long-running process, establishes a bidirectional WebSocket channel with the IDE, uses JWT for authentication, routes IDE actions (editing files, viewing diffs, permission confirmation, initiating sessions) to the CLI core, and streams the results back to the IDE.

From the user's perspective: you open the Claude Code panel in the IDE, type a message, and click an "Apply edits" button—these actions are not executed by the IDE extension itself, but are handed over to the local CLI through this bridge channel. After the CLI finishes, it pushes changes, diffs, and pending permissions back to the IDE. This module explains how this channel is established, how authentication works, how messages travel back and forth, and how sessions are started and torn down.

Input
Connection requests initiated by the IDE extension (including a JWT token) + subsequent operation commands (start a session, send a user message, approve/reject permissions, view diffs, etc.)
Output
A persistent bidirectional channel: every action submitted by the IDE is executed by the CLI, and results (streaming text, file changes, permission requests, errors) flow back to the IDE panel in real time

1. Bridge Mode Startup: From Command to Channel Ready

When the user types claude remote-control in the terminal (or clicks "Connect Remote Control" in the IDE extension), the CLI does not enter an interactive REPL but takes the bridgeMain path. Its work can be broken into several steps: first confirm that this machine is allowed to run bridge (logged in, workspace trusted, multi-session rollout enabled), then register an "environment" with the server, obtain an environmentId + secret pair, and finally enter a long-polling loop waiting for the server to dispatch "work items".

Note that BRIDGE_MODE is a feature flag (GrowthBook rollout). When disabled, the entire bridge command exits directly without even registering—so this capability is rolled out gradually by account/rollout.

  • Pre-creating a session (createBridgeSession) is non-blocking—failure does not block startup; the user just has to wait for the server to dispatch the first work item.

2. Authentication: Two Tokens, Two Layers

This is the most confusing part of the module, but it has clear layers. Bridge involves two credentials:

  • OAuth access token: the one you get from /login, used by bridge itself to call server-side management APIs directly (register environment, poll for work, stop work, archive session, unregister environment). This uses withOAuthRetry: on 401, it first tries to refresh the OAuth token once; if refresh succeeds, the request is retried once, and only a second 401 is treated as fatal.
  • Session ingress JWT (session_ingress_token): when the server dispatches each work item, it sends a short-lived JWT packed into the work secret (base64url-encoded JSON). This JWT is what proves "I am the legitimate worker for this session" on the session communication channel (WebSocket / SSE read stream + POST write stream). It expires (about 5 hours); after expiry the server dispatches the work again, and bridge gets a new JWT and reconnects.

There is also a trusted device token layer: when the tengu_sessions_elevated_auth_enforcement rollout is on, bridge includes it in the X-Trusted-Device-Token header when calling the registration API, so the server can require "elevated sessions must come from a registered trusted device". This token is enrolled at /login via POST /auth/trusted_devices, stored in the system keychain, and rolls over every 90 days.

  • The JWT is not verified for signature; bridge only base64url-decodes the payload to read exp and schedules the refresh timer (jwtUtils.ts: decodeJwtExpiry). Signature verification is the server's job.
  • v1 (Session-Ingress) channels can accept either OAuth or JWT; bridge prefers OAuth because it has a standard refresh flow; v2 (CCR /worker/*) must use JWT because register_worker validates the session_id claim in the JWT, which OAuth does not have.
  • On 401/403 (unless it is a suppressible cosmetic 403), bridge fatally exits without retrying—authentication does not fix itself. 404/410 are treated as environment/session expired.
  • URL safety check: non-localhost base URLs must be HTTPS to prevent credentials leaking in plaintext.

3. Bidirectional Message Flow: How IDE Clicks Reach the CLI and CLI Output Returns to the IDE

This is the core of the module. The channel is not actually IDE-to-CLI direct; both sides connect to the server, which relays traffic:

Upstream (CLI → server → IDE): the child CLI process runs with --print --output-format stream-json. Every NDJSON line it emits (assistant text, tool_use, result, etc.) is read by the bridge parent process from its stdout, parsed into an activity summary ("Editing src/foo.ts") to update the status line; at the same time these SDK messages are POSTed to session-ingress via the transport's writeBatch, and the server distributes them to IDE/Web clients subscribed to that session.

Downstream (IDE → server → CLI): when the user types a message in the IDE, clicks "Apply edits", or approves a permission, the IDE extension sends it to the server, which wraps "there is work to do" as a work item and drops it into this environment's queue. The bridge polling loop pollForWork pulls the work, decodes the JWT, acks it, and connects to the ingress channel (v1 WebSocket or v2 SSE). After that, the server pushes user messages and control_request (interrupt, set_model, can_use_tool permission questions) down through this channel; bridge forwards them to the child process or to local permission callbacks.

  • Echo deduplication is critical: messages bridge POSTs itself are echoed back by the server. BoundedUUIDSet (ring buffer, capacity 2000) records recently sent UUIDs; echoed ones are dropped.
  • Redelivery deduplication: the server redelivers historical prompts when transport switches or sequence-number negotiation fails; recentInboundUUIDs provides a safety net filter.
  • Permissions form a bidirectional loop: child CLI wants to use a tool → sends control_request(can_use_tool) → bridge forwards to server → IDE shows dialog → user chooses → control_response returns → bridge forwards back to child stdin.
  • Attachment handling: files uploaded from the Web get a file_uuid and are sent down with the message; bridge uses OAuth to call /api/oauth/files/{uuid}/content to download them, writes them to ~/.claude/uploads/{sessionId}/, prepends the @"path" prefix to the message content, and the child CLI's Read tool can read them.
  • v1/v2 selection is decided by the server per session (via use_code_sessions in the work secret), not by bridge itself; CLAUDE_BRIDGE_USE_CCR_V2 is an internal Anthropic override switch.

4. Disconnections, Session Expiry, and System Sleep: Recovery Paths

Long-lived connections break. This module's recovery strategy has several layers, with the key goals of not losing user messages, not retrying forever, and distinguishing "transient network jitter" from "environment is dead".

graph TD
  pollErr["pollForWork polling error"] --> isAborted{"signal aborted?"}
  isAborted -->|Yes| break["Exit loop"]
  isAborted -->|No| isFatal{"BridgeFatalError?"}
  isFatal -->|Yes| classifyFatal{"401/403 auth?"}
  classifyFatal -->|Yes| fatalExit["Fatal exit<br/>no retry"]
  classifyFatal -->|404/410 expired| fatalExit
  isFatal -->|No| isConn{"Connection error?<br/>(ECONNRESET/ETIMEDOUT...)"}
  isConn -->|Yes| sleepDetect{"Interval far greater than<br/>max backoff?<br/>(system sleep)"}
  sleepDetect -->|Yes| resetBudget["Reset error budget<br/>fresh retry"]
  sleepDetect -->|No| connBackoff["Exponential backoff 2s→120s<br/>give up after 10min"]
  connBackoff --> heartbeatDuring["Keep heartbeating during backoff<br/>to keep 300s lease"]
  isConn -->|No| generalBackoff["Generic error backoff<br/>give up after 10min"]
  connBackoff --> reconnectedOk{"Recovered?"}
  reconnectedOk -->|Yes| logReconnect["Print Reconnected"]
  reconnectedOk -->|No| giveUp["Give up after 10min<br/>fatal exit"]

  wsClose["ingress channel onClose"] --> closeCode{"Close code"}
  closeCode -->|Normal 1000| teardown["Tear down"]
  closeCode -->|Abnormal| wakePoll["Wake polling loop<br/>quickly re-pull work"]
  wakePoll --> freshWork["Server re-dispatches work<br/>with new JWT"]
  freshWork --> reconnect["onWorkReceived<br/>reconnect channel with new token"]

  env404["poll returns 404<br/>(environment deleted)"] --> envLost["onEnvironmentLost"]
  envLost --> strategy1{"Strategy 1:<br/>re-register with same envId<br/>+ reconnectSession"}
  strategy1 -->|Success| sameSession["Keep original session<br/>URL still valid"]
  strategy1 -->|Failure| strategy2["Strategy 2:<br/>archive old session<br/>create new session on new env"]
  strategy2 --> freshSession["currentSessionId updated<br/>reset sequence numbers / dedup sets"]
  strategy2 --> retryLimit{"Reconstruction failed > 3 times?"}
  retryLimit -->|Yes| failed["Status → failed<br/>trigger onFatalError"]
  retryLimit -->|No| continue["Continue polling"]
  • System sleep detection: if the interval between two polling errors is far greater than the maximum backoff (default 240s), the machine is judged to have slept, and the error budget is reset; otherwise it would be abandoned immediately on wake due to "already exceeded 10min".
  • Heartbeat to keep lease: work items have a 300s lease TTL; even while the channel is backing off and reconnecting, heartbeatActiveWorkItems continues to send heartbeats to prevent the server from reclaiming the lease.
  • JWT expiry → heartbeat receives 401/403 → bridge does not treat this as fatal but calls reconnectSession to have the server re-enqueue this work; the next poll gets the same work with a new JWT, and onWorkReceived reconnects the channel with the new token (existingHandle path).
  • Environment reconstruction has a 3-time limit to prevent infinite loops; the counter resets on success, so a failure hours later is not wrongly killed by historical accumulation.
  • Perpetual mode (daemon) teardown is "local exit, server stays"—no archiving, no stopping work, relying on the server's 300s lease to naturally return to pending; the next startup reads the pointer from reconnectSession to resume.

5. Session Lifecycle: From Dispatch to Teardown

From the moment a session is dispatched to when it ends, bridge manages its directory, timeout, title, worktree cleanup, and reporting stop to the server. This is the Lifecycle perspective supplement.

graph TD
  workReceived["work item arrives<br/>type=session"] --> existing{"Is this sessionId<br/>already running?"}
  existing -->|Yes| updateToken["Update ingress token<br/>reschedule refresh timer<br/>push new token to child stdin"]
  existing -->|No| atCap{"Capacity limit reached?"}
  atCap -->|Yes| skipSpawn["Do not spawn<br/>wait for capacity"]
  atCap -->|No| spawn["safeSpawn child CLI<br/>--print --sdk-url ..."]
  spawn --> spawnFail{"Spawn threw?"}
  spawnFail -->|Yes| stopWorkFail["stopWork report stop<br/>clean worktree"]
  spawnFail -->|No| register["Record in activeSessions<br/>set start time/timeout/refresh timer"]
  register --> timeout["Set 24h timeout watchdog"]
  register --> title["Fetch session title async<br/>derive title on first user message"]
  register --> status["Start real-time status line refresh<br/>(tool trace / elapsed time)"]
  status --> running["Session running"]
  running -->|"Child process close"| done["onSessionDone"]
  done --> status2{"Exit status"}
  status2 -->|completed| logComplete["Print complete"]
  status2 -->|failed| logFail["Print failed<br/>(not shutdown-induced)"]
  status2 -->|interrupted| skipLog["Skip stop report<br/>(server-initiated or shutdown)"]
  logComplete --> stopWork["stopWorkWithRetry report stop"]
  logFail --> stopWork
  stopWork --> worktree["Clean worktree<br/>(worktree mode)"]
  worktree --> mode{"Spawn mode?"}
  mode -->|single-session| archiveExit["Archive session<br/>abort loop → exit"]
  mode -->|multi-session| archiveIdle["Archive session<br/>return to idle for new work"]
  • Spawn mode is one of three: single-session (exit after one session), same-dir (multiple sessions share cwd), or worktree (each session has its own git worktree, no file collisions). Pressing w while running toggles between same-dir and worktree.
  • Title strategy: a server-set name (--name/Web rename) takes priority; otherwise wait for the first real user message, deriveSessionTitle truncates to 80 characters, and PATCHes it back to the server so the IDE list can display it.
  • 24h timeout watchdog: when triggered, SIGKILL the child process, mark timedOut, and onSessionDone reclassifies interrupted as failed, still reporting stop and cleaning up.
  • Graceful shutdown sequence: SIGTERM → wait 30s grace → SIGKILL if still alive → stop all work → clean worktrees → archive sessions → unregister environment → clear crash pointer. For single-session non-fatal exits, the session and environment are intentionally left alive, printing claude remote-control --continue so the user can resume.
  • Crash recovery pointer (bridgePointer): written to .claude/bridge_pointer immediately after spawn, mtime refreshed hourly; cleared on normal teardown, left behind on crash, and read on next --continue to resume.

6. REPL Bridge: Another Connection Style Inside an Interactive Session

The above describes the standalone claude remote-control process mode. There is another form: the user is already running a normal interactive claude REPL and wants to also control the same session from Web/mobile—this goes through initReplBridge / initBridgeCore. It shares bridgeApi, workSecret, bridgeMessaging, and jwtUtils with standalone mode, but differs in that: the transport lives in this process (HybridTransport/SSETransport), no child process is spawned; environment/session lifecycle is bound to the REPL process; and there is extra reconnect-in-place logic for environment loss.

  closeOld --> pickV{"v1 or v2?"}
  pickV -->|v1| hybrid["HybridTransport<br/>WS read + POST write<br/>use OAuth"]
  pickV -->|v2| sse["createV2ReplTransport<br/>SSE read + CCRClient write<br/>use JWT, registerWorker"]
  hybrid --> wire["wireTransport<br/>attach onConnect/onData/onClose"]

  wire --> flush["onConnect: first history flush<br/>cap 200 messages"]
  flush --> ready["onStateChange connected<br/>channel ready"]
  ready --> twoWay["Bidirectional message flow<br/>(same as section 3)"]
  twoWay --> envLost2{"poll 404?"}
  envLost2 -->|Yes| reconnect["doReconnect<br/>strategy 1 in-place / strategy 2 new"]
  reconnect --> startPoll
  twoWay --> teardown2["teardown: send result → stopWork → archive → unregister"]
  • Carrying over the SSE sequence number (lastTransportSequenceNum) is a key detail: when switching transports, the old high-water mark is carried over, and the new SSE first connection sends from_sequence_num; otherwise the server would replay the entire session history from 0—treating every historical prompt as a new message.
  • The initial history flush uses a flushGate: new messages are queued during the flush to avoid arriving out of order with historical POSTs on the server.
  • Perpetual mode (daemon) teardown does not send result, stop work, or close transport—local exit, server stays, relying on the 300s lease to return to pending; next --continue reads the pointer from reconnectSession to resume.
  • Outbound-only mode: only POST upstream, does not receive downstream control_request, and returns errors for mutable requests (interrupt/set_model)—used for "you may only watch my local results, not control me from outside" scenarios.
