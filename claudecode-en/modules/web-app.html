Web Client (claude.ai/code-style browser interface)

When the user opens this Next.js page in the browser, they get a chat interface similar to claude.ai/code: a conversation list on the left and the current conversation's message stream plus input box on the right. The user types in the input box and presses Enter to send; messages stream in chunk by chunk; they can stop generation midway, switch/create/delete conversations, change theme, adjust font, search history, and export conversations. Behind the scenes it forwards requests to the real backend Claude Code session (default http://localhost:3001) through a Next.js API route, and passes the backend's SSE stream straight through to the browser.

This subproject actually contains two backend-facing client implementations: one is the lightweight streamChat in lib/api.ts, which is what the UI currently uses, going through Next.js's /api/chat proxy route; the other is the fuller client under lib/api/ (client.ts with retries/timeouts/request deduplication, stream.ts with reconnect and progress callbacks, conversations.ts/messages.ts/files.ts wrapping various resource operations), but the current UI has not been wired to this fuller layer yet—it is a "prepared, more complete layer". The data flow below focuses primarily on the lightweight path that is actually in use, with supplementary notes on the fuller layer.

Input
Text the user types into the input box + history messages of the current conversation + model/API address selected in settings
Output
Claude replies rendered chunk by chunk (with Markdown rendering, code highlighting, and tool-call block display), plus conversation records persisted to localStorage

Main Data Flow: Sending a Message to Receiving a Reply

After the user finishes typing and presses Enter (or clicks the send button), the frontend first writes the user message into the local state store (zustand's useChatStore), immediately creates a "placeholder" empty assistant message (status marked streaming), and then initiates a fetch request with an AbortController. The request hits the same-origin Next.js route /api/chat, which forwards the request body as-is to the real backend (NEXT_PUBLIC_API_URL, default localhost:3001) and streams the backend's SSE response straight back to the browser. The browser reads the stream line by line, appending text deltas to that placeholder assistant message; the interface grows chunk by chunk. After reading [DONE] or when the user interrupts midway, the message status is changed to complete.

  user-type["User types in input box and presses Enter"] --> input-guard{"Input empty or already generating?"}
  input-guard -->|Yes| no-op["Do nothing"]
  input-guard -->|No| add-user-msg["Write user message into store"]
  add-user-msg --> add-placeholder["Create empty assistant message(status=streaming)"]
  add-placeholder --> build-msgs["Build history message array to send to backend"]
  build-msgs --> fetch-stream["fetch /api/chat starts streaming request(with AbortController)"]
  fetch-stream --> proxy-route["Next.js /api/chat route proxies to backend"]
  proxy-route --> backend{"Backend response OK?"}
  backend -->|Non-200| yield-error["Yield error chunk"]
  backend -->|200| read-sse["Browser reads SSE stream, parses data: lines"]
  read-sse --> chunk-type{"Chunk type?"}
  chunk-type -->|text| append-text["Append text delta to assistant message, UI grows in real time"]
  chunk-type -->|done| finish-done["End loop"]
  chunk-type -->|error| mark-error["Mark assistant message error and show error text"]
  append-text --> read-sse
  finish-done --> mark-complete["Mark assistant message complete"]
  mark-error --> cleanup
  mark-complete --> cleanup["Clear isStreaming/abortRef"]
  yield-error --> cleanup
  • Clicking the "Stop generation" button calls abortRef.current.abort(), cancelling the fetch; the caught AbortError preserves the text already generated and marks it complete (not a failure).
  • The /api/chat route is a pure pass-through proxy: it does not parse the stream content, only returns the backend's response.body as the Next.js response body and preserves Content-Type. The backend address is determined by NEXT_PUBLIC_API_URL and may include ANTHROPIC_API_KEY for Bearer auth.
  • History messages are taken directly from the messages array of the current conversation in the zustand store and assembled into the request body—meaning "memory" is entirely maintained on the browser side, and the full history is resent every request.

Message Rendering: From Markdown to Code Highlighting

Assistant messages go to MessageBubble: user messages are rendered as plain text (preserving line breaks), while assistant messages are handed to MarkdownContent which uses react-markdown + remark-gfm to render rich text. If the message status is still streaming, a blinking cursor block is shown at the end of the bubble. Code block syntax highlighting is handled by the SyntaxHighlight component—it lazily loads Shiki on the main thread (imported only on first use), generates highlighted HTML by language/theme and injects it via dangerouslySetInnerHTML; unknown languages fall back to plain text. Note: lib/workers/highlight.worker.ts contains a parallel "move highlighting to Web Worker" implementation, but the current UI does not instantiate it anywhere with new Worker, so highlighting currently runs on the main thread.

  • lib/workers/markdown.worker.ts is likewise "prepared but not wired": it performs lightweight Markdown pre-segmentation (classifying lines into heading/code-fence/list, etc.), but no UI consumes it; current Markdown is parsed by react-markdown on the main thread.

Theme Switching

The theme is provided by ThemeProvider (lib/theme.tsx) through React Context, supporting dark / light / system. When system is selected, it listens to the prefers-color-scheme media query and follows the system's light/dark changes. The actual effect is achieved by adding/removing the light class on the <html> root element (default is dark, the layout uses <html class="dark">), and Tailwind's dark: variants switch accordingly. The selection is stored in localStorage (key ui-theme) and read back next time. Note this theme only controls UI colors; the code highlighting light/dark theme is independent on the Shiki side as github-dark/github-light.

Search: Full-Text Search of History Conversations in the Browser

Search uses client-search.ts—pure frontend, running on the main thread, directly iterating over the conversation array in the store. It first tokenizes the query (lowercase, split by non-word characters), then scores each message: title hit weight ×1.5, body score by token hit count + word-boundary bonus + multiple-occurrence bonus, and partial matches are also penalized proportionally. Besides keywords, it can filter by date, conversation, model, role, content type (text/tool_use/code/file), and tags. Results are sorted by total score descending, each conversation keeps at most 5 matches, and highlighted snippets are included. lib/search/highlighter.ts is responsible for excerpt snippets and wrapping hit words in highlight markers.

There is also a "prepared but not wired" parallel implementation: lib/workers/search.worker.ts maintains an inverted index (term → set of message indices), designed to be fed data, queried, and deleted via postMessage, but currently no UI starts it with new Worker. lib/search/search-api.ts is a stub reserved for a future backend search endpoint—the comments say "replace clientSearch calls with apiSearch once the backend search endpoint is ready".

Export Conversations

In the export panel the user picks a format (markdown / json / html / plaintext; pdf uses client-side window.print()) and options (whether to include tool calls, thinking process, timestamps, file contents, etc.). After clicking export, the frontend POSTs the entire Conversation object + options to /api/export. This route calls the corresponding conversion function (lib/export/markdown.ts, etc.) by format, uses the conversation title slug as the filename, and returns the file as a download with Content-Disposition: attachment. All conversion happens in the server-side route; the frontend receives the finished file.

Auxiliary View: Message State Machine

Each Message has a status field with values pending / streaming / complete / error, which determines how it is displayed and how it participates in subsequent requests. State transitions mainly happen in ChatInput.handleSubmit: when sending, the newly created assistant message starts at streaming, each received text chunk updates its content while keeping it streaming, the stream ending (done) moves it to complete, an error chunk or non-abort exception moves it to error; when the user actively interrupts (AbortError) it is treated as complete—the generated content is preserved.

graph TD
  new-user["Create user message"] --> user-complete["status=complete(immediately)"]
  new-assistant["Create placeholder assistant message"] --> st-streaming["status=streaming"]
  st-streaming --> on-text["Receive text chunk: accumulate content, stay streaming"]
  on-text --> st-streaming
  st-streaming --> on-done["Receive done chunk"] --> st-complete["status=complete"]
  st-streaming --> on-err-chunk["Receive error chunk"] --> st-error["status=error, show error text"]
  st-streaming --> on-abort["User interrupts (AbortError)"] --> st-complete
  st-streaming --> on-fail["Other exception"] --> st-error

Auxiliary View: Permissions and Lifecycle

Permissions: Settings include a permission configuration—switches for auto-approval by tool type (file_read / file_write / bash / web_search) plus a restricted directory list. This configuration lives in the zustand store's settings and is persisted to localStorage, but it currently only exists as local configuration: the chat request body sent to the backend only contains {messages, model, stream}; permission settings are neither sent to the backend nor validated when sending messages. The UI only has the PermissionSettings component reading and writing it. It is essentially "switches prepared now, to be wired once the backend can understand them".

Lifecycle: Almost the entire data lifecycle of this web subproject is tied to browser localStorage. The zustand store uses the persist middleware to persist conversations, activeConversationId, settings, pinnedIds, recentSearches, and tags (key claude-code-chat), reading them back after refresh; while pure UI states like isSearchOpen, settingsOpen, sidebarTab, and selectedConversationIds are forcibly reset to defaults during merge and are not persisted. The conversation "memory" is also entirely in the local store—each message sends the full history of the current conversation from the store to the backend; the backend does not keep context for the browser. This means clearing localStorage or switching browsers wipes the history and context.

Relationship with the main project's src/server/web: The main project's src/server/web/ is another independent terminal web service (pty-server, session-manager, terminal.js, etc., using xterm to embed a real terminal in the webpage). The web/ subproject does not reuse code from src/server/web—it uses the SSE streaming conversation path rather than the terminal PTY path. The two only intersect at the configuration layer: web/ points to the backend via NEXT_PUBLIC_API_URL (default localhost:3001). Which exact backend service that is (src/server or another service) is decided by deployment and is not hard-coded. lib/terminal-compat.ts is a stub "for moving Ink-based terminal components from src/components into the browser"—providing browser-side default implementations for process.stdout.columns/rows, process.exit, useInput/useApp and other Ink/Node APIs—but currently no file in web/ imports it; it is a shim reserved for future porting of terminal components.

  • The /api/share route (generates shareable links) and /api/files/read|write routes are another group of backend proxies, following the same pattern as /api/chat: receive frontend requests and forward them to the real backend. Sharing supports public/unlisted/password visibility and multiple expiration durations.
  • PDF is a special case in the export route: the server side does not handle it, and directly returns 400 indicating "handled by client-side window.print()".
