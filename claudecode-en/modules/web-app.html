Web App (Browser UI in the style of claude.ai/code)

When the user opens this Next.js page in a browser, they get a chat interface similar to claude.ai/code: a conversation list on the left and the message stream plus input box for the current conversation on the right. The user types in the input box and presses Enter to send; messages render chunk by chunk in a streaming fashion. They can stop generation midway, switch/create/delete conversations, change themes, adjust fonts, search history, and export conversations. Behind the scenes, it forwards requests to the real backend Claude Code session via a Next.js API route (default http://localhost:3001), proxying the backend's SSE stream back to the browser unchanged.

This subproject actually contains two sets of client code for talking to the backend: one is the lightweight streamChat in lib/api.ts, which is what the UI currently uses, going through Next.js's /api/chat proxy route; the other is a more complete client under lib/api/ (client.ts with retry/timeout/request deduplication, stream.ts with reconnection and progress callbacks, conversations.ts/messages.ts/files.ts wrapping various resource operations), but the current UI has not yet wired this up — it's a "more complete layer kept ready." The data flow below focuses mainly on the lightweight path actually in use, with supplementary notes on the complete layer.

Input
The text the user typed into the input box + the current conversation's history + settings such as selected model and API address
Output
Claude replies rendered as a streaming sequence (including Markdown rendering, code highlighting, and tool-use block display), plus conversation records persisted to localStorage

Main data flow: sending a message to receiving a reply

After the user finishes typing and presses Enter (or clicks the send button), the frontend first writes the user message into the local state store (zustand's useChatStore), then immediately creates a "placeholder" empty assistant message (status set to streaming), and initiates a fetch request with an AbortController. The request hits the same-origin Next.js route /api/chat, which forwards the request body unchanged to the real backend (NEXT_PUBLIC_API_URL, default localhost:3001) and proxies the backend's SSE stream back to the browser. The browser reads the stream while appending text increments to that placeholder assistant message, so the interface grows chunk by chunk; after reading [DONE] or being interrupted by the user, the message status is changed to complete.

  • Clicking the "Stop generation" button calls abortRef.current.abort(), canceling the fetch; the AbortError is caught, and any text already generated is preserved and marked as complete (not a failure).
  • The /api/chat route is a pure pass-through proxy: it does not parse stream content, but returns the backend's response.body as the Next.js response body while preserving Content-Type. The backend address is determined by NEXT_PUBLIC_API_URL and can include ANTHROPIC_API_KEY for Bearer authentication.
  • Conversation history is taken directly from the messages array of the current conversation in the zustand store and assembled into the request body — meaning "memory" is entirely maintained in the browser, and the full history is resent with each request.

Message rendering: from Markdown to code highlighting

Assistant messages go to MessageBubble: user messages are rendered as plain text (preserving line breaks), and assistant messages are handed to MarkdownContent for rich-text rendering with react-markdown + remark-gfm. If the message status is still streaming, a blinking cursor block is shown at the end of the bubble. Code-block syntax highlighting is handled by the SyntaxHighlight component — it lazily loads Shiki in the main thread (imported only on first use), generates highlighted HTML by language/theme, and injects it directly via dangerouslySetInnerHTML; unknown languages fall back to plain text. Note: lib/workers/highlight.worker.ts contains a parallel implementation for "moving highlighting to a Web Worker," but the current UI never instantiates it with new Worker, so highlighting currently runs on the main thread.

  • lib/workers/markdown.worker.ts is likewise "ready but unwired": it performs lightweight Markdown pre-segmentation (classifying lines into heading/code-fence/list etc.), but no UI consumes it; current Markdown is parsed by react-markdown on the main thread.

Theme switching

The theme is provided by ThemeProvider (lib/theme.tsx) through React Context, supporting dark / light / system. When system is selected, it listens to the prefers-color-scheme media query and follows the system light/dark changes. The actual effect is achieved by adding/removing the light class on the <html> root element (default is dark, <html class="dark"> in layout), and Tailwind's dark: variants switch accordingly. The selection is stored in localStorage (key ui-theme) and read back on next load. Note this theme only controls UI colors; the code highlighting light/dark theme is separate, handled by Shiki as github-dark/github-light.

Search: full-text search of conversation history in the browser

Search is handled by client-search.ts — purely frontend, running on the main thread, directly iterating over the conversation array in the store. It first tokenizes the query (lowercased, split on non-word characters), then scores each message: title matches get a ×1.5 weight, body scoring is based on token hit count + word-boundary bonus + multiple-occurrence bonus, and results missing all tokens are penalized proportionally. Beyond keywords, it can filter by date, conversation, model, role, content type (text/tool_use/code/file), and tags. Results are sorted by total score descending, each conversation keeps at most 5 matches, and highlight summaries are included. lib/search/highlighter.ts is responsible for extracting snippets and wrapping hit terms with highlight markers.

There is also a "ready but unwired" parallel implementation: lib/workers/search.worker.ts maintains an inverted index (term → set of message indices), designed to be fed data, queried, and deleted via postMessage, but currently no UI starts it with new Worker. lib/search/search-api.ts is a stub reserved for a future backend search endpoint — comments note "replace clientSearch with apiSearch once the backend search endpoint is ready."

Export conversation

In the export panel, the user selects a format (markdown / json / html / plaintext; pdf uses client-side window.print()) and options (whether to include tool calls, thinking process, timestamps, file contents, etc.). On export, the frontend POSTs the entire Conversation object + options to /api/export. This route calls the corresponding conversion function (lib/export/markdown.ts etc.) to generate content, uses the conversation title slug as the filename, and returns it as a download with Content-Disposition: attachment. All conversion happens in the server route; the frontend receives the finished file.

Supporting view: message state machine

Each Message has a status field with values pending / streaming / complete / error, which determines how it is displayed and how it participates in subsequent requests. The state transitions mainly happen in ChatInput.handleSubmit: when sending a message, the newly created assistant message starts as streaming; each received text chunk updates its content but keeps it streaming; the stream ending (done) transitions to complete; an error chunk or non-abort exception transitions to error; when the user actively interrupts (AbortError), it is instead treated as complete — the generated content is preserved.

  new-assistant["Create placeholder assistant message"] --> st-streaming["status=streaming"]
  st-streaming --> on-text["Receive text chunk: accumulate content, keep streaming"]
  on-text --> st-streaming
  st-streaming --> on-done["Receive done chunk"] --> st-complete["status=complete"]
  st-streaming --> on-err-chunk["Receive error chunk"] --> st-error["status=error, show error text"]
  st-streaming --> on-abort["User interrupts(AbortError)"] --> st-complete
  st-streaming --> on-fail["Other exception"] --> st-error

Supporting view: permissions and lifecycle

Permissions: Settings include a permission configuration — toggles for auto-approval by tool type (file_read / file_write / bash / web_search), plus a restricted directory list. This configuration lives in the zustand store's settings and is persisted to localStorage, but currently remains at the local configuration level only: the chat request body sent to the backend only contains {messages, model, stream}; permission settings are neither sent to the backend nor validated when sending messages. The UI only has the PermissionSettings component reading/writing it. It's essentially "switches prepared now, to be wired up once the backend can understand them."

Lifecycle: Almost the entire web subproject's data lifecycle is tied to browser localStorage. The zustand store uses the persist middleware to persist conversations, activeConversationId, settings, pinnedIds, recentSearches, and tags (key claude-code-chat), reading them back after refresh; while pure UI states such as isSearchOpen, settingsOpen, sidebarTab, and selectedConversationIds are forcibly reset to defaults during merge and are not persisted. The conversation "memory" is also entirely in the local store — every message sent resends the full history of the current conversation from the store to the backend; the backend does not keep context for the browser. This means clearing localStorage or switching browsers wipes history and context.

Relationship with main project's src/server/web: The main project's src/server/web/ is a separate standalone terminal web service (pty-server, session-manager, terminal.js, etc., using xterm to embed a real terminal in the browser). The web/ subproject does not reuse src/server/web code — it follows the SSE streaming conversation path, not the terminal PTY path. The two only intersect at the configuration layer: web/ points to the backend via NEXT_PUBLIC_API_URL (default localhost:3001). Whether that backend is src/server or another service is decided by deployment, not hardcoded. lib/terminal-compat.ts is a shim prepared for "porting terminal components using Ink from src/components to the browser" — providing browser-side defaults for process.stdout.columns/rows, process.exit, useInput/useApp, and other Ink/Node APIs — but currently no file in web/ imports it, so it's a "spacer reserved for future terminal component porting."

  • The /api/share route (generating shareable links) and /api/files/read|write routes are another set of backend proxies, following the same pattern as /api/chat: receive frontend requests, forward them to the real backend. Sharing supports public/unlisted/password visibility and multiple expiration durations.
  • pdf is a special case in the export route: the server side doesn't handle it, returning 400 with a message that it's handled by client-side window.print().
