Permission System

Every time Claude is about to execute an operation that could change the environment (run a bash snippet, write a file, call an MCP tool, etc.), the system first passes it through a "gate" that decides how to handle the action: allow it directly, pause and ask the user, or reject it outright. This module is that gate. It combines four runtime modes (default / plan / bypassPermissions / auto, plus the internal ant mode dontAsk), user-configured allow / deny / ask rules, tool-reported command-level checks, and (in auto mode) an AI classifier to produce a final verdict for each pending tool call: allow / ask / deny.

Input
A pending tool call (tool name + input arguments) + the current session's permission context (mode, allow/deny/ask rules from each source, additional working directories) + the current conversation history (used by the auto-mode classifier).
Output
A verdict: allow (possibly with user-modified input), ask (popup to the user, possibly with a "suggested rule" and concurrent classifier check), or deny (with a reason). The verdict is also logged and recorded for analytics.
  • The four modes are not flat switches; they form a "pre-processing + post-processing" layering: rule checks run first (deny / ask / safetyCheck cannot be bypassed), then the result is routed by mode (bypass allows directly, auto invokes the classifier, dontAsk turns ask into deny, headless turns ask into deny / uses hooks).
  • User-configured rules are grouped by source (userSettings/projectSettings/localSettings/flagSettings/policySettings/cliArg/command/session). The rule syntax is ToolName or ToolName(content); parentheses support escaping. MCP tools are matched by their full name mcp__server__tool, and mcp__server__* is supported for whole-server granularity.
  • Safety checks (safetyCheck), e.g. touching .git/, .claude/, or shell config files, are "bypass-immune" — even in bypassPermissions mode the user must be asked. Among these, entries with classifierApprovable=true (sensitive file paths) can be adjudicated by the classifier in auto mode, while false entries (Windows path bypass, cross-machine bridge messages) require human confirmation in every mode.
  • Auto mode has a circuit breaker: after 3 consecutive denials or 20 cumulative denials in the session, it falls back to human confirmation. If the classifier API is unavailable, the tengu_iron_gate_closed switch decides whether to fail-closed (deny and prompt retry) or fail-open (fall back to human). An overly long transcript is a deterministic error and falls back to human confirmation, or aborts in headless mode.

Main decision chain: from tool call to final verdict

This is the core data flow. From the user's perspective, the trigger is: Claude produces a tool_use in conversation (for example, it wants to run rm). Before the tool system actually executes it, it calls hasPermissionsToUseTool, and the following chain begins. First, "hard rule checks that every mode must pass" run, then the result is routed by mode, and finally some post-processing is applied (dontAsk flips to deny, auto runs the classifier, headless falls back to deny).

Auto-mode classifier flow

When the mode is auto (or auto is temporarily activated in plan mode) and the inner result is ask, this flow is entered. Its goal is to replace human confirmation with an independent AI classifier: feed the current conversation history plus the pending action to the classifier, and let it decide whether the action should be allowed. However, there are several fast paths that avoid invoking the classifier whenever possible (saving tokens and latency).

User interaction: from popup to resolved verdict

When hasPermissionsToUseTool returns ask, useCanUseTool takes over and pushes the permission request into the UI queue to wait for a user response. This section describes what the user can do in the popup and how the system converts the user's choice into a final verdict. Three sources of "allow" (manual yes, PreToolUse hook, concurrent bash classifier) race for the same verdict; an atomic createResolveOnce claim ensures only the first winner takes effect.

graph TD
  ask-result["hasPermissionsToUseTool returns ask"] --> coord{"awaitAutomatedChecksBeforeDialog?"}
  coord -->|Yes| coordinator["Run coordinatorHandler first<br/>(hook / bash classifier quick-answer)"]
  coordinator --> coord-decided{"Produced a verdict?"}
  coord-decided -->|Yes| done-coord["Apply that verdict"]
  coord-decided -->|No| swarm["Then run swarmWorkerHandler"]
  coord -->|No| swarm
  swarm --> swarm-decided{"Produced a verdict?"}
  swarm-decided -->|Yes| done-swarm["Apply that verdict"]
  swarm-decided -->|No| interactive["handleInteractivePermission<br/>push to UI queue and wait for user"]
  interactive --> user-resp{"User response"}
  user-resp -->|"Allow (may modify input / add rule)"| on-allow["handleUserAllow<br/>persist rule + log"]
  user-resp -->|"Deny (may include feedback)"| on-deny["Log denial"]
  user-resp -->|Esc abort| on-abort["cancelAndAbort<br/>abort this tool call"]
  on-allow --> done-allow2["Apply allow"]
  on-deny --> done-deny["Apply deny"]
  on-abort --> done-abort["Apply ask (abort message)"]
  • If the concurrent bash classifier (BASH_CLASSIFIER) produces an allow while the popup is open, it auto-allows before the user clicks; if the user starts keyboard interaction, the auto-answer is canceled after a 200ms grace period.
  • When the user chooses "Allow and always allow this kind of operation", persistPermissions writes the new rule to the appropriate source (userSettings/projectSettings/localSettings/session), so the next matching call goes straight through the allow-rule path.
  • Tools denied by the classifier in auto mode trigger a UI notification "Denied by auto mode", and the denial is recorded in recordAutoModeDenial for later display.

Mode-switch state machine (Shift+Tab cycle)

When the user presses Shift+Tab to cycle modes, getNextPermissionMode decides the next mode. default / plan / acceptEdits / bypassPermissions are the four user-visible modes; auto is only exposed to internal ant users, and only when the TRANSCRIPT_CLASSIFIER feature flag is on and the auto gate (isAutoModeGateEnabled, including circuit-breaker state) passes. dontAsk is not exposed in the cycle. When entering auto, transitionPermissionMode also strips dangerous permission rules (to prevent the auto + dangerous-rule combination).

Rule matching decision table

Rules are bucketed by behavior: allow / deny / ask. Matching separates "whole-tool" rules from "content" rules: ToolName (no parentheses) matches the whole tool; ToolName(content) is interpreted by the tool's own checkPermissions (bash does command-prefix matching, Write/Edit does glob path matching). Whole-tool deny/ask run before the tool's self-check; whole-tool allow runs after the tool's self-check — this order is deliberate: deny has the highest priority, ask forces a popup, and allow only takes effect when no deny/ask objects.

graph TD
  start["Rule matching entry"] --> d1{"Whole-tool deny rule?<br/>toolMatchesRule (no content)"}
  d1 -->|Hit| out-deny["deny, reason=rule"]
  d1 -->|No| d2{"Whole-tool ask rule?"}
  d2 -->|Hit and not sandbox allow| out-ask["ask, reason=rule"]
  d2 -->|No / sandbox allow| d3{"Tool checkPermissions<br/>returns deny / content ask / safetyCheck?"}
  d3 -->|deny| out-tool-deny["deny (tool self-report)"]
  d3 -->|"content ask (rule)"| out-ask-content["ask, reason=rule<br/>(bypass-immune)"]
  d3 -->|safetyCheck| out-ask-safety["ask, reason=safetyCheck<br/>(bypass-immune)"]
  d3 -->|passthrough/allow| d4{"mode=bypass?"}
  d4 -->|Yes| out-allow-bypass["allow, reason=mode"]
  d4 -->|No| d5{"Whole-tool allow rule?<br/>toolAlwaysAllowedRule"}
  d5 -->|Hit| out-allow-rule["allow, reason=rule"]
  d5 -->|No| out-ask-default["passthrough→ask<br/>(hand to mode post-processing)"]
  • Rule source priority is merged by permissionsLoader (policy > userSettings > projectSettings > localSettings > ...), ultimately collapsed into the three buckets attached to ToolPermissionContext. Within the same bucket, earlier sources shadow later sources with the same content (shadowedRuleDetection is responsible for warning about shadowed rules).
  • Rule string parsing happens in permissionRuleValueFromString: find the first unescaped ( and the last unescaped ); the content inside is ruleContent; Bash(*) and Bash() degenerate into whole-tool rules; parentheses themselves can be escaped with \( \).
  • Agent-type denial is a special syntax Agent(Explore); filterDeniedAgents collects all denied agent types at once and then filters, avoiding O(agent×rule) repeated parsing.
