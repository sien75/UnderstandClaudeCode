Permission System

Every time Claude is about to perform an operation that could change the environment (run a bash snippet, write a file, call an MCP tool, etc.), the system first runs a "gate check" to decide how to handle the action: allow it directly, pause and ask the user, or reject it outright. This module is that gate. It combines four runtime modes (default / plan / bypassPermissions / auto, plus the internal ant mode dontAsk), user-configured allow/deny/ask rules, tool-self-reported command-level checks, and (in auto mode) an AI classifier to deliver a final verdict for each pending tool call: allow / ask / deny.

Input
A pending tool call (tool name + input parameters) + the current session's permission context (mode, allow/deny/ask rule lists from each source, additional working directories) + the current conversation history (used by the auto-mode classifier).
Output
A verdict: allow (permit, possibly with user-edited input), ask (pop-up asking the user, possibly with a "suggested rule" and concurrent classifier checks), or deny (rejected, with a reason). The verdict is also logged and recorded for analytics.
  • The four modes are not flat switches; they form a "pre-processing + post-processing" layering: rule checks run first (deny/ask/safetyCheck bypasses cannot skip these), then the mode branches (bypass allows directly, auto calls the classifier, dontAsk turns ask into deny, headless turns ask into deny / goes through hooks).
  • User-configured rules are grouped by source (userSettings/projectSettings/localSettings/flagSettings/policySettings/cliArg/command/session). The rule syntax is ToolName or ToolName(content); parentheses support escaping. MCP tools are matched by their full mcp__server__tool name, and mcp__server__* is supported for whole-server granularity.
  • Safety checks (safetyCheck), such as touching .git/, .claude/, or shell config files, are "bypass-immune" — even in bypassPermissions mode the system must stop and ask. Among these, entries with classifierApprovable=true (sensitive file paths) can be judged by the classifier in auto mode; those with false (Windows path bypasses, cross-machine bridge messages) require human confirmation in every mode.
  • Auto mode has a circuit breaker: after 3 consecutive classifier denials or 20 cumulative denials in the session, it falls back to human confirmation; if the classifier API is unavailable, the tengu_iron_gate_closed switch decides fail-closed (deny and prompt retry) or fail-open (fall back to human); transcript-too-long is a deterministic error that falls back to human or, in headless mode, aborts.

Main Decision Chain: From Tool Call to Final Verdict

This is the core data flow. From the user's point of view, the trigger is: Claude produces a tool_use in the conversation (for example, it wants to run rm). Before the tool system actually executes it, it calls hasPermissionsToUseTool, and the following chain begins. It first runs "hard rule checks that every mode must pass", then branches by mode, and finally performs some post-processing (dontAsk turns ask to deny, auto runs the classifier, headless falls back to deny).

Auto-Mode Classifier Flow

When the mode is auto (or auto is temporarily active in plan mode) and the inner result is ask, this flow is entered. Its goal is to use an independent AI classifier in place of human confirmation: feed the current conversation history plus the action about to be taken into the classifier, and let it decide whether the action should be allowed. However, there are several fast-path checks that avoid calling the classifier whenever possible (saving tokens and latency).

User Interaction: From Ask Popup to Verdict Commitment

When hasPermissionsToUseTool returns ask, useCanUseTool takes over and pushes the permission request into the UI queue to await the user's response. This section describes what the user can do in the popup and how the system commits the user's choice back to a verdict. Three sources of "allow" (the user clicking yes, a PreToolUse hook, and a concurrent bash classifier) race for the same verdict; createResolveOnce's atomic claim guarantees only the first winner takes effect.

graph TD
  ask-result["hasPermissionsToUseTool returns ask"] --> coord{"awaitAutomatedChecksBeforeDialog?"}
  coord -->|Yes| coordinator["Run coordinatorHandler first<br/>(hook / bash classifier snipes answer)"]
  coordinator --> coord-decided{"Got a verdict?"}
  coord-decided -->|Yes| done-coord["Commit that verdict"]
  coord-decided -->|No| swarm["Then run swarmWorkerHandler"]
  coord -->|No| swarm
  swarm --> swarm-decided{"Got a verdict?"}
  swarm-decided -->|Yes| done-swarm["Commit that verdict"]
  swarm-decided -->|No| interactive["handleInteractivePermission<br/>Push to UI queue and wait for user"]
  interactive --> user-resp{"User response"}
  user-resp -->|"Allow (possibly edit input / add rule)"| on-allow["handleUserAllow<br/>Persist rule + log"]
  user-resp -->|"Deny (possibly with feedback)"| on-deny["Log denial"]
  user-resp -->|Esc abort| on-abort["cancelAndAbort<br/>Abort this tool call"]
  on-allow --> done-allow2["Commit allow"]
  on-deny --> done-deny["Commit deny"]
  on-abort --> done-abort["Commit ask (abort message)"]
  • If the concurrent bash classifier (BASH_CLASSIFIER) returns allow during the popup, it can auto-allow before the user clicks; once the user begins keyboard interaction (after a 200ms grace period), this auto-snipe is canceled.
  • When the user selects "Allow and always allow this kind of operation", persistPermissions writes the new rule to the corresponding source (userSettings/projectSettings/localSettings/session). Next time, the same kind of call goes directly through the allow-rule path.
  • Tools denied by the classifier in auto mode surface a UI notification "Rejected by auto mode" and record the denial in recordAutoModeDenial for later display.

Mode Switching State Machine (Shift+Tab Cycle)

The user presses Shift+Tab to cycle through modes; getNextPermissionMode decides the next mode. default/plan/acceptEdits/bypassPermissions are the four user-visible modes; auto is only shown to internal ant users, and only when the TRANSCRIPT_CLASSIFIER feature flag is on and the auto gate (isAutoModeGateEnabled, including circuit-breaker status) passes; dontAsk is not exposed in the cycle. Entering auto via transitionPermissionMode strips dangerous permission rules (preventing the dangerous auto + dangerous rule combination).

Rule Matching Decision Table

Rules are divided into three buckets by behavior: allow / deny / ask. Matching separates "whole tool" from "with content": ToolName (no parentheses) matches the whole tool; ToolName(content) is interpreted by the tool's own checkPermissions (bash uses command-prefix matching, Write/Edit uses glob path matching). Whole-tool deny/ask run before the tool's self-reported check; whole-tool allow runs after the tool's self-reported check — this order is intentional: deny has the highest priority, ask forces a popup, and allow only takes effect when no deny/ask objects.

graph TD
  start["Rule matching entry"] --> d1{"Whole-tool deny rule?<br/>toolMatchesRule (no content)"}
  d1 -->|Hit| out-deny["deny, reason=rule"]
  d1 -->|No| d2{"Whole-tool ask rule?"}
  d2 -->|Hit and not sandbox allow| out-ask["ask, reason=rule"]
  d2 -->|No/sandbox allow| d3{"Tool checkPermissions<br/>returns deny/content-level ask/safetyCheck?"}
  d3 -->|deny| out-tool-deny["deny (tool self-reported)"]
  d3 -->|"content-level ask(rule)"| out-ask-content["ask, reason=rule<br/>(bypass-immune)"]
  d3 -->|safetyCheck| out-ask-safety["ask, reason=safetyCheck<br/>(bypass-immune)"]
  d3 -->|passthrough/allow| d4{"Mode=bypass?"}
  d4 -->|Yes| out-allow-bypass["allow, reason=mode"]
  d4 -->|No| d5{"Whole-tool allow rule?<br/>toolAlwaysAllowedRule"}
  d5 -->|Hit| out-allow-rule["allow, reason=rule"]
  d5 -->|No| out-ask-default["passthrough→ask<br/>(Hand to mode post-processing)"]
  • Rule source precedence is merged by permissionsLoader (policy > userSettings > projectSettings > localSettings > ...), finally collapsing into the three rule buckets attached to ToolPermissionContext. Within the same bucket, earlier-source rules shadow later-source rules with the same content (shadowedRuleDetection warns about shadowed rules).
  • Rule string parsing happens in permissionRuleValueFromString: find the first unescaped ( and the last unescaped ); the inside is ruleContent; Bash(*) and Bash() degenerate into whole-tool rules; parentheses themselves can be escaped with \( \).
  • Agent-type denial is special syntax Agent(Explore); filterDeniedAgents collects all denied agent types at once and then filters, avoiding O(agent×rule) repeated parsing.
