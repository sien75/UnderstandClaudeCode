Terminal UI Library (@earendil-works/pi-tui)

A terminal UI library with differential rendering: it turns raw stdin bytes into structured key events, lets components consume those events to update their own state, then diffs the new render tree against the previous frame and writes only the changed rows to stdout. The coding-agent's interactive mode mounts a component tree (editor + markdown output + autocomplete), feeds stdin into it, and reads back submitted lines.

Module
Raw terminal bytes in → decoded key events → component state mutations → diffed render buffer out to stdout.
Input
Raw bytes from process.stdin in raw mode: key presses, paste (bracketed-paste wrapped), mouse sequences, Kitty keyboard CSI-u sequences, resize events, and terminal query responses (OSC 11 background color, DSR color-scheme, cell-size replies).
Output
An ANSI escape-sequence buffer written to process.stdout that repositions the cursor, clears only the changed lines, re-renders them, and optionally places a hardware cursor for IME — all inside a synchronized-output wrapper (ESC[?2026h…l) to prevent tearing.
Notes
The library is transport-agnostic at the edges: ProcessTerminal talks to real stdin/stdout, but the Terminal interface could back a test or in-memory terminal. The core TUI class is a Container of components; each component implements render(width) returning string[] and optionally handleInput(data). Differential rendering is the central optimization: unchanged lines are never rewritten.

What a caller can do with this library

The coding-agent interactive mode (the primary external caller) uses the library to:

  • Mount a component tree — add an Editor (line input), Markdown (streamed agent output), Loader (spinner), SelectList, or overlays (autocomplete menu, settings) to a TUI instance.
  • Feed stdin — call tui.start(); the library takes over process.stdin in raw mode, decodes keys, and routes each event to the focused component's handleInput.
  • Render frames — any component calls tui.requestRender(); the TUI batches renders to at most one per 16ms tick, diffs against the previous frame, and writes only changed lines.
  • Read line input — the Editor component exposes onSubmit/onChange callbacks; when the user presses Enter (matched via keybindings), the submitted text flows out to the caller.
  • Drive an editor — the Editor supports multi-line editing, history (up/down), undo/redo, kill-ring (Emacs yank), word navigation, and autocomplete (slash commands + file paths).
  • Query the terminal — queryTerminalBackgroundColor, queryTerminalColorScheme, and cell-size queries return promises; responses are intercepted from stdin and resolved.
  • Show overlays — showOverlay positions a modal component (autocomplete, settings) on top of base content with anchor/percentage sizing and focus management.

The action that triggers the whole chain is tui.start(): it flips stdin into raw mode, enables bracketed paste, queries the Kitty keyboard protocol, and wires the data callback to TUI.handleInput.

Data Flow: Input Decode (stdin bytes → key event → focused component)

This is the path a single key press travels. Bytes arrive on process.stdin, get split into complete escape sequences by StdinBuffer, are inspected for Kitty-protocol negotiation or terminal query responses by ProcessTerminal, then forwarded to TUI.handleInput which routes them to the focused component (after global interceptors: OSC 11 replies, color-scheme reports, cell-size replies, debug key, overlay focus restore).

Data Flow: Render / Diff (component tree → changed stdout bytes)

This is the path a render frame travels. A component calls requestRender; the TUI coalesces into a nextTick + 16ms-throttled timer; doRender renders all children + composites overlays, extracts the IME cursor marker, normalizes lines, then either does a full redraw (first render, width/height change, clear-on-shrink, viewport misalignment) or a differential update that writes only the firstChanged..lastChanged range inside synchronized output.

graph TD
    req[requestRender called by component] --> force{Force flag?}
    force -->|yes| force-reset[Reset previousLines/width/height, nextTick doRender]
    force -->|no| already{renderRequested already true?}
    already -->|yes| req-stop[Return, coalesce]
    already -->|no| sched-next[renderRequested true, nextTick scheduleRender]
    sched-next --> timer{renderTimer active?}
    timer -->|yes| sched-stop[Return]
    timer -->|no| delay[Compute 16ms minus elapsed]
    delay --> fire[setTimeout fires, doRender]
    force-reset --> fire

The scheduling phase above coalesces repeated requestRender calls into a single doRender invocation throttled to one frame per 16ms tick. Once fire (the doRender entry point) is reached, execution continues into the render and diff phase below, which renders the component tree, composites overlays, and writes only the changed line range to stdout.

Data Flow: Editor (key event → text state → render lines)

The Editor is the primary input-consuming component. handleInput consults the global KeybindingsManager to dispatch each key to an action; actions mutate state.lines/cursorLine/cursorCol (with undo snapshots, kill-ring accumulation); render wraps text to width, applies scroll offset, renders the cursor as a highlighted grapheme, and appends the autocomplete list if active. Flag state machines: jumpMode (forward/backward awaiting next char), paste buffering, autocomplete navigation, undo stack, kill-ring yank-pop cycling.

The dispatch phase above resolves jump mode, paste, copy, undo, and autocomplete interactions; most non-terminal paths converge on the tab-trigger decision node, which is the entry point to the key-action dispatch phase below. The ac-submit node (slash-command confirm) also falls through to the submit node in the lower diagram.

State machines flagged

  • Editor jumpMode (editor.ts:595-613): null | "forward" | "backward". A jump hotkey sets the mode; the next printable char performs jumpToChar and clears it; a control char or repeat hotkey cancels.
  • Editor paste buffering (editor.ts:616-639, input.ts:48-84): isInPaste toggles on ESC[200~, accumulates until ESC[201~, then handlePaste + recurse on remainder.
  • Editor autocomplete navigation (editor.ts:653-711): when autocompleteState is set, up/down/cancel/tab/confirm are intercepted by the SelectList child; confirm on a slash command falls through to submit.
  • Undo stack (undo-stack.ts, editor.ts pushUndoSnapshot/undo): clone-on-push snapshots; pop returns the detached clone. Cleared on submit.
  • Kill ring (kill-ring.ts, editor.ts yank/yankPop): ring buffer; consecutive kills accumulate (prepend/append); yank peeks the last; yankPop rotates to cycle older entries.
  • Overlay focus restore (tui.ts:241-258, 370-470): inactive | eligible | blocked{overlay, blockedBy, resume}. Blocked transitions track when a non-overlay steals focus from a focused overlay; resume resolves to restore-overlay or focus-target.
  • Kitty keyboard protocol negotiation (terminal.ts:220-301): query → buffer fragments → parse kitty-flags vs device-attributes → enable Kitty (flags != 0) or modifyOtherKeys fallback; 150ms fragment timeout flushes as input.
  • StdinBuffer sequence completeness (stdin-buffer.ts:29-78): per-slice state machine complete | incomplete | not-escape for CSI/OSC/DCS/APC/SS3/meta; 10ms timeout flushes stuck incomplete remnants.
