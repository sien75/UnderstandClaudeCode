Interactive Coding Agent CLI (`pi`)

This module is the user-facing package @earendil-works/pi-coding-agent — the pi command line tool. It is the integration layer that wires the rest of the monorepo together: it owns the CLI entry, the three runtime modes (interactive TUI, one-shot print, headless JSON-RPC), the built-in filesystem/shell tools (read, bash, edit, write, plus optional grep/find/ls), session and config management, and the system prompt. It drives the @earendil-works/pi-agent-core runtime, which in turn talks to LLM providers via @earendil-works/pi-ai.

What a user can do with pi:

  • Interactive chat: run pi in a terminal to get a full-screen TUI with streaming model output, file/bash/edit/write tool calls rendered inline, slash commands, skills, prompt templates, model cycling (Ctrl+P), and mid-stream steering.
  • One-shot prompt: pi -p "list .ts files" (or pi --mode json "...") runs a single prompt non-interactively, prints the final answer (or a JSON event stream), and exits. Piped stdin is treated as part of the prompt.
  • Headless RPC: pi --mode rpc turns the process into a JSON-line server on stdin/stdout — an orchestrator (editor extension, CI runner, another agent) sends commands like prompt, steer, abort, switch_session, fork and receives streamed session events.
  • Session management: resume (--continue / --resume), open by id/path (--session), fork (--fork), name (--name), or run ephemeral (--no-session).
  • Slash commands / skills: in a prompt, text starting with / is an extension command; /skill:name args inlines a skill file; /template args expands a prompt template.
  • Package/config subcommands: pi install <source>, pi remove, pi update, pi list, pi config manage extensions and settings; these exit before any agent runtime is built.
  • Export: pi --export session.jsonl [output.html] renders a saved session to HTML and exits.

The chain is triggered the moment the user runs pi ...: cli.ts hands process.argv to main(), which parses args, builds a session + agent runtime, then dispatches to one of the three modes. The mode feeds user text to AgentSession.prompt(), which drives the Agent loop in pi-agent-core; the Agent calls the LLM provider, executes tool calls against the filesystem/shell, and loops until the model stops requesting tools. Streaming events flow back through AgentSession to the mode's listener, which renders them (TUI) or serializes them (print/rpc).

Module
The pi CLI turns a user's typed request (plus files, piped stdin, or JSON-RPC commands) into a series of LLM turns interleaved with filesystem/shell tool executions, ending in a rendered answer — interactive, printed, or streamed as JSON events.
Input
CLI args + a prompt (typed text, @file references, piped stdin, or JSON-RPC prompt commands), evaluated against the current working directory.
Output
For interactive: a live TUI. For print: final assistant text on stdout (or a JSON event stream). For rpc: a JSON-line event/response stream on stdout. Side effects: files read/written/edited, shell commands executed.
Notes
The package itself does not implement the LLM turn loop; that lives in @earendil-works/pi-agent-core (Agent). This package supplies the tools, system prompt, auth/settings/model registry, session persistence, extension hooks, and the user-facing modes. External boundaries: the LLM provider is reached via streamSimple in the streamFn configured on the Agent; the filesystem and shell are reached only through the built-in tools.

Diagram 3 — The prompt loop (AgentSession → Agent → tools → LLM)

This is the heart of the data flow. AgentSession.prompt() is the entry called by all three modes. It expands commands/templates, validates auth, and calls _runAgentPrompt, which calls agent.prompt() and then loops on agent.continue() while post-run handling (auto-retry, compaction, queued follow-ups) says to continue. The Agent itself (in pi-agent-core) owns the LLM call and the tool-call dispatch loop; the coding-agent package supplies the streamFn (which calls streamSimple with provider auth/headers) and the tool definitions whose execute functions touch the filesystem/shell.

State machines flagged

  • Agent streaming state (agent.state.isStreaming): idle → streaming (during agent.prompt/agent.continue) → idle. AgentSession.isStreaming gates whether prompt() queues via steer/followUp or sends immediately. This is the core state machine governing user input routing.
  • Session lifecycle (AgentSessionRuntime): a runtime is created, then can be torn down and replaced via newSession / switchSession / fork / importFromJsonl / reload / dispose. Each replacement emits session_before_switch, tears down (emits session_shutdown with reason new/resume/fork/reload/quit), re-runs the factory, and rebinds.
  • Auto-retry state (_retryAttempt): on a retryable error (overloaded/rate-limit/server, but not context overflow), emits auto_retry_start with exponential backoff, continues the agent; on exhaustion emits auto_retry_end with success=false.
  • Compaction state: when the last assistant message overflows context, _checkCompaction runs a compaction turn (emits compaction_start/compaction_end) and continues the agent with the compacted context. Context-overflow errors are routed to compaction, not retry.
  • Project trust state: untrusted → trusted, cached per-cwd in projectTrustByCwd; --approve/--no-approve override; interactive mode can prompt.
  • RPC shutdown state (shutdownRequested/shuttingDown): a shutdown command or stdin-end sets the flag; after the current command finishes, checkShutdownRequested runs the full teardown.

Notes on external boundaries

  • LLM provider: reached only inside streamFn configured on the Agent in sdk.ts. It calls modelRegistry.getApiKeyAndHeaders(model) for credentials, merges provider-attribution headers, lets extensions adjust headers via the before_provider_headers hook, then calls streamSimple from @earendil-works/pi-ai. The onPayload/onResponse hooks let extensions observe/modify the outgoing request and the incoming response status/headers.
  • Filesystem & shell: reached only through the seven built-in tools. The bash tool spawns a real shell (spawn with getShellEnv and configurable shellPath/commandPrefix), tracks the process tree for abort/timeout, and streams stdout/stderr. read/edit/write/grep/find/ls operate on real files under cwd. File-mutating tools are serialized through withFileMutationQueue.
  • TUI rendering: InteractiveMode subscribes to AgentSession events and renders streaming text, tool calls, and approvals through @earendil-works/pi-tui components. The TUI also captures user input (steering, slash commands, model cycling) and feeds it back into session.prompt.
  • Session persistence: SessionManager appends every agent event to a JSONL session file under the session directory; on resume/continue/fork it rebuilds the message context from that file. This is what makes --continue, --resume, and /fork work.
  • Extensions: loaded by DefaultResourceLoader; they can register slash commands, custom tools, prompt-template/skill/theme providers, model providers, and hook into input, tool_call, tool_result, before_agent_start, before_provider_request, after_provider_response, session_start/session_shutdown, session_before_switch/session_before_fork, and context transform events.
