Multi-Agent Orchestrator

The @earendil-works/pi-orchestrator package is a supervisor process that spawns and manages multiple coding-agent child processes, routes tasks and RPC traffic between a caller (CLI or external client) and those workers, and coordinates their lifecycle. From the user's perspective, you run orchestrator serve once to start the long-running supervisor daemon; then every other command — list, spawn, status, stop, rpc, rpc-stream — is a thin client that connects to that daemon over a Unix socket, sends a JSON-Lines request, and prints the JSON response. The supervisor itself spawns each agent as a child process and speaks to it over its stdin/stdout using the same JSON-Lines RPC protocol defined by @earendil-works/pi-coding-agent. Optionally, the supervisor also registers itself and each worker with a remote "Radius" presence service so other machines can discover and reach the orchestrator.

cli.ts — command dispatcher
Parses process.argv and routes to either serve() (start the daemon) or one of the client commands that call sendIpcRequest. For rpc-stream it opens a raw bidirectional socket and pipes stdin JSON-Lines into it. Input: argv flags. Output: printed JSON response or a running daemon.
serve.ts — daemon entry
Starts the IPC socket server, recovers persisted instance state after a restart, optionally starts Radius presence, then blocks forever until a signal or fatal error triggers a graceful shutdown that closes the server, stops all workers, and unlinks the socket. Input: nothing (side effect: a running socket). Output: a long-lived process.
ipc/server.ts — socket server
Accepts Unix-socket connections, reads newline-delimited JSON, parses each line into an OrchestratorRequest, dispatches it to the handler, and writes the JSON response back. For rpc_stream requests it upgrades the connection into a long-lived bidirectional bridge that fans agent session events and UI requests out to the client and queues inbound RPC commands serially. Input: JSON-Lines requests on a socket. Output: JSON-Lines responses and streamed events.
ipc/client.ts — socket client
One-shot helper used by every non-serve CLI subcommand: opens a connection to the orchestrator socket, writes one request, reads one response, and returns it. Input: an OrchestratorRequest. Output: an OrchestratorResponse.
ipc/protocol.ts — wire format
Defines the request/response discriminated unions and the encodeMessage/parseRequestLine/parseResponseLine helpers. Every byte on the socket is JSON.stringify(msg) + "\n".
handler.ts — request router
Switches on request.type and delegates to the matching supervisor method, mapping InstanceRecord to the wire-friendly InstanceSummary. Also exposes openRpcStream which adapts the supervisor's stream handle into the socket-facing callback shape.
supervisor.ts — lifecycle and routing core
Holds the in-memory liveInstances map (one LiveInstance per worker, binding a persisted InstanceRecord to the live RpcProcessInstance, subscriber set, and UI-request handler). Owns spawn, stop, RPC routing, stream subscription, crash recovery, and shutdown. Persists instance metadata to instances.json and refreshes it after session-changing RPCs. Wires itself as the Radius coordinator.
rpc-process.ts — child process wrapper
Spawns one pi --mode rpc (or pi-coding-agent/rpc-entry) child process per worker, frames JSON-Lines over stdin/stdout, correlates responses to pending requests by id, fans session events and extension_ui_request messages to listeners, and detects child exit/error. Input: RpcCommand. Output: RpcResponse + async AgentSessionEvent stream. External boundary: a child Node process.
storage.ts — persistence
Reads/writes instances.json and machine.json under the orchestrator dir. Pure JSON file IO, no caching.
config.ts — paths and environment
Resolves the orchestrator directory (PI_ORCHESTRATOR_DIR or ~/.pi/orchestrator), the socket path, the instances/machine/auth paths, the package version, and the isBunBinary flag that picks the child spawn command.
radius.ts — remote presence
Optional integration with the Radius service: registers this machine and each Pi (worker), runs heartbeats with exponential backoff + jitter, and re-registers after repeated 404s. External boundary: the Radius HTTP API.
types.ts — shared types
InstanceStatus ("starting" | "online" | "stopping" | "stopped" | "error"), InstanceRecord, MachineRecord, RadiusRegistration.
