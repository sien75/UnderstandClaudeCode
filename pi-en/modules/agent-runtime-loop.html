Agent Runtime Loop

This module is the heart of @earendil-works/pi-agent-core: it turns a caller's prompt plus a set of tools into a multi-turn conversation with an LLM, executing tool calls the LLM requests and feeding the results back until the agent has nothing left to do. Think of it as a small runtime: you hand it a goal and some capabilities, it loops "ask the model, run what the model asked, repeat" until the model stops calling tools and there are no more queued messages.

There are two layers. The lower layer (agent-loop.ts) is a pure function — given a context, a config, and an event sink, it runs the loop and returns the new messages. The upper layer (agent.ts) is a stateful Agent class that owns the transcript, manages run lifecycle, and exposes queueing for "steering" (interrupt mid-turn) and "follow-up" (run again after the agent would otherwise stop). A richer AgentHarness (in harness/) layers session persistence, hooks, skills, and prompt templates on top.

What the caller can do

Start a prompt
Agent.prompt(text|messages) or AgentHarness.prompt(text) kicks off a fresh run. This is the main trigger for the loop.
Continue from the existing transcript
Agent.continue() resumes the loop from the last message without adding a new user prompt (used after retries or when the last message is already a user/tool-result).
Steer mid-run
steer(message) queues a message injected before the next assistant response — lets the user redirect the agent while it is working.
Follow up
followUp(message) queues a message processed only after the agent would otherwise stop — a "one more thing" mechanism.
Invoke a skill or prompt template
AgentHarness.skill(name) / promptFromTemplate(name, args) format a skill or template into a prompt and run a turn.
Abort
abort() cancels the active run; the loop surfaces an aborted stop reason.
Compact or navigate the session tree
AgentHarness.compact() summarizes old context; navigateTree(targetId) moves the session leaf to another branch. These are out-of-loop operations on stored history.

How data flows through one run

The conversation is held as AgentMessage[] — a union of normal LLM messages (user/assistant/toolResult) plus app-specific custom messages (bash output, compaction summaries, branch summaries, arbitrary "custom" messages). The loop only converts this transcript to the LLM's native Message[] shape at the very edge, right before calling the model. This separation lets apps carry UI-only or structured data in the transcript that never reaches the provider.

State lives in three places: the Agent's _state (transcript, model, tools, streaming flags), the per-run AgentContext snapshot (system prompt + messages + active tools), and — under AgentHarness — the Session tree, which is the durable source of truth that each turn's context is rebuilt from.

Tool execution data flow

When the assistant message contains tool calls, the loop executes them as a batch. Each tool call goes through three phases: prepare (find the tool, validate arguments, run the beforeToolCall hook which can block it), execute (call tool.execute, streaming partial updates), and finalize (run afterToolCall which can override the result). Tools run sequentially by default or in parallel — but even in parallel mode, preparation is sequential and only execution is concurrent. A batch terminates early only when every finalized result sets terminate: true.

State machine note

The loop is itself a natural state machine. Per run, the AgentHarness tracks an explicit phase of idle, turn, compaction, branch_summary, or retry (defined in harness/types.ts). Within a turn, the implicit states are: streaming assistant response → executing tool batch → turn settled → either next turn (if tool calls, steering, or follow-ups exist) or agent ended. The streamed assistant message itself also cycles through sub-states emitted as events: message_start → message_update (text/thinking/tool-call deltas) → message_end. Tool calls cycle tool_execution_start → tool_execution_update → tool_execution_end.

External boundaries

  • LLM provider call — the streamFn (default streamSimple from @earendil-works/pi-ai) is the single external boundary where the transcript leaves the process. Its contract: never throw; encode failures as a final assistant message with stopReason: "error" | "aborted". The streamProxy variant routes this call through a server that holds auth and proxies to providers, reconstructing the partial message client-side from bandwidth-optimized events.
  • Tool execution — each tool.execute is an external boundary (tools may run shell commands, hit APIs, touch the filesystem). Tools signal failure by throwing; the loop catches and converts to an error tool result.
  • Session storage — under AgentHarness, every message_end and turn boundary is persisted through the Session (JSONL or in-memory repo), so a run can be resumed or forked.

Key extension points

convertToLlm
Controls which transcript messages reach the model and in what shape. Custom message types (bash output, summaries, custom notifications) are mapped to user messages here; excludeFromContext bash messages are filtered out entirely.
transformContext
Runs just before convertToLlm on the AgentMessage[] level — the hook for context-window management (pruning, compaction) and injecting external context. Under the harness, the context hook can rewrite messages here.
beforeToolCall / afterToolCall
Per-tool-call interception. before can block a call; after can rewrite the result (content, details, error flag, terminate hint). The harness surfaces these as tool_call / tool_result events.
prepareNextTurn
Between turns, can replace the context, model, or thinking level. The harness uses this to flush pending session writes and rebuild a fresh turn state from the session.
shouldStopAfterTurn
Requests a graceful stop after the current turn — e.g. before the context window fills up — without aborting in-flight tool execution.
