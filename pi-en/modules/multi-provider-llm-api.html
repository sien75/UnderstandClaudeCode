Multi-Provider LLM API

This is the @earendil-works/pi-ai package — a single library that lets an application talk to many different LLM providers (Anthropic, OpenAI, Google, AWS Bedrock, and roughly 25 others) through one uniform streaming interface. The business value is: a coding agent or CLI tool can swap the model behind a request without rewriting call sites, and auth, retry, prompt-caching, and "thinking" controls are handled once for every provider. This report walks the system the way a request actually travels — from the caller choosing a model, through auth resolution and provider routing, down to the HTTP call and the streaming events handed back. Two side doors exist: a CLI for OAuth login, and a legacy compat entrypoint that preserves the old global API for apps that haven't migrated yet.

What the caller can do here

Stream a chat completion
The primary action. The caller (an agent runtime, or a higher-level app) picks a Model from a Models collection, hands over a Context (system prompt + conversation messages + optional tool definitions), and receives an AssistantMessageEventStream — an async iterable that emits partial text, thinking, and tool-call deltas, then terminates with a final AssistantMessage. Triggered by models.stream(model, context, options) or models.complete(...) (which just awaits the stream's result).
Stream with a "reasoning level"
streamSimple / completeSimple take a human-friendly reasoning: "low" | "medium" | "high" | ... instead of provider-specific thinking knobs. The provider maps it to whatever the upstream API expects.
Log in to an OAuth provider
From the CLI (npx @earendil-works/pi-ai login <provider>) a user runs the Anthropic / GitHub Copilot / OpenAI Codex OAuth flow and saves credentials to a local auth.json. This is the only interactive, human-facing entry point.
Generate images
A parallel image-generation pipeline (ImagesModels) with the same shape — currently only OpenRouter images is wired up.

Everything below is from the data-flow perspective: where a prompt enters, how it is authenticated, routed, transformed into a provider payload, sent over the wire, and how the streaming response is normalized back into pi's event protocol. State-machine details (OAuth token refresh) and structural details (the provider-adapter hierarchy) are flagged where they naturally appear.

Notes on this section:

  • Synchronous stream, asynchronous setup. Models.stream() must return an AssistantMessageEventStream synchronously (callers shouldn't await before they can start consuming). Auth resolution and lazy module loading are async, so lazyStream hands back an empty stream immediately and pushes events into it once setup completes. If setup throws (unknown provider, OAuth refresh failure), the stream is terminated with an error event carrying an empty AssistantMessage — never a thrown exception to the caller.
  • Auth is resolved per-request, not per-provider. resolveProviderAuth prefers an explicit options.apiKey override, then a stored credential, then ambient sources (env vars, AWS profiles, ADC files). A stored credential "owns" the provider: ambient env is consulted only when nothing is stored. This prevents silent env fallback after a failed refresh.
  • OAuth refresh is double-checked-locked. The optimistic expiry check runs unlocked; if expired, the refresh runs inside CredentialStore.modify(), which serializes writes per provider (in-memory via a promise chain; a persistent store would use a file lock). Under the lock it re-checks expiry so concurrent requests don't double-refresh. Refresh failure throws ModelsError("oauth") and preserves the stored credential for retry / re-login.
  • Provider routing is a two-level dispatch. Models finds the Provider by model.provider; the provider (built by createProvider) then dispatches on model.api to the right ProviderStreams implementation. A provider with a single API passes one api; a mixed-API provider passes a map keyed by model.api. If no implementation matches the API, the stream errors with ModelsError("stream").

Inside the provider HTTP call (Anthropic as the reference impl)

Every API implementation module under src/api/ exports a stream and streamSimple function with the same shape. They differ in payload construction and event decoding. The Anthropic messages implementation (api/anthropic-messages.ts) is the most thoroughly observed here; OpenAI-completions, OpenAI-responses, Google, Bedrock, and the rest follow the same skeleton with provider-specific transformations.

Notes on the provider call:

  • External boundary. client.messages.create() is the call that leaves the process — an authenticated HTTP POST to https://api.anthropic.com/v1/messages with an SSE response body. Everything before it is local transformation; everything after it is decoding a remote stream. Same boundary exists for the other providers (OpenAI responses endpoint, Google generateContent, Bedrock Converse, etc.).
  • Three auth flavors, one client constructor. createClient branches on (a) GitHub Copilot — bearer token + Copilot dynamic headers; (b) OAuth token — bearer auth plus Claude Code identity headers (claude-code-20250219, oauth-2025-04-20 beta flags, user-agent: claude-cli/...); (c) plain API key — x-api-key header. The OAuth "stealth" path exists so a Claude Pro/Max subscription token works against the same API as a paid API key.
  • onPayload / onResponse hooks. Callers can inspect or replace the final provider payload right before it ships, and observe raw HTTP response status/headers right after they arrive. This is how higher layers do request logging, redaction, and A/B routing without touching provider internals.
  • Streaming JSON repair. Tool-call argument deltas arrive as partial JSON fragments. parseStreamingJson incrementally parses them so the partial arguments object is always readable; parseJsonWithRepair fixes malformed SSE data. These live in utils/json-parse.ts.
  • Usage and cost are computed inline. Token counts arrive split across message_start (input/cache) and message_delta (output, reasoning). calculateCost(model, usage) multiplies by the model's per-million prices on every update; the final AssistantMessage carries a fully costed Usage. Anthropic's 1h cache writes are charged at 2x base input, handled specially in calculateCost.
  • Abort and error semantics. An AbortSignal is checked inside the SSE loop; on abort or any thrown error, the partial AssistantMessage is finalized with stopReason: "aborted" | "error" and an errorMessage, then emitted as an error event. The stream never throws to the caller — failures are values.

This is the other entry point — the CLI's login command, and the OAuthAuth.login interface that backs it. The Anthropic flow (authorization code + PKCE) is the reference; Copilot and OpenAI Codex follow the same shape with their own endpoints.

The matching refresh state machine (run lazily on first authenticated request after expiry, under the credential-store lock) is the oauthExpiry → oauthRefresh branch in the master diagram. refreshAnthropicToken POSTs grant_type=refresh_token to the same token endpoint and returns a new OAuthCredentials; the 5-minute skew on expires (expires_in * 1000 - 5 * 60 * 1000) makes the next request refresh proactively rather than racing the boundary.

The provider-adapter hierarchy (structure view)

Structurally, every built-in provider is built by createProvider() from the same parts: an id, a baseUrl, an auth block (one or both of ApiKeyAuth / OAuthAuth), a static model list (or a refreshModels fetcher for dynamic providers), and an api implementation. The built-in catalog is assembled in providers/all.ts, which wires ~30 provider factories into a fresh Models via builtinModels(). Most providers are thin: e.g. anthropicProvider() just declares env vars ANTHROPIC_OAUTH_TOKEN/ANTHROPIC_API_KEY, a lazy OAuth loader, the Anthropic model catalog, and the lazy anthropicMessagesApi() streams. The interesting structural variations:

  • Amazon Bedrock (providers/amazon-bedrock.ts) is the ambient-credential case: its ApiKeyAuth.resolve returns an empty auth object (no apiKey) for AWS profile / IAM / ECS / IRSA sources — the AWS SDK's default credential chain does the actual SigV4 signing inside the API implementation. A stored bearer token is the only non-ambient path.
  • Mixed-API providers (OpenAI, Google, Google Vertex) pass an api map keyed by model.api so a single provider can serve both openai-completions and openai-responses, or google-generative-ai vs google-vertex, dispatching per model.
  • Lazy wrappers everywhere. api/*.lazy.ts files wrap each API implementation module in lazyApi(() => import("./real.ts")) so the heavy provider SDKs (Anthropic SDK, AWS Bedrock SDK, Google SDK) are only loaded when a request for that API actually starts. auth/helpers.ts → lazyOAuth does the same for OAuth flow modules, keeping Node-only http.createServer code out of browser bundles.

The legacy compat entrypoint

compat.ts preserves the older global API (stream(), complete(), getModel()) that existed before the Models/Provider redesign. It maintains a separate in-process apiProviderRegistry keyed by API id, auto-registers the nine built-in API implementations on import, and routes a call by either delegating to a builtinModels() instance (when the model matches the built-in catalog) or by dispatching through the registry with env API-key injection (withEnvApiKey). It is explicitly marked for deletion once the coding-agent finishes migrating to createModels(). New code should not use it; it's documented here only because existing apps still import @earendil-works/pi-ai/compat.

Environment-key resolution

env-api-keys.ts answers "is this provider configured via environment?" without touching credentials. findEnvKeys maps provider ids to their conventional env vars (OPENAI_API_KEY, ANTHROPIC_API_KEY, etc.); getEnvApiKey goes further for ambient-credential providers: for google-vertex it checks ADC file existence plus GOOGLE_CLOUD_PROJECT/GOOGLE_CLOUD_LOCATION; for amazon-bedrock it recognizes the full AWS credential chain (profile, access keys, bearer token, ECS URIs, web identity). It returns the sentinel "<authenticated>" for ambient-configured providers so callers can show "configured" in status UIs without seeing a real secret. This module is the fallback path inside ApiKeyAuth.resolve implementations and the compat env-injection layer; the Models path uses the provider's own auth.apiKey.resolve instead.

Image generation (parallel pipeline)

images-models.ts mirrors the chat side: ImagesProvider / ImagesModels / createImagesProvider with the same auth-resolution and option-merging logic, but the contract is a single Promise<AssistantImages> rather than a stream (image generation isn't streamed). Failures are returned as an AssistantImages with stopReason: "error" — again, failures-as-values, no throws to the caller. Only openrouter-images is wired as a built-in image provider.

Module summary

packages/ai/src/index.ts
Barrel re-export of the core, side-effect-free public API. Provider factories and API implementations live under subpath exports; this file intentionally pulls in no generated catalogs or OAuth implementations so browser bundles stay lean.
packages/ai/src/cli.ts
The npx @earendil-works/pi-ai binary. Supports login <provider> and list; loads/saves OAuth credentials to auth.json in the cwd. No chat commands — it's purely an auth helper.
packages/ai/src/models.ts
The runtime heart. Defines Provider, Models, createModels, createProvider, and the auth-application + per-request dispatch logic. A prompt + context turns into an AssistantMessageEventStream here.
packages/ai/src/auth/resolve.ts
Auth decision tree: explicit override → stored credential (OAuth with locked refresh, or API key) → ambient. Wraps failures in ModelsError with codes auth/oauth.
packages/ai/src/auth/types.ts
The auth contracts: CredentialStore (serialized read-modify-write), ApiKeyAuth / OAuthAuth (with the refresh/toAuth split that lets Models own the locked refresh pattern).
packages/ai/src/api/anthropic-messages.ts
Reference API implementation. Converts a Context to Anthropic MessageCreateParams, builds the SDK client with the right auth flavor, decodes SSE, and emits pi's unified event protocol. The other api/*.ts files follow this skeleton.
packages/ai/src/api/lazy.ts
lazyStream + lazyApi — the synchronous-stream/async-setup bridge that every Models.stream and lazy API wrapper goes through. Setup failures become stream error events.
packages/ai/src/env-api-keys.ts
Provider → env-var mapping for status/availability checks; recognizes ambient AWS and Google ADC configurations. Used by compat and by status UIs, not by the Models request path.
packages/ai/src/compat.ts
Deprecated global API surface with an API-provider registry and env-key injection. Routes to builtinModels() when possible, else dispatches through the registry. Slated for deletion.
packages/ai/src/utils/oauth/anthropic.ts
Anthropic OAuth (Claude Pro/Max): PKCE + local callback server + manual-code-paste race, token exchange, and the anthropicOAuth OAuthAuth adapter (login/refresh/toAuth).
packages/ai/src/providers/all.ts
Assembles all built-in provider factories into a Models / ImagesModels collection and exposes typed static-catalog reads (getBuiltinModel).
