多提供商 LLM API

这是 @earendil-works/pi-ai 包 —— 一个让应用通过统一的流式接口与多种不同 LLM 提供商（Anthropic、OpenAI、Google、AWS Bedrock 等约 25 家）对话的单一库。业务价值在于：编码智能体或 CLI 工具可以在不重写调用点的情况下更换请求背后的模型，并且认证、重试、提示缓存和 “thinking” 控制对所有提供商只实现一次。本报告沿着请求的真实路径展开 —— 从调用方选择模型，到认证解析、提供商路由，再到 HTTP 调用以及返回的流式事件。此外有两条旁路：一条用于 OAuth 登录的 CLI，以及一个保留旧全局 API 的 compat 入口，供尚未迁移的应用使用。

调用方在这里能做什么

流式聊天补全
主要操作。调用方（智能体运行时或上层应用）从 Models 集合中挑选一个 Model，传入 Context（系统提示词 + 对话消息 + 可选工具定义），然后获得一个 AssistantMessageEventStream —— 一个异步可迭代对象，会发出部分文本、thinking 和工具调用增量，最后以一条最终的 AssistantMessage 结束。入口为 models.stream(model, context, options) 或 models.complete(...)（后者只是等待流的结果）。
带“推理级别”的流式调用
streamSimple / completeSimple 接受人类友好的 reasoning: "low" | "medium" | "high" | ...，而不是各提供商特定的 thinking 参数。提供商把它映射到上游 API 所需的值。
登录 OAuth 提供商
通过 CLI（npx @earendil-works/pi-ai login <provider>）运行 Anthropic / GitHub Copilot / OpenAI Codex 的 OAuth 流程，并将凭据保存到本地 auth.json。这是唯一一个面向人的交互式入口。
生成图片
并行的图片生成流水线（ImagesModels）具有同样的形态 —— 目前只接入了 OpenRouter images。

下面的内容全部从数据流视角展开：提示词从哪进入、如何认证、如何被路由、如何被转换成提供商负载、如何通过网络发送，以及流式响应如何被标准化回 pi 的事件协议。状态机细节（OAuth token 刷新）和结构细节（provider-adapter 层级）会在自然出现的位置标注。

本节说明：

  • 同步返回流，异步准备。 Models.stream() 必须同步返回一个 AssistantMessageEventStream（调用方不应在消费前 await）。认证解析和懒加载模块都是异步的，因此 lazyStream 立即返回一个空流，准备完成后把事件推入其中。如果准备阶段抛出异常（未知 provider、OAuth 刷新失败），流会以一个携带空 AssistantMessage 的 error 事件结束 —— 不会向调用方抛出异常。
  • 认证按请求解析，而不是按提供商解析。 resolveProviderAuth 优先使用显式的 options.apiKey 覆盖，其次是已存凭据，最后是环境来源（环境变量、AWS profile、ADC 文件）。已存凭据“拥有”该 provider：只有在没有已存凭据时才会 consult 环境变量。这避免了刷新失败后静默回退到环境变量。
  • OAuth 刷新采用双检锁。 乐观过期检查不加锁；若发现过期，刷新在 CredentialStore.modify() 内部执行，该函数按 provider 串行化写操作（内存中通过 promise 链；持久化存储会使用文件锁）。在锁内会重新检查过期时间，防止并发请求重复刷新。刷新失败抛出 ModelsError("oauth")，并保留已存凭据以便重试 / 重新登录。
  • 提供商路由是两级分发。 Models 按 model.provider 找到 Provider；provider（由 createProvider 构建）再按 model.api 分发到正确的 ProviderStreams 实现。单 API 的 provider 只传递一个 api；混合 API 的 provider 传递一个以 model.api 为键的 map。如果没有实现匹配该 API，流会以 ModelsError("stream") 报错。

Provider HTTP 调用内部（以 Anthropic 为参考实现）

Provider 调用说明：

  • 外部边界。 client.messages.create() 是离开进程的调用 —— 一个带 SSE 响应体的认证 HTTP POST，地址为 https://api.anthropic.com/v1/messages。此前全部是本地转换；此后全部是解码远程流。其他提供商也有同样的边界（OpenAI responses 端点、Google generateContent、Bedrock Converse 等）。
  • 三种认证风味，一个 client 构造器。 createClient 按三种情况分支：(a) GitHub Copilot — bearer token + Copilot 动态 headers；(b) OAuth token — bearer auth 加上 Claude Code identity headers（claude-code-20250219、oauth-2025-04-20 beta flags、user-agent: claude-cli/...）；(c) 普通 API key — x-api-key header。OAuth “隐身”路径使得 Claude Pro/Max 订阅 token 能像付费 API key 一样调用同一 API。
  • onPayload / onResponse 钩子。 调用方可以在最终 provider payload 发出前检查或替换它，并在原始 HTTP 响应状态/headers 到达后立即观察。这样上层就能在不触碰 provider 内部的情况下完成请求日志、脱敏和 A/B 路由。
  • 流式 JSON 修复。 工具调用参数增量以部分 JSON 片段到达。parseStreamingJson 增量解析它们，使部分 arguments 对象始终可读；parseJsonWithRepair 修复损坏的 SSE 数据。它们位于 utils/json-parse.ts。
  • 用量和成本在线计算。 Token 计数分散在 message_start（input/cache）和 message_delta（output、reasoning）中到达。calculateCost(model, usage) 在每次更新时按模型的百万 token 价格相乘；最终的 AssistantMessage 携带完整计费的 Usage。Anthropic 的 1h cache writes 按 2 倍基础输入计费，在 calculateCost 中特殊处理。
  • 中止与错误语义。 SSE 循环内部会检查 AbortSignal；一旦中止或抛出任何错误，部分 AssistantMessage 会以 stopReason: "aborted" | "error" 和 errorMessage 完成，然后作为 error 事件发出。流不会向调用方抛出 —— 失败被当作值传递。

配套的刷新状态机在过期后的第一次认证请求时懒执行，并在凭据存储锁下运行，即主图中的 oauthExpiry → oauthRefresh 分支。refreshAnthropicToken 向同一 token 端点 POST grant_type=refresh_token 并返回新的 OAuthCredentials；对 expires 预留 5 分钟偏移（expires_in * 1000 - 5 * 60 * 1000），让下一次请求主动刷新，而不是在边界竞速。

Provider-adapter 层级（结构视图）

从结构上看，每个内置 provider 都由 createProvider() 用相同部件构建：一个 id、一个 baseUrl、一个 auth 块（ApiKeyAuth / OAuthAuth 中的一种或两种）、静态模型列表（或动态 provider 的 refreshModels 拉取器），以及一个 api 实现。内置目录在 providers/all.ts 中组装，它把约 30 个 provider 工厂接入一个新的 Models，通过 builtinModels()。大多数 provider 很薄：例如 anthropicProvider() 只声明环境变量 ANTHROPIC_OAUTH_TOKEN/ANTHROPIC_API_KEY、一个懒加载 OAuth loader、Anthropic 模型目录，以及懒加载的 anthropicMessagesApi() 流。有意思的结构变体：

  • Amazon Bedrock（providers/amazon-bedrock.ts）是环境凭据场景：它的 ApiKeyAuth.resolve 返回空 auth 对象（没有 apiKey），因为 AWS profile / IAM / ECS / IRSA 等来源的 AWS SDK 默认凭据链会在 API 实现内部完成 SigV4 签名。只有已存 bearer token 才是非环境路径。
  • 混合 API provider（OpenAI、Google、Google Vertex）传递一个以 model.api 为键的 api map，使单个 provider 可同时服务 openai-completions 和 openai-responses，或 google-generative-ai 与 google-vertex，按模型分发。
  • 到处是懒加载包装。 api/*.lazy.ts 文件把每个 API 实现模块包装在 lazyApi(() => import("./real.ts")) 中，因此沉重的 provider SDK（Anthropic SDK、AWS Bedrock SDK、Google SDK）只在真正发起该 API 请求时才加载。auth/helpers.ts → lazyOAuth 对 OAuth 流程模块也做同样处理，把 Node 专属的 http.createServer 代码排除在浏览器打包之外。

Legacy compat 入口

compat.ts 保留了旧的全局 API（stream()、complete()、getModel()），存在于 Models/Provider 重构之前。它维护一个独立的进程内 apiProviderRegistry，以 API id 为键，在导入时自动注册 9 个内置 API 实现，并通过 env API-key 注入（withEnvApiKey）进行分发。显式标记为待删除，等 coding-agent 迁移到 createModels() 后移除。新代码不应使用它；这里仅作记录，因为现有应用仍可能导入 @earendil-works/pi-ai/compat。

环境变量 key 解析

env-api-keys.ts 在不触碰凭据的前提下回答“该 provider 是否通过环境配置？”。findEnvKeys 把 provider id 映射到常规环境变量（OPENAI_API_KEY、ANTHROPIC_API_KEY 等）；getEnvApiKey 对环境凭据 provider 更进一步：对 google-vertex 检查 ADC 文件存在性以及 GOOGLE_CLOUD_PROJECT/GOOGLE_CLOUD_LOCATION；对 amazon-bedrock 识别完整 AWS 凭据链（profile、access keys、bearer token、ECS URIs、web identity）。它对已配置环境凭据的 provider 返回哨兵值 "<authenticated>"，让调用方在状态 UI 中显示“已配置”而不暴露真实密钥。该模块作为 ApiKeyAuth.resolve 实现和 compat 环境注入层的后备路径；Models 主路径使用 provider 自身的 auth.apiKey.resolve。

图片生成（并行流水线）

images-models.ts 镜像了聊天侧：ImagesProvider / ImagesModels / createImagesProvider 具有同样的认证解析和选项合并逻辑，但契约是单个 Promise<AssistantImages> 而不是流（图片生成不是流式）。失败作为 stopReason: "error" 的 AssistantImages 返回 —— 仍然是失败即值，不向调用方抛出。目前只有 openrouter-images 作为内置图片 provider 接入。

模块摘要

packages/ai/src/index.ts
核心公共 API 的桶式重新导出，无副作用。provider 工厂和 API 实现位于子路径导出；本文件故意不引入生成的目录或 OAuth 实现，以保持浏览器包体积精简。
packages/ai/src/cli.ts
npx @earendil-works/pi-ai 二进制。支持 login <provider> 和 list；在 cwd 的 auth.json 中加载/保存 OAuth 凭据。没有聊天命令 —— 它只是一个认证辅助工具。
packages/ai/src/models.ts
运行时核心。定义 Provider、Models、createModels、createProvider，以及认证应用 + 每次请求分发逻辑。提示词 + 上下文在这里变成 AssistantMessageEventStream。
packages/ai/src/auth/resolve.ts
认证决策树：显式覆盖 → 已存凭据（带锁刷新 OAuth 或 API key）→ 环境。失败包装为 ModelsError，错误码为 auth/oauth。
packages/ai/src/auth/types.ts
认证契约：CredentialStore（序列化读-改-写）、ApiKeyAuth / OAuthAuth（含 refresh/toAuth 拆分，让 Models 拥有锁刷新模式）。
packages/ai/src/api/anthropic-messages.ts
参考 API 实现。将 Context 转换为 Anthropic MessageCreateParams，用正确的认证风味构建 SDK client，解码 SSE，并发出 pi 的统一事件协议。其他 api/*.ts 文件遵循此骨架。
packages/ai/src/api/lazy.ts
lazyStream + lazyApi —— 同步流/异步准备桥梁，每个 Models.stream 和懒加载 API 包装都经过这里。准备失败变成流 error 事件。
packages/ai/src/env-api-keys.ts
Provider → env-var 映射，用于状态/可用性检查；识别环境 AWS 和 Google ADC 配置。供 compat 和状态 UI 使用，不参与 Models 请求路径。
packages/ai/src/compat.ts
已弃用的全局 API 表面，含 API-provider 注册表和环境 key 注入。尽可能路由到 builtinModels()，否则通过注册表分发。计划删除。
packages/ai/src/utils/oauth/anthropic.ts
Anthropic OAuth（Claude Pro/Max）：PKCE + 本地回调服务器 + 手动粘贴 code 竞速、token 交换，以及 anthropicOAuth OAuthAuth 适配器（login/refresh/toAuth）。
packages/ai/src/providers/all.ts
把所有内置 provider 工厂组装成 Models / ImagesModels 集合，并暴露带类型的静态目录读取（getBuiltinModel）。
