智能体运行时循环

本模块是 @earendil-works/pi-agent-core 的核心：它把调用方的提示词和一组工具转换成与 LLM 的多轮对话，执行 LLM 请求的 tool call，并将结果反馈回去，直到智能体无事可做。可以把它看作一个小型运行时：你交给它一个目标和一些能力，它会循环执行“询问模型、运行模型要求的内容、重复”，直到模型停止调用工具且没有更多排队消息。

它分为两层。底层（agent-loop.ts）是一个纯函数——给定上下文、配置和事件接收器，它运行循环并返回新消息。上层（agent.ts）是一个有状态的 Agent 类，拥有对话记录、管理运行生命周期，并为“steering”（中途干预）和“follow-up”（在智能体将要停止后再次运行）提供排队机制。更丰富的 AgentHarness（位于 harness/）在其之上叠加了会话持久化、hook、skill 和 prompt template。

调用方可以做什么

启动提示词
Agent.prompt(text|messages) 或 AgentHarness.prompt(text) 启动一次新运行。这是循环的主要触发器。
从已有对话记录继续
Agent.continue() 从最后一条消息恢复循环，而不添加新的用户提示词（用于重试后，或最后一条消息已经是 user/tool-result 的情况）。
运行中干预
steer(message) 在下一轮 assistant 回复前注入一条排队消息——让用户在智能体工作时重新定向它。
跟进
followUp(message) 仅在智能体本来会停止后才处理排队消息——一种“再补充一点”的机制。
调用 skill 或 prompt template
AgentHarness.skill(name) / promptFromTemplate(name, args) 将 skill 或 template 格式化成提示词并运行一轮。
中止
abort() 取消当前运行；循环以 aborted 的 stop reason 结束。
压缩或导航会话树
AgentHarness.compact() 总结旧上下文； navigateTree(targetId) 将会话叶子移动到另一分支。这些是针对已存储历史的循环外操作。

单次运行的数据流

对话以 AgentMessage[] 形式保存——它是普通 LLM 消息（user/assistant/toolResult）与应用特定的自定义消息（bash 输出、压缩摘要、分支摘要、任意“custom”消息）的并集。循环仅在调用模型的最后一刻才将此对话记录转换为 LLM 原生的 Message[] 形态。这种分离让应用可以在对话记录中携带仅用于 UI 或结构化的数据，而这些数据永远不会发给模型提供商。

状态存在于三处：Agent 的 _state（对话记录、模型、工具、流式标志）、每次运行的 AgentContext 快照（system prompt + 消息 + 激活的工具），以及——在 AgentHarness 下——Session 树，它是持久化的真相来源，每一轮的上下文都从这里重建。

工具执行数据流

当 assistant 消息包含 tool call 时，循环会批量执行它们。每个 tool call 都经历三个阶段：prepare（查找工具、验证参数、运行可阻止它的 beforeToolCall hook）、execute（调用 tool.execute，流式返回部分更新）和 finalize（运行可覆盖结果的 afterToolCall）。工具默认顺序执行，也可以并行执行——但即使在并行模式下，prepare 仍然是顺序的，只有 execute 是并发的。只有当每个 finalized 结果都设置了 terminate: true 时，批次才会提前终止。

状态机说明

循环本身就是一个自然的状态机。每次运行中，AgentHarness 显式跟踪一个 phase，取值为 idle、turn、compaction、branch_summary 或 retry（定义在 harness/types.ts 中）。在一轮之内，隐式状态为：流式 assistant 响应 → 执行 tool batch → 一轮结束 → 要么 下一轮（如果存在 tool call、steering 或 follow-up），要么 agent 结束。流式 assistant 消息本身也会经历作为事件发出的子状态：message_start → message_update（text/thinking/tool-call 增量）→ message_end。Tool call 的状态循环为 tool_execution_start → tool_execution_update → tool_execution_end。

外部边界

  • LLM 提供商调用 —— streamFn（默认来自 @earendil-works/pi-ai 的 streamSimple）是 transcript 离开进程的唯一外部边界。其契约是：绝不抛出；将失败编码为最终 assistant 消息，并带有 stopReason: "error" | "aborted"。streamProxy 变体将此调用路由到一台持有认证并代理到提供商的服务器，客户端从带宽优化的事件中重建部分消息。
  • 工具执行 —— 每个 tool.execute 都是外部边界（工具可能运行 shell 命令、调用 API、触碰文件系统）。工具通过抛出异常来报告失败；循环捕获异常并转换为错误 tool result。
  • 会话存储 —— 在 AgentHarness 下，每个 message_end 和轮次边界都通过 Session 持久化（JSONL 或内存 repo），因此一次运行可以被恢复或分叉。

关键扩展点

convertToLlm
控制哪些对话记录消息以及以何种形态进入模型。自定义消息类型（bash 输出、摘要、自定义通知）会在这里映射为 user 消息；带有 excludeFromContext 的 bash 消息会被完全过滤掉。
transformContext
在 convertToLlm 之前于 AgentMessage[] 层级运行——是上下文窗口管理（剪枝、压缩）和注入外部上下文的钩子。在 harness 下，context hook 可以在这里重写消息。
beforeToolCall / afterToolCall
针对每个 tool call 的拦截。before 可以阻止调用；after 可以重写结果（content、details、error 标志、terminate 提示）。Harness 将它们作为 tool_call / tool_result 事件暴露出来。
prepareNextTurn
在轮次之间，可以替换 context、model 或 thinking 级别。Harness 利用它刷出 pending 的会话写入，并从会话重建一个干净的新轮次状态。
shouldStopAfterTurn
请求在当前轮次后优雅停止——例如在上下文窗口填满前——但不会中止正在执行的 tool。
