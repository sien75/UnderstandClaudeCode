终端 UI 库 (@earendil-works/pi-tui)

一个支持差分渲染的终端 UI 库：它将原始 stdin 字节转换为结构化的按键事件，让组件消费这些事件来更新自身状态，然后将新的渲染树与上一帧进行差分对比，只把发生变化的行写入 stdout。编码智能体的交互模式会挂载一个组件树（编辑器 + Markdown 输出 + 自动补全），将 stdin 喂给它，并读取提交的行。

模块
原始终端字节输入 → 解码为按键事件 → 组件状态变更 → 差分渲染缓冲区输出到 stdout。
输入
raw 模式下 process.stdin 的原始字节：按键、粘贴（由 bracketed-paste 包裹）、鼠标序列、Kitty 键盘 CSI-u 序列、大小调整事件，以及终端查询响应（OSC 11 背景色、DSR 配色方案、单元格大小回复）。
输出
写入 process.stdout 的 ANSI 转义序列缓冲区：重新定位光标、只清除变化的行、重新渲染这些行，并可选择为 IME 放置硬件光标 —— 全部在同步输出包装器（ESC[?2026h…l）内完成，以防止画面撕裂。
说明
该库在边界上是传输无关的：ProcessTerminal 对接真实 stdin/stdout，但 Terminal 接口也可以由测试或内存终端实现。核心 TUI 类是一个组件容器；每个组件实现 render(width) 返回 string[]，并可选择实现 handleInput(data)。差分渲染是核心优化：未改变的行永远不会被重写。

调用方可以用这个库做什么

编码智能体交互模式（主要的外部调用方）使用这个库来：

  • 挂载组件树 —— 向 TUI 实例添加 Editor（行输入）、Markdown（流式智能体输出）、Loader（转圈）、SelectList，或覆盖层（自动补全菜单、设置）。
  • 接入 stdin —— 调用 tui.start()；库会接管 raw 模式下的 process.stdin，解码按键，并将每个事件路由到焦点组件的 handleInput。
  • 渲染帧 —— 任意组件调用 tui.requestRender()；TUI 将渲染批量合并为每 16ms 最多一次，与上一帧做差分，只写入变化的行。
  • 读取行输入 —— Editor 组件暴露 onSubmit/onChange 回调；当用户按下 Enter（通过按键绑定匹配）时，提交的文本会流回调用方。
  • 驱动编辑器 —— Editor 支持多行编辑、历史记录（上/下）、撤销/重做、kill-ring（Emacs yank）、单词导航，以及自动补全（斜杠命令 + 文件路径）。
  • 查询终端 —— queryTerminalBackgroundColor、queryTerminalColorScheme 和单元格大小查询返回 Promise；响应从 stdin 中截获并解析。
  • 显示覆盖层 —— showOverlay 将一个模态组件（自动补全、设置）定位在基础内容之上，支持锚点/百分比尺寸和焦点管理。

触发整个链路的关键动作是 tui.start()：它将 stdin 切换到 raw 模式，启用 bracketed paste，查询 Kitty 键盘协议，并将数据回调绑定到 TUI.handleInput。

数据流：输入解码（stdin 字节 → 按键事件 → 焦点组件）

这是单个按键按下所经过的路径。字节到达 process.stdin 后，由 StdinBuffer 拆分为完整的转义序列；ProcessTerminal 检查 Kitty 协议协商或终端查询响应，然后转发给 TUI.handleInput；TUI 将数据路由到焦点组件（在此之前会经过全局拦截器：OSC 11 回复、配色方案报告、单元格大小回复、调试键、覆盖层焦点恢复）。

数据流：渲染 / 差分（组件树 → 变化的 stdout 字节）

这是渲染帧经过的路径。组件调用 requestRender；TUI 合并为 nextTick + 16ms 节流定时器；doRender 渲染所有子组件并合成覆盖层，提取 IME 光标标记，规范化行，然后执行完整重绘（首次渲染、宽高变化、收缩时清空、视口错位）或差分更新（只写入 firstChanged..lastChanged 范围内的行，位于同步输出内）。

graph TD
    req[requestRender called by component] --> force{Force flag?}
    force -->|yes| force-reset[Reset previousLines/width/height, nextTick doRender]
    force -->|no| already{renderRequested already true?}
    already -->|yes| req-stop[Return, coalesce]
    already -->|no| sched-next[renderRequested true, nextTick scheduleRender]
    sched-next --> timer{renderTimer active?}
    timer -->|yes| sched-stop[Return]
    timer -->|no| delay[Compute 16ms minus elapsed]
    delay --> fire[setTimeout fires, doRender]
    force-reset --> fire

上面的调度阶段将重复的 requestRender 调用合并为一次 doRender 调用，节流为每 16ms 一帧。一旦到达 fire（doRender 入口点），执行就进入下面的渲染与差分阶段：渲染组件树、合成覆盖层，并只将变化的行范围写入 stdout。

数据流：编辑器（按键事件 → 文本状态 → 渲染行）

Editor 是主要的输入消费组件。handleInput 会咨询全局 KeybindingsManager，将每个按键分派到一个动作；动作会修改 state.lines/cursorLine/cursorCol（并创建撤销快照、累积 kill-ring）；render 将文本按宽度换行、应用滚动偏移、将光标渲染为高亮的字素，并在激活时追加自动补全列表。标志状态机包括：jumpMode（正向/反向等待下一个字符）、粘贴缓冲、自动补全导航、撤销栈、kill-ring yank-pop 循环。

标记的状态机

  • Editor jumpMode（editor.ts:595-613）：null | "forward" | "backward" 。按下 jump 热键进入该模式；下一个可打印字符执行 jumpToChar 并清除模式；控制字符或重复热键则取消。
  • Editor paste buffering（editor.ts:616-639、input.ts:48-84）：isInPaste 在 ESC[200~ 时开启，累积直到 ESC[201~，然后调用 handlePaste 并递归处理剩余部分。
  • Editor autocomplete navigation（editor.ts:653-711）：当 autocompleteState 存在时，上/下/取消/Tab/确认由 SelectList 子组件拦截；斜杠命令的确认会下落到提交。
  • Undo stack（undo-stack.ts、editor.ts pushUndoSnapshot/undo）：入栈时克隆快照；出栈返回 detached 克隆。提交时清空。
  • Kill ring（kill-ring.ts、editor.ts yank/yankPop）：环形缓冲区；连续的 kill 会累积（前置/追加）；yank 查看最后一个；yankPop 旋转以循环更早的条目。
  • Overlay focus restore（tui.ts:241-258、370-470）：inactive | eligible | blocked{overlay, blockedBy, resume}。Blocked 转换跟踪非覆盖层从有焦点的覆盖层窃取焦点的情况；resume 解析为 restore-overlay 或 focus-target。
  • Kitty keyboard protocol negotiation（terminal.ts:220-301）：查询 → 缓冲片段 → 解析 kitty-flags 与 device-attributes → 启用 Kitty（flags != 0）或 modifyOtherKeys 回退；150ms 片段超时后将剩余内容作为输入刷新。
  • StdinBuffer sequence completeness（stdin-buffer.ts:29-78）：针对 CSI/OSC/DCS/APC/SS3/meta 的逐片段状态机 complete | incomplete | not-escape；10ms 超时刷新卡住的不完整残留。
