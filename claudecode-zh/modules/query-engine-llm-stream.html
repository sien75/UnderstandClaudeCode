对话引擎与 LLM 流式调用

这是 Claude Code 里"一次提问到最终回复"的核心通道。用户在 REPL 敲一句提问（或 SDK 调一次 ask），引擎要做的不是简单地转发给模型——它得先把用户输入里的斜杠命令、附件、技能发现都消化掉，再带着完整上下文去调 Claude API；模型流式吐字的同时，引擎边收边解析 token、判断是否要调工具，工具结果要喂回去再调一轮，如此循环直到模型给出最终回复；中途还要处理 thinking、限流重试、模型降级、prompt-too-long/超长输出恢复、上下文压缩、预算和轮数上限。这条链路上任何一步失败都不能让对话"卡死"，必须有明确的退出和上报。

输入
用户的一句提问（文本或带附件的内容块）+ 当前会话历史 + 工具/MCP/技能/权限上下文
输出
一条 result 消息（成功/超预算/超轮数/执行错误），附带最终文本、累计 token 用量、成本、模型用量、权限拒绝记录
  • 外层 QueryEngine.submitMessage 负责"会话级"事务：拼系统提示、跑用户输入预处理、把 query() 吐出的消息转成 SDK 消息、写 transcript、盯预算/轮数/结构化输出重试上限。
  • 内层 query.ts 的 queryLoop 是真正的"模型-工具"循环：while(true) 每轮先压缩上下文，再调模型流式，再决定"收工"还是"跑工具后下一轮"。
  • 最底层的 claude.ts:queryModel 负责把请求送进 Anthropic SDK 的流式接口，逐事件解析并攒成 assistant 消息；withRetry 在外层包了重试/降级/限流退避。

一、整体数据流：从用户提问到 result 消息

先把全链路串一遍。用户敲完回车后，消息先进 QueryEngine.submitMessage 做会话级准备（系统提示、用户输入处理、transcript 落盘），然后进入 query() 的循环。循环每一轮要么"模型没要工具→收尾返回"，要么"模型要了工具→执行工具→带着结果回到循环顶部"。循环结束后，submitMessage 再做一次结果校验（isResultSuccessful），失败给 error_during_execution，成功给 success。

二、QueryEngine.submitMessage：会话级编排

submitMessage 是 SDK/REPL 调用的入口（ask 函数只是它的一次性包装）。它不直接调模型，而是把"调模型前的所有准备"和"调模型后的所有收尾"揽下来。关键设计：每个 QueryEngine 实例对应一个会话，mutableMessages、totalUsage、readFileState 跨轮持久；submitMessage 每调一次开一个新 turn。

几个值得说的点：① canUseTool 在这里被包一层，用来偷偷记录权限拒绝（最后塞进 result 给 SDK 上报）。② 用户消息在进 query 循环前就写 transcript——如果用户发完消息几秒后点 Stop，进程被杀，resume 仍能从这条消息恢复，而不是"找不到会话"。③ 进入 query() 的 for await 后，每条 assistant/user/compact_boundary 消息都会 push 到 mutableMessages 并写 transcript；stream_event 用来攒 token 用量和 stop_reason。④ 循环每步都查预算上限和结构化输出重试上限，超了立即 yield 对应 error result 并 return。

三、query() 主循环：模型-工具的 while(true)

这是引擎的心脏。query() 只是 queryLoop 的薄壳，负责在正常返回时把消费掉的命令 UUID 标记 completed。queryLoop 是一个 while(true)，每一轮干三件事：① 压缩上下文（snip→microcompact→context-collapse→autocompact，四者按序、互不排斥）；② 调模型流式拿 assistant 消息；③ 看模型有没有要工具——不要就走收尾路径（含 prompt-too-long/超长输出恢复、stop hooks、token budget 续写），要就跑工具、收集附件/内存预取/技能预取，拼成下一轮的 messages 继续。

状态在迭代间通过 state: State 对象传递（messages、turnCount、压缩跟踪、恢复计数、上一轮的 transition 原因）。每个 continue 点都显式写一个新的 State，避免散落的赋值。几个恢复路径值得记：① 模型降级：流式过程中抛 FallbackTriggeredError，切到 fallbackModel 重试同一请求，并清掉已生成的 assistant 消息（用 tombstone 让 UI 移除）。② prompt-too-long：流式时先把错误"扣留"不吐给上层，收尾时先试 context-collapse 排空，再试 reactive compact，都不行才把错误吐出来并 early return（不进 stop hooks，否则死循环）。③ max_output_tokens：先试把 max_tokens 从默认 8k 升到 64k 重试一次，再试"续写消息"恢复最多 3 次，都耗尽才把扣留的错误吐出来。④ streaming 空响应：流式结束但没拿到 message_start 或没有任何 content block，抛错走非流式 fallback。

四、流式解析：claude.ts queryModel

queryModelWithStreaming 经 withStreamingVCR 包装后调 queryModel。queryModel 干两件事：① 构造请求参数（model 解析、beta 头、tool schema、tool search 过滤、prompt caching、cached microcompact、advisor、task_budget 等），② 用 withRetry 拿到一个 Stream，然后 for await 逐事件解析。

流式事件的处理是状态机式的：message_start 建 partialMessage 和初始 usage；content_block_start 按 block 类型（text/thinking/tool_use/server_tool_use/advisor_tool_result）初始化一个可变 block；content_block_delta 把 text/thinking/input_json/signature 增量累加进对应 block；content_block_stop 把 block 封装成一条 AssistantMessage yield 出去（所以一条 assistant 消息可能被拆成多条 yield，每个 content block 一条）；message_delta 把最终 usage 和 stop_reason 写回最后一条已 yield 的消息（直接 mutate，不替换对象，因为 transcript 写队列还持有引用），并在这里算成本、检测 max_tokens/context_window_exceeded 并 yield 对应错误；message_stop 收尾。

健壮性方面有两个看门狗：① stream idle watchdog——若 N 毫秒（默认 90s）没收到任何 chunk，主动 abort 流并转非流式 fallback；② stall 检测——chunk 间隔超 30s 记录但不中断。流结束后还会校验"是否真的拿到了 message_start 且至少有一个 content block 或 stop_reason"，否则视为代理返回了空响应，抛错走非流式 fallback。

五、重试与降级：withRetry

withRetry 是请求级别的重试外壳，包在 anthropic.beta.messages.create(stream:true) 外面。默认最多 10 次重试，529（容量过载）最多 3 次连续。重试决策：① 529 在非前台来源上直接放弃（避免级联放大）；② 529 连续 3 次且配了 fallbackModel，抛 FallbackTriggeredError 让上层切模型；③ shouldRetry 判断错误是否可重试（限流、网关 5xx 等）；④ max_tokens 上下文溢出错误会调 parseMaxTokensContextOverflowError 算出可用空间，调小下轮的 max_tokens 继续；⑤ persistent retry 模式针对 429/529 长退避（最高 5 分钟，封顶 6 小时），期间每 30s yield 一次 api_retry 系统消息让上层/用户知道在等。

退避时长优先用服务端的 retry-after 头，否则指数退避（500ms × 2^n，封顶 32s）。每次重试前都会 yield 一条 SystemAPIErrorMessage，这条消息在 QueryEngine.submitMessage 里被转成 api_retry SDK 消息吐给前端。客户端实例在鉴权错误后会重新获取，stale connection（ECONNRESET/EPIPE）会关 keep-alive 重试。

六、工具执行循环

当模型在流式响应里产出 tool_use 块，queryLoop 标记 needsFollowUp=true。工具执行有两条路：① StreamingToolExecutor——在模型还在流式吐字时就把已确定的 tool_use 块丢进去并行跑（仅并发安全的只读工具），getCompletedResults 在流式循环里边收边 yield；流结束后 getRemainingResults 收尾。② runTools——流式执行器关闭时的串行/并发批量执行：partitionToolCalls 把工具分批，并发安全批次用 runToolsConcurrently，写操作批次用 runToolsSerially。每条工具结果作为 user 消息（含 tool_result 块）push 进 toolResults，下一轮作为 messages 喂回模型。

权限在这里实打实落地：canUseTool（在 submitMessage 里被包了一层记拒绝）会被调用，决定 allow/deny/ask；deny 的工具会产生一个 tool_result 带 tool_use_error，模型下一轮会看到。中断处理：流式/工具执行中若 abort 信号触发，executor 会为已排队但没跑完的工具生成 synthetic tool_result（避免 tool_use 没有 tool_result 配对导致 API 400）。

七、上下文压缩链

压缩不是单一机制，而是四道工序按固定顺序在每轮循环顶部跑，互不排斥（前道释放的 token 会让后道变成 no-op）：① snip（HISTORY_SNIP 特性门）——移除"僵尸"消息和过期标记，SDK 长会话用来限制内存增长。② microcompact——按 tool_use_id 替换大块 tool_result 内容（图片/长输出）为占位符，可基于时间或缓存编辑；cached MC 会延迟 boundary 消息到 API 响应后，用真实的 cache_deleted_input_tokens。③ context-collapse——把历史段落折叠成粒度摘要（读时投影，不删原始消息），溢出时 recoverFromOverflow 排空已暂存的折叠。④ autocompact——阈值触发（上下文接近窗口上限时）跑一次完整的 forked-agent 摘要，把整段历史压成一条 summary 消息 + 附件，产出 compact_boundary。autocompact 有熔断器：连续失败 N 次后停止重试。

除了这四个"主动"压缩，还有 reactive compact——在模型流式返回 prompt-too-long 或媒体过大错误时才触发（错误在流式阶段被扣留，收尾时试一次压缩后重试本轮）。所有压缩成功后都会 yield post-compact 消息（summary + 附件 + hook 结果），并把 messagesForQuery 替换成压缩后数组，下一轮从压缩后的上下文继续。task_budget 在压缩时会扣减压缩前的最终上下文 token 数。

八、token / 成本计数

token 和成本的累积点在 claude.ts:queryModel 的 message_delta 分支：每次拿到带 usage 的 message_delta，先 updateUsage 攒本条消息的 usage，再 calculateUSDCost 算这条消息的美元成本，调 addToTotalSessionCost 加进会话总账。QueryEngine.submitMessage 在 stream_event 的 message_start/message_delta/message_stop 上分别 updateUsage / accumulateUsage，把每条消息的 usage 累加进 totalUsage，最终塞进 result 消息。cost-tracker.ts 维护按模型拆分的用量（input/output/cacheRead/cacheCreation/webSearch/costUSD），并暴露 getTotalCost/getModelUsage 给 result 和预算检查用。

token 估算（tokenEstimation.ts）用于压缩阈值判断和阻塞限制检查：在不调 API 的前提下用 roughTokenCountEstimationForMessages 估一个粗略值（按字节/类型换算），autocompact 的 shouldAutoCompact 和 queryLoop 顶部的 blocking limit 检查都靠它。注意：snip 释放的 token 数要显式从估算里减掉（snipTokensFreed），否则 snip 已经把上下文压到阈值以下、但 stale 的 usage 还显示超阈值，会误触发阻塞。

  • 状态机视角：queryLoop 的 state.transition.reason（next_turn / collapse_drain_retry / reactive_compact_retry / max_output_tokens_escalate / max_output_tokens_recovery / stop_hook_blocking / token_budget_continuation）是一个隐式状态机——每个 continue 点都带一个 reason，便于测试断言"哪条恢复路径被触发了"。
  • 权限视角：canUseTool 在 submitMessage 被包一层记拒绝；工具执行时实调；deny 不抛错而是产 tool_use_error 让模型看到；权限上下文（allowedTools、additionalWorkingDirectories）在用户输入处理后更新到 AppState。
  • 生命周期视角：QueryEngine 一会话一实例，state 跨 turn 持久；submitMessage 一 turn 一次；queryLoop 一轮一次 state；withRetry 一请求多次 attempt；流式一请求一 stream。中断信号（abortController）贯穿全层，每层都有自己的 abort 处理（流式补 synthetic 结果、工具补 synthetic 结果、stop hooks 跳过）。
  • 转录持久化：用户消息进 query 前就写 transcript（防 kill 后 resume 失败）；assistant 消息 fire-and-forget 写（避免阻塞 message_delta）；compact_boundary 前先 flush 保留段尾部；result yield 前在 cowork/eager 模式下强制 flush。
  • 非流式 fallback：流式空响应、idle 超时、代理返回非 SSE 体等都会触发 executeNonStreamingRequest——用 messages.create(stream:false) 重试，超时默认 300s（remote 120s），结果再走相同的 usage/cost 累积路径。
