工具系统框架与文件/Shell 工具

当 Claude（大模型）决定"去读一个文件、改一段代码、跑一条 shell 命令"时，它会产出一个 tool_use 请求块。本模块讲的就是这个请求从落地到回传给 LLM 的完整生命周期：怎么找到对应的工具、怎么校验入参、怎么过权限关、怎么真正执行、执行完怎么把结果拼回消息流让模型继续推理。范围限定在内置的文件工具（Read/Edit/Write）、Shell 工具（Bash）和两个搜索工具（Glob/Grep，后者封装 ripgrep）。

输入
模型在助手消息里发出的一个 tool_use 块：工具名 + 参数（如 {file_path, old_string, new_string} 或 {command}），外加当前会话上下文（权限规则集、文件读取缓存、中止信号等）。
输出
一条用户消息，里面带一个 tool_result 块（成功则是工具返回的内容/摘要，失败则是带 is_error 的错误说明），供模型在下一轮对话里看到并据此决策。
  • 工具不是散装函数，而是一组统一契约（Tool 接口）的实例：每个工具自带 validateInput、checkPermissions、call、mapToolResultToToolResultBlockParam 等方法，框架按固定顺序调用它们。
  • 并发安全是关键分水岭：标了 isConcurrencySafe=true 的只读工具（Read/Glob/Grep、Bash 只读命令）会被批量并行跑，写操作和不确定的命令则串行执行，避免互相覆盖。
  • 权限不是单一闸门，而是"工具自检 + 规则匹配 + PreToolUse 钩子 + canUseTool 决策"多层叠加，任何一层 deny 就直接返回错误结果给模型。
  • 文件编辑工具有"陈旧检查"：必须先 Read 过、且磁盘 mtime 没变过才允许写，防止模型基于旧内容覆盖用户最新的改动。

一、工具注册与工具池装配

所有内置工具在 tools.ts 里集中登记。一个工具是否"可见"由几道筛子决定：环境/特性开关（feature flag）决定要不要装载、isEnabled() 决定是否启用、权限上下文里的 deny 规则决定是否对模型隐藏。最终交给模型的工具列表，是内置工具与 MCP 工具合并、按名字去重、deny 规则过滤后的结果。

  feat-gate -->|否| skip["跳过该工具(如 SleepTool/WorkflowTool)"]
  feat-gate -->|是| enabled-check["调用 tool.isEnabled()"]
  enabled-check --> deny-check{"被 deny 规则整名命中?"}
  deny-check -->|是| hide["从工具池移除:模型根本看不到"]
  deny-check -->|否| keep["保留进 allowedTools"]

要点：getAllBaseTools 是"理论上全部工具"的唯一来源，注释强调它必须和 Statsig 上的系统缓存配置保持同步，否则跨用户的 prompt 缓存会失效。filterToolsByDenyRules 用的是和运行时权限判定同一套匹配器，所以 mcp__server 这种前缀规则会在模型看到工具之前就把整组工具屏蔽掉，而不只是调用时挡掉。

二、工具契约：Tool 接口与 buildTool

src/Tool.ts 定义了所有工具的统一形状。一个工具就是一个对象，带着名字、输入 schema（Zod）、prompt 模板，以及一组方法。其中几个关键方法构成框架调用的固定管线：

  • validateInput：纯逻辑校验（路径格式、参数范围、deny 路径预判），不碰需要权限的 I/O。失败直接返回错误，不进入权限流程。
  • checkPermissions：工具自报的权限意见（allow/ask/deny），只在 validateInput 通过后调用。它和通用权限系统（hasPermissionsToUseTool）叠加，工具说 ask 不代表最终一定问用户。
  • isConcurrencySafe：决定本工具能否和其他只读工具并行批量跑。默认 false（保守），Read/Glob/Grep 返回 true，Bash 仅当命令被判为只读时返回 true。
  • isReadOnly / isDestructive：语义标签，影响 UI 折叠、自动分类器输入等。
  • call：真正干活的函数，拿到已通过权限的 input 和上下文，返回 ToolResult。
  • mapToolResultToToolResultBlockParam：把 call 的输出序列化成发给模型的 tool_result 文本/图片块。
  • renderToolUseMessage / renderToolResultMessage：纯 UI 渲染，不影响发给模型的内容。

buildTool(def) 是个工厂：把工具定义和一组"安全默认值"合并（isEnabled 默认 true、isConcurrencySafe 默认 false、checkPermissions 默认 allow），这样单个工具可以省略常用桩方法，调用方也永远拿到完整对象。默认值是"fail-closed"取向——不确定就当不安全、当需要权限。

三、单次工具调用的完整数据流

这是本模块的核心。模型一轮里可能同时发好几个 tool_use 块。runTools 先把它们按"是否并发安全"分批：连续的只读工具合成一个并发批，每个写/不确定的工具各自成串行批。然后对每个块走 runToolUse → checkPermissionsAndCallTool 这条主管线。下图把一次调用的关键阶段、分支和错误出口都画出来。

几个容易被忽略的细节：

  • 入参先过 Zod 再过业务校验。模型其实经常生成类型不对的入参（数组写成字符串之类），Zod 这一关会返回带"schema 未发送"提示的错误，引导模型先用 ToolSearch 加载延迟工具再重试。
  • backfill 在权限/钩子之前。backfillObservableInput 把 ~/相对路径展开成绝对路径，目的是不让钩子的 allowlist 被波浪号绕过。但它只改"给钩子看的副本"，call() 拿到的仍是模型原始字段——这样工具结果里嵌的路径字符串（"File created at: X"）和 transcript/VCR 哈希才稳定。
  • 权限决策有三个来源叠加：PreToolUse 钩子可以直接给 allow/deny/改 input；resolveHookPermissionDecision 优先采纳钩子结论；没有钩子结论才走 canUseTool（即 hasPermissionsToUseTool）。
  • 权限拒绝也要跑钩子：auto 模式下分类器判 deny 后，会跑 PermissionDenied 钩子；如果钩子说 retry，会额外塞一条"该命令现已批准，可重试"的元消息给模型。
  • 错误也跑钩子：runPostToolUseFailureHooks 让钩子能在工具抛异常时介入（比如自动修复）。中止（AbortError）和 Shell 错误有专门分支，不会无脑 logError。
  • 结果可能比 30K/100K 字符大：超过 maxResultSizeChars 会被持久化到磁盘文件，模型只收到带预览的 persisted-output 包装；Read 工具例外（设为 Infinity，避免"读文件→又触发读文件"的环）。

四、并发分批：为什么只读能并行

partitionToolCalls 把模型本轮的 tool_use 序列扫一遍，对每个块调 tool.isConcurrencySafe(input)。连续返回 true 的合并进同一批走 runToolsConcurrently（用 all 工具按上限 10 并发拉起），遇到 false 就断开，那个写操作单独走 runToolsSerially。串行批里每个工具跑完才会更新 currentContext（因为写操作可能改 readFileState 等共享状态），并发批则把 contextModifier 攒到批末统一应用——读操作之间互不影响，所以可以放心并行。

注意：isConcurrencySafe 解析失败（如 shell-quote 解析抛异常）会被 catch 成 false，保守地降级为串行。Bash 的并发安全等于"命令是否只读"：isReadOnly 调 checkReadOnlyConstraints 判定，含 cd 的复合命令会被特殊处理。

五、权限闸门内部：规则 + 钩子 + 分类器

hasPermissionsToUseTool（src/utils/permissions/permissions.ts）是通用权限层。它综合工具自报的 checkPermissions、用户配置的 allow/ask/deny 规则、PreToolUse 钩子决策，以及（auto 模式下）AI 分类器结论，最终给出 allow/ask/deny。模式开关会影响出口：dontAsk 模式把 ask 直接转成 deny（不问用户）；auto 模式把 ask 交给分类器而不是弹交互框。

Bash 的权限更复杂，在 bashToolHasPermission 里：先用 tree-sitter 做 AST 级安全解析（检测命令注入、eval、zsh 内建等），解析成 SimpleCommand[] 后再走"精确匹配 → 沙箱自动放行 → 前缀/wildcard 规则 → deny/ask 分类器"。AST 判为 too-complex（含命令替换/控制流）会尊重 deny 规则后降级为 ask，绝不悄悄放行。沙箱开启且命令适合沙箱时，会自动 allow（仍尊重显式 deny/ask）。

六、FileReadTool：读文件的去重与多类型

用户视角："Claude 要看一个文件"。Read 工具的 call 先查 readFileState 缓存：如果同一文件、同一 offset/limit 区间已经读过、且磁盘 mtime 没变，就直接返回 file_unchanged 桩（"文件未变"，不重发全文，省 cache token）。否则按扩展名分流：文本（带行号 + cyber-risk 提醒）、图片（base64 image 块）、PDF（按页提取图片或文档块）、notebook（映射 cell）。ENOENT 时会尝试找相似文件名给"Did you mean"提示。

  dedup-lookup --> dedup-hit{"同区间且 mtime 未变?"}
  dedup-hit -->|是| unchanged-stub["返回 file_unchanged 桩"]
  dedup-hit -->|否| skill-discover["后台发现/激活该路径下的 skills"]
  skill-discover --> call-inner["进 callInner 按扩展名分流"]
  call-inner --> ext-switch{"文件类型?"}
  ext-switch -->|文本| text-branch["读字节→探测编码→规范化换行→加行号→附 cyber-risk 提醒"]
  ext-switch -->|图片| image-branch["base64 编码为 image 块"]
  ext-switch -->|PDF| pdf-branch["按 pages 参数提取页→Document/图片块"]

关键点：validateInput 在权限之前就挡掉 deny 路径、UNC 路径（防 NTLM 凭据泄漏）、二进制扩展名、会阻塞的设备文件——这些都不需要权限就能判。checkPermissions 则委托给通用的 checkReadPermissionForTool。maxResultSizeChars: Infinity 是有意为之，避免"读大文件→持久化→模型再 Read 这个持久化文件"的环。

七、FileEditTool / FileWriteTool：陈旧检查与原子写

两个写工具的核心防御是"陈旧检查"（staleness check）：模型必须先 Read 过这个文件，并且自上次 Read 后磁盘 mtime 没变，才允许写。这防止模型基于几轮前的旧内容覆盖用户刚改的代码。Edit 走 read-modify-write：读原内容→用 findActualString 处理引号风格→生成 patch→写盘→更新 readFileState→通知 LSP（didChange/didSave）和 VSCode diff 视图。Write 是整文件替换，额外要求文件必须先 Read 且非 partial view。

graph TD
  write-start["Edit/Write.call"] --> mkdir["确保父目录存在(在临界区外)"]
  mkdir --> backup["fileHistory 备份旧内容(按内容 hash 幂等)"]
  backup --> load-current["读当前文件内容+编码+换行"]
  load-current --> stale-check{"已 Read 且 mtime 未变?"}
  stale-check -->|否| stale-err["抛 FILE_UNEXPECTEDLY_MODIFIED_ERROR"]
  stale-check -->|是| compute-new["Edit: getPatchForEdit 生成新内容 / Write: 直接用 input.content"]
  compute-new --> write-disk["writeTextContent 原子写盘"]
  write-disk --> lsp-notify["通知 LSP didChange/didSave + VSCode"]
  lsp-notify --> update-state["更新 readFileState(写后 mtime)"]
  update-state --> return-diff["返回 filePath/patch/userModified 等"]

几个重要约束：

  • 临界区不能有 await：从"读当前内容"到"写盘"之间不允许让出事件循环，否则并发编辑会交错。mkdir、备份都显式放在临界区外。
  • Write 不改写换行符：模型在 content 里给什么换行就用什么，避免把 bash 脚本在 Linux 上写成 CRLF。
  • UNC 路径跳过 I/O 校验：validateInput 对 UNC 路径直接 return true，把判定让给权限层——因为 fs.existsSync 在 Windows UNC 路径上会触发 SMB 认证、可能泄露凭据。
  • team memory 防泄密：写入 team memory 文件前会扫 secrets，命中就拒绝。

八、BashTool：从命令到 shell 进程

用户视角："Claude 跑一条 shell 命令"。BashTool 的 call 先处理特殊的"模拟 sed 编辑"（直接落盘，保证用户预览的就是写的），否则进 runShellCommand 这个 async generator：用 exec 起一个 shell 进程，边跑边通过 onProgress 回调把最新输出喂给 generator yield 成 bash_progress 事件。超时、显式 run_in_background、或 assistant 模式下的阻塞预算超限都会把命令转后台跑，返回 backgroundTaskId 让模型后续用 TaskOutput 取结果。

Bash 的并发安全取决于命令是否只读：isReadOnly 调 checkReadOnlyConstraints，isConcurrencySafe 直接复用 isReadOnly 的结果。所以 ls/cat 这类会和别的只读工具并行，git push 这种则串行。preparePermissionMatcher 会把复合命令拆成子命令列表，让 Bash(git *) 这种钩子规则能命中 ls && git push 里的 git 子命令——拆不动就 fail-safe 让钩子跑。

九、GlobTool / GrepTool：ripgrep 封装

两个搜索工具都是只读、并发安全、走 checkReadPermissionForTool。Glob 调内部的 glob 函数按 pattern 找文件名；Grep 真正调 ripgrep（rg）做内容搜索。Grep 的 call 把模型参数翻译成 rg 命令行：--hidden、排除 VCS 目录、--max-columns 500（防 base64/压缩文件刷屏）、-l/-c/-n/-i/-C/-A/-B/-U 等，再叠加权限上下文里的 ignore pattern 和插件缓存排除项。路径用绝对路径调 ripgrep，结果再把绝对路径相对化成 cwd 相对路径省 token。

  add-ignore --> call-rg["ripgrep(args, absolutePath, abortSignal)"]

Grep 的 validateInput 只做路径存在性检查（UNC 跳过）。head_limit 在相对化路径之前应用，避免给上千行刷掉只剩几十行的宽模式做无用的逐行处理。Glob 的 call 类似但更简单：调 glob 取文件列表，相对化，返回 filenames + 是否截断。
