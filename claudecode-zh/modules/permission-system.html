权限系统

每次 Claude 准备执行一个可能改动环境的操作（跑一段 bash、写文件、调 MCP 工具……）之前，系统都要先"过一道关卡"决定怎么处理这个动作：是直接放行、停下来问用户、还是当场拒绝。这个模块就是那道关卡。它综合四种运行模式（default / plan / bypassPermissions / auto，外加 ant 内部的 dontAsk）、用户配置的允许/拒绝/必问规则、工具自报的命令级检查、以及（在 auto 模式下）一个 AI 分类器，给每一个待执行的工具调用给出一个最终裁决：allow / ask / deny。

输入
一个待执行的工具调用（工具名 + 输入参数）+ 当前会话的权限上下文（模式、各来源的允许/拒绝/必问规则列表、附加工作目录）+ 当前对话历史（auto 模式分类器要用）
输出
一个裁决：allow（放行，可能带用户改过的输入）、ask（弹窗问用户，可能附"建议加哪条规则"和并发分类器检查）、deny（拒绝，带原因）。裁决同时被记进日志和埋点。
  • 四种模式不是平级的开关，而是一套"预处理 + 后处理"分层：先做规则检查（deny/ask/safetyCheck 这几类 bypass 也躲不掉），再按模式分流（bypass 直接放行、auto 调分类器、dontAsk 把 ask 翻成 deny、headless 把 ask 翻成 deny/走 hook）。
  • 用户配的规则按来源分（userSettings/projectSettings/localSettings/flagSettings/policySettings/cliArg/command/session），规则语法是 ToolName 或 ToolName(content)，括号内支持转义；MCP 工具按 mcp__server__tool 全名匹配，也支持 mcp__server__* 整服务器粒度。
  • 安全检查（safetyCheck，比如碰 .git/、.claude/、shell 配置文件）是"bypass 免疫"的——哪怕在 bypassPermissions 模式也得停下来问；其中 classifierApprovable=true 的那些（敏感文件路径）在 auto 模式下可以让分类器去判，false 的（Windows 路径绕过、跨机桥接消息）则任何模式都必须人工确认。
  • auto 模式有熔断：连续 3 次或本会话累计 20 次被分类器拒绝就降级回人工确认；分类器 API 不可用时按 tengu_iron_gate_closed 开关决定 fail-closed（拒并提示重试）还是 fail-open（回退人工）；transcript 超长是确定性错误，直接回退人工或 headless 下中止。

主决策链：从工具调用到最终裁决

这是核心数据流。用户视角的触发点：Claude 在对话里产生了一个 tool_use（比如要跑 rm）。工具系统在真正执行前调用 hasPermissionsToUseTool，于是下面这条链开始跑。先做"无论什么模式都得过的硬规则检查"，再按模式分流，最后做一些后处理（dontAsk 翻 deny、auto 跑分类器、headless 兜底拒绝）。

graph TD
  abort-check["会话已被中止?"] -->|是| throw-abort["抛 AbortError"]
  abort-check -->|否| deny-rule["整工具是否有 deny 规则?<br/>getDenyRuleForTool"]
  deny-rule -->|命中| ret-deny-rule["返回 deny + 命中规则"]
  deny-rule -->|未命中| ask-rule["整工具是否有 ask 规则?<br/>getAskRuleForTool"]
  ask-rule -->|命中且非沙箱自动放行| ret-ask-rule["返回 ask + 命中规则"]
  ask-rule -->|未命中/可沙箱放行| tool-check["调用工具自报 checkPermissions<br/>(bash 子命令规则等)"]
  tool-check --> tool-deny{"工具返回 deny?"}
  tool-deny -->|是| ret-tool-deny["返回该 deny"]
  tool-deny -->|否| ask-content{"ask 且 reason=rule<br/>(内容级必问)?"}
  ask-content -->|是| ret-ask-content["返回该 ask<br/>(bypass 也躲不掉)"]
  ask-content -->|否| safety{"ask 且 reason=safetyCheck?<br/>(敏感路径)"}
  safety -->|是| ret-safety["返回该 ask<br/>(bypass 免疫)"]
  safety -->|否| mode-branch["按模式分流"]
  mode-branch --> bypass{"bypassPermissions?<br/>(或 plan+原 bypass)"}
  bypass -->|是| ret-allow-bypass["返回 allow<br/>reason=mode"]
  bypass -->|否| allow-rule{"整工具有 allow 规则?<br/>toolAlwaysAllowedRule"}
  allow-rule -->|是| ret-allow-rule["返回 allow<br/>reason=rule"]
  allow-rule -->|否| passthrough-to-ask["passthrough→ask<br/>准备交给后处理"]
  passthrough-to-ask --> postproc["进入后处理 hasPermissionsToUseTool"]

后处理（hasPermissionsToUseTool 包裹层）拿到上面的结果后，对 ask 结果再做一次模式相关的转换，对 allow 顺手重置 auto 模式的连续拒绝计数。下面是 ask 之后的那段后处理分流。

graph TD
  inner-result["内层裁决结果"] --> allow{"behavior=allow?"}
  allow -->|是| reset-denial["auto 模式下重置连续拒绝计数<br/>recordSuccess"] --> done-allow["返回 allow"]
  allow -->|否| ask2{"behavior=ask?"}
  ask2 -->|否| done-other["返回原结果(deny 等)"]
  ask2 -->|是| dontask{"模式=dontAsk?"}
  dontask -->|是| ret-deny-dontask["翻成 deny<br/>'不该问的工具不问'"]
  dontask -->|否| auto-decide{"模式=auto<br/>(或 plan+auto 激活)?"}
  auto-decide -->|否| headless{"shouldAvoidPermissionPrompts?<br/>(headless/后台)"}
  auto-decide -->|是| auto-flow["进入 auto 分类器流程"]
  headless -->|是| hook-headless["跑 PermissionRequest hook<br/>runPermissionRequestHooksForHeadlessAgent"]
  headless -->|否| ret-ask-user["返回 ask<br/>交给 useCanUseTool 弹窗"]
  hook-headless --> hook-decided{"hook 给了裁决?"}
  hook-decided -->|是| ret-hook["返回 hook 裁决"]
  hook-decided -->|否| ret-deny-headless["返回 deny<br/>'此环境无法弹权限窗'"]

auto 模式分类器流程

当模式是 auto（或 plan 模式下 auto 临时激活）且内层结果是 ask 时，进入这段。它的目标是用一个独立的 AI 分类器代替人工确认：把当前对话历史 + 这一步要做的动作喂给分类器，让它判这个动作该不该放行。但前面有几条快速通道，能不调分类器就不调（省 token、省延迟）。

graph TD
  auto-enter["auto 模式 + ask 结果"] --> safety-immune{"safetyCheck 且<br/>classifierApprovable=false?"}
  safety-immune -->|是| ret-ask-safety["保持 ask<br/>(敏感安全检查不交给分类器)"]
  safety-immune -->|否| needs-interact{"工具需用户交互?"}
  needs-interact -->|是| ret-ask-interact["保持 ask"]
  needs-interact -->|否| ps-guard{"PowerShell 且未开<br/>POWERSHELL_AUTO_MODE?"}
  ps-guard -->|是| ret-ask-ps["保持 ask<br/>(PS 需人工确认)"]
  ps-guard -->|否| accept-edits-fast["快速通道1: 假装 acceptEdits<br/>再问一次工具 checkPermissions"]
  accept-edits-fast --> ae-allow{"acceptEdits 下会 allow?"}
  ae-allow -->|是| ret-allow-ae["返回 allow<br/>fastPath=acceptEdits"]
  ae-allow -->|否| allowlist-fast["快速通道2: 工具在<br/>安全白名单 isAutoModeAllowlistedTool?"]
  allowlist-fast -->|是| ret-allow-wl["返回 allow<br/>fastPath=allowlist"]
  allowlist-fast -->|否| call-classifier["调用分类器<br/>classifyYoloAction"]
  call-classifier --> cls-result{"分类器结果"}
  cls-result -->|shouldBlock=false| ret-allow-cls["返回 allow<br/>reason=classifier"]
  cls-result -->|transcriptTooLong| ret-fallback-tl["回退人工 ask<br/>(headless 则中止)"]
  cls-result -->|unavailable| iron-gate{"tengu_iron_gate_closed?"}
  cls-result -->|shouldBlock=true| record-denial["记录拒绝计数<br/>recordDenial"]
  iron-gate -->|"true(默认)"| ret-deny-closed["返回 deny<br/>'分类器不可用,请重试'"]
  iron-gate -->|false| ret-ask-open["回退人工 ask"]
  record-denial --> denial-limit{"达上限?<br/>连续3/累计20"}
  denial-limit -->|是| ret-fallback-limit["回退人工 ask<br/>(headless 则中止)"]
  denial-limit -->|否| ret-deny-cls["返回 deny<br/>reason=classifier"]

用户交互：ask 之后弹窗到裁决落地

当 hasPermissionsToUseTool 返回 ask 时，useCanUseTool 接手把权限请求推进 UI 队列等用户响应。这段描述的是用户看到弹窗后能做什么、系统怎么把用户的选择落回裁决。三种来源的"放行"（用户手动 yes、PreToolUse hook、并发 bash 分类器）会竞争同一个裁决，靠 createResolveOnce 的原子 claim 保证只有第一个赢家生效。

graph TD
  ask-result["hasPermissionsToUseTool 返回 ask"] --> coord{"awaitAutomatedChecksBeforeDialog?"}
  coord -->|是| coordinator["先跑 coordinatorHandler<br/>(hook / bash 分类器抢答)"]
  coordinator --> coord-decided{"有裁决?"}
  coord-decided -->|是| done-coord["落地该裁决"]
  coord-decided -->|否| swarm["再跑 swarmWorkerHandler"]
  coord -->|否| swarm
  swarm --> swarm-decided{"有裁决?"}
  swarm-decided -->|是| done-swarm["落地该裁决"]
  swarm-decided -->|否| interactive["handleInteractivePermission<br/>推 UI 队列等用户"]
  interactive --> user-resp{"用户响应"}
  user-resp -->|"允许(可能改输入/加规则)"| on-allow["handleUserAllow<br/>持久化规则+记日志"]
  user-resp -->|"拒绝(可能带反馈)"| on-deny["记拒绝日志"]
  user-resp -->|Esc 中止| on-abort["cancelAndAbort<br/>中止本工具调用"]
  on-allow --> done-allow2["落地 allow"]
  on-deny --> done-deny["落地 deny"]
  on-abort --> done-abort["落地 ask(中止消息)"]
  • 弹窗期间如果 bash 并发分类器（BASH_CLASSIFIER）跑出 allow，会抢在用户点击前自动放行；用户一开始用键盘交互（200ms 宽限期后）就取消这个自动抢答。
  • 用户选"允许并总是允许这类操作"时，会通过 persistPermissions 把新规则写到对应来源（userSettings/projectSettings/localSettings/session 之一），下次同类调用直接走 allow 规则通道。
  • auto 模式下被分类器 deny 的工具会在 UI 上弹一条通知"被 auto 模式拒绝"，并把拒绝记进 recordAutoModeDenial 供后续展示。

模式切换状态机（Shift+Tab 循环）

用户按 Shift+Tab 在模式间循环，getNextPermissionMode 决定下一个模式。default/plan/acceptEdits/bypassPermissions 是用户可见的四个；auto 仅 ant 内部用户、且 TRANSCRIPT_CLASSIFIER 开关打开、且 auto 门禁（isAutoModeGateEnabled，含熔断状态）通过才出现；dontAsk 不在循环里暴露。进入 auto 时 transitionPermissionMode 会顺手剥掉危险权限规则（防止 auto + 危险规则组合）。

  def["default"] -->|Shift+Tab| def-next{"ant 用户?"}
  def-next -->|是| def-bypass{"bypass 可用?"}
  def-next -->|否| ae["acceptEdits"]
  def-bypass -->|是| bp["bypassPermissions"]
  def-bypass -->|否| def-auto{"auto 门禁开?"}
  def-auto -->|是| auto["auto"]
  def-auto -->|否| def["default"]
  ae -->|Shift+Tab| plan["plan"]
  plan -->|Shift+Tab| plan-bp{"bypass 可用?"}
  plan-bp -->|是| bp
  plan-bp -->|否| plan-auto{"auto 门禁开?"}
  plan-auto -->|是| auto
  plan-auto -->|否| def
  bp -->|Shift+Tab| bp-auto{"auto 门禁开?"}
  bp-auto -->|是| auto
  bp-auto -->|否| def
  auto -->|Shift+Tab| def

规则匹配决策表

规则按行为分三桶：allow / deny / ask。匹配时"整工具"和"带内容"分开看：ToolName（无括号）匹配整工具；ToolName(content) 由工具自己的 checkPermissions 解释 content（bash 是命令前缀匹配，Write/Edit 是 glob 路径匹配）。整工具级的 deny/ask 在工具自报检查之前跑，allow 在工具自报检查之后跑——这个顺序是刻意设计的：deny 优先级最高，ask 强制弹窗，而 allow 只在没有 deny/ask 反对时才生效。

graph TD
  start["规则匹配入口"] --> d1{"整工具 deny 规则?<br/>toolMatchesRule(无 content)"}
  d1 -->|命中| out-deny["deny, reason=rule"]
  d1 -->|否| d2{"整工具 ask 规则?"}
  d2 -->|命中且不沙箱放行| out-ask["ask, reason=rule"]
  d2 -->|否/可沙箱| d3{"工具 checkPermissions<br/>返回 deny/内容级 ask/safetyCheck?"}
  d3 -->|deny| out-tool-deny["deny(工具自报)"]
  d3 -->|"内容级 ask(rule)"| out-ask-content["ask, reason=rule<br/>(bypass 免疫)"]
  d3 -->|safetyCheck| out-ask-safety["ask, reason=safetyCheck<br/>(bypass 免疫)"]
  d3 -->|passthrough/allow| d4{"模式=bypass?"}
  d4 -->|是| out-allow-bypass["allow, reason=mode"]
  d4 -->|否| d5{"整工具 allow 规则?<br/>toolAlwaysAllowedRule"}
  d5 -->|命中| out-allow-rule["allow, reason=rule"]
  d5 -->|否| out-ask-default["passthrough→ask<br/>(交给模式后处理)"]
  • 规则来源优先级在 permissionsLoader 里合并（policy > userSettings > projectSettings > localSettings > ...），最终压成三桶规则列表挂在 ToolPermissionContext 上；同一桶里前来源的规则会遮蔽后来源的同内容规则（shadowedRuleDetection 负责提示被遮蔽的规则）。
  • 规则字符串解析在 permissionRuleValueFromString：找第一个未转义的 ( 和最后一个未转义的 )，括号内为 ruleContent；Bash(*) 和 Bash() 退化成整工具规则；括号本身可用 \( \) 转义。
  • Agent 类型拒绝是特殊语法 Agent(Explore)，filterDeniedAgents 一次性收集所有被拒 agent 类型再过滤，避免 O(agent×rule) 重复解析。
