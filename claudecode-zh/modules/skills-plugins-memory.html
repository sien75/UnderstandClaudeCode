技能、插件与记忆系统

这个模块讲 Claude Code 里三件"扩展能力"的事：① 用户或 Claude 调用一个 skill——预置好的、打包了提示词和工具配置的可复用工作流；② 安装一个 plugin 来扩展能力——plugin 怎么被发现、安装、加载，又如何贡献工具和命令；③ Claude 怎么"记住"东西——基于 CLAUDE.md 的持久记忆，分项目级、用户级、自动抽取、团队同步四条线。三套机制各自独立，但都围绕同一个主题：让 Claude 在本次会话之外仍能复用经验、能力和上下文。

一、Skill 加载与调用

用户视角：在会话里输入 /skills 会列出当前所有可用技能；Claude 自己也可以通过 SkillTool 主动调用某个技能。一个 skill 本质上是一个目录，里面放了一份 SKILL.md（带 YAML frontmatter 描述元信息：名字、描述、允许用的工具、参数等，正文是给 Claude 看的提示词）。系统启动时会从四个来源扫描这些目录：策略级（managed）、用户级（~/.claude/skills）、项目级（项目目录向上一直走到家目录的 .claude/skills）、以及额外的 --add-dir 目录。此外还有内置 skill（随 CLI 编译进去）和 MCP 服务器贡献的 skill。

输入
启动时的 cwd + 各级 .claude/skills 目录；运行时用户/Claude 选择的 skill 名 + 可选参数
输出
一份展开好的提示词（替换了 $ARGUMENTS、${CLAUDE_SKILL_DIR} 等），注入回对话；同时按需调整本次调用的工具白名单、模型、effort

  • 带 paths frontmatter 的 skill 是"条件技能"，平时不加载，等用户操作命中路径模式（gitignore 风格匹配）才激活，省 token
  • 动态发现：运行中读/写到 cwd 下子目录的文件时，会向上找 .claude/skills 并加载；gitignored 目录会被跳过
  • MCP skill 来自远程服务器，不可信——其正文里的 !`...` 内嵌 shell 命令永远不会被执行（安全边界）
  • 内置 skill 可携带参考文件，首次调用时懒提取到带 nonce 的私有目录（O_NOFOLLOW + 0o600 防符号链接攻击）
  • fork 模式：技能在独立子 agent 里跑，有自己的 token 预算，结果文本回传；inline 模式：直接把展开后的提示词塞回主对话
  • 权限：只含"安全属性"的 skill 自动放行；含 hooks、allowed-tools 等敏感字段的 skill 要问用户；用户也可配置 Skill(name) allow/deny 规则

二、Plugin 生命周期

用户视角：插件是比 skill 更大粒度的扩展包——一个 plugin 可以同时贡献 skill、命令、hook、MCP 服务器、LSP 服务器、agent 定义等多种能力。用户通过 /plugin 打开管理界面浏览/安装/启停，或在命令行用 claude plugin install foo@marketplace。插件来源有三种：① marketplace（插件市场，本身是一个 git 仓库，列着若干插件条目）；② 内置插件（随 CLI 编译，用户可开关）；③ --plugin-dir 临时会话插件（指向本地目录，仅本次有效）。安装作用域分 user/project/local/managed 四级，managed 由企业管理员通过策略锁死。

输入
用户选择的插件标识（name@marketplace）+ 作用域；或配置文件里声明的 marketplace 仓库
输出
一个已加载到内存的 LoadedPlugin——它的 skill/命令/hook/MCP/LSP/agent 都被注册进相应子系统，下次 Claude 能用上

  • 加载优先级：--plugin-dir 会话插件 > marketplace 安装的插件 > 内置插件；但 managed 策略会盖过 --plugin-dir（管理员意图优先于本地开发便利）
  • 启动时有两条加载路径：loadAllPlugins（会联网 clone 缺失的 marketplace）和 loadAllPluginsCacheOnly（只读缓存，不联网），后者用于不阻塞启动的消费者；前者用于 /plugins、/reload-plugins 等显式刷新
  • 安装是非原地更新：下到临时目录→算版本（git commit SHA 或 manifest version）→拷到版本化缓存目录→更新 installed_plugins.json；内存里的旧版本要等重启才换
  • 卸载时若仍有其他 enabled 插件依赖它，只警告不阻断（避免 delisted 插件卡住整个依赖图）
  • project scope 的卸载会引导用户用 local scope 禁用，因为 .claude/settings.json 是团队共享的
  • 插件可贡献 MCP/LSP 服务器，refreshActivePlugins 会 bump pluginReconnectKey 触发 MCP 重连，并 reinitialize LSP manager

三、记忆的读写与同步

用户视角：Claude 能"记住"东西，靠的是基于 CLAUDE.md 的持久记忆系统。记忆分四条线：① 项目级 CLAUDE.md——项目根目录的 .claude/CLAUDE.md，团队共享，进项目就加载；② 用户级 CLAUDE.md——~/.claude/CLAUDE.md，跨项目生效；③ 自动记忆（auto memory）——Claude 在对话中自己判断哪些值得记，写到 ~/.claude/projects/<slug>/memory/，下次自动注入；④ 团队记忆同步（team memory）——把本地记忆 push 到远端服务器，队友 pull 下来，实现记忆共享。用户用 /memory 命令可以打开编辑器手动编辑记忆文件。

输入
对话过程中的用户消息、纠正、偏好、项目背景；或用户在编辑器里手写的记忆内容
输出
落到 memory 目录的 .md 文件（带 frontmatter：type/description）+ MEMORY.md 索引；团队场景下还会同步到远端

  • 记忆分四类（user/feedback/project/reference），可从当前项目状态推导出的内容（代码模式、架构、git 历史）明确排除——这些不该记
  • MEMORY.md 是索引不是记忆本身：每条一行 - [标题](file.md) — 一句话钩子；超过 200 行或 25KB 会被截断并附警告
  • 回忆机制：扫目录读 frontmatter 生成清单 → 用 Sonnet 做相关性选择（独立 sideQuery，不污染主对话）→ 选中文件注入；已注入过的会过滤掉避免重复
  • 自动抽取（extractMemories）在每轮对话结束（模型产出无工具调用的最终回复）时触发，用 forked agent 共享主对话缓存；与主 agent 直接写互斥——主 agent 写过记忆就跳过本轮抽取
  • 有抽取频率节流（tengu_bramble_lintel，默认每轮都抽）和游标（lastMemoryMessageUuid）保证只看新增消息
  • team memory 同步：watcher 监听 team 目录写入 → 防抖 2s → push；push 用 content hash 算 delta，只传变化的条目；冲突（HTTP 412）时 pull 刷新 serverChecksums 后重试
  • 密钥安全：push 前用 gitleaks 规则扫描，命中密钥的文件被排除不上传，只记 rule id 不记值
  • session memory 是另一条线：用于 compaction 时压缩上下文，只在 auto-compact 开启时激活，写入专用 session memory 文件
  • KAIROS 模式：长会话改用追加式日志（按天一个文件），由夜间 /dream 技能蒸馏成 MEMORY.md + 主题文件
  • /memory 命令：弹出选择器让用户选要编辑哪个记忆文件，用 $EDITOR/$VISUAL 打开
