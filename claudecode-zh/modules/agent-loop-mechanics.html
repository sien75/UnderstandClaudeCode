Agent 循环机制（状态机·预算·钩子·历史）

这一模块回答的是「一次 agent 会话的循环到底怎么转」。当用户在 REPL 输入一条提示，或 SDK / -p 脚本调用进入一次 query，系统并不是简单地把请求丢给模型就完事——它会进入一个多轮循环：每一轮调用模型、执行工具、检查预算、跑终止钩子、决定是继续下一轮还是结束。本模块覆盖循环的五个核心控制面：终止/继续状态机（transitions）、上下文窗口预算与续写（tokenBudget）、终止钩子（stopHooks）、对话历史落盘（history）、会话状态地基与 plan 模式切换（bootstrap/state）。钩子系统的总执行链在 src/utils/hooks.ts，这里补全其主执行函数与匹配逻辑。

输入
用户的一条提示 + 当前会话状态（cwd、权限模式、agent 身份、历史消息）+ 运行时开关快照（QueryConfig）
输出
一条结束原因（Terminal）或一条继续原因（Continue）+ 可能的多轮工具执行结果与落盘的对话历史
  • transitions.ts 是循环的「状态枚举」，本身不含逻辑，但它是 query 主循环里所有 if/else 分支的判别依据——读懂它就读懂了循环能走到哪些终态。
  • tokenBudget 只在「主会话、非 agent、且配置了预算」时启用；子 agent 直接走 stop。
  • Stop 钩子在主线程与子 agent 行为不同：ComputerUse 锁、模板任务分类、记忆抽取都只对主线程跑，避免子 agent 污染主时间线。

一、终止/继续状态机

从外部调用者视角：一次 query 调用最终只会以两种方式离开循环——「Terminal」（循环返回，会话这一段结束）或「Continue」（再迭代一轮）。每种方式下又有一组业务原因。transitions.ts 把这些原因列成两个 union type，相当于状态机的状态集定义。下面这张图把所有可达状态及其业务语义画出来，并标注触发条件。

  after-model -->|用户/系统中断流| terminal-aborted["aborted_streaming / aborted_tools 中断"]

二、Token 预算与续写

业务背景：长任务（如大规模重构、长链路研究）单轮跑不完模型上下文窗口。系统允许配置一个「全局 token 预算」（按整个会话累计 token 计），当本轮累计还没烧到预算 90% 时，循环会在模型自然停下后注入一条「继续」nudge 消息，让模型接着干；接近预算或收益递减时停止。这个机制只对主会话、非子 agent 生效——子 agent 没有跨轮续写的语义。

graph TD
  budget-entry["checkTokenBudget 入口"] --> gate-check{"有 agentId 或 无预算?"}
  gate-check -->|是| stop-null["直接 stop（无事件）"]
  gate-check -->|否| calc-pct["算 pct=turnTokens/budget 累计 delta"]
  calc-pct --> diminishing-check{"连续≥3轮 且 最近两轮 delta 都 < 500?"}
  diminishing-check -->|是| stop-diminishing["stop 收益递减 产出 completionEvent"]
  diminishing-check -->|否| threshold-check{"turnTokens < budget*0.9?"}
  threshold-check -->|是| continue-nudge["continue 注入 nudge 消息 累加 continuationCount"]
  threshold-check -->|否| stop-budget["stop 预算用满 产出 completionEvent"]
  continue-nudge --> done-cont["返回 ContinueDecision"]
  stop-diminishing --> done-stop["返回 StopDecision"]
  stop-budget --> done-stop
  stop-null --> done-stop
  • COMPLETION_THRESHOLD = 0.9：累计 token 没到预算 90% 才允许续写。
  • DIMINISHING_THRESHOLD = 500：连续 3 轮、每轮新增 token 都不到 500，认定模型在「原地打转」，停。
  • BudgetTracker 是有状态的：continuationCount、上次 delta、上次累计 token 都存在 tracker 里跨轮传递。
  • completionEvent 会带 diminishingReturns 标志和 durationMs，用于上报分析。

三、Stop 钩子执行链

从用户视角：当模型本轮不再调工具、循环准备收尾时，会触发 Stop 钩子——用户可能在 settings.json 配了「停止时跑某条命令」（比如格式化、跑测试、发通知）。钩子可以阻断继续、可以塞 blocking 错误回灌给模型、也可以纯粹做副作用。handleStopHooks 是这个流程的编排器：先做一堆主线程专属的收尾（任务分类、记忆抽取、auto-dream、ComputerUse 解锁），再真正跑用户配的 Stop 钩子，最后如果是 teammate 身份还会跑 TaskCompleted 与 TeammateIdle 钩子。

  • 主线程判定靠 !toolUseContext.agentId——子 agent 跳过任务分类、记忆抽取、autoDream、ComputerUse 清理，避免污染主时间线或误释放进程级锁。
  • Stop 钩子结果有四种形态：成功无输出、成功有输出、非阻塞错误、执行中错误；后者两类都计入 hasOutput 与 hookErrors，用于摘要消息。
  • teammate 的 TaskCompleted 钩子按 in_progress 任务逐个跑，每个任务独立生成 toolUseID。
  • 整个流程包了 try/catch：钩子自身崩溃不会让 query 循环挂掉，只会产出一条 warning 系统消息和一条 tengu_stop_hook_error 上报。

四、对话历史持久化

从用户视角：当用户在 REPL 敲下回车提交一条提示，这条提示会进入「上箭头历史」和「ctrl+r 搜索」。系统把每条提示按 JSONL 行追加写到全局文件 ~/.claude/history.jsonl，每行带 display 文本、项目根、sessionId、时间戳、粘贴内容。读取时按逆序读文件，保证最新优先。粘贴内容（图片、大段文本）单独走 paste store，历史行里只存内联小文本或哈希引用，避免历史文件膨胀。

  skip-check -->|是| skip["跳过 不写"]
  skip-check -->|否| reg-cleanup["首次注册 cleanup 钩子 等待 in-flight flush + 最终兜底 flush"]

  paste-loop -->|大文本| hash["hashPastedText 算哈希 存 contentHash fire-and-forget 写 paste store"]

  remaining -->|是| sleep500["sleep 500ms 退避递归 retries+1"]
  remaining -->|否| done["完成"]
  sleep500 --> flush
  • 读取侧 makeLogEntryReader 先吐 pendingEntries（未落盘的），再逆序读 history.jsonl；保证 Up 箭头立刻看到刚提交的提示。
  • getHistory 把当前 session 的条目优先吐、其它 session 暂存后吐，避免多 session 交错；上限 MAX_HISTORY_ITEMS=100。
  • removeLastFromHistory 配合 Esc 撤销：pending 命中就 pop，否则把时间戳加入 skippedTimestamps 让读取侧跳过——保证已落盘的撤销条目不在 Up 箭头里出现两次。
  • lockfile 超时 stale=10s、重试 3 次、minTimeout 50ms，防止多进程并发追加互相踩。
  • 大文本粘贴 fire-and-forget 写盘，哈希引用立即进历史，不阻塞用户提交。

五、会话状态地基与 Plan 模式切换

bootstrap/state.ts 是整个会话的「单例状态盒」。它不直接驱动循环，但循环里几乎每个判断都要读它：当前 cwd、原始 cwd、稳定项目根、sessionId、累计 token 与成本、本轮预算基线、post-compaction 标志、注册的 hook matchers、plan/auto 模式切换附件标志。这里只展开与 agent loop 直接相关的两段——cwd/项目根三元组 与 plan/auto 模式切换——OTel provider 段 mapped 不展开。

cwd 三元组的业务含义：originalCwd 是会话启动时的工作目录（EnterWorktree 工具会改它）；cwd 是当前工作目录（每次工具切换就更新）；projectRoot 是稳定项目根，不受 mid-session worktree 影响，专门给历史/技能/session 锚定身份用。三者分离是为了「进临时 worktree 干活时，历史和技能还指向原项目」。

Plan 模式切换的逻辑很轻：切到 plan 时清掉待发的 plan_mode_exit 附件标志（防止快速来回切时同时发 plan_mode 和 plan_mode_exit）；切出 plan 时置 needsPlanModeExitAttachment=true，由后续消息组装侧消费。Auto 模式同理，但 auto↔plan 双向跳过，交给 prepareContextForPlanMode / ExitPlanMode 处理。

token 预算相关状态也在这一层：snapshotOutputTokensForTurn 在每轮开始拍下「截至本轮开始的累计输出 token」基线 + 设本轮预算 + 重置 continuationCount；getTurnOutputTokens = 总输出 - 基线，给 tokenBudget.ts 用作 turnTokens。

六、钩子匹配与执行总链（utils/hooks.ts 补全）

这一节补全 src/utils/hooks.ts 的主执行链。每个 execute*Hooks 函数（PreToolUse / PostToolUse / Stop / TeammateIdle / TaskCompleted / UserPromptSubmit / SessionStart 等）结构都一样：构造 hookInput → hasHookForEvent 快速短路 → 调 executeHooks。executeHooks 是核心：先做信任检查与 SIMPLE 模式短路，再 getMatchingHooks 拉到本事件匹配的钩子，按类型分组并行执行（callback/function/command/prompt/agent/http），逐个 yield 进度消息与结果。

匹配逻辑：getMatchingHooks 按 hookEvent 决定 matchQuery（PreToolUse/PostToolUse 用 tool_name，SessionStart 用 source，Notification 用 notification_type，FileChanged 用 basename 等），再用 matchesPattern 与每条 matcher 的 matcher 字段比对。matchesPattern 支持三种语义：通配 '*' 或空 → 全匹配；纯字母数字下划线竖线 → 精确匹配或竖线分隔多选；其它 → 当正则。同时还会对 legacy 工具名也试一遍正则，保证旧模式如 ^Task$ 仍能命中。
