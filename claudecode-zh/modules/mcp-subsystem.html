MCP 子系统

MCP（Model Context Protocol）是 Claude Code 接入外部能力的核心通道。这个子系统有两个对称的方向：① Claude Code 作为客户端，连接用户配置的各种 MCP server（本地 stdio 进程、远程 SSE/HTTP/WebSocket、claude.ai 代理连接器、IDE 内嵌服务），发现它们暴露的工具/资源/提示词，并在对话中转发调用；② Claude Code 自己也作为一个 MCP server，通过 stdio 把自己的内置工具暴露给别的 AI 客户端调用（claude mcp serve）。整个链路围绕"连接 → 发现 → 调用 → 处理结果 → 失败恢复"这条数据流展开，并叠加重连、会话过期、OAuth 授权、URL 引导授权（elicitation）等状态机分支。

输入
用户配置的 MCP server 列表（.mcp.json / 配置文件 / claude.ai 连接器 / IDE 锁文件 / SDK 动态注入），加上对话中模型决定调用的某个 MCP 工具名与参数；server 模式下则是外部 MCP 客户端发来的 list_tools / call_tool JSON-RPC 请求。
输出
客户端模式：把外部 MCP server 的工具/资源/提示词接入 Claude 的工具集，模型调用后返回的内容（文本/图片/音频/二进制）被规范化成消息块送回模型；server 模式：把 Claude Code 内置工具以 MCP 协议形式暴露给外部客户端并返回调用结果。

方向一：Claude Code 作为 MCP 客户端

启动时（或配置变更时），useManageMCPConnections 收集所有生效的 server 配置，调用 getMcpToolsCommandsAndResources 批量建立连接并发现能力。每个 server 通过 connectToServer（按配置序列化的 key 做 memoize 缓存）建立一条长连接：根据 type 选择 StdioClientTransport / SSEClientTransport / StreamableHTTPClientTransport / WebSocketTransport / SdkControlClientTransport / 进程内 InProcessTransport（Chrome、Computer Use 等重型 server 直接在主进程内跑，避免拉起 ~325MB 子进程）。连接带 30 秒超时（MCP_TIMEOUT），超时即关闭 transport 并抛出超时错误。

对远程 server（SSE/HTTP/claude.ai-proxy），连接前会查"needs-auth 缓存"（15 分钟 TTL）和"探测过但没有 token"标记——命中则跳过连接、直接给该 server 挂一个 McpAuthTool 伪工具，避免每 15 分钟对一批注定 401 的 server 反复探测。远程 server 的 fetch 经过三层包装：最内层 wrapFetchWithStepUpDetection（捕获 403 提权请求）、中层 wrapFetchWithTimeout（每个 POST 60 秒新鲜超时 + 强制带 Streamable HTTP 的 Accept 头）、外层是 claude.ai 代理专用的 createClaudeAiProxyFetch（带 OAuth bearer、401 时强制刷新一次并重试）。

连接成功后，fetchToolsForClient / fetchResourcesForClient / fetchCommandsForClient（均按 server 名 LRU 缓存）分别请求 tools/list、resources/list、prompts/list。发现的 MCP 工具被包装成内部 Tool 接口（名字形如 mcp__<server>__<tool>），其 call 方法真正执行时再走 callMCPToolWithUrlElicitationRetry → callMCPTool。若 server 声明支持 resources 且当前还没有资源工具，就补挂 ListMcpResourcesTool 和 ReadMcpResourceTool。任何 server 只要进入 needs-auth 状态，就额外挂一个 McpAuthTool 让模型可以触发 OAuth 流。

调用一个 MCP 工具时，callMCPTool 用 client.callTool 发起 JSON-RPC 调用，自带超时（默认 MCP_TOOL_TIMEOUT，约 27.8 小时——事实上无限，因为很多 MCP 工具是长任务）和 30 秒一次的进度日志。返回结果先经 processMCPResult 规范化：文本直接保留，图片压缩到 API 尺寸限制，二进制 blob 写盘后只回路径（避免 base64 灌进上下文），超大输出走 persistToolResult 落盘并返回读取指引。若 server 返回 isError: true，抛出 McpToolCallError（携带 _meta 以便 SDK 消费者仍能拿到元数据）。若返回 401/UnauthorizedError，抛 McpAuthError 让上层把 server 状态翻成 needs-auth；若返回 404 + JSON-RPC -32001（会话过期）或派生的 -32000 Connection closed，则清掉连接缓存并抛 McpSessionExpiredError，fetchToolsForClient 内的调用循环会自动重试一次（拿新 client 重连）。

连接掉线靠 client.onerror / client.onclose 检测。onclose 会清掉 connectToServer / fetch* 的 memoize 缓存，使下一次调用自动重连。onerror 区分终端错误（ECONNRESET/ETIMEDOUT/EPIPE/…）和瞬态错误：终端错误连续累计到 3 次就主动 close transport 让挂起的 callTool 失败并触发重连；HTTP 类还会识别"SSE 重连耗尽"和"会话过期"两种 server 主动放弃的信号。stdio server 的 cleanup 走 SIGINT → 100ms → SIGTERM → 400ms → SIGKILL 的逐级升级，保证 Docker 容器等需要显式信号的 server 也能优雅退出。

  • 缓存策略：connectToServer 按 ${name}-${JSON(config)} 做 memoize；fetchTools/Resources/Commands 按 server 名做 LRU（容量 20）。onclose / clearServerCache / reconnectMcpServerImpl 都会同时清这三类缓存，保证重连后不返回旧数据。
  • needs-auth 缓存：远程 server 一次 401 就写入 mcp-needs-auth-cache.json，15 分钟内不再尝试连接；McpAuthTool 完成 OAuth 或 /mcp 手动认证后会 clearMcpAuthCache。
  • 会话过期自愈：Streamable HTTP 协议下 server 返回 404 + -32001 表示 session 失效，客户端清缓存后下一次调用自动重连拿新 session，对调用方最多重试一次。
  • URL elicitation：MCP 工具可返回 -32042 要求用户打开一个 URL（OAuth 授权、外部确认等）。系统先跑 elicitation hook，hook 不处理则在 REPL 排队等用户操作、在 print/SDK 模式走 structuredIO 回调；最多重试 3 次。
  • 结果规范化：图片走压缩+下采样，二进制 blob 写盘后只回路径，超大输出落盘并返回读取指引——目的是把上下文体积压到最小。
  • 连接健康监测：useMcpConnectivityStatus 钩子在 mcpClients 变化时检查 failed/needs-auth 计数，分别给本地 server、claude.ai 连接器各发一条通知（"/mcp"），让用户知道有 server 掉线或需要授权。
  • 项目级 server 审批：handleMcpjsonServerApprovals 在启动时检查 .mcp.json 里 status=pending 的项目 server，单个走单选对话框、多个走多选对话框，用户批准后才会进入连接流程——防止项目仓库偷偷拉起不受信的 server。

方向二：Claude Code 作为 MCP server

当外部 AI 客户端想把 Claude Code 的能力当 MCP 工具用时，运行 claude mcp serve（src/cli/handlers/mcp.tsx 动态导入 src/entrypoints/mcp.ts 的 startMCPServer）。这条链路很直：创建一个 stdio MCP server，声明 tools 能力，注册 ListToolsRequest 和 CallToolRequest 两个 handler。

ListTools 把 getTools 返回的内置工具逐个转成 MCP 协议的 Tool 描述：调 tool.prompt() 取描述、zodToJsonSchema 转换 inputSchema、outputSchema（仅当根节点是 type: object 时才带，避免 anyOf/oneOf 在 SDK 端被拒）。CallTool 收到 {name, arguments} 后：找到对应工具 → 检查 isEnabled → 调 validateInput → 用 hasPermissionsToUseTool 做权限校验 → 调 tool.call。成功把结果序列化成 text content 返回；任何阶段抛错都走 getErrorParts 拆出错误文本，返回 isError: true 的结构化错误（不让异常打断 stdio 会话）。

server 模式构造的 ToolUseContext 是一个"无状态壳"：mcpClients: []、messages: []、isNonInteractiveSession: true、thinkingConfig: disabled——MCP 客户端不读消息历史，参数即一切。文件读取状态用 100 文件/25MB 上限的 LRU 缓存，防止 server 长时间运行内存无限增长。注释里有两个 TODO："Also re-expose any MCP tools"——目前 server 模式只暴露 Claude Code 自身内置工具，不会把客户端模式连的 MCP 工具再转发出去（避免循环依赖）。

  • stdio 传输：server 模式只用 StdioServerTransport，stdin/stdout 跑 JSON-RPC，stderr 留给日志——这是 MCP server 最通用的接入方式，任何 MCP 客户端都能直接 claude mcp serve 拉起。
  • 无状态上下文：ToolUseContext 里 mcpClients/messages 全是空，setAppState 是空函数——server 模式不维护会话状态，每次 call_tool 都是独立调用。
  • 权限校验：调用前过 hasPermissionsToUseTool，沿用客户端模式同一套权限规则；外部客户端的请求受同一套审批/规则约束。
  • 错误不杀会话：CallTool 的 try/catch 把任何异常转成 isError: true 的结构化返回，避免一次工具失败就让 stdio 会话崩掉。
  • 暂不转发 MCP 工具：代码注释明确标注 TODO，未来可能把客户端连的 MCP 工具也透传出去，但目前为避免循环/性能问题只暴露内置工具。

OAuth 授权流（客户端模式鉴权）

远程 server 首次连接或 token 过期时走 performMCPOAuthFlow。如果配置了 oauth.xaa（跨应用授权），走 performMCPXaaAuth——只要 IdP 的 id_token 还在缓存里，整条链是零交互的：用 id_token 换取 MCP server 的 access/refresh token，存进同一个 keychain 槽位，后续 ClaudeAuthProvider.tokens() 完全无感。

标准 OAuth 流程：找一个可用端口建 localhost 回调 server → 用 PKCE 生成 authorization URL（state + code_verifier + code_challenge）→ 打开浏览器让用户在 AS 上授权 → 回调收到 code → 换 token → 存 keychain。ClaudeAuthProvider 实现了 MCP SDK 的 OAuthClientProvider 接口：tokens() 读 keychain，过期则用 refresh_token 刷新（带进程内去重，并发请求复用同一个 refresh promise）；saveTokens 写 keychain 并清掉 step-up 标记；invalidateCredentials('tokens') 在 refresh 失败时清 token 但保留 discovery 信息，下次连接重新走授权。遇到 403 insufficient_scope 时，fetch 包装层会调 markStepUpPending(scope)，让 tokens() 故意省略 refresh_token——RFC 6749 §6 禁止用 refresh 提权，必须重新走授权拿到更大 scope 的 token。

  • XAA 静默授权：组织级 IdP 一次登录后所有 XAA server 共享凭证，避免每个 server 弹一次浏览器。
  • 跨进程 token 一致：keychain 读取有缓存 TTL，reconnectMcpServerImpl 会先 clearKeychainCache，确保 VS Code 扩展等别的进程改了 token 后本进程能看到。
  • State/nonce 防护：日志里对 state/nonce/code_verifier/code 等 OAuth 敏感参数做 redact，防 CSRF/会话固定。
