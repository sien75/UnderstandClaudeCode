智能体、任务与多智能体编排

这是 Claude Code 的"并行执行大脑"。当主对话里的 Claude 想把一部分工作交给别人去做——派一个子智能体去搜代码、起一个后台 shell 跑测试、或者拉起一支多智能体团队分工合作——所有这些请求都汇聚到这里，再分流到具体的执行通道。一条请求从进入到最后把结果送回主对话，会经历：选派谁去做 → 在什么隔离环境跑 → 任务状态如何登记与流转 → 完成或失败时如何通知主循环 → 中途怎么停掉或继续。

从用户/外部调用方的视角，能做的事有四类：(1) 派一个子智能体（同步等结果，或后台跑完再通知）——通过 Agent 工具；(2) 起一个后台 shell 任务（bash 监控、长命令）——通过 Bash 工具的 run_in_background 或 Task 工具族；(3) 组建多智能体团队（一个 leader 带若干 teammate，各自独立进程/会话，互发消息）——通过 TeamCreate + Agent 带 name 参数；(4) 对正在跑的任务查进度/取输出/停止——通过 TaskOutput、TaskList、TaskStop。本模块覆盖这四条链路的源码实现。

输入
主对话产生的一个工具调用：一段自然语言任务描述（prompt）、可选的子智能体类型/模型/隔离模式/后台标志；或一个 shell 命令；或一条发给某 teammate 的消息。
输出
一个结构化的工具结果回到主对话：子智能体的最终文本回复 + token/工具调用统计；或后台任务的 async_launched 句柄（agentId + 输出文件路径）；或 shell 任务的退出码 + 输出；或 teammate 的"已派发"确认。终态通知以 <task-notification> XML 形式注入回主循环消息队列。

一、Agent 工具：派子智能体的总入口

用户视角："我想让 Claude 并行处理子任务"——主对话的 Claude 调用 Agent 工具，传入 prompt、可选 subagent_type/model/run_in_background/name。Agent 工具是这一切的调度枢纽，它要回答几个问题：这次是要派一个普通子智能体、还是拉一个 teammate 进团队、还是 fork 出一个共享父上下文的 worker？走同步还是后台？需不需要 worktree 隔离？

  • 入口在 AgentTool.tsx 的 call()，整个调度逻辑约 1000 行，是本模块最核心的一段。
  • "应跑后台吗"这一步是分水岭：同步子智能体共享父对话的 abortController（用户按 ESC 会一起取消），后台子智能体拿一个独立 abortController（ESC 取消主循环不会误杀后台）。
  • 前台路径有个独特的"竞速循环"：每轮把"下一条消息"和"被转入后台的信号"做 Promise.race——一旦收到 background 信号就立即转后台，不阻塞等待下一轮。
  • worktree 隔离：选 isolation: "worktree" 时，给子智能体开一个临时 git worktree，跑完若无改动自动清理，有改动则保留并在结果里带 worktreePath。

二、子智能体执行循环：runAgent

一旦 Agent 工具决定"派谁、以什么隔离跑"，真正的执行交给 runAgent()。它是一个 async generator，逐条 yield 子智能体产生的消息。它负责把子智能体武装起来：解析它该用哪些工具、装上系统提示、连上它自己声明的 MCP 服务器、注册它的 frontmatter hooks、预加载它要用的 skills，然后把它喂进主 query() 推理循环。

  • 工具解析 resolveAgentTools（在 agentToolUtils.ts）按权限模式过滤：async 子智能体只能用 ASYNC_AGENT_ALLOWED_TOOLS 白名单里的工具；in-process teammate 额外放行 Agent 和任务协调工具以便它再派同步子智能体。
  • fork 路径特殊：用 useExactTools=true，直接拿父的工具数组与系统提示，目的是让 API 请求前缀字节级一致以命中 prompt cache——所以 fork worker 不能嵌套 fork（递归保护在 AgentTool.call 里靠 querySource + 消息扫描双重判断）。

  • finally 里的清理是"不留尾巴"的关键：杀掉这个 agent 自己 spawn 的 shell 任务、注销 session hooks、释放 perfetto 追踪、清空 todos 键——否则长会话里几百个 agent 会慢慢泄漏内存。

三、任务状态机：LocalAgentTask

每个被派出去的子智能体在 AppState.tasks 里都有一条 LocalAgentTaskState。状态字段 status 取自 TaskStatus 联合类型，流转有严格约束——这是本模块最典型的状态机视角。

graph TD
  pending["pending<br/>createTaskStateBase 初始态"] -->|"spawn 时直接置"| running["running<br/>registerAsyncAgent / registerAgentForeground"]
  running -->|"正常跑完<br/>completeAgentTask"| completed["completed<br/>带 result + endTime"]
  running -->|"抛异常<br/>failAsyncAgent"| failed["failed<br/>带 error"]
  running -->|"用户/TaskStop kill<br/>killAsyncAgent"| killed["killed<br/>abort + 清理"]
  running -->|"前台转后台<br/>backgroundAgentTask"| runningBg["running (isBackgrounded=true)<br/>前台循环退出，后台续跑"]
  runningBg -->|"后台跑完"| completed
  runningBg -->|"后台失败"| failed
  runningBg -->|"后台被 kill"| killed
  completed -->|"evictAfter 到期<br/>非 retain"| evicted["从 AppState 移除"]
  failed --> evicted
  killed --> evicted
  running -->|"SendMessage 续跑<br/>resumeAgentBackground"| running
  killed -->|"SendMessage 复活<br/>resumeAgentBackground"| running
  completed -.->|"retain=true 时<br/>UI 持有"| retained["保留在 AppState<br/>不进 evictAfter"]
  • 状态机定义在 src/Task.ts：TaskStatus = pending | running | completed | failed | killed，isTerminalTaskStatus() 把后三个标为终态——防止往死掉的 teammate 注入消息、防止清理路径重入。
  • notified 标志是去重关键：任务进入终态后通过 enqueueAgentNotification 向主循环注入 <task-notification> XML。但 TaskStop 也会主动标 notified 以抑制冗余通知。所有终态转换函数（complete/fail/kill）都先 updateTaskState 再 evictTaskOutput——状态先翻，磁盘后清，保证 TaskOutput 的 block 等待能立刻解除。
  • retain 字段控制 UI 是否"按住"这个任务：retained 任务不设 evictAfter，不会被后台指标清理，且会触发磁盘 bootstrap 把 sidechain 转录 UUID-merge 进 messages 供面板查看。
  • pendingMessages 队列：SendMessage 给一个正在跑的 agent 发消息时，消息进 task.pendingMessages，在工具轮次边界被 drainPendingMessages 抽干注入 agent 的 API 输入——不打断当前轮。
  • 面板可见性谓词 isPanelAgentTask：所有 pill/panel 过滤器必须共用这一个谓词，否则会不一致——agentType 为 'main-session' 的不算面板 agent。

四、后台生命周期驱动：runAsyncAgentLifecycle

无论从 Agent 工具直接后台派发，还是前台转后台，还是 SendMessage 复活一个已停止的 agent，最终都汇入 runAsyncAgentLifecycle()（在 agentToolUtils.ts）。它把"跑 agent + 追踪进度 + 终态转换 + 通知"这套流程封装成一个共享驱动器。

  makeTracker --> onCache["onCacheSafeParams 回调<br/>（启用摘要服务）"]
  onCache --> loop["for await makeStream() 消息"]

  retainCheck -->|"否"| progress["更新 progress<br/>emit task_progress SDK 事件"]
  appendLive --> progress
  progress --> loop
  loop -->|"流结束"| finalize["finalizeAgentTool<br/>提取最后一条 assistant 文本 + 统计"]
  finalize --> complete["completeAsyncAgent<br/>状态翻 completed（先于通知）"]
  complete --> handoff{"auto 模式<br/>需交接分类？"}
  handoff -->|"是"| classify["classifyHandoffIfNeeded<br/>安全分类器审查子智能体产出"]

  • 先翻状态再发通知是硬约束（gh-20236）：completeAsyncAgent/failAsyncAgent/killAsyncAgent 立刻把 status 翻成终态，这样 TaskOutput(block=true) 的 waitForTaskCompletion 立刻解除阻塞；而分类器调用、worktree 清理这种可能挂起的步骤放在状态翻转之后——它们不能阻塞状态机推进。
  • extractPartialResult：被 kill 时从已收集的 assistant 消息里倒序提取第一条有文本的，作为 finalMessage 通知——即使中途取消，父 agent 也能看到子智能体做到哪了。
  • classifyHandoffIfNeeded：仅 auto 权限模式 + TRANSCRIPT_CLASSIFIER 特性开启时触发，用一个独立分类器模型审查子智能体的动作是否违反安全策略——不可用时放行但加警告，阻断时返回 SECURITY WARNING 给父 agent。

五、停止任务：TaskStop → stopTask

用户或主对话想中途停掉一个后台任务（shell 或 agent），调用 TaskStop 工具。它是个薄封装，真正逻辑在 tasks/stopTask.ts 的 stopTask()。

六、多智能体协作：团队、teammate、消息

上面讲的是"主对话 vs 一个子智能体"的二元关系。多智能体团队是另一套机制：一个 leader 通过 TeamCreate 建团，再用 Agent 带 name 参数派 teammate；teammate 之间通过 SendMessage 互发消息。这套由 TeamCreateTool / TeamDeleteTool / SendMessageTool / spawnMultiAgent 协作实现。

  • in-process teammate用 AsyncLocalStorage 隔离，不是独立进程，生命周期绑在 leader 进程上——所以它不能再派后台 agent、不能再 spawn teammate（团队花名册是扁平的，嵌套会乱）。这些约束在 AgentTool.call 入口硬校验。

七、Coordinator 模式：leader 不亲自下场

普通模式下主对话的 Claude 自己用工具干活；Coordinator 模式（COORDINATOR_MODE feature flag + CLAUDE_CODE_COORDINATOR_MODE 环境变量）下，Claude 变成纯调度者——只用 Agent/SendMessage/TaskStop，所有实际工作派给 worker 子智能体。这套由 coordinator/coordinatorMode.ts 配置。

  gate -->|"否"| off["isCoordinatorMode()=false<br/>普通模式"]
  gate -->|"是"| on["isCoordinatorMode()=true"]
  on --> sysPrompt["getCoordinatorSystemPrompt()<br/>注入 leader 角色提示"]
  on --> userCtx["getCoordinatorUserContext()<br/>告知 worker 可用工具集 + scratchpad"]
  on --> forceAsync["Agent 工具里 shouldRunAsync=true<br/>所有 spawn 强制后台"]
  on --> escKill["ESC 取消时<br/>killAllRunningAgentTasks 全杀"]
  on --> summary["启用 AgentSummary<br/>周期性摘要 worker 进度"]
  match["matchSessionMode<br/>--resume 时"] --> switch{"session 模式 ≠ 当前？"}
  switch -->|"是"| flip["翻转 env var<br/>返回提示语"]
  switch -->|"否"| noop["无操作"]
  • Coordinator 的系统提示（coordinatorMode.ts 111-369）是一份完整的工作手册：定义 leader 角色、Agent/SendMessage/TaskStop 工具用法、<task-notification> 格式、研究→综合→实现→验证四阶段、并发策略（只读并行、写串行）、continue vs spawn 决策表、worker prompt 撰写规范。
  • 关键约束：leader 不设 model 参数（worker 用默认模型干实事）、不在一个 worker 里查另一个 worker（worker 完成会自动通知）、不臆造结果（结果作为独立消息到达）。
  • matchSessionMode 处理 resume 时模式不一致：旧会话存了 coordinator 模式但当前没开，就翻转 env 让 isCoordinatorMode() 返回正确值——因为 isCoordinatorMode() 是实时读 env 不缓存。
  • scratchpad 目录（tengu_scratch gate）给 worker 们一个免权限提示的共享知识库——leader 在 userContext 里告诉 worker 路径，worker 们在此读写交接信息。

八、任务工具族：Create/Get/List/Output/Update/Stop

除了 Agent（派智能体）和 SendMessage（智能体间通信），还有一组面向"任务清单"的工具，让多智能体团队共享一个待办列表来协调分工。这组工具操作的是磁盘上的任务列表（utils/tasks.ts，团队=项目=任务列表），与上面的"后台执行任务"是两套概念——这里是一个 TODO 看板。

  • TaskCreate：在当前任务列表（getTaskListId()，leader 用团队名、普通会话用 sessionId）建一条 pending 任务，触发 TaskCreated hooks；若 hook 返回 blockingError 就回滚删除。auto-expand 任务面板。
  • TaskUpdate：改任务状态（pending→in_progress→completed）、owner、依赖关系。用于 teammate 领活、标记完成。
  • TaskList/TaskGet：列出全部 / 查单条，供 teammate 看板协调。
  • TaskOutput：读后台执行任务（LocalAgent/LocalShell/Remote）的输出。block=true 时 waitForTaskCompletion 阻塞到终态或 timeout——这是主对话"等某个后台 agent 跑完"的同步原语，依赖第三节的状态翻转及时解除阻塞。
  • TaskStop：见第五节，多态 kill。

九、Remote 与 Dream：另两类任务

除了 LocalAgent（本机子智能体）和 LocalShell（本机 bash），任务系统还有两个成员：

  • RemoteAgentTask（tasks/RemoteAgentTask/）：把 agent 派到远程 CCR 环境跑（ant-only，isolation: "remote" 或 ultraplan/review 等场景）。registerRemoteAgentTask 建任务态，remoteTaskMetadata 持久化到 sidecar 供 --resume 重连还活着的远程会话；restoreRemoteAgentTasks 在启动时从 CCR 拉取最新状态恢复任务。kill 通过 RemoteAgentTask.kill 多态分发。
  • DreamTask（tasks/DreamTask/）：一种特殊的多轮迭代任务（DreamPhase: starting/updating），addDreamTurn 累加轮次、completeDreamTask/failDreamTask 终态转换，同样实现 Task 接口参与多态 kill。
  • 所有具体 Task 实现都注册在 getAllTasks()（tasks.ts）：LocalShellTask、LocalAgentTask、RemoteAgentTask、DreamTask 始终在列，LocalWorkflowTask/MonitorMcpTask 受 feature flag 控制按需 require（避免循环依赖 + 死代码消除）。
  • 任务 ID 前缀按 type 分配（local_bash→b、local_agent→a、remote_agent→r、in_process_teammate→t、dream→d），后接 8 位 36 进制随机——36^8≈2.8 万亿组合，抗符号链接暴力攻击。

关键设计要点回顾

  • 状态先翻，通知后发：所有终态转换函数先 updateTaskState 再做可能挂起的清理/分类/通知，保证 TaskOutput 的阻塞等待能立刻解除（gh-20236）。
  • notified 去重：enqueueAgentNotification 用 notified 标志原子地防止重复通知；TaskStop 也标 notified 抑制 shell 的 exit 噪声。
  • 前台↔后台可逆：前台 agent 跑到一半可以转后台（backgroundSignal race），后台 agent 被 SendMessage 续跑或复活——同一套 runAsyncAgentLifecycle 驱动。
  • 隔离层次：同步子智能体共享父 abortController；后台子智能体独立 abortController（ESC 不误杀）；in-process teammate 用 AsyncLocalStorage 同进程隔离；tmux teammate 独立进程；remote agent 独立 CCR 环境。
  • 权限沿继承链收窄：worker 的工具池用 worker 自己的 permissionMode 重新组装（assembleToolPool），不继承父的工具限制；allowedTools 提供时替换所有 session 级规则防泄漏，但保留 cliArg（SDK --allowedTools）。
  • fork 缓存对齐：fork worker 继承父系统提示 + 父工具数组 + 父 thinkingConfig，目的是 API 请求前缀字节级一致以命中 prompt cache——所以禁止嵌套 fork，递归保护靠 querySource（autocompact 不改 context.options）+ 消息扫描双保险。
  • 清理不留遗：runAgent 的 finally 杀掉 agent 自己 spawn 的 shell 任务、注销 hooks、清 todos 键、释放 perfetto——长会话里几百个 agent 不至于慢慢泄漏。
