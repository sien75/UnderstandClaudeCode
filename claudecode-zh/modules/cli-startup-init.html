CLI 启动与会话初始化

这个模块讲的是：用户在终端敲下 claude 回车之后，到屏幕上出现可交互的 REPL 输入框之前，程序内部到底走了一条什么样的路。整条链路的核心目标是"又快又稳地进入对话状态"——快体现在大量并行预取和快速通道（fast-path）提前 return，避免无谓地加载整个 CLI；稳体现在信任对话框、权限沙箱校验、配置错误处理等一道道关卡。从数据流的角度看，输入是用户敲下的一条命令行（argv），输出要么是一个正在跑的交互式 REPL 进程，要么是一次性的 headless 结果打印后退出。

输入
用户在终端敲的 claude 及其后跟的参数（子命令、flags、prompt 文本、stdin 管道内容）
输出
一个就绪的可交互 REPL（屏幕渲染完成、上下文装配好、权限模式确定），或一次性打印完回复后进程退出（--print 模式）

上面这张图里，每一个"是"分支都是一次"提前 return"，意味着对应场景下根本不会去加载庞大的 main.tsx，这是启动速度优化的关键设计——比如 --version 连一个动态 import 都不需要。只有所有快速通道都不命中，才会落到最底下加载完整 CLI、调用 main()。需要留意 --update/--upgrade 会被改写成 update 子命令再继续往下走（而非直接 return）。

第二段进入 main.tsx 的 main() 与 run()：main() 做一批"必须在任何代码执行前完成"的安全与状态初始化，并把一些特殊调用形式（cc:// URL、ssh、assistant）从 argv 里剥离暂存到全局 pending 对象里，让后续默认 action handler 来接管；run() 则构建 commander 程序并注册一个 preAction 钩子——这个钩子在任意命令真正执行前运行，负责把配置系统、MDM/keychain 预取、init()、日志 sinks、迁移、远程设置等"公共初始化"集中跑掉。

注意 init()（在 entrypoints/init.ts，memoize 单次执行）才是真正"打开配置系统"的地方：它 enableConfigs()、应用"安全"环境变量（信任对话框之前的子集）、应用额外 CA 证书、挂优雅退出、初始化 1P 事件日志与 GrowthBook、配置 mTLS/全局 HTTP agent、preconnect Anthropic API、CCR upstreamproxy、JetBrains/GitHub 仓库检测等。配置解析错误时会区分交互/非交互分别走对话框或 stderr 退出。preAction 之后才真正进入具体命令的 action handler。

第三段是默认 action handler 的前半段——这一段把用户选项解构清楚、做一堆校验、并行启动若干预取，然后调用 setup() 把工作目录/会话内存/git/worktree 等运行时地基铺好。这一段是"启动优化"最密集的区域：很多 IO 被并行化或延迟到首帧之后。

这里有个关键的并行设计：setup()（src/setup.ts）和 getCommands()/getAgentDefinitionsWithOverrides() 是同时 kick 出去的——因为 setup() 主要是 UDS socket 绑定（约 20ms）不抢磁盘 IO，而 commands 读取是文件 IO，两者天然可重叠。但 --worktree 会让 setup() 调 process.chdir()，所以那种情况下 commands 必须等 chdir 完成再读，于是被门控为 null、在 setup 完成后重新取。setup() 内部还有几道值得知道的关卡：bypassPermissions 模式会校验"不能以 root/sudo 跑、必须在无网的 Docker/沙箱里"（否则 exit），tengu_started 事件在 sinks 挂好后立即发出（作为发布健康监控的最早可靠"进程已启动"信号）。

几个值得新人知道的细节：--init-only 是个隐藏的"只跑 Setup + SessionStart:startup hooks 然后退出"的开关，用于环境初始化脚本；--print 模式信任是隐式的（帮助文本里也写明了"仅在信任目录使用"），所以它跳过信任屏但会做 applyConfigEnvironmentVariables 应用完整（含潜在危险）环境变量；交互模式的信任屏一旦被拒，process.exitCode 会被置位，后续所有可能触发代码执行的步骤（包括 apiKeyHelper）都被跳过，这是为了防止在未信任目录里执行插件代码。launchRepl（src/replLauncher.tsx）本身非常薄——动态 import App 和 REPL 组件后交给 renderAndRun 渲染，真正的 REPL 逻辑在 screens/REPL 里，不在本模块范围。

关于辅助视角的自然观察：状态机层面最关键的状态分叉是"交互 vs 非交互"（isNonInteractiveSession，由 -p/--init-only/--sdk-url/!stdout.isTTY 决定），它决定信任是走对话框还是隐式、MCP/插件初始化是 fire-and-forget 还是 await；其下又有 continue/resume/connect/ssh/assistant/new 等会话来源子状态。权限层面，permissionMode 由 initialPermissionModeFromCLI 根据 --permission-mode/--dangerously-skip-permissions 解析，bypass 模式在 setup() 里有专门的沙箱校验关卡（root/docker/互联网），auto 模式有 GrowthBook 门控异步校验。生命周期层面，tengu_started 事件是"进程已启动"的最早可靠信号（sinks 挂好后立即发），tengu_timer{event:'startup'} 在首屏阻塞对话框渲染前记录启动耗时，会话结束时 setup() 还会从 projectConfig 读上一会话的 tengu_exit 指标并发出。

  • 整个启动链路极度依赖"动态 import + 并行/延迟"来压首帧时间：cli.tsx 几乎所有分支都是 await import(...)，--version 甚至零 import；main.tsx action handler 里 setup/commands/agents/MCP/git/user/model 预取大量并行。
  • --bare（等价 CLAUDE_CODE_SIMPLE=1）是一个"最小模式"总开关：跳过 UDS messaging、teammate 快照、iTerm/Terminal 备份恢复、插件预取、attribution hooks、release notes、startup prefetches 等，专为脚本化调用瘦身。
  • 信任（trust）是一道硬边界：交互模式下 showSetupScreens 之前的代码刻意避免在未信任目录执行插件 LSP / apiKeyHelper；非交互模式信任隐式但要求"仅在你信任的目录使用 -p"。
  • cli.tsx 里的 feature('XXX') 是构建期 dead-code-elimination 门控，外部构建会把这些 ant-only/实验性分支整段裁掉，所以同一份源码对外对内行为不同。
  • 覆盖说明：本模块聚焦"敲下 claude 到 REPL 就绪"的主数据流。cli/print.ts（5596 行）的 runHeadless 内部细节、bridgeMain/daemonMain/templatesMain 等快速通道的内部实现、screens/REPL 渲染后的交互逻辑均未深读，仅在主流程中标为出口节点；main.tsx 中大量纯选项注册行（968-1005、3811-3884 的子命令/option 链）与 4395-4684 的尾部子命令 action 仅 mapped 理解职责。
