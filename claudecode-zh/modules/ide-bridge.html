IDE Bridge（IDE 集成）

当用户在 VS Code 或 JetBrains 里装了 Claude Code 扩展后，扩展需要连上本地的 CLI 后端来真正干活。这个模块就是 CLI 以 "bridge 模式" 启动后的整套通信骨架：它让 CLI 变成一个长驻进程，和 IDE 之间通过 WebSocket 建立双向通道，用 JWT 做鉴权，把 IDE 里点的各种操作（改文件、看 diff、权限确认、发起会话）路由到 CLI 核心，再把结果回流给 IDE 展示。

从用户视角看：你在 IDE 里打开 Claude Code 面板、敲一句话、点一个"应用编辑"按钮——这些动作并不是 IDE 扩展自己执行的，而是通过这条 bridge 通道递交给本地 CLI，CLI 跑完再把改动、diff、待确认权限推回 IDE。本模块讲的就是这条通道怎么建立、怎么鉴权、消息怎么来回跑、会话怎么跑起来又怎么收尾。

输入
IDE 扩展发起的连接请求（含 JWT token）+ 后续一系列操作指令（发起会话、发送用户消息、确认/拒绝权限、查看 diff 等）
输出
一个持久双向通道：IDE 提交的每个动作都被 CLI 执行，结果（流式文本、文件改动、权限询问、错误）实时回流到 IDE 面板

一、Bridge 模式启动：从敲下命令到通道就绪

用户在终端敲 claude remote-control（或在 IDE 扩展里点"连接 Remote Control"），CLI 不进交互式 REPL，而是走 bridgeMain 这条分支。它要做的事可以拆成几步：先确认你这台机器有资格跑 bridge（已登录、工作区已信任、多会话灰度放行），再向服务端注册一个"环境（environment）"，拿到一对 environmentId + secret，然后进入一个长轮询循环，等服务端把"工作项（work item）"派下来。

注意 BRIDGE_MODE 是个 feature flag（GrowthBook 灰度），关掉时整个 bridge 命令直接退出，连注册都不做——所以这套能力是按账号/灰度逐步开放的。

  • 预创建会话（createBridgeSession）是非阻塞的——失败也不挡启动，只是用户得等服务端派下来第一条 work 才能开始。

二、鉴权：两套 token，各管一段

这是这个模块最容易绕晕的地方，但它有清晰分层。Bridge 一共涉及两种凭证：

  • OAuth 访问令牌：你 /login 拿到的那个，用于 bridge 自己直接调服务端管理类接口（注册环境、轮询 work、停 work、归档会话、注销环境）。这套走 withOAuthRetry：遇 401 先尝试刷新一次 OAuth，刷新成功就重试一次请求，再 401 才当致命错误。
  • 会话 ingress JWT（session_ingress_token）：服务端在派发每个 work item 时，把一个短寿命 JWT 塞进 work secret（base64url 编码的 JSON）一起发下来。这个 JWT 才是用来在 会话通信通道（WebSocket / SSE 读流 + POST 写流）上证明"我是这个会话的合法 worker"的。它会过期（约 5 小时），过期后服务端重新派发一次 work，bridge 拿到新 JWT 重连。

另外还有一层 可信设备令牌（trusted device token）：当 tengu_sessions_elevated_auth_enforcement 灰度开着，bridge 调注册接口时会在 X-Trusted-Device-Token 头里带上它，服务端借此要求"高权限会话必须来自已登记的可信设备"。这个 token 在 /login 时通过 POST /auth/trusted_devices 登记，存在系统钥匙串，90 天滚动有效。

  • URL 安全校验：非 localhost 的 base URL 必须是 HTTPS，防凭证明文外泄。

三、消息双向流：IDE 点的按钮怎么到 CLI，CLI 的输出怎么回 IDE

这是这个模块的核心。整个通道其实不是 IDE 直连 CLI，而是双方都连服务端中转：

上行（CLI → 服务端 → IDE）：子 CLI 进程以 --print --output-format stream-json 跑，每产生一条 NDJSON 行（assistant 文本、tool_use、result 等），bridge 父进程读它的 stdout，解析出活动摘要（"Editing src/foo.ts"）更新状态行；同时这些 SDK 消息通过 transport 的 writeBatch POST 到 session-ingress，服务端再分发给订阅了该会话的 IDE/Web 客户端。

下行（IDE → 服务端 → CLI）：用户在 IDE 里敲一句话、点"应用编辑"、批准一个权限——这些动作被 IDE 扩展发到服务端，服务端把"有活干"封装成一个 work item 投到这个 environment 的队列。bridge 的轮询循环 pollForWork 拉到 work，解码出 JWT，ack 确认，然后连上 ingress 通道（v1 WebSocket 或 v2 SSE）。之后服务端通过这条通道把 user 消息、control_request（interrupt、set_model、can_use_tool 权限询问）推下来，bridge 转发给子进程或本地权限回调处理。

  • 权限是个双向环：子 CLI 想用某工具 → 发 control_request(can_use_tool) → bridge 转发给服务端 → IDE 弹框 → 用户选择 → control_response 回来 → bridge 转回子 CLI stdin。
  • 附件处理：Web 端上传文件拿到 file_uuid，随消息发下来；bridge 用 OAuth 调 /api/oauth/files/{uuid}/content 下载，写到 ~/.claude/uploads/{sessionId}/，把 @"path" 前缀拼到消息内容前，子 CLI 的 Read 工具就能读到。
  • v1/v2 选择是服务端按会话决定的（work secret 里 use_code_sessions），不是 bridge 自己选；CLAUDE_BRIDGE_USE_CCR_V2 是 ant 内部强制开关。

四、连接断了 / 会话过期 / 系统休眠：恢复路径

长连接难免会断。这个模块的恢复策略分几层，关键是不丢用户消息、不无限重试、能区分"短暂网络抖动"和"环境已死"。

graph TD
  pollErr["轮询 pollForWork 抛错"] --> isAborted{"signal 已 abort?"}
  isAborted -->|是| break["退出循环"]
  isAborted -->|否| isFatal{"BridgeFatalError?"}
  isFatal -->|是| classifyFatal{"401/403 认证?"}
  classifyFatal -->|是| fatalExit["致命退出<br/>不重试"]
  classifyFatal -->|404/410 过期| fatalExit
  isFatal -->|否| isConn{"连接错误?<br/>(ECONNRESET/ETIMEDOUT...)"} 
  isConn -->|是| sleepDetect{"间隔远大于<br/>最大退避?<br/>(系统休眠)"}
  sleepDetect -->|是| resetBudget["重置错误预算<br/>fresh 重试"]
  sleepDetect -->|否| connBackoff["指数退避 2s→120s<br/>最多 10min 放弃"]
  connBackoff --> heartbeatDuring["退避期间仍心跳<br/>保 300s 租约"]
  isConn -->|否| generalBackoff["通用错误退避<br/>最多 10min 放弃"]
  connBackoff --> reconnectedOk{"恢复?"}
  reconnectedOk -->|是| logReconnect["打印 Reconnected"]
  reconnectedOk -->|否| giveUp["超 10min 放弃<br/>致命退出"]

  closeCode -->|异常| wakePoll["唤醒轮询循环<br/>快速重拉 work"]
  wakePoll --> freshWork["服务端重派 work<br/>带新 JWT"]
  freshWork --> reconnect["onWorkReceived<br/>用新 token 重连通道"]

  env404["poll 返回 404<br/>(环境被删)"] --> envLost["onEnvironmentLost"]
  envLost --> strategy1{"策略1:<br/>同 envId 重注册<br/>+ reconnectSession"}
  strategy1 -->|成功| sameSession["保留原会话<br/>URL 仍有效"]
  strategy1 -->|失败| strategy2["策略2:<br/>归档旧会话<br/>新 env 上建新会话"]
  strategy2 --> freshSession["currentSessionId 更新<br/>重置序列号/去重集"]
  strategy2 --> retryLimit{"重建超 3 次?"}
  retryLimit -->|是| failed["状态 → failed<br/>触发 onFatalError"]
  retryLimit -->|否| continue["继续轮询"]
  • 系统休眠检测：如果两次轮询错误的间隔远大于最大退避上限（默认 240s），判定机器睡了，重置错误预算，否则醒来会立刻因"已超 10min"被放弃。
  • 心跳保租约：work item 有 300s 租约 TTL，即使通道在退避重连，heartbeatActiveWorkItems 也会继续打 heartbeat，防止租约到期被服务端回收。
  • JWT 过期 → 心跳会收到 401/403 → bridge 不当作致命错误，而是调 reconnectSession 让服务端把这条 work 重新入队，下一轮 poll 拿到带新 JWT 的同一条 work，onWorkReceived 用新 token 重连通道（existingHandle 路径）。
  • 环境重建有 3 次上限，防止死循环；成功后重置计数，所以几小时后另一次失败不会因为历史累计而被误杀。
  • perpetual 模式（daemon）teardown 是"本地退、服务端留"——不归档、不停 work，靠服务端 300s 租约自然回 pending，下次启动读指针 reconnectSession 接上。

五、会话生命周期：从派发到收尾

一个会话从被派发到结束，bridge 要管它的目录、超时、标题、worktree 清理、向服务端报停。这是 Lifecycle 视角的补充。

  existing -->|是| updateToken["更新 ingress token<br/>重排刷新定时器<br/>子进程 stdin 推新 token"]
  existing -->|否| atCap{"已达容量上限?"}
  atCap -->|是| skipSpawn["不派生<br/>等容量"]
  atCap -->|否| spawn["safeSpawn 启子 CLI<br/>--print --sdk-url ..."]
  spawn --> spawnFail{"spawn 抛错?"}
  spawnFail -->|是| stopWorkFail["stopWork 报停<br/>清理 worktree"]
  spawnFail -->|否| register["记入 activeSessions<br/>设开始时间/超时/刷新定时器"]
  register --> timeout["设 24h 超时看门狗"]
  register --> title["异步拉会话标题<br/>首条用户消息时推导标题"]
  register --> status["开始状态行实时刷新<br/>(工具轨迹/耗时)"]
  status --> running["会话运行中"]
  running -->|"子进程 close"| done["onSessionDone"]
  done --> status2{"退出状态"}
  status2 -->|completed| logComplete["打印完成"]
  status2 -->|failed| logFail["打印失败<br/>(非关停导致)"]
  status2 -->|interrupted| skipLog["跳过报停<br/>(服务端发起或关停)"]
  logComplete --> stopWork["stopWorkWithRetry 报停"]
  logFail --> stopWork
  stopWork --> worktree["清理 worktree<br/>(worktree 模式)"]
  worktree --> mode{"spawn 模式?"}
  mode -->|single-session| archiveExit["归档会话<br/>abort 循环 → 退出"]
  mode -->|多会话| archiveIdle["归档会话<br/>回到 idle 接新 work"]
  • spawn 模式三选一：single-session（一条会话跑完即退）、same-dir（多会话共享 cwd）、worktree（每会话独立 git worktree，互不踩文件）。运行中按 w 可在 same-dir/worktree 间切。
  • 标题策略：服务端设过名（--name/Web 改名）优先；否则等首条真实用户消息，deriveSessionTitle 截 80 字符，PATCH 回服务端，让 IDE 列表里能显示。
  • 24h 超时看门狗：到点 SIGKILL 子进程，标记 timedOut，onSessionDone 据此把 interrupted 改记为 failed，照常报停+清理。
  • 优雅关停顺序：SIGTERM → 等 30s 宽限 → 仍不死就 SIGKILL → 停所有 work → 清 worktree → 归档会话 → 注销环境 → 清崩溃指针。single-session 且非致命退出时，故意留会话和环境活着，打印 claude remote-control --continue 让用户能恢复。
  • 崩溃恢复指针（bridgePointer）：spawn 后立刻写到 .claude/bridge_pointer，每小时刷新 mtime；正常 teardown 清掉，崩溃则留下，下次 --continue 读它恢复。

六、REPL bridge：交互式会话里的另一种接法

上面讲的是 claude remote-control 独立进程模式。还有一种形态：用户已经在跑正常交互式 claude REPL，又想顺手从 Web/手机控制同一条会话——这时走 initReplBridge / initBridgeCore。它和独立模式共用 bridgeApi、workSecret、bridgeMessaging、jwtUtils，但区别是：transport 在本进程内（HybridTransport/SSETransport），不 spawn 子进程；环境/会话生命周期和 REPL 进程绑定；多了一套环境丢失后原地重建（reconnect-in-place）逻辑。

  registerEnv --> createSess["创建/恢复会话<br/>写崩溃指针"]

  pickV -->|v2| sse["createV2ReplTransport<br/>SSE 读 + CCRClient 写<br/>用 JWT, registerWorker"]

  ready --> twoWay["双向消息流<br/>(同图三)"]
  twoWay --> envLost2{"poll 404?"}

  reconnect --> startPoll
  twoWay --> teardown2["teardown: 发 result → stopWork → 归档 → 注销"]
  • SSE 序列号携带（lastTransportSequenceNum）是关键细节：换 transport 时把旧的高水位带上，新 SSE 首次连接发 from_sequence_num，否则服务端会把整段会话历史从 0 重放一遍——每条历史 prompt 都当新消息再砸一遍。
  • 初始历史刷写有 flushGate：刷历史期间新消息排队，避免和历史的 POST 交错到达服务端导致顺序乱。
  • perpetual 模式（daemon）teardown 不发 result、不停 work、不关 transport——本地退、服务端留，靠 300s 租约回 pending，下次 --continue 读指针 reconnectSession 接上。
  • outbound-only 模式：只 POST 上行、不收下行 control_request，对 mutable 请求（interrupt/set_model）回错——用于"我只许你看我本地跑的结果，不许从外面操控我"的场景。
