后台 Forked 子智能体与远程 Agent 执行

这个模块讲两件事：一是 Claude Code 主对话每跑完一轮，会在后台"分叉"出若干个子智能体——它们复用主线程的 prompt 缓存，偷偷干一些不打扰用户的活，比如巩固记忆、生成下一句建议、维护 Magic Doc、给协调器子智能体写进度摘要；二是当会话跑在远程 CCR 容器里时，本地 CLI 通过 WebSocket 订阅远程会话事件、转发权限请求、把 curl/gh 的出站流量经 CONNECT-over-WebSocket 隧道中转到 CCR 入口、由服务端做 MITM TLS 并注入凭据。两部分的共同主线是"数据流"：forked 子智能体的核心是缓存命中这条数据流，远程执行的核心是消息与字节在 CLI、WebSocket、CCR 容器之间的双向流动。

输入
主对话每轮结束后的 REPLHookContext（含 systemPrompt、tools、messages、toolUseContext）——后台 fork 子智能体都从这里取经；远程侧则是 CCR 服务端推来的 SDK 消息流，以及本地 curl/gh 等发往 127.0.0.1 隧道端口的 CONNECT 请求。
输出
后台侧：巩固后的记忆文件、Magic Doc 更新、会话记忆 markdown、prompt suggestion 文本、协调器摘要——全部回写到磁盘或 AppState，不阻塞主对话。远程侧：本地 REPL 渲染远程 agent 的流式消息、权限请求经本地用户决策后回传 CCR、子进程流量被透明代理到真实上游。
  • "forked"的本质：用主线程一模一样的 system/tools/messages 前缀发起一次新 query，让 Anthropic API 的 prompt cache 命中，把后台活儿几乎变成"只付输出 token"的廉价操作。
  • 所有后台 fork 子智能体的共同模式：工具定义留在请求里（保 cache key 一致），但通过 canUseTool 回调按需放行或拒绝——该只读的全 deny，该改特定文件的才 allow。
  • 远程侧全链路 fail-open：WebSocket 断了有重连预算，重连失败降级为"会话结束"；upstreamproxy 任一步出错都禁用代理、绝不拖垮会话。

一、Fork 机制核心：runForkedAgent + CacheSafeParams

主对话每跑完一轮，handleStopHooks 会把当前轮的 CacheSafeParams 存到模块级 lastCacheSafeParams 槽位。后台 fork 子智能体（promptSuggestion、postTurnSummary、/btw 等）直接从这个槽位取参，不用各自穿透参数——这是缓存命中的第一道保证。CacheSafeParams 携带五个影响 cache key 的字段：systemPrompt、userContext、systemContext、toolUseContext（含 tools/model/options）、forkContextMessages（消息前缀）。

thinking config 也是 cache key 的一部分，所以 maxOutputTokens 这个参数被特别警告：它会在 claude.ts 里钳制 budget_tokens，从而改变 thinking config、击穿缓存。只有"不需要共享父缓存"的场景（比如紧凑摘要）才允许设它。

几个对业务很关键的细节：createSubagentContext 默认把所有可变状态隔离——readFileState 克隆、abortController 新建（但链到父，父 abort 会传播）、所有 mutation 回调置 no-op。这样 fork 写文件不会污染父线程的文件状态视图。例外是 setAppStateForTasks：它必须穿透到根 store，否则后台 bash 任务永远注册不上、杀不掉，变成 PPID=1 的僵尸。contentReplacementState 默认克隆而非新建——fork 处理父消息里带着的 tool_use_ids 时，新建状态会把它们当未见过、做出不同的替换决策、wire 前缀就不同、cache miss；克隆则做相同决策、cache hit。

二、autoDream：记忆巩固的时间+会话数双闸门

用户视角：什么都不用做。每轮对话结束后，autoDream 偷偷检查"距离上次巩固记忆过了多久"和"这段时间积累了多少个会话"。两边都过线，就 fork 一个跑 /dream 提示的子智能体，让它读会话 transcript、合并/修订/裁剪记忆文件。这是为了让 Claude 在新会话里能快速 orient——把近期学到的沉淀成持久记忆。

闸门顺序是"最便宜的先查"：先一个 stat 读锁文件 mtime（=lastConsolidatedAt），过时间闸门再扫会话目录、过会话数闸门，最后才尝试拿锁。每个闸门失败都直接 return，不浪费后续成本。

锁的设计很讲究：锁文件本身就是 lastConsolidatedAt——它的 mtime 就是上次巩固时间，body 是持有者 PID。拿锁就是覆盖写自己的 PID，然后回读验证"我写的是不是我的 PID"（两个 reclaim 竞争时，最后写的赢、输的 bail）。失败时回退 mtime 到拿锁前的值，让时间闸门下次还能过。HOLDER_STALE_MS=1h 防 PID 复用：即便 PID 还活着，超过 1h 也认 stale 抢占。崩溃场景：mtime 卡住、PID 是死的，下个进程看到 dead PID 直接 reclaim。

三、后台 fork 服务群（AgentSummary / PromptSuggestion / MagicDocs / SessionMemory）

这四个服务共享同一个 runForkedAgent 模式，但各自做不同的活。共同点：都用 CacheSafeParams 共享父缓存、都用 canUseTool 回调控制权限、都 skipTranscript 或按需记录。区别在"什么时候 fork、fork 出来干什么、工具放行到什么程度"。

PromptSuggestion 是这套里逻辑最重的。它先做一堆抑制检查（aborted、早期会话不足 2 轮、上一条是 API 错误、父缓存冷启动超过 10k 未缓存 token、权限待决、elicitation 队列、plan 模式、external 用户限流），任何一条命中就抑制并打点。fork 出来后还要过 12 道内容过滤——done/meta_text/括号包裹/API 错误前缀/单字过短/超长/多句/带格式/evaluative 评价语/Claude 口吻（"Let me..."）——把这些"模型说错的话"滤掉，保证只把真正像用户会打的字递上去。

speculation 是 PromptSuggestion 的进阶版：fork 出来不是只生成文本，而是预跑用户可能输入的下一句——用 overlay 文件系统做 copy-on-write 隔离写操作、只放行只读工具和 read-only bash、碰到文件编辑/非只读 bash/未知工具就 abort 并记 boundary。用户接受预跑时，把 overlay 里的改动 copy 回主目录、把推测的消息注入主对话。关键约束：fork 用的是 createChildAbortController（链到父），父 abort 会传播下来；onMessage 边收消息边累计 toolUseCount，超过 100 条消息或 20 轮就强制 abort。

AgentSummary 是协调器模式（coordinator agent）下的进度摘要：每 30s 用主线程的 CacheSafeParams fork 一次，从 transcript 拉最新消息，全 deny 工具，生成"Reading runAgent.ts"这种 3-5 词现在进行时摘要。它特意不传 forkContextMessages 进闭包——每次 tick 都从 getAgentTranscript 重读，避免原始 fork 消息被定时器钉住一辈子。

MagicDocs 是唯一不用 runForkedAgent 的——它用 runAgent（因为要带自己的 agent definition 和受限工具集）。它在 FileReadTool 监听器里检测 # MAGIC DOC: [title] 头和紧随的斜体指令行，注册跟踪；对话空闲时（上一轮无 tool call）sequential 串行地更新每个 tracked doc。canUseTool 只放行 Edit 且 file_path 必须精确等于该 doc 路径。

SessionMemory 维护一个会话级 markdown 笔记，触发条件是"token 阈值且工具调用阈值"或"token 阈值且上一轮无 tool call"——token 阈值是硬性的，防止过度抽取。它 fork 出来只允许 Edit 那一个记忆文件。

四、远程 Agent 执行：消息流与权限桥接

用户视角：在本地 claude CLI 里连一个远程 CCR 会话，看到的就是远程 agent 的对话——打字、看流式输出、对权限请求按 y/n。背后是 RemoteSessionManager 协调一条 WebSocket 订阅 + 一条 HTTP POST 上行通道。

数据流上分三类消息：SDKMessage（assistant/user/stream_event/system/result/tool_progress 等渲染消息）、control_request（CCR 发来问"这个工具能不能用"）、control_response（CLI 回答"能/不能"）。SDKMessage 经 sdkMessageAdapter 转成 REPL 内部 Message 类型渲染；control_request 经 RemoteSessionManager 路由到 onPermissionRequest 回调、最终由 remotePermissionBridge 造一个合成 AssistantMessage 喂给本地权限 UI。

SDK 消息适配器要处理一个微妙：远程发来的 user 消息可能是工具结果（content 里有 tool_result 块）。在直连模式（convertToolResults=true）要把这些转成 UserMessage 才能像本地工具结果一样折叠渲染；CCR 实时模式则忽略——因为本地 REPL 已经自己加了用户消息。parent_tool_use_id 不可靠（agent 端 normalizeMessage 把它硬编码成 null），所以靠 content shape 判断。success 类型的 result 消息被当噪声忽略（isLoading=false 已经够了），只有 error 才显示。

权限桥接的关键：本地根本没跑那个工具（工具在 CCR 容器里跑），所以 ToolUseConfirm 需要的 AssistantMessage 是合成的——用 request 里的 tool_use_id/name/input 伪造一条 assistant 消息。如果 CCR 装了本地 CLI 不认识的工具（比如某个 MCP 工具），createToolStub 造一个最小 Tool 桩，路由到 FallbackPermissionRequest。

这条 WebSocket 是远程会话的生命线。close code 决定了下一步行为——是永久放弃、还是带预算重连、还是瞬时重连。

六、upstreamproxy：CONNECT-over-WebSocket 中转

用户视角：远程 CCR 容器里的 agent 跑 curl/gh/kubectl 时，这些子进程的 HTTPS 流量被透明代理——经本地 127.0.0.1 的 CONNECT relay、过 WebSocket 隧道到 CCR 服务端、服务端 MITM TLS 并注入组织凭据（如 DD-API-KEY）、再转给真实上游。容器里的 agent 看到的 HTTPS_PROXY 环境变量指向这个本地 relay。

为什么用 WebSocket 而不是裸 CONNECT：CCR 入口是 GKE L7 路径前缀路由，没有 connect_matcher；sessions/tunnel 已经用这个模式。字节流用 UpstreamProxyChunk protobuf（单字段 bytes）封装，跟服务端 gateway.NewWebSocketStreamAdapter 兼容。

  ws-close --> cleanup["cleanupConn: 清 pinger, 关 ws"]

这里有一堆对抗边界情况的工程细节：Bun 的 sock.write() 是部分写、内核缓冲满时静默丢字节，所以 Bun relay 自己维护 writeBuf 尾队列、drain 回调刷新；Node 的 net.Socket 无条件缓冲、不用尾队列。TCP 会把 CONNECT 头和 ClientHello 合并到一个包里，所以 pending 缓冲要接住"CRLFCRLF 之后跟着的字节"；ws.onopen 之前如果 socket.data 又回调了，那些字节也进 pending。established 标志位一旦置 true（收到服务端首字节），就再也不能写明文 502——会污染客户端的 TLS 流，只能直接 close。closed 标志位防 ws.onerror 后 ws.onclose 二次 sock.end()。

安全设计也值得点出：prctl(PR_SET_DUMPABLE, 0) 通过 libc FFI 调用，防同 UID 的 ptrace 抓堆——这是为了让 prompt-injected 的 gdb -p $PPID 抓不走 session token。token 文件在 relay 起来之后才 unlink，保证 CA 下载或 listen 失败时 supervisor 重启还能重试。NO_PROXY 列表把 Anthropic API、GitHub、各包 registry 都排除——MITM 会破坏非 Bun 运行时（Python httpx/certifi 不信伪造 CA）。子进程环境继承有个继承分支：如果父进程已经设了 HTTPS_PROXY+SSL_CERT_FILE（说明父跑了 relay），子 CLI 也直接继承，因为 token 文件已被父 unlink、子无法重新 init relay。

整个 upstreamproxy 是 fail-open 典范：env 没设→禁用、token 没了→禁用、CA 下载失败→禁用、relay 监听失败→禁用，每一步都只 logForDebugging warn 级别，绝不抛、绝不拖垮会话。

七、远程会话历史分页

sessionHistory.ts 是远程会话的事件历史分页——给"查看更早的消息"用。走 HTTP GET /v1/sessions/{id}/events，axios 带 OAuth 头 + anthropic-beta: ccr-byoc-2025-07-29 + x-organization-uuid。两个游标模式：fetchLatestEvents 用 anchor_to_latest=true 拿最新一页；fetchOlderEvents 用 before_id 拿指定游标之前的更老一页。每页 100 条，has_more 表示还有更老的。validateStatus: () => true + .catch(() => null) 让任何 HTTP 错误都降级成 null 返回，调用方自己处理——也是 fail-open 风格。createHistoryAuthCtx 把 auth 头准备一次复用，避免每页重新 prepareApiRequest。
