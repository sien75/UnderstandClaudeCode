MCP 探索服务器（让外部 AI 客户端浏览本仓库源码）

这是一个独立部署的 MCP（Model Context Protocol）服务器，作用很直接：任何兼容 MCP 的 AI 客户端（Claude Desktop、Cursor、VS Code Copilot 等）连上它之后，就能像翻书一样浏览本仓库的源码——列目录、读文件、按正则搜索、查看架构总览，甚至能列出 Claude Code 内置的全部"工具"和"斜杠命令"并读取它们的实现。换句话说，它把一个本地源码树变成了一组 AI 可调用的工具，让远端客户端不用 clone 仓库也能"看代码"。

它本身和 Claude Code 主项目互不耦合：主项目是它"展示"的对象，而它自己只负责对外暴露能力。部署形态有两种——本地 STDIO（客户端直接拉起一个 node 进程通信）和远程 HTTP（部署在 Railway/Vercel/Docker 上，多客户端通过 Bearer Token 鉴权复用）。三种传输协议并存：STDIO、Streamable HTTP（现代 MCP）、Legacy SSE（兼容老客户端）。

输入
来自外部 AI 客户端的 MCP 请求：调用某个工具（如 read_source_file 传一个相对路径）、读取某个 resource URI、或渲染某个 prompt 模板。HTTP 模式下还附带 Bearer Token 鉴权。
输出
对应工具的执行结果：文件内容片段、目录列表、搜索命中行、架构 JSON、工具/命令清单等，按 MCP 协议封装后返回给客户端，由客户端展示给用户或喂给它自己的 LLM。

入口与传输层

用户（外部 AI 客户端）触发的链路有三种起点，但都汇到同一个"共享 MCP 服务器"上。本地场景最简单：客户端在配置里写一条 command，拉起 node dist/index.js，进程通过 STDIO 跟客户端收发 JSON-RPC 消息。远程场景则是一个常驻 Express 进程，客户端按 /mcp（现代）或 /sse + /messages（老协议）发 HTTP 请求。

STDIO 入口只有 25 行，干完三件事就退场：校验源码根目录、创建共享 MCP 实例、把 StdioServerTransport 接上去。HTTP 入口复杂一些，核心是三块中间件/路由：鉴权、健康检查、两种传输协议的会话管理。

  • 鉴权是可选的：只有设了 MCP_API_KEY 才生效，且 /health 永远免鉴权（方便部署平台探活）。
  • Streamable HTTP 用一个 Map<sessionId, transport> 在内存里维护多会话，POST /mcp 创建/复用会话，GET /mcp 拉 SSE 流，DELETE /mcp 主动收尾。
  • Legacy SSE 走老协议：客户端先 GET /sse 建立长连接拿到 sessionId，后续 POST /messages?sessionId=xxx 发指令。
  • 每个 HTTP 会话都 createServer() 一次——也就是说每个会话拿到的是独立的 MCP 实例，互不串扰；STDIO 则全程只有一个实例。

共享 MCP 服务器：请求如何进来、如何读源码、如何返回

所有传输协议最终都把 JSON-RPC 请求交给同一个工厂函数 createServer() 造出来的 Server 实例。这个实例在构造时就把三类能力的 handler 一次性注册好：resources（被动可读的数据，按 URI 取）、tools（可调用的函数，带输入 schema）、prompts（预制的提示词模板，注入源码上下文后交给客户端的 LLM）。客户端先调 list 系列接口拿到菜单，再调具体的 call/read/get 取实际内容。

所有源码读取的底层动作只有三类：fs.readFile（读单个文件）、fs.readdir（列目录）、walkFiles（递归走整棵树用于搜索）。所有用户传进来的路径都要先过一道 safePath()——它用 path.resolve 把相对路径拼到 SRC_ROOT 下，再校验结果是否仍以 SRC_ROOT 开头，否则返回 null。这一步挡住了 ../../etc/passwd 之类的目录穿越攻击，是整个服务唯一的权限边界。

输入
三条 list 请求（ListTools / ListResources / ListPrompts）只返回静态菜单；三条 call 请求（CallTool / ReadResource / GetPrompt）携带工具名/URI/prompt 名加参数，参数里多半是一个相对路径或工具名。
输出
MCP 协议格式的响应：tools 返回 content: [{type:"text", text:...}]；resources 返回 contents: [{uri, mimeType, text}]；prompts 返回 messages: [{role:"user", content:{type:"text", text}}]。文本内容就是文件原文、目录列表、搜索命中行或拼好的提示词。

主数据流：工具调用（CallTool）

客户端调一个工具时，请求带 name 和 arguments，server 用一个 switch 分发到 8 个工具之一。每个工具的核心都是"定位文件 → 读内容 → 拼成文本返回"，区别只在定位逻辑。

资源读取流（ReadResource）

资源是"按 URI 直接取"的只读数据。server 注册了 3 个固定 URI 和 1 个模板 URI claude-code://source/{path}。模板 URI 走的也是 safePath，跟 read_source_file 工具其实是同一条安全通路，只是协议层不同。

graph TD
  readres["客户端 ReadResource(uri)"] --> arch?{"uri = architecture?"}
  arch -->|"是"| read-readme["读 SRC_ROOT/../README.md"]
  read-readme --> out-arch["返回 markdown"]
  arch -->|"否"| tools?{"uri = tools?"}
  tools -->|"是"| call-gettools["getToolList 扫目录"]
  call-gettools --> out-tools["返回工具清单 JSON"]
  tools -->|"否"| cmds?{"uri = commands?"}
  cmds -->|"是"| call-getcmds["getCommandList 扫目录"]
  call-getcmds --> out-cmds["返回命令清单 JSON"]
  cmds -->|"否"| src?{"uri 以 source/ 开头?"}
  src -->|"是"| safe-res["safePath 校验相对路径"]
  safe-res -->|"非法"| err-res["抛错: Invalid path"]
  safe-res -->|"OK"| read-res["fs.readFile 读文件"]
  read-res --> out-res["返回 text/plain 全文"]
  src -->|"否"| err-unknown["抛错: Unknown resource"]

提示词渲染流（GetPrompt）

5 个 prompt 模板的作用是"替客户端的 LLM 把源码上下文喂好"。server 收到请求后，按 prompt 名定位相关源码文件、读出来、塞进一段预设的提问文本里，整个组装成一条 user 消息返回。客户端拿到的就是一条可以直接喂给 LLM 的 prompt。

  pswitch -->|"explain_tool"| p-tool["定位 tools/工具名 主文件"]
  pswitch -->|"explain_command"| p-cmd["定位 commands/命令名"]

  pswitch -->|"how_does_it_work"| p-how["查 featureMap 找相关路径"]
  pswitch -->|"compare_tools"| p-cmp["读两个工具主源码"]
  p-tool --> p-tool-read["读主文件源码"]
  p-tool-read --> p-tool-out["拼成 6 维度分析提问"]
  p-cmd --> p-cmd-find["尝试目录/单文件三种候选"]
  p-cmd-find --> p-cmd-out["拼成命令解释提问"]
  p-arch --> p-arch-out["拼成架构导览提问"]
  p-how --> p-how-map{"feature 在 featureMap 中?"}
  p-how-map -->|"是"| p-how-read["逐路径读文件/列目录(前200行)"]
  p-how-map -->|"否"| p-how-empty["提示用 search/read 自行查找"]
  p-how-read --> p-how-out["拼成子系统讲解提问"]
  p-how-empty --> p-how-out

  • how_does_it_work 内置了一张 featureMap，把人类口语（"permission system"、"mcp client"、"bridge"）映射到一组源码路径——这是这个服务里唯一带"业务知识"的地方，其它工具都是纯文件系统操作。
  • 所有 prompt 返回的都是单条 user 消息，不带 system 消息；客户端可以再叠自己的 system prompt。
  • 读源码时 how_does_it_work 只取前 200 行，避免 prompt 过长；read_source_file 工具则支持 startLine/endLine 精确切片，客户端可自行翻页。

辅助视图：权限边界

整个服务的权限模型非常薄，只有两层：HTTP 层的 Bearer Token（可选）、文件层的 safePath（强制）。

graph TD
  req["传入请求"] --> http-layer{"HTTP 模式 & 设了 MCP_API_KEY?"}
  http-layer -->|"是"| bearer{"Authorization: Bearer 匹配?"}
  http-layer -->|"否"| pass-http["放行"]
  bearer -->|"否"| deny-401["401 Unauthorized"]
  bearer -->|"是"| pass-http
  pass-http --> tool-call["进入工具/resource handler"]
  tool-call --> path-arg{"参数含相对路径?"}
  path-arg -->|"是"| safepath["safePath: resolve + startsWith 校验"]
  path-arg -->|"否"| direct["直接扫固定目录"]
  safepath -->|"穿越出 SRC_ROOT"| deny-null["返回 null → 抛错"]
  safepath -->|"在 SRC_ROOT 内"| allow-read["允许 fs.readFile/readdir"]
  • /health（和 Vercel 的 /api）永远免鉴权，给部署平台探活用。
  • STDIO 模式没有 HTTP 鉴权——能拉起这个进程的客户端本来就视为可信。
  • safePath 用 resolved.startsWith(SRC_ROOT) 判断；注意 SRC_ROOT 本身是 path.resolve 出来的绝对路径，没有结尾斜杠，所以正好等于 SRC_ROOT 的路径也通过。
  • 没有按工具粒度做权限区分——任意已鉴权客户端可调用全部 8 个工具、读 SRC_ROOT 下任意文件。这是个"只读探索"服务，不写不改，所以模型够用。

辅助视图：会话生命周期与状态机

HTTP 模式下每个会话有独立的生命周期。Streamable HTTP 用 mcp-session-id 头标识会话，Legacy SSE 用 query 参数 sessionId。两者都在内存 Map 里维护 transport，进程重启即丢失。

graph TD
  new-req["新请求(无 sessionId)"] --> create-sess["createServer + 新 transport"]
  create-sess --> gen-id["生成 sessionId(UUID)"]
  gen-id --> store-map["存入 transports Map"]
  store-map --> state-active["状态: 活跃"]
  reuse-req["带 sessionId 的请求"] --> lookup{"Map 中存在?"}
  lookup -->|"是"| reuse-sess["复用 transport 处理"]
  lookup -->|"否"| reject-400["400 Invalid/Missing session"]
  reuse-sess --> state-active
  state-active -->|"客户端 DELETE /mcp"| close-sess["transport.close()"]
  state-active -->|"transport.onclose 触发"| del-map["从 Map 删除"]
  close-sess --> del-map
  del-map --> state-gone["状态: 已销毁"]
  state-gone -->|"客户端再来"| reject-400
  • 每个会话独立 createServer()，意味着每个会话有自己的 MCP Server 实例，但底层都读同一份磁盘源码——无状态业务，天然支持并发。
  • Vercel 部署是 serverless：Map 不跨调用持久化，Streamable HTTP 的会话复用基本失效；老 SSE 和无状态工具调用仍可用。README 明确建议生产用 Railway/Docker。
  • STDIO 模式没有"会话"概念，全程一个 transport，进程退出即结束。

部署形态一览

同一套 createServer() 内核，外面套不同壳就成了不同部署方式：本地 STDIO（src/index.ts）、远程 HTTP（src/http.ts）、Vercel serverless（api/vercelApp.ts 导出 Express app）。Dockerfile 把整个仓库 COPY 进镜像，构建出 dist/ 后用 node dist/http.js 启动，CLAUDE_CODE_SRC_ROOT=/app/src 指向镜像内的源码副本。Railway 用同一 Dockerfile，配 healthcheckPath=/health 自动探活。

  • Docker 镜像里同时打包了 mcp-server/dist 和主项目 src/——这个 server 探索的"源码"在部署时就被固化进镜像，不依赖宿主文件系统。
  • src/index.ts.new 是 index.ts 的同名副本（内容完全一致），疑似开发中遗留，未参与构建。
