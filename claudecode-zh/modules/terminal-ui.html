终端 UI 层（React + Ink）

用户在终端里看到的一切——输入框、流式回复、工具调用卡片、权限弹窗、diff 展示——都由这一层渲染。它是一个用 React + Ink 跑在终端里的单页应用：用户敲键 → 输入框组件更新本地状态 → 提交后把用户消息塞进消息数组 → 发起一次查询 → 流式事件不断追加助理消息和工具进度 → 每一帧 Ink 把 React 树渲染成字符画写到 stdout。整条数据流的主轴是"消息数组的单向流动"：几乎所有状态变更最终都落在 messages 这个数组上，渲染层只是它的一个投影。

输入
用户的键盘输入（文本、粘贴、斜杠命令、vim 按键）+ 来自查询引擎的流式事件（文字增量、工具调用块、思考块、进度 tick）
输出
终端屏幕上持续刷新的字符界面：一段对话转录 + 底部输入框，中间穿插权限弹窗、工具结果卡片、spinner 动画

一、从用户敲键到提交查询

用户在底部 PromptInput 里打字。输入框内部维护着自己的 buffer 和光标状态（含 vim 模式状态机）。按下回车后，PromptInput 调用 REPL 传下来的 onSubmit。onSubmit 不是直接发请求，而是先做几件事：如果是斜杠命令且标记为 immediate，就绕过队列立即执行（例如 /btw、/config 这类本地弹窗命令）；否则把输入交给 handlePromptSubmit 统一处理——它会展开粘贴引用、判断是普通对话还是斜杠命令、必要时排队（当 Claude 正忙时输入会被暂存，等当前回合结束再提交）。最终，对于需要发请求的输入，handlePromptSubmit 调 onQuery，由它把用户消息追加进 messages 并启动查询引擎。

二、流式回复的渲染

查询引擎开始返回流式事件后，每一段增量都会经过 onQueryEvent 进入 handleMessageFromStream。这个函数把原始事件翻译成对 messages 数组的变更操作：普通增量就 append；压缩边界消息会替换整个数组（保留少量历史做滚动回看）；进度 tick 会原地替换上一条同工具的 tick（避免 sleep/bash 每秒一条把数组撑到上万条）。setMessages 是一个特殊包装：它同步更新 messagesRef（让后续逻辑立刻读到最新值，不等 React 批处理），再交给 React 真正 setState。Messages 组件拿到新数组后，用 VirtualMessageList 做虚拟滚动只渲染视口内的行，由 Ink reconciler → yoga 布局 → paint 到 Screen buffer → 写到 stdout。

三、工具权限弹窗

当模型要调用一个需要确认的工具时，查询引擎调用 canUseTool（由 useCanUseTool 提供）。它先跑 hasPermissionsToUseTool 看配置里是否已经允许/拒绝；如果结果是 allow 就直接放行，deny 就直接拒绝。只有结果是 ask 时才需要弹窗：先尝试协调器自动检查、swarm worker 代理、bash 分类器投机判定（2 秒竞速），这些都拿不到结论才走到 handleInteractivePermission，它把一个 ToolUseConfirm 推进 toolUseConfirmQueue。REPL 渲染层看到队列非空，就在 focusedInputDialog === 'tool-permission' 分支挂载 PermissionRequest 组件。用户选择后，onDone 把队首出队，resolve 那个挂起的 Promise，查询引擎继续或中止。

四、回合生命周期与状态机

一个回合的加载状态由 queryGuard（一个简易状态机：idle → running）驱动。isLoading 不再是简单的 React state，而是从 queryGuard 的外部 store 派生而来，保证"正在处理"这个事实不会被 React 批处理延迟。onQuery 进入时 tryStart() 原子地把 idle 翻成 running 并返回一个 generation 号；如果已经在跑，就拒绝并把输入入队。finally 块里 queryGuard.end(thisGeneration) 原子地核对 generation 后翻回 idle，随后 resetLoadingState 清理临时状态、onTurnComplete 回调通知外部。如果用户中途 Esc 取消，forceEnd 强制结束并在满足条件时自动回退用户消息、恢复输入框内容。

五、全局状态与重渲染

全局状态放在一个极简的 Zustand 风格 store 里（createStore，不到 35 行）：getState、setState、subscribe。setState 用 Object.is 判等跳过无变更，然后调 onChangeAppState 做副作用（同步权限模式到 CCR、持久化 verbose/expandedView 到全局配置、清认证缓存），最后通知所有订阅者。组件通过 useAppState(selector) 订阅切片，selector 返回值变了才重渲染。这样 REPL 这个巨型组件不会因为无关字段变动而无谓重算。messages 数组本身是 REPL 的本地 React state（不是全局 store），但通过 messagesRef 同步镜像，让 onQuery、getToolUseContext 等闭包能立刻读到最新值。

  • setMessages 是真相源：它先同步写 messagesRef，再把计算好的值交给 React。这是为了绕过 React 批处理延迟——onQuery 里 setMessages 之后立刻读 messagesRef.current 能拿到刚追加的用户消息，不用等 commit。注释里明说这是 Zustand 模式：ref 是真相，React state 是投影。
  • 进度 tick 原地替换：sleep/bash 每秒发一个 tick，如果无脑 append 会让 messages 数组涨到 1.3 万条、transcript 文件 120MB。onQueryEvent 检测到同类 tick 就替换最后一条，useLogMessages 还会因长度不变而跳过磁盘写入。
  • 虚拟滚动是分水岭：transcript 模式下若开了虚拟滚动（默认全屏），用 FullscreenLayout + AlternateScreen 包裹，VirtualMessageList 只挂载视口内行；没开就退化为 30 条上限 + 原生终端回滚。两种模式共用同一个 scrollRef，互斥不会同时挂载两个 ScrollBox。
  • 即时命令绕过队列：斜杠命令标了 immediate: true 的（/btw、/model、/mcp）即使 Claude 正忙也立即执行，渲染成底部固定浮层而非滚动区里的内容——避免主对话流式刷新时把它们挤得乱跳。
  • 权限模式同步单一咽喉：onChangeAppState 是权限模式变更的唯一同步点。以前 8+ 个改模式的地方只有 2 个会通知 CCR，现在 hook 在 diff 上，任何路径改了模式都会自动同步到 CCR 和 SDK 状态流。
  • vim 是独立状态机：useVimInput 把按键交给 vim/transitions.ts 的 transition()，按当前状态（idle/count/operator/find 等）和输入决定下一状态和要执行的动作，是数据流主轴旁的一个独立小机器。
  • 渲染管线：Ink 的渲染走 React reconciler → yoga 布局算尺寸 → renderNodeToOutput 把节点画到 Screen 字符格 → diff 上一帧输出增量 → 写 stdout。renderToScreen 还提供了一个隔离渲染单条消息的入口，专给搜索功能用——渲染一条消息扫字符位置，约 1-3ms。
