沙箱机制（OS 级执行隔离）

当 Claude 准备跑一条 bash 命令、读写文件、或发起网络请求时，系统要决定一件事：这条命令要不要被关进操作系统级沙箱里跑。沙箱是一道硬安全边界——它由 @anthropic-ai/sandbox-runtime 在 OS 层实现（macOS 用 sandbox-exec，Linux/WSL2 用 bubblewrap+socat），限制进程能写哪些文件、能连哪些主机。违规不会让 Claude 飙到不该去的地方，而是被拦下来、记一笔、必要时弹窗问用户。

这个模块的入口不是某个 UI 按钮，而是 agent 自主执行工具的那一瞬间：BashTool 在真正 spawn 子进程前，会问 shouldUseSandbox(input)，得到 yes 就把原命令交给 SandboxManager.wrapWithSandbox 包一层沙箱启动器再 spawn。命令跑完后，cleanupAfterCommand 清掉临时目录、扫掉被攻击者植入的 bare-git 文件。整个生命周期由 SandboxManager（adapter）统一对外暴露，内部转发给外部包的 BaseSandboxManager。

输入
用户/agent 触发的一条 bash 命令（含 dangerouslyDisableSandbox 标志）+ 当前合并好的 settings（permissions allow/deny 规则、sandbox.* 配置）+ 平台/依赖检测结果
输出
一条要么原样、要么被沙箱启动器包裹的命令字符串；执行结果里被标注过的 stderr；可能的违规事件、权限弹窗、以及（用户同意后）持久化到本地 settings 的 WebFetch(domain) 规则
  • 沙箱不是"可选美化"：当 autoAllowBashIfSandboxed 打开时，沙箱命中 = 自动放行 bash（绕过 ask 规则）。所以"要不要关沙箱"直接决定了"要不要弹权限"，二者是同一个决策点。
  • 外部包 @anthropic-ai/sandbox-runtime 是真正的 OS 级实现，本模块只做"CLI settings → runtime config"的翻译和事件回流，不读它的源码。
  • 有三道独立闸门：平台/依赖是否支持、settings 是否 enabled、单条命令是否被排除/dangerouslyDisableSandbox 豁免。任一不满足都不进沙箱。
  • 违规和权限是两条回流路径：网络违规走 ask callback 弹窗；FS/网络违规走 violation store 沉淀到 UI 展开。

图一：一条 bash 命令的沙箱决策与执行流

从用户视角看：Claude 决定跑 npm install。BashTool 在 spawn 之前会先问"这条要不要进沙箱"，问的顺序是"全局开关 → 显式豁免 → 命令是否在排除名单 → 默认进沙箱"。一旦决定进沙箱，命令字符串会被 sandbox-runtime 包成 bwrap ... -- sh -c '原命令'（macOS 类似），spawn 后由 OS 层强制限制读写和网络。命令结束后清场，并把 stderr 里被 sandbox-runtime 标记的违规行回填到给 Claude 看的输出里。

图三：违规事件处理与权限回流

沙箱在 OS 层拦截后，回流有两条独立的路。网络路：sandbox-runtime 跑命令时遇到一个没在 allowedDomains 里的主机，会回调 SandboxAskCallback——这个 callback 在 REPL 里被装成"排队一个 SandboxPermissionRequest 弹窗"，用户选 Yes/不再问/No。选"不再问"会把 WebFetch(domain:host) 作为 allow 规则持久化到 localSettings，并立即 refreshConfig 让在途请求也享受到新规则。FS/通用路：违规被写进 SandboxViolationStore，annotateStderrWithSandboxFailures 把它缝进命令输出告诉 Claude，UI 端两个组件订阅 store 展示计数和最近 10 条。另外还有一条启动硬门：若 failIfUnavailable=true 且沙箱起不来，REPL 直接退出不肯跑——这是给企业部署的硬安全闸。

  • autoAllowBashIfSandboxed 把"沙箱命中"和"权限自动放行"焊在一起：在 hasPermissionsToUseToolInner 里，只要 shouldUseSandbox 返回 true 且 auto-allow 开着，就跳过 Bash 的 ask 规则直接 allow，但显式 deny/ask 规则仍优先（含子命令级检查，防 echo ok && rm -rf / 逃逸）。
  • excludedCommands 是便利特性不是安全边界——代码注释明确写了真正的安全控制是会弹窗的权限系统。它按 prefix/exact/wildcard 匹配，且对复合命令按 && 拆分逐个查，避免首段命中就整条逃逸。
  • 路径解析两套语义是高频坑：Edit(/foo) 的 /foo 是"相对 settings 文件目录"，而 sandbox.filesystem.allowWrite: ["/foo"] 的 /foo 是绝对路径。后者是 #30067 的修复——用户写在 sandbox 配置里时合理期待绝对路径。 //foo 是历史逃逸写法，两套都把它当绝对路径。
  • 启动时若 failIfUnavailable=true 且沙箱起不来（缺依赖/不支持平台/不在 enabledPlatforms），REPL 直接 gracefulShutdownSync(1) 退出，拒绝在无沙箱状态下运行——给 NVIDIA 这类企业部署的硬闸。否则只发一条 sandbox disabled 通知指向 /sandbox。
  • 裸 git 仓逃逸（#29316）：攻击者在 cwd 植入 HEAD/objects/refs 让 Claude 后续的未沙箱 git 把 cwd 当 bare 仓 + 触发 core.fsmonitor。adapter 在装配置时 stat 这些文件，存在的 denyWrite（沙箱内 ro-bind），不存在的加入 bareGitRepoScrubPaths，在 cleanupAfterCommand 里删掉，让未沙箱的 git 看不到。
  • settings 变更通过 settingsChangeDetector 订阅动态刷新，权限弹窗"不再问"还会额外手动 refreshConfig 一次防竞态（在 settings 文件落盘事件回来前，pending 请求可能滑过去用旧配置）。
  • swarm 场景下 worker 的网络权限请求通过 mailbox 转给 leader 审批；leader 端 useInboxPoller 收到 SandboxPermissionRequestMessage 后弹同一个 SandboxPermissionRequest 组件。
